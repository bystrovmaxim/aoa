# План трансформации обработчиков плагинов — согласование

## Исходные требования

1. Вместо `action_name` (строки) передавать в событие **тип действия** (`type[BaseAction]`). Это позволит плагинам использовать `issubclass` и `isinstance` для фильтрации, а также получать метаданные действия через координатор.

2. Создать **уникальные типы событий** (классы) для каждого типа события, а не объединять их в один класс с опциональными полями. Каждый тип события содержит ровно те поля, которые нужны для этого события.

3. Декоратор `@on` должен принимать **третий параметр** — класс события (или несколько классов через `Union` / перегрузку). Этот параметр указывает, какие типы событий может обрабатывать метод. При этом декоратор также принимает `event_type` (строка) и `action_filter` (регулярное выражение).

4. Сигнатура метода-обработчика должна принимать **базовый класс события** (например, `BasePluginEvent`), чтобы в метод могли передаваться события разных конкретных типов (полиморфизм). При этом mypy будет проверять, что передаваемый объект совместим с аннотацией.

5. Возникает конфликт: у нас есть **три фильтра**:
   - `event_type` (строка, regex) — по имени события.
   - `action_filter` (строка, regex) — по имени действия (строковому).
   - `event_class` (тип) — по классу события.
   
   Нужно решить, как они соотносятся. Возможные варианты:
   - Отказаться от `event_type` и `action_filter`, оставить только `event_class` (и, возможно, дополнительный фильтр по действию на основе его типа, а не строки).
   - Оставить `event_type` и `action_filter` как строковые маски, а `event_class` использовать как дополнительную проверку (класс должен быть совместим с ожидаемым).
   - Объединить: `event_class` определяет тип события, а `event_type` и `action_filter` — это устаревшие строковые фильтры, которые можно убрать, но для обратной совместимости (если нужно) оставить как опциональные.

## Предлагаемое решение (консенсус)

Мы строим систему с нуля (без обратной совместимости), поэтому можем принять радикальное решение:

### 1. Отказаться от строковых `event_type` и `action_filter`

Вместо них использовать:
- Фильтр по классу события (обязательный).
- Фильтр по действию — через **предикат**, принимающий `type[BaseAction]` (или экземпляр). Например, можно передать класс действия, список классов, или функцию `Callable[[type[BaseAction]], bool]`. Но для простоты оставим `action_class` (один класс или кортеж классов) или `action_filter` как функцию.

**Итог:** декоратор `@on` принимает:
- `event_class: type[BasePluginEvent]` — класс события, который этот обработчик может принимать (обязательно).
- `action_class: type[BaseAction] | tuple[type[BaseAction], ...] | None = None` — ограничение по классу действия (опционально). Если `None` — любые действия.
- `ignore_exceptions: bool = True` — как раньше.

### 2. Сигнатура обработчика

```python
async def handler(self, state, event: BasePluginEvent, log) -> dict
```

- `event` имеет тип `BasePluginEvent` (базовый). Конкретный тип события будет передан через `event` (например, `GlobalStartEvent`), но mypy позволит обращаться только к полям базового класса. Если нужно работать со специфичными полями, обработчик должен использовать `isinstance` или явное приведение. Это не очень удобно, но зато типобезопасно.

Альтернатива: использовать `Generic` в декораторе, чтобы тип события был привязан к конкретному классу, но тогда метод-обработчик должен аннотировать `event` конкретным типом, и декоратор будет проверять совместимость. Это сложнее, но возможно. Пока предлагаю первый вариант.

### 3. Как регистрируются обработчики

`MetadataBuilder` при сборке плагина находит методы с `_on_subscriptions`. В `SubscriptionInfo` теперь храним:
- `event_class: type[BasePluginEvent]`
- `action_class: type[BaseAction] | tuple[type[BaseAction], ...] | None`
- `ignore_exceptions: bool`

При эмиссии события `PluginRunContext.emit_event` создаёт конкретный объект события (например, `GlobalStartEvent`) и передаёт его всем обработчикам, подписанным на совместимый `event_class` (проверка через `issubclass(event_class, handler.event_class)`). Также проверяется соответствие действия (`action_class`).

### 4. Решение конфликта: отказываемся от строковых масок

- `event_type` (строка) больше не нужна, потому что класс события однозначно определяет тип.
- `action_filter` (строка, regex) заменяем на явную проверку по классу действия. Это типобезопасно, но менее гибко (нельзя фильтровать по имени модуля, только по иерархии классов). Если нужна фильтрация по имени, можно добавить опциональный `action_name_filter: str | None` (регулярка), но это уже компромисс.

Предлагаю: `action_class` — основной фильтр, а если нужна фильтрация по имени, можно использовать `action_class` с базовым классом и дополнительный предикат `action_predicate: Callable[[type[BaseAction]], bool] | None`. Но для простоты пока оставляем только `action_class`.

### 5. Пример использования

```python
@on(GlobalFinishEvent, action_class=OrderAction, ignore_exceptions=False)
async def on_finish(self, state, event, log):
    # event — это GlobalFinishEvent, но тип event: BasePluginEvent
    # приходится делать явное приведение или проверку:
    if isinstance(event, GlobalFinishEvent):
        duration = event.duration
```

Можно улучшить: декоратор может быть дженериком, и тогда сигнатура метода будет конкретной. Но это требует более сложной типизации.

## Подробное описание параметров декоратора `@on`

Декоратор `@on` предназначен для подписки метода плагина на события ActionMachine. Новая версия декоратора (без обратной совместимости) принимает следующие параметры:

```python
@on(
    event_class: type[BasePluginEvent],
    action_class: type[BaseAction] | tuple[type[BaseAction], ...] | None = None,
    ignore_exceptions: bool = True,
)
async def handler(self, state, event: BasePluginEvent, log) -> dict:
    ...
```

### 1. `event_class` — класс события (обязательный)

- **Тип:** `type[BasePluginEvent]` — класс, унаследованный от `BasePluginEvent`.
- **Назначение:** определяет, какие конкретные типы событий будет обрабатывать этот метод. При эмиссии события `PluginRunContext` проверяет, является ли созданный объект события экземпляром `event_class` (или наследником). Если да — обработчик вызывается.
- **Почему класс, а не строка:** класс даёт статическую типизацию, проверку совместимости на этапе сборки метаданных и исключает опечатки в именах событий.
- **Примеры значений:** `GlobalStartEvent`, `GlobalFinishEvent`, `BeforeAspectEvent`, `AfterAspectEvent`, `OnErrorEvent`, `BeforeCompensateEvent`, `AfterCompensateEvent`.

### 2. `action_class` — фильтр по классу действия (опциональный)

- **Тип:** `type[BaseAction] | tuple[type[BaseAction], ...] | None`
  - Если передан один класс — обработчик будет вызван только для действий, являющихся экземплярами этого класса или его наследников.
  - Если передан кортеж классов — обработчик будет вызван для действий, соответствующих любому из классов в кортеже.
  - Если `None` (значение по умолчанию) — фильтр не применяется, обработчик вызывается для всех действий.
- **Назначение:** позволяет плагину реагировать только на определённые действия (например, только на `OrderAction`). Это заменяет старый строковый `action_filter` с регулярными выражениями, но теперь фильтрация идёт по иерархии типов, что безопаснее и проверяется статически.
- **Проверка в рантайме:** `PluginRunContext` при эмиссии события проверяет, что тип действия `action` соответствует `action_class` (через `isinstance`). Если нет — обработчик не вызывается.

### 3. `ignore_exceptions` — подавление ошибок в обработчике (опциональный)

- **Тип:** `bool`, значение по умолчанию `True`.
- **Назначение:** определяет, как система реагирует на исключения, возникшие внутри обработчика плагина.
  - `True` (по умолчанию) — ошибка перехватывается, логируется (если есть логгер), но не прерывает выполнение других плагинов и основного действия. Это стандартное поведение для наблюдателей.
  - `False` — ошибка пробрасывается наружу, прерывая выполнение цепочки плагинов и, возможно, всего действия (если не обработана выше).
- **Совместимость:** поведение полностью аналогично старому параметру `ignore_exceptions`.

## Как обрабатываются несколько типов событий одним методом

Если плагин должен реагировать на несколько разных типов событий (например, на `GlobalStartEvent` и `GlobalFinishEvent`), можно:
- Использовать **несколько декораторов `@on`** на одном методе, каждый со своим `event_class`. В этом случае метод будет вызван для каждого события отдельно, и параметр `event` будет иметь соответствующий тип (в рантайме). Сигнатура метода должна быть совместима с `BasePluginEvent` (что она и есть).
- Или подписаться на **базовый класс** `BasePluginEvent`, но тогда метод будет получать все события, и внутри придётся различать их через `isinstance`. Это менее предпочтительно, но возможно.

**Пример с несколькими декораторами:**

```python
@on(GlobalStartEvent, action_class=OrderAction)
@on(GlobalFinishEvent, action_class=OrderAction)
async def track_order(self, state, event, log):
    if isinstance(event, GlobalStartEvent):
        # старт заказа
    elif isinstance(event, GlobalFinishEvent):
        # финиш заказа, есть event.result и event.duration
```

## Проверки на этапе сборки метаданных

`MetadataBuilder` при обработке класса плагина:

1. Находит все методы с атрибутом `_on_subscriptions` (который заполняет декоратор).
2. Для каждого метода проверяет, что у него второй параметр (после `self` и `state`) имеет аннотацию типа, совместимую с `BasePluginEvent` (или конкретным классом, но мы рекомендуем `BasePluginEvent` для гибкости).
3. Если аннотация отсутствует или несовместима — ошибка сборки.
4. Проверяет, что `event_class` является подклассом `BasePluginEvent`.
5. Проверяет, что `action_class`, если указан, является подклассом `BaseAction` или кортежем таких подклассов.
6. Сохраняет подписку в `ClassMetadata.subscriptions` как `SubscriptionInfo(event_class, action_class, ignore_exceptions)`.

## Эмиссия события в `PluginRunContext`

При вызове `emit_event`:

1. По имени события и переданным параметрам создаётся конкретный объект события (например, `GlobalStartEvent`).
2. Для каждого плагина получается список подписанных обработчиков через `plugin.get_handlers(event_name, action_class_name)` — но теперь этот метод должен работать с типами, а не со строками.
   - Вместо `event_name` (строки) передаём класс события `event.__class__`.
   - Вместо `action_class_name` (строки) передаём тип действия `type(action)`.
3. `plugin.get_handlers(event_class, action_class)` возвращает обработчики, у которых:
   - `issubclass(event_class, handler.event_class)`
   - Если у обработчика указан `action_class`, то `isinstance(action, handler.action_class)`.
4. Далее обработчики вызываются с объектом события, их состоянием и логгером.

## Преимущества нового подхода

- **Типобезопасность:** mypy проверит, что в обработчике используются правильные поля (через `isinstance` или явное приведение).
- **Отсутствие магии со строками:** классы событий — это явные сущности, их легко найти и проанализировать.
- **Проверка при старте:** несовместимость типов ловится до запуска приложения.
- **Гибкость:** можно фильтровать действия по их типу, а не по имени модуля.
