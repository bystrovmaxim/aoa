# ActionMachine — жизненный цикл и пайплайн

ActionMachine — это сердце ActionEngine. Она не содержит бизнес-логики, не принимает решений и не знает что именно делает действие. Её задача одна — обеспечить предсказуемый, детерминированный и безопасный конвейер выполнения любого действия. И именно она хранит полную метамодель каждого действия — всё что известно о нём до первого запуска [1].

---

## Единственная точка входа

Любое действие запускается только одним способом:

```python
result = await machine.run(action, params)
```

Нет других способов войти в конвейер. Нельзя вызвать аспект напрямую. Нельзя пропустить проверку ролей. Нельзя войти в середину пайплайна. Это архитектурная гарантия а не соглашение [1].

---

## Полный жизненный цикл вызова run

Когда вызывается machine.run происходит следующее строго по порядку:

Первый шаг. Увеличение уровня вложенности nest_level. Это позволяет плагинам видеть глубину дерева вызовов.

Второй шаг. Проверка ролей через CheckRoles. Если декоратор отсутствует — немедленная ошибка. Если роли не совпадают — AuthorizationException. Эта проверка происходит до запуска любого аспекта [1].

Третий шаг. Создание DependencyFactory для данного вызова. Фабрика собирает все зависимости объявленные через @depends и готовит их к созданию.

Четвёртый шаг. Событие global_start. Все плагины подписанные на это событие получают уведомление с params и начальным состоянием.

Пятый шаг. Выполнение regular-аспектов строго по порядку. Для каждого аспекта: событие before — плагины уведомляются до выполнения, выполнение самого аспекта — бизнес-логика, применение чекеров к результату, событие after — плагины уведомляются после выполнения с duration.

Шестой шаг. Выполнение summary-аспекта. Он обязан вернуть Result. Если вернул что-то другое — TypeError.

Седьмой шаг. Событие global_finish. Плагины получают итоговый result и полное время выполнения.

Восьмой шаг. Уменьшение уровня вложенности и возврат Result.

## Как машина находит и сортирует аспекты

При первом вызове действия машина анализирует его класс через метод _collect_aspects [1]:

Первое. Перебирает все методы класса. Второе. Ищет методы с атрибутом _is_aspect. Третье. Разделяет на regular-аспекты и summary-аспект. Четвёртое. Сортирует regular-аспекты по номеру строки в исходном файле через co_firstlineno.

```python
aspects.sort(key=lambda m: m.__code__.co_firstlineno)
```

Это гарантирует что порядок выполнения всегда совпадает с порядком объявления в файле. Не алфавитный порядок. Не порядок добавления в словарь. Именно строки кода [1].

Машина проверяет что summary-аспект ровно один. Если их нет или больше одного — ошибка при первом же вызове.

Результат кешируется в _aspect_cache. Повторный анализ класса не происходит.

---

## ActionMachine как хранилище метамодели

Здесь важно понять нечто что выходит за рамки просто "исполнителя". После первого анализа класса машина хранит в своём кеше не только порядок аспектов — она хранит полную машиночитаемую спецификацию каждого действия.

Для любого Action можно получить программно:

Список зависимостей с описаниями — через атрибут _dependencies класса. Ролевую модель — через _role_spec. Полный упорядоченный список шагов — через _is_aspect и co_firstlineno. Контракт каждого шага — через _result_checkers. Объявленные соединения с описаниями — через _connections.

Эти данные не устаревают и не расходятся с кодом — потому что без них машина не запустит действие. @CheckRoles обязателен. summary-аспект обязателен. Это означает что метаданные верифицированы самим фактом работающей системы.

Из этого вытекает возможность статического анализа бизнес-архитектуры. Простой скрипт обходящий все Action-классы способен построить матрицу доступа Action × Role, граф зависимостей между операциями, обнаружить действия без зависимостей которые вызывают вложенные действия, найти действия с подозрительными ролями на платёжных операциях. Это не анализ кода — это анализ бизнес-намерений которые формализованы принудительно [1].

## Проверка ролей

CheckRoles — обязательный декоратор для каждого Action. Машина проверяет его через _check_action_roles [1].

Если декоратор отсутствует — ошибка конфигурации. Это не предупреждение. Действие не будет выполнено.

Логика проверки:

CheckRoles.NONE означает что аутентификация не требуется — доступно всем. CheckRoles.ANY означает что пользователь должен быть аутентифицирован — любая роль подходит. Список ролей означает что у пользователя должна быть хотя бы одна из указанных ролей.

Проверка происходит до создания DependencyFactory и до вызова первого плагина [1].

---

## DependencyFactory — создание и кеширование зависимостей

Фабрика создаётся для каждого вызова run. Она живёт ровно столько сколько живёт этот вызов.

При вызове deps.get(SomeClass) происходит следующее:

Первое. Проверяется кеш _instances. Если объект уже создавался — возвращается он же. Второе. Проверяется что SomeClass объявлен через @depends. Если не объявлен — ошибка. Третье. Если есть factory — вызывается она. Иначе вызывается конструктор SomeClass(). Четвёртое. Результат сохраняется в кеш.

Это означает что все аспекты одного действия работают с одним и тем же экземпляром каждой зависимости [1].

---

## Плагины — архитектура вызова

Машина хранит кеш обработчиков плагинов в _plugin_cache. Для каждого события и каждого класса действия список подходящих обработчиков вычисляется один раз и кешируется.

При наступлении события машина вызывает _run_plugins_async который:

Первое. Берёт список обработчиков из кеша. Второе. Для каждого обработчика передаёт текущее состояние плагина state_plugin. Третье. Ограничивает количество одновременных вызовов через Semaphore. Четвёртое. Сохраняет новое состояние плагина.

Если обработчик бросил исключение и ignore_exceptions=True — ошибка пишется в лог и выполнение продолжается. Бизнес-логика не останавливается из-за ошибки плагина [1].

---

## Применение чекеров

После каждого regular-аспекта машина вызывает _apply_checkers [1]. Чекеры привязаны к аспекту и проверяют результат:

Первое. Проверяется что результат является dict. Если нет — TypeError немедленно. Второе. Проверяется что в результате нет лишних ключей относительно объявленных чекерами. Третье. Для каждого чекера проверяется тип и наличие значения.

Если нарушение найдено — ValidationFieldException. Следующий аспект не вызывается.

## Вложенные вызовы через run_action

Из аспекта можно вызвать другое действие:

```python
result = await deps.run_action(
    OtherAction,
    OtherAction.Params(...)
)
```

Что происходит внутри:

Первое. DependencyFactory создаёт экземпляр OtherAction. Второе. Вызывается machine.run с тем же контекстом и плагинами. Третье. nest_level увеличивается. Четвёртое. Плагины получают все события вложенного действия с увеличенным nest_level. Пятое. После завершения nest_level уменьшается.

Плагины видят полное дерево вызовов. Каждое вложенное действие трассируется полностью автоматически [1].

---

## Валидация connections

Если действие объявило зависимости через @connection — машина проверяет соответствие до запуска аспектов через _check_connections [1]:

Каждый ключ из @connection должен присутствовать в переданном словаре connections. Каждый ключ в переданном словаре должен быть объявлен через @connection. Лишние ключи — ошибка. Недостающие ключи — ошибка.

Это предотвращает ситуацию когда аспект обращается к connections["key"] и получает KeyError во время выполнения.

## Две реализации машины

ActionProductMachine — полная production-реализация. Плагины, роли, чекеры, вложенность, измерение времени, кеширование аспектов.

ActionTestMachine — тестовая реализация. Полностью повторяет логику production-машины но позволяет подменять зависимости через словарь моков. Автоматически оборачивает BaseResult в MockAction. Автоматически оборачивает функции в MockAction с side_effect [1].

```python
machine = ActionTestMachine({
    ChildAction: ChildAction.Result(doubled=42),
    EmailService: FakeEmailService(),
    SomeAction: lambda params: SomeAction.Result(value=params.num * 2)
})
```

Оба вида машины используют одинаковый пайплайн. Разница только в источнике зависимостей.

---

## Что машина не делает

Машина не содержит бизнес-логики. Она не знает что делает конкретное действие. Она не управляет транзакциями автоматически. Она не решает когда делать commit или rollback. Она не меняет params или result. Она не влияет на то что возвращает summary-аспект.

Всё это — ответственность самих действий и ресурсных менеджеров.

---

## Измерение времени

Машина измеряет время выполнения каждого аспекта и всего действия целиком. Значение duration передаётся:

В событие after — время выполнения конкретного аспекта. В событие global_finish — полное время выполнения действия включая все аспекты и вложенные вызовы.

Это даёт плагинам данные для метрик и профилирования без дополнительной инструментации. Плагин на 50 строк способен записывать семантическое дерево выполнения с duration каждого аспекта, nest_level и state на каждом узле. Такое дерево можно передать языковой модели для объяснения что именно произошло в этом конкретном вызове и почему [1].
