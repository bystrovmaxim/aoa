# Обзор архитектуры AOA

ActionEngine построен на принципах Action-Oriented Architecture. Бизнес-логика выражена через действия, инфраструктура вынесена в ресурсные менеджеры и плагины, а выполнение любого сценария проходит через строгий конвейер аспектов под управлением ActionMachine. Такая структура даёт предсказуемость, простоту тестирования и высокую изоляцию слоёв [1].

---

## Три уровня архитектуры

Архитектура разделена на три уровня.

Первый уровень — ядро бизнес-логики. Actions, Params, Result, Aspects. Здесь принимаются решения.

Второй уровень — инфраструктурный слой. Ресурсные менеджеры, адаптеры, репозитории, порты. Здесь выполняются эффекты.

Третий уровень — исполнитель и наблюдатели. ActionMachine и Plugins. Здесь управляется конвейер и собирается наблюдаемость.

Все элементы взаимодействуют предсказуемо и однонаправленно. Бизнес-логика не знает о деталях реализации инфраструктуры, а инфраструктура не влияет на поведение действий [1].

---

## Поток данных

Прохождение запроса в системе происходит по плоскому детерминированному конвейеру:

```
Транспорт (HTTP / MCP / CLI)
        ↓
AuthCoordinator → Context
        ↓
ActionMachine.run(action, params)
        ↓
[global_start] → Plugins
        ↓
[before:aspect] → Plugin
[aspect()] → бизнес-шаг
[after:aspect] → Plugin
        ↓  ... повторяется для каждого аспекта
        ↓
summary_aspect() → Result
        ↓
[global_finish] → Plugins
        ↓
Result → Transport Layer
```

Поток всегда линейный. Сверху вниз, без скрытых переходов и магии [1].

---

## Роль ActionMachine

ActionMachine координирует весь процесс. Она не содержит бизнес-логики — только правила конвейера.

Она отвечает за:

Выбор и выполнение аспектов строго в порядке определения. Применение чекеров результата. Ролевую модель через CheckRoles. Вызовы плагинов на каждом этапе. Вложенность вызовов через deps.run_action. Создание DependencyFactory для каждого вызова. Контроль времени выполнения [1].

---

## Actions — атомы логики

Action — главный строительный блок архитектуры. Каждая бизнес-операция оформляется как отдельный класс.

Params задаёт входные данные. Аспекты разбивают процесс на шаги. Summary-аспект возвращает результат. Зависимости приходят через @depends.

Actions не имеют состояния и не знают ничего об инфраструктуре. Это делает их идеальными объектами для юнит-тестов [1].

---

## Аспекты — линейный конвейер

Аспекты — этапы исполнения действия. Они выполняются в порядке определения в файле. Каждый аспект принимает params, state, deps и возвращает новый state. Аспекты вызываются строго последовательно и могут тестироваться изолированно.

ActionMachine сортирует аспекты по строкам объявления через co_firstlineno. Это исключает поведение по случайности [1].

---

## Ресурсные менеджеры

Ресурсные менеджеры — инфраструктурные адаптеры, скрывающие реальное взаимодействие с БД, API, файлами и внешними сервисами.

Признаки ресурса:

Управляет состоянием — соединение, сессия, кеш. Не содержит бизнес-логики. Предоставляет домену унифицированный интерфейс — порт. Живёт между вызовами. Открыт к замене без изменения Actions.

В ActionEngine все публичные менеджеры — BaseConnectionManager, PostgresConnectionManager, CsvConnectionManager — имеют строгую иерархию и реализуют паттерн порт-адаптер [1].

---

## DI — зависимость через декларацию

Action указывает зависимости декларативно через @depends. DependencyFactory создаёт нужные объекты, кеширует их в рамках одного вызова, поддерживает вложенные действия через run_action и подменяется ActionTestMachine в тестах.

Это не контейнер зависимостей. Это контракт Action с инфраструктурой [1].

---

## Плагины — слой наблюдения

Плагины получают события global_start, before, after, global_finish. Они могут логировать, отправлять метрики, трассировать, аудировать — но не могут менять бизнес-данные.

Ошибки в плагинах изолируются и не останавливают конвейер при ignore_exceptions=True. Observability встроена в архитектуру как фундаментальный принцип, а не добавлена сверху [1].

---

## Context — только для инфраструктуры

Context содержит данные пользователя, запроса и окружения. Он передаётся только в механизм проверки ролей и в плагины. Actions физически не имеют доступа к Context.

Это исключает зависимость бизнес-логики от транспорта и предотвращает превращение Context в God-object [1].

---

## Вложенные действия

Action может вызвать другое действие через deps.run_action. ActionMachine обрабатывает вложенный вызов полностью — увеличивает nest_level, вызывает плагины, повторяет конвейер. Плагины видят полное дерево вызовов автоматически.

Это основной механизм композиции в AOA. Вместо сервисных монолитов — набор маленьких переиспользуемых действий [1].

## Полная архитектурная схема

```
┌─────────────────────────────┐
│     Transport Layer          │
│  HTTP / MCP / CLI / Queue    │
└──────────────┬───────────────┘
               ↓
┌─────────────────────────────┐
│      AuthCoordinator         │
│  Extractor → Auth → Assembler│
│      Context ready           │
└──────────────┬───────────────┘
               ↓
┌─────────────────────────────┐
│       ActionMachine          │
│  Roles / Pipeline / Plugins  │
│  DI / Nesting / Checkers     │
└──────────────┬───────────────┘
               ↓
┌─────────────────────────────┐
│           Action             │
│  Aspects → Summary → Result  │
└──────────────┬───────────────┘
               ↓
┌─────────────────────────────┐
│      Resource Managers       │
│   Adapters / Ports / DB / API│
└─────────────────────────────┘

          ↕ наблюдение
┌─────────────────────────────┐
│          Plugins             │
│  Logging / Metrics / Audit   │
└─────────────────────────────┘
```

---

## Почему такая архитектура удобна

Одна точка входа для любого сценария — machine.run. Строгий порядок выполнения. Неизменяемые данные. Отсутствие побочных эффектов внутри Actions. Лёгкая тестируемость через ActionTestMachine. Высокая повторная используемость через deps.run_action. Простое расширение через плагины. Естественное соответствие гексагональной архитектуре [1].
