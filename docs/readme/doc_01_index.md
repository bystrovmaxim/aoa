# ActionEngine и AOA: Введение

ActionEngine — это архитектура и фреймворк, которые позволяют строить бизнес‑логику как последовательность предсказуемых, изолированных и тестируемых шагов. Она основана на концепции AOA — Action‑Oriented Architecture, архитектурной парадигме, в которой действия становятся ядром системы, а вся инфраструктура работает вокруг них, не вмешиваясь в бизнес‑решения.

Но AOA — это не просто способ структурировать код. Это первая архитектура в Python-экосистеме где каждая бизнес-операция является полностью машиночитаемой спецификацией самой себя.

---

## Что такое AOA

AOA — это подход в котором бизнес‑логика оформлена в виде действий, каждое действие выполняется через линейный пайплайн аспектов, входные и выходные данные неизменяемы, зависимости объявляются декларативно, внешние системы доступны через ресурсные менеджеры, инфраструктура расширяется через плагины, а все эффекты контролируются ActionMachine.

AOA формирует пространство где разработчик пишет только бизнес‑логику, а всё остальное — получение зависимостей, вызов аспектов, порядок выполнения, наблюдение, логирование — выполняется автоматически.

---

## Зачем нужны Actions

Action — это атомарная бизнес‑операция. Он определяет чёткие входные данные через Params, определяет ожидаемый результат через Result, состоит из аспектов — небольших шагов выполнения, не хранит состояние, не знает про HTTP, БД, API или контекст запроса, получает зависимости только через декларативный DI.

Это позволяет легко тестировать логику без инфраструктуры, видеть весь сценарий целиком, переиспользовать действия в других действиях и избегать скрытых связей и побочных эффектов.

---

## Скрытая суперсила: каждый Action — это живая спецификация

Когда вы пишете Action в AOA, вы не просто пишете код. Вы создаёте машиночитаемую спецификацию бизнес-операции. Прямо сейчас, без единой строчки дополнительного кода, для любого Action можно извлечь:

Список зависимостей с описаниями — из атрибута _dependencies. Ролевую модель — из _role_spec. Полный упорядоченный список шагов — через _is_aspect и co_firstlineno. Контракт каждого шага — из _result_checkers. Объявленные соединения с описаниями — из _connections.

Это не документация которая устаревает. Это не комментарии которые врут. Это живые верифицированные данные потому что без них код не запустится. Машина физически не выполнит Action у которого нет CheckRoles, нет summary-аспекта или нарушен контракт чекера.

Из этого факта вытекает нечто важное: AOA позволяет строить статический анализ бизнес-архитектуры. Не анализ кода — анализ бизнес-намерений. Матрицу доступа Action × Role которая всегда актуальна. Граф зависимостей между операциями. Автоматические предупреждения об архитектурных проблемах. Diff бизнес-логики между версиями — не "изменено 47 файлов", а "добавлен шаг проверки фрода, расширен доступ на операторов".

Ни один другой Python-фреймворк не даёт этого потому что ни в одном другом фреймворке бизнес-намерения не формализованы в машиночитаемом виде.

---

## Что такое аспекты

Аспект — это этап бизнес‑операции: валидация, подготовка данных, вычисления, взаимодействие с внешним миром, обработка результата. Они вызываются линейно — сверху вниз, в порядке объявления. Это делает поведение действия предсказуемым: каждый шаг можно увидеть, протестировать и проанализировать отдельно.

---

## Что такое ресурсные менеджеры

Ресурсные менеджеры — это адаптеры внешних систем: баз данных, сторонних API, файловых хранилищ, очередей, сервисов отправки писем. Они управляют состоянием, соединениями, сессиями, ошибками транспорта. Actions используют их через интерфейсы не зная деталей реализации.

Так достигается разделение домена и инфраструктуры: бизнес‑логика не знает как именно хранятся данные.

---

## Что такое ActionMachine

ActionMachine — исполнитель действий. Она запускает аспекты по очереди, вызывает плагины, обеспечивает вложенность вызовов, применяет чекеры и ролевые требования, собирает зависимости, измеряет время выполнения.

Благодаря ActionMachine у всех действий единая точка входа — machine.run(action, params).

И именно ActionMachine хранит метамодель каждого действия. Кеш аспектов, зависимости, ролевые спецификации — всё это живёт в атрибутах класса и доступно через getattr в любой момент, не только во время выполнения.

---

## Плагины

Плагины — это наблюдатели за исполнением действий. Они получают события, могут логировать, собирать метрики, строить трассировки, создавать аудит. Они никогда не влияют на результат выполнения.

Но плагины могут делать нечто большее. Плагин на 50 строк способен записывать семантическое дерево выполнения в JSON — не плоский лог, а структурированный граф с state на каждом узле, duration каждого аспекта и полной вложенностью вызовов. Такое дерево можно передать языковой модели и получить объяснение что именно произошло, где возникла проблема и почему.

Это создаёт безопасный слой расширений не нарушающий чистоту бизнес‑логики.

---

## Почему это проще чем традиционный подход

Потому что нет смешивания инфраструктуры и логики, нет сервисных монолитов, нет непредсказуемых побочных эффектов, нет дублирования логики по контроллерам, есть единый шаблон для всех бизнес‑операций, тестируемость встроена в архитектуру, поведение действий видно как на ладони.

AOA заставляет писать код структурированно и держать бизнес‑логику отдельно от всего остального. Обязательные декораторы — это не ограничения. Это принудительная формализация бизнес-намерений которая делает систему анализируемой, аудируемой и понятной машине.
