# Формальные гарантии AOA

AOA — это не просто набор рекомендаций. Это архитектура с жёсткими формальными гарантиями, которые обеспечиваются не договорённостями в команде, а самим устройством ActionEngine. Нарушить эти гарантии случайно практически невозможно — машина выбросит ошибку до того, как некорректный код успеет навредить [1].

---

## Гарантия единственной точки входа

Любая бизнес-операция запускается только одним способом:

```python
machine.run(action, params)
```

Нет возможности войти в середину конвейера, пропустить аспекты или вызвать summary-аспект напрямую. Это исключает ситуации когда разные части кода запускают одну и ту же логику разными путями и получают разные результаты [1].

---

## Гарантия линейного детерминированного конвейера

ActionMachine выполняет аспекты строго в порядке их определения в классе. Порядок определяется по номеру строки в исходном файле через co_firstlineno и не зависит от имён методов, алфавитного порядка или любых других факторов [1].

Если аспект validate определён до аспекта calculate — он всегда выполнится первым. Всегда. Без исключений. Без зависимости от версии Python или окружения.

---

## Гарантия отсутствия состояния в Actions

Action не может накапливать состояние в атрибутах экземпляра. Всё состояние конвейера проходит через state — словарь который существует только внутри одного вызова run и не переживает его завершение.

Это исключает классическую проблему когда один запрос видит остатки данных от предыдущего [1].

---

## Гарантия неизменяемости входных и выходных данных

Params и Result объявляются как frozen dataclass. Python физически запрещает их изменение после создания. Аспект не может случайно изменить входные данные и тем самым повлиять на работу следующего аспекта.

Это исключает целый класс ошибок связанных с мутацией данных в конвейере.

---

## Гарантия изоляции Context от Actions

Context содержит данные пользователя, запроса и окружения. ActionMachine передаёт Context только плагинам и механизму проверки ролей. Actions физически не имеют доступа к Context.

Это означает что бизнес-логика не может случайно начать зависеть от IP-адреса клиента, заголовков запроса или имени хоста. Она работает только с тем что явно передано в Params [1].

---

## Гарантия read-only плагинов

Плагины получают события и могут читать params, state, result, context. Но архитектура не предоставляет им механизма изменения этих данных. Плагин наблюдает за процессом но не участвует в нём.

При этом ошибки в плагине при ignore_exceptions=True не прерывают выполнение бизнес-логики. Сломанный плагин не ломает продакшн [1].

---

## Гарантия обязательной авторизации

Каждый Action обязан иметь декоратор CheckRoles. Если декоратор отсутствует — ActionMachine выбросит ошибку до запуска первого аспекта. Нельзя случайно создать Action без политики доступа [1].

Это означает что авторизация встроена в архитектуру, а не является опциональным дополнением.

---

## Гарантия строгой типизации аспектов

Regular-аспект обязан вернуть dict. Summary-аспект обязан вернуть экземпляр Result. Если аспект вернул что-то другое — ActionMachine выбросит TypeError немедленно.

Это исключает ситуацию когда аспект ничего не вернул, вернул None или случайно вернул число [1].

---

## Гарантия валидации state через чекеры

Если к аспекту привязаны чекеры — они проверяют результат после каждого выполнения. Лишние поля в state, неверные типы, нарушение контракта — всё это немедленно вызывает ValidationFieldException.

Контракт аспекта защищён не только статически через TypedDict, но и динамически через чекеры во время выполнения [1].

---

## Гарантия декларативного DI

Зависимость можно использовать только если она объявлена через @depends. Попытка получить незадекларированную зависимость через deps.get() вызовет ошибку. Нет возможности тихо получить зависимость в обход объявленного контракта.

DependencyFactory кеширует зависимости в рамках одного вызова. Все аспекты одного действия работают с одним и тем же экземпляром сервиса [1].

---

## Гарантия корректной вложенности

При вызове deps.run_action() уровень вложенности nest_level увеличивается. Плагины получают корректный уровень вложенности для каждого события. Трассировка вложенных вызовов работает автоматически без дополнительной конфигурации.

---

## Гарантия атомарности действия

Каждое действие имеет ровно один summary-аспект. Если summary-аспект отсутствует или их больше одного — машина выбросит ошибку при первом же запуске. Это проверяется при сборке конвейера, а не в момент выполнения [1].

---

## Гарантия тестируемости

ActionTestMachine воспроизводит поведение ActionProductMachine полностью. Любые зависимости можно заменить моками через словарь. MockAction запоминает все вызовы и параметры. Поведение тестовой машины детерминировано и предсказуемо.

Это означает что тест написанный сегодня будет работать одинаково через год, потому что у него нет случайных зависимостей от окружения.

---

## Гарантия безопасности транзакций при вложенности

При передаче соединения в дочернее действие через connections оно автоматически оборачивается в прокси. Прокси блокирует open, commit, rollback и разрешает только execute. Дочернее действие физически не может закрыть или зафиксировать транзакцию родителя.

Это гарантируется архитектурой, а не соглашением [1].

---

## Гарантия живой документации

Это гарантия которой нет ни в одном другом Python-фреймворке.

Каждый Action является полностью машиночитаемой спецификацией самого себя. Список зависимостей с описаниями доступен через _dependencies. Ролевая модель доступна через _role_spec. Упорядоченный список шагов — через _is_aspect и co_firstlineno. Контракт каждого шага — через _result_checkers. Объявленные соединения — через _connections.

Эти данные не расходятся с кодом и не устаревают потому что без них код не запустится. Машина физически не выполнит Action у которого нет CheckRoles, нет summary-аспекта или нарушен контракт чекера.

Это открывает возможности недоступные в других архитектурах: матрица доступа Action × Role собираемая из кода автоматически, граф зависимостей между операциями, статический анализ архитектурных проблем, diff бизнес-логики между версиями. Не анализ кода — анализ бизнес-намерений, которые формализованы принудительно и верифицированы машиной.

---

## Гарантия детерминированности

Одинаковые входные данные при одинаковых зависимостях дают одинаковый результат. Конвейер не содержит случайных элементов, скрытого глобального состояния или недетерминированных ветвлений.

Это делает систему предсказуемой и легко отлаживаемой.
