# Версия 1.0.0: Единый протокол доступа к данным

## Мотивация

Работа с данными в ActionMachine была неоднородной:
- Параметры действий часто реализовались как `dataclass` (доступ через атрибуты).
- Результаты могли быть как `dataclass`, так и простыми словарями.
- Промежуточное состояние (`state`) между аспектами оставалось бесформенным — никто строго не контролировал его структуру, что приводило к ошибкам времени выполнения.
- Плагины вынуждены были угадывать способ доступа (`params.attr` или `params["key"]`), что делало их хрупкими и сложными в поддержке.

**Цель рефакторинга** – ввести единый, строго типизированный интерфейс для чтения и записи данных, который будет одинаково работать для любых реализаций (dataclass, TypedDict, простые dict). Это позволит:
- Плагинам и ядру работать с данными предсказуемо.
- Разработчикам получать все преимущества статической типизации (mypy, IDE-подсказки) и чёткие контракты.
- Упростить внутреннее устройство и повысить надёжность.

## Ключевые изменения

### 1. Протоколы доступа

Введены два протокола (`Core/Protocols.py`):

- `ReadableDataProtocol` – требует реализации `__getitem__`, `__contains__`, `get`, `keys`, `values`, `items`. Все методы возвращают `object` (совместимо с mypy --strict).
- `WritableDataProtocol` – расширяет читаемый протокол методом `__setitem__`.

Любой объект, удовлетворяющий этим протоколам, может использоваться как `Params` или `Result`. Это открывает возможность использовать не только dataclass, но и TypedDict, и даже пользовательские классы, реализующие нужные методы.

### 2. Миксины для dataclass

Чтобы существующие dataclass-модели не ломались, созданы миксины (`Core/DataAccessMixins.py`):

- `ReadableMixin` – реализует `ReadableDataProtocol` через атрибуты (использует `getattr`, `hasattr`, `vars`).
- `WritableMixin` – добавляет `__setitem__`, используя `setattr`.

Теперь достаточно унаследовать класс параметров от `BaseParams` (который включает `ReadableMixin`), и он автоматически получает dict-доступ. Аналогично для результатов.

### 3. State = TypedDict

Самое важное изменение для аспектов: **state теперь всегда словарь** (точнее, объект, соответствующий `ReadableDataProtocol`, но на практике – `dict` или `TypedDict`). Это гарантирует:

- Максимальную производительность (нет накладных расходов на прокси).
- Полную совместимость с плагинами (они работают с `event.state_aspect` как с обычным dict).
- Возможность строгой типизации через `TypedDict`.

Каждое действие должно определить свой `TypedDict` для state (даже пустой). Аспекты аннотируются этим типом и работают только через dict-доступ.

Пример:
```python
class MyState(TypedDict, total=False):
    loaded: bool
    data: dict

@aspect("Загрузка")
async def load(self, params, state: MyState, deps, connections) -> MyState:
    state["loaded"] = True
    state["data"] = await deps.get(Loader).fetch()
    return state

### 4. Обновление плагинов

Плагины больше не могут полагаться на атрибутный доступ к params или state_aspect. Все обращения должны идти через квадратные скобки или методы get(), keys() и т.д.

Сигнатура get_initial_state изменена с Any на object – это сигнал, что состояние плагина может быть любым, но обычно это словарь.

Класс PluginEvent теперь имеет строгие типы:

python
params: ReadableDataProtocol
state_aspect: Optional[dict[str, object]]
result: Optional[WritableDataProtocol]
### 5. Типизация всего проекта
Убрано подавляющее большинство Any – заменено на object или конкретные типы.

Везде, где требуется безопасное приведение, используется cast из typing.

mypy --strict теперь проходит без ошибок на всех 42 модулях.
