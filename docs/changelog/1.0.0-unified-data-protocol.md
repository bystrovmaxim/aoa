### Added
- **Единый протокол доступа к данным** – введены интерфейсы `ReadableDataProtocol` и `WritableDataProtocol`, которые стандартизируют работу с параметрами, результатами и состоянием действий.

## Статья: Версия 1.0.0 – Единый протокол доступа к данным и асинхронная надёжность

### Мотивация

Работа с данными в ActionMachine была неоднородной:
- Параметры действий часто реализовались как `dataclass` (доступ через атрибуты).
- Результаты могли быть как `dataclass`, так и простыми словарями.
- Промежуточное состояние (`state`) между аспектами оставалось бесформенным — никто строго не контролировал его структуру, что приводило к ошибкам времени выполнения.
- Плагины вынуждены были угадывать способ доступа (`params.attr` или `params["key"]`), что делало их хрупкими и сложными в поддержке.
- Кроме того, некоторые компоненты (аутентификация, инициализация плагинов) использовали синхронные методы внутри асинхронного контекста, создавая риск блокировки event loop.

**Цель рефакторинга** – ввести единый, строго типизированный интерфейс для чтения и записи данных, который будет одинаково работать для любых реализаций (dataclass, TypedDict, простые dict), а также устранить все выявленные асинхронные узкие места. Это позволит:
- Плагинам и ядру работать с данными предсказуемо.
- Разработчикам получать все преимущества статической типизации (mypy, IDE-подсказки) и чёткие контракты.
- Упростить внутреннее устройство и повысить надёжность.
- Гарантировать неблокирующее выполнение в асинхронных сценариях.

## Ключевые изменения

### 1. Протоколы доступа

Введены два протокола (`Core/Protocols.py`):

- `ReadableDataProtocol` – требует реализации `__getitem__`, `__contains__`, `get`, `keys`, `values`, `items`. Все методы возвращают `object` (совместимо с mypy --strict).
- `WritableDataProtocol` – расширяет читаемый протокол методом `__setitem__`.

Любой объект, удовлетворяющий этим протоколам, может использоваться как `Params` или `Result`. Это открывает возможность использовать не только dataclass, но и TypedDict, и даже пользовательские классы, реализующие нужные методы.

### 2. Миксины для dataclass

Чтобы существующие dataclass-модели не ломались, созданы миксины (`Core/DataAccessMixins.py`):

- `ReadableMixin` – реализует `ReadableDataProtocol` через атрибуты (использует `getattr`, `hasattr`, `vars`).
- `WritableMixin` – добавляет `__setitem__`, используя `setattr`.

Теперь достаточно унаследовать класс параметров от `BaseParams` (который включает `ReadableMixin`), и он автоматически получает dict-доступ. Аналогично для результатов.

### 3. State = TypedDict

Самое важное изменение для аспектов: **state теперь всегда словарь** (точнее, объект, соответствующий `ReadableDataProtocol`, но на практике – `dict` или `TypedDict`). Это гарантирует:

- Максимальную производительность (нет накладных расходов на прокси).
- Полную совместимость с плагинами (они работают с `event.state_aspect` как с обычным dict).
- Возможность строгой типизации через `TypedDict`.

Каждое действие должно определить свой `TypedDict` для state (даже пустой). Аспекты аннотируются этим типом и работают только через dict-доступ.

Пример:
```python
class MyState(TypedDict, total=False):
    loaded: bool
    data: dict

@aspect("Загрузка")
async def load(self, params, state: MyState, deps, connections) -> MyState:
    state["loaded"] = True
    state["data"] = await deps.get(Loader).fetch()
    return state
```

### 4. Асинхронная аутентификация

Интерфейсы пакета `Auth` (`Authenticator`, `CredentialExtractor`, `ContextAssembler`) и класс `AuthCoordinator` переведены на асинхронные методы. Это позволяет реализациям выполнять I/O-операции (например, проверку токена через внешний API) без блокировки event loop.

**Изменения в коде:**
- Все абстрактные методы теперь объявлены как `async def`.
- `AuthCoordinator.process()` стал асинхронным и использует `await` для вызовов компонентов.
- Вызывающий код (например, FastAPI-зависимости) должен использовать `await coordinator.process(...)`.

**Пример обновлённого координатора:**
```python
async def process(self, request_data: Any) -> Optional[Context]:
    credentials = await self.extractor.extract(request_data)
    if not credentials:
        return None
    user_info = await self.authenticator.authenticate(credentials)
    if not user_info:
        return None
    metadata = await self.assembler.assemble(request_data)
    req_info = RequestInfo(**metadata)
    return Context(user=user_info, request=req_info, environment=None)
```

### 5. Безопасная инициализация плагинов

Ранее синхронный метод `Plugin.get_initial_state()` вызывался напрямую внутри асинхронного конвейера, что могло привести к блокировке, если реализация выполняла длительные операции. Теперь вызов перенесён в executor:

```python
async def _init_plugin_states(self) -> None:
    loop = asyncio.get_running_loop()
    for plugin in self._plugins:
        plugin_id = id(plugin)
        if plugin_id not in self._plugin_states:
            state = await loop.run_in_executor(None, plugin.get_initial_state)
            self._plugin_states[plugin_id] = state
```

Это гарантирует, что даже «тяжёлая» инициализация не повлияет на производительность event loop.

### 6. Улучшение `sync_run`

Метод `BaseActionMachine.sync_run()` получил более точное сообщение об ошибке при вызове из async-контекста, помогающее разработчикам быстро диагностировать проблему.

### 7. Типизация всего проекта

- Убрано подавляющее большинство `Any` – заменено на `object` или конкретные типы.
- Везде, где требуется безопасное приведение, используется `cast` из `typing`.
- `mypy --strict` теперь проходит без ошибок на всех 42 модулях.
- `pylint` вернулся к идеальным 10.00/10.

## Руководство по миграции

### Шаг 1: Обновить базовые классы

Замените старые импорты (файлы обновлены, импорты остаются теми же):
```python
from ActionMachine.Core.BaseParams import BaseParams
from ActionMachine.Core.BaseResult import BaseResult
```

### Шаг 2: Определить TypedDict для state

Для каждого действия, которое использует регулярные аспекты, создайте TypedDict:
```python
from typing import TypedDict

class MyActionState(TypedDict, total=False):
    some_field: str
    another: int
```

### Шаг 3: Переписать аспекты на dict-доступ

- Заменить `params.field` на `params["field"]`.
- Заменить `state.field` на `state["field"]`.
- Присваивания: `state["field"] = value`.
- При возврате state из аспекта использовать словарь (можно `dict`, можно экземпляр TypedDict).

### Шаг 4: Обновить плагины

В методах плагинов:
- Заменить `event.params.field` на `event.params["field"]` (или `event.params.get("field")`).
- Для `event.state_aspect` аналогично.
- Убедиться, что `get_initial_state` возвращает объект (обычно dict). Если метод выполняет I/O, его всё равно можно оставить синхронным – он будет вызван в executor.

### Шаг 5: Обновить реализации аутентификации

Если вы используете собственные реализации `Authenticator`, `CredentialExtractor`, `ContextAssembler`, добавьте `async` перед `def` и, при необходимости, используйте `await` для внутренних вызовов. Если реализация не выполняет I/O, просто добавьте `async` – это не повлияет на производительность.

### Шаг 6: Адаптировать вызовы `AuthCoordinator`

Там, где вызывается `coordinator.process()`, добавьте `await`. Если вызов происходит из синхронного кода, используйте `asyncio.run()` или обёртку.

### Шаг 7: Тесты

Адаптировать тесты: создавать параметры как dataclass или TypedDict; state в вызовах аспектов передавать как dict; проверять через dict-доступ; для асинхронных компонентов использовать `pytest.mark.anyio`.

### Шаг 8: Проверка

Запустить `mypy --strict`, `pylint`, `pytest`. Все должно проходить.
