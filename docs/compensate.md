# План внедрения компенсации (Sagas) в ActionMachine

## Преамбула

### Что такое компенсация

В распределённых транзакциях и длительных бизнес-процессах часто невозможно использовать двухфазный коммит. Вместо этого применяется паттерн Saga: каждая операция имеет компенсирующую операцию, которая отменяет её эффекты. При сбое на любом шаге выполняются компенсации уже выполненных операций в обратном порядке.

ActionMachine изначально поддерживает конвейер регулярных аспектов. Если на каком-то шаге возникает ошибка, вся предыдущая работа не откатывается. Компенсация заполняет этот пробел.

### Цель

Добавить в ActionMachine механизм, позволяющий объявить для каждого регулярного аспекта метод-компенсатор. При возникновении ошибки в любом аспекте все успешно выполненные аспекты (включая аспекты вложенных действий) будут откачены в обратном порядке. После отката вызывается штатный обработчик `@on_error`.

### Область действия

- Компенсаторы определяются только для регулярных аспектов (не для summary).
- Компенсаторы могут использовать `@context_requires`.
- Компенсаторы поддерживают вложенные вызовы через `box.run()` без единого глобального стека (каждый уровень имеет свой стек).
- Компенсаторы не наследуются.
- Ошибки в компенсаторах **не должны прерывать размотку стека**. Они никак не обрабатываются (не логируются, не пробрасываются) — просто игнорируются, и размотка продолжается со следующим компенсатором.

### Мотивация

**Без компенсации:**
- Частично выполненное действие оставляет систему в неконсистентном состоянии.
- Разработчикам приходится вручную писать сложный код отката в `@on_error`.
- Невозможно безопасно выполнять длительные цепочки шагов.

**С компенсацией:**
- Декларативное описание отката.
- Автоматическая размотка в обратном порядке.
- Чёткое разделение: компенсатор отменяет эффект аспекта, `@on_error` обрабатывает бизнес-ошибку.

## План

### 1. Декоратор `@compensate`

- Создать декоратор `@compensate(regular_method, description)`, где:
  - `regular_method` — ссылка на метод-аспект (unbound), который компенсируется. Обязательный параметр.
  - `description` — текстовая строка. Обязательный параметр.
- Декоратор применяется к методу-компенсатору.
- При применении декоратор проверяет:
  - `regular_method` существует и является регулярным аспектом (имеет `_new_aspect_meta` с `type="regular"`).
  - `description` — непустая строка.
  - Метод-компенсатор — асинхронный (`async def`).
  - Имя метода-компенсатора заканчивается на `_compensate`.
  - Сигнатура метода-компенсатора соответствует пункту 2.
- Декоратор записывает в метод-компенсатор атрибут `_compensate_meta`:
  ```python
  func._compensate_meta = {
      "target_method": regular_method,
      "description": description,
  }
  ```
- Запретить повторное применение `@compensate` для одного и того же `regular_method` (один аспект — строго один компенсатор).

### 2. Сигнатура компенсатора

Метод-компенсатор должен иметь следующую сигнатуру (без учёта `self`):

**Без `@context_requires`:**
```python
async def compensate_method(self, params, state_before, state_after, box, connections, error)
```

**С `@context_requires`:**
```python
async def compensate_method(self, params, state_before, state_after, box, connections, error, ctx)
```

**Параметры:**
- `params` — входные параметры действия (`BaseParams`).
- `state_before` — состояние **до** выполнения аспекта (`BaseState`).
- `state_after` — состояние **после** выполнения аспекта (`BaseState | None`). Равно `None`, если аспект упал до того, как вернул dict.
- `box` — экземпляр `ToolsBox` (тот же, что использовался в аспекте).
- `connections` — словарь ресурсных менеджеров.
- `error` — исключение, которое вызвало размотку.
- `ctx` — `ContextView` (только при наличии `@context_requires`).

Количество параметров проверяется декоратором. Имена параметров не важны, важен порядок.

### 3. Связь компенсатора с регулярным аспектом

- Компенсатор ссылается на реальный метод регулярного аспекта через аргумент `regular_method` декоратора.
- Координатор при сборке метаданных:
  - Находит все методы с `_compensate_meta`.
  - Для каждого проверяет, что целевой метод действительно является аспектом в том же классе.
  - Сохраняет связь в `ClassMetadata.compensators` (словарь: имя аспекта → `CompensatorMeta`).
- Если целевой аспект не найден или не является регулярным — ошибка сборки.

### 4. Именование

- Декоратор называется `@compensate`.
- Метод-компенсатор должен иметь суффикс `_compensate` (например, `rollback_payment_compensate`).
- Проверка суффикса выполняется декоратором и координатором.

### 5. Уникальность

- Для одного регулярного аспекта может быть определён **не более одного** компенсатора.
- Если в классе встречается второй `@compensate` с тем же `regular_method` — ошибка сборки.

### 6. Наследование

- Компенсаторы **не наследуются** от родительского класса.
- При сборке метаданных `MetadataBuilder` собирает компенсаторы только из текущего класса (`vars(cls)`), игнорируя MRO.
- Если дочерний класс хочет иметь компенсатор для унаследованного аспекта, он должен объявить его явно.

### 7. Ошибки в компенсаторах

- Если при вызове компенсатора возникает исключение, оно **игнорируется** (не логируется, не пробрасывается).
- Размотка стека продолжается со следующим компенсатором.
- Никакие события для таких ошибок не генерируются.

### 8. Стек компенсации в `_run_internal`

- Каждый вызов `_run_internal` создаёт локальный список `saga_stack: list[SagaFrame]`.
- `SagaFrame` содержит:
  - `compensator: Optional[CompensatorMeta]`
  - `params: BaseParams`
  - `state_before: BaseState`
  - `state_after: BaseState | None`
  - `connections: dict[str, BaseResourceManager]`
  - `context: Context`
  - `aspect_name: str`
- При успешном выполнении регулярного аспекта (получен dict, чекеры пройдены) в стек добавляется фрейм с `state_before` (старое состояние) и `state_after` (новое состояние).
- При ошибке в аспекте фрейм не добавляется, начинается размотка стека (пункт 9).
- При вложенном вызове `box.run()` дочерний `_run_internal` создаёт **новый пустой стек**. Родительский стек не передаётся.

### 9. Размотка стека (вызов компенсаторов)

- При ошибке в любом аспекте текущего уровня вызывается функция `_rollback_saga(saga_stack, error, box, action)`.
- `_rollback_saga` проходит по стеку **в обратном порядке**.
- Для каждого фрейма:
  - Если у фрейма нет компенсатора (`compensator is None`) — пропуск.
  - Если компенсатор есть — вызывается метод-компенсатор с параметрами из фрейма и исходной ошибкой.
  - Если компенсатор выбрасывает исключение — оно **игнорируется**, размотка продолжается со следующим фреймом.
- После размотки стека управление передаётся `@on_error` (пункт 12).

### 10. Особенности для вложенных вызовов

- Если аспект вызвал `box.run(ChildAction)` и в дочернем действии произошла ошибка:
  - Дочерний `_run_internal` разматывает свой стек, затем пробрасывает исключение.
  - В родительском `_run_internal` это исключение перехватывается.
  - Родительский аспект, содержащий `box.run()`, **не добавляется** в стек родителя (он не завершился успешно).
  - Затем разматывается родительский стек (только те аспекты, которые успели выполниться до вызова `box.run()`).

### 11. События плагинов для компенсаторов

- Плагины могут подписываться на события:
  - `before:compensate:{aspect_name}` — перед вызовом компенсатора.
  - `after:compensate:{aspect_name}` — после вызова компенсатора (даже если он упал).
- В `PluginEvent` для этих событий передаются:
  - `event_name`
  - `action_name`
  - `params`
  - `state_aspect` (содержит `state_before` из фрейма)
  - `state_after` (новое поле в `PluginEvent`)
  - `error` (исходная ошибка)
  - `duration` (для after-события)
  - `nest_level`
- Scope логгера для компенсатора (если плагин использует `log`): `machine`, `mode`, `action`, `aspect=frame.aspect_name`, `event="before:compensate"`, `nest_level`.

### 12. Порядок: компенсаторы → `@on_error`

- При ошибке в регулярном аспекте:
  1. Вызываются компенсаторы (размотка стека).
  2. После завершения размотки вызывается штатный `@on_error` обработчик (если есть) с исходной ошибкой.
- Если `@on_error` возвращает `Result` — это результат действия.
- Если `@on_error` выбрасывает исключение — оно пробрасывается наверх (в родительский вызов или наружу).

## Результат

После реализации этого плана в ActionMachine появится полноценный механизм компенсации, соответствующий паттерну Saga, с поддержкой вложенных вызовов, независимыми стеками на каждом уровне, отсутствием наследования компенсаторов и игнорированием ошибок в компенсаторах.
