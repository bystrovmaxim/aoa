## 1. Проблемы, которые решают шлюзы

В ActionMachine существует множество компонентов, которые должны быть зарегистрированы, организованы и доступны для выполнения: аспекты, зависимости, соединения, чекеры, роли, плагины. Ранние версии фреймворка использовали «магические» подходы:

- **Скрытые атрибуты** – декораторы добавляли специальные поля (`_is_aspect`, `_dependencies`, `_connections`), которые затем читались машиной.
- **Смешанная ответственность** – машина сама собирала компоненты, используя рефлексию (`inspect`), и кэшировала их. Это нарушало принцип единственной ответственности.
- **Сложность тестирования** – чтобы подменить набор компонентов, нужно было либо мокать машину, либо использовать сложные механизмы.
- **Отсутствие инкапсуляции** – любой код мог читать или даже изменять атрибуты компонентов.
- **Неединообразие** – каждый тип компонентов обрабатывался по-своему, без общего контракта.

**Шлюзы (gates)** – это архитектурный паттерн, призванный заменить «магию» на явное, типобезопасное и тестируемое управление компонентами.

---

## 2. Что такое шлюз?

Шлюз – это контейнер, который хранит компоненты определённого типа. Он предоставляет единый интерфейс для:

- **Регистрации** – добавления компонента с возможными метаданными (описание, тип, приоритет).
- **Удаления** – удаления компонента по ссылке.
- **Получения списка** – получения всех зарегистрированных компонентов (часто в порядке регистрации).

Шлюз **не знает**, для чего используются компоненты – он только управляет коллекцией. Это позволяет использовать один и тот же механизм для разных сущностей: аспектов, зависимостей, соединений, чекеров, ролей и т.д.

Шлюз может быть:

- **Классовым** – общим для всех экземпляров.
- **Экземплярным** – уникальным для каждого объекта.
- **Неизменным** – после заполнения компоненты не меняются (что упрощает понимание и исключает гонки).
- **Динамическим** – с возможностью добавления/удаления в рантайме (например, для тестов).

## 3. Базовый интерфейс `BaseGate`

В ActionMachine все шлюзы наследуются от абстрактного класса `BaseGate` (находится в `action_machine/Core/base_gate.py`):

```python
from abc import ABC, abstractmethod
from typing import TypeVar, Any

T = TypeVar('T')

class BaseGate[T](ABC):
    @abstractmethod
    def register(self, _component: T, **metadata: Any) -> T:
        """Зарегистрировать компонент."""
        pass

    @abstractmethod
    def unregister(self, _component: T) -> None:
        """Удалить компонент."""
        pass

    @abstractmethod
    def get_components(self) -> list[T]:
        """Вернуть все зарегистрированные компоненты (порядок значим)."""
        pass
```

Этот интерфейс параметризован типом `T` – классом компонента, которым управляет шлюз. Такой подход обеспечивает строгую типизацию и позволяет статическим анализаторам (mypy) отлавливать ошибки.

## 4. Пример: `AspectGate` – управление аспектами действия

Один из первых шлюзов, реализованных в ActionMachine, – `AspectGate`. Он управляет аспектами (методами действия). Конкретная реализация добавляет два дополнительных метода для удобства:

```python
class AspectGate(BaseGate[AspectMethodProtocol]):
    def __init__(self):
        self._regular: list[tuple[AspectMethodProtocol, str]] = []
        self._summary: tuple[AspectMethodProtocol, str] | None = None

    def register(self, _component: AspectMethodProtocol, **metadata: Any) -> AspectMethodProtocol:
        description = metadata.get("description")
        aspect_type = metadata.get("type", "regular")

        if aspect_type == "regular":
            self._regular.append((_component, description))
        else:  # summary
            if self._summary is not None:
                raise ValueError("Only one summary aspect allowed")
            self._summary = (_component, description)
        return _component

    def unregister(self, component: AspectMethodProtocol) -> None:
        # удаление из списков

    def get_components(self) -> list[AspectMethodProtocol]:
        # возвращает все аспекты (сначала обычные, потом summary)

    def get_regular(self) -> list[tuple[AspectMethodProtocol, str]]:
        """Возвращает список обычных аспектов."""
        return self._regular.copy()

    def get_summary(self) -> tuple[AspectMethodProtocol, str] | None:
        """Возвращает summary-аспект."""
        return self._summary
```

`AspectGate` хранит два набора: обычные аспекты (в порядке регистрации) и один summary-аспект. Он не знает, как эти методы будут вызваны – только обеспечивает хранение и извлечение.

## 5. Интеграция с классами: `GateHost` – как присоединить шлюз

Чтобы класс действия мог использовать шлюз, он должен включать его в себя. Для этого используется миксин `AspectGateHost` (в `aspect_gate_host.py`). Этот миксин:

- Создаёт экземпляр шлюза при создании класса (в `__init_subclass__`).
- Собирает методы, помеченные декораторами, и регистрирует их в шлюзе.
- Предоставляет публичный метод `get_aspects()`, который возвращает уже закешированные данные (обычные аспекты и summary-аспект).
- **Не позволяет изменять шлюз после создания класса** – нет методов `add_regular_aspect` и т.п. Это сознательное решение: структура аспектов должна быть фиксированной.

Аналогичные хосты могут быть созданы для других типов компонентов. Например, `DependencyGateHost` для управления зависимостями, `ConnectionGateHost` для соединений, `RoleGateHost` для ролей. Все они будут использовать один и тот же паттерн:

- В `__init_subclass__` собирать компоненты из атрибутов/декораторов.
- Регистрировать их в шлюзе.
- Кэшировать результат.
- Предоставлять метод `get_dependencies()` или `get_connections()` для машины.

## 6. Кэширование и неизменность

Важное улучшение, которое стало возможным благодаря шлюзам, – **кэширование данных на уровне класса**. После того как класс создан, его шлюз заполняется, и больше не меняется. Поэтому мы можем извлечь из шлюза копии списков и сохранить их в приватных атрибутах класса. Например, в `AspectGateHost`:

```python
class AspectGateHost:
    __regular_aspects: list[tuple[AspectMethodProtocol, str]] | None = None
    __summary_aspect: tuple[AspectMethodProtocol, str] | None = None

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        gate = AspectGate()
        # регистрируем аспекты в gate
        cls.__class_gate = gate
        # кэшируем данные
        cls.__regular_aspects = gate.get_regular()
        cls.__summary_aspect = gate.get_summary()

    def get_aspects(self):
        return self.__class__.__regular_aspects or [], self.__class__.__summary_aspect
```

Такой подход:

- Убирает необходимость держать шлюз в памяти после инициализации (можно даже удалить `__class_gate`).
- Исключает повторные вызовы `get_regular()`/`get_summary()`.
- Делает данные доступными за O(1) без косвенных обращений.

Если бы шлюзы были изменяемыми, пришлось бы каждый раз обращаться к ним или инвалидировать кэш. Но наша архитектура предполагает неизменность, что значительно упрощает жизнь.

## 7. Масштабирование: другие шлюзы в ActionMachine

Концепция шлюзов не ограничивается аспектами. В ActionMachine уже есть или могут быть созданы:

### 7.1 `DependencyGate` – управление зависимостями
Вместо того чтобы хранить зависимости в списке `_dependencies` и создавать фабрику, можно зарегистрировать каждую зависимость в шлюзе. Шлюз будет хранить информацию о классе, фабрике и описании. Машина будет получать список зависимостей через `get_dependencies()`.

### 7.2 `ConnectionGate` – управление соединениями
Сейчас соединения хранятся в списке `_connections`. Шлюз позволил бы единообразно регистрировать их, проверять уникальность ключей и получать список для валидации.

### 7.3 `RoleGate` – управление ролями
Ролевая спецификация (`_role_spec`) – это по сути один компонент, но может быть расширена до набора правил. Шлюз позволил бы регистрировать несколько проверок ролей и комбинировать их (AND/OR).

### 7.4 `CheckerGate` – управление чекерами полей
Чекеры сейчас прикрепляются к методам через декораторы. Можно было бы создать шлюз, который хранит все чекеры для класса или метода, и машина применяла бы их в правильном порядке.

### 7.5 Плагины
Плагины уже управляются через `PluginCoordinator`, который по сути является шлюзом: хранит список плагинов, умеет получать обработчики по событиям. Можно было бы сделать его наследником `BaseGate`, чтобы унифицировать API.

## 8. Преимущества для тестирования и расширения

### 8.1 Тестирование
- **Замена шлюза** – в тестах можно подменить классовый шлюз на пустой или предопределённый, чтобы изолировать тестируемый компонент.
- **Мокирование** – легко замокать метод `get_aspects()` или `get_dependencies()`.
- **Проверка регистрации** – можно проверить, что определённый компонент был зарегистрирован (например, через `assert component in gate.get_components()`).

### 8.2 Расширение
- **Добавление новых типов компонентов** – достаточно создать новый шлюз, наследующий `BaseGate`, и соответствующий хост-миксин. Весь механизм сбора и кэширования остаётся одинаковым.
- **Динамическая конфигурация** – хотя мы выбрали неизменность для аспектов, шлюзы могут быть сделаны изменяемыми, если это необходимо (например, для плагинов). Интерфейс позволяет это.
- **Гетерогенные коллекции** – шлюзы могут хранить компоненты разных типов (с помощью `Union` или обобщений), но лучше оставить однородными.

---

## 9. Заключение

Шлюзы – это не просто замена «магии» на явные структуры данных. Это архитектурный паттерн, который:

- **Стандартизирует** способ регистрации и получения компонентов.
- **Улучшает типизацию** и безопасность кода.
- **Упрощает тестирование** за счёт лёгкой подмены коллекций.
- **Снижает связность** – машина не знает, как именно компоненты собраны, она только вызывает методы шлюза.
- **Позволяет легко масштабировать** систему на новые типы компонентов.

В ActionMachine шлюзы уже применяются для аспектов, и планируется постепенный перевод на эту модель всех остальных компонентов (зависимости, соединения, чекеры, роли). Это делает фреймворк более прозрачным, надёжным и готовым к будущим расширениям.
