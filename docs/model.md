# Финальный план ActionMachine: Entity Model (v1.0)

## Полная преамбула: архитектурные принципы и решения

Этот документ фиксирует не только порядок разработки, но и **почему** каждое решение принято. Любой разработчик, открывший план через полгода, должен понять логику, не читая переписку.

### Что мы строим

Декларативную модель предметной области на Python. Модель описывает сущности, их поля, связи между ними и жизненные циклы — и больше ничего. Она не знает про базы данных, HTTP, файлы или любой другой внешний мир. Это чистое ядро в терминах гексагональной архитектуры.

Из этой модели автоматически извлекается вся информация, необходимая для построения ArchiMate‑диаграмм с правильными типами связей (Composition, Aggregation, Association), для генерации OCEL 2.0 метаданных (типы объектов, типы связей, жизненные циклы), и для статической проверки целостности архитектуры до запуска приложения.

### Почему модель живёт внутри ActionMachine

Entity Model — не отдельный пакет и не внешняя зависимость. Она живёт в `src/action_machine/domain/` как часть ядра ActionMachine. Причина: модель и действия — две стороны одной системы. Модель описывает статику (что есть), действия описывают динамику (что происходит). Разделять их в разные пакеты — создавать искусственную границу, которая порождает проблемы с версионированием, циклические зависимости и дублирование типов. Единый репозиторий, единая версия, единые тесты. Координатор модели и координатор действий работают рядом и могут обмениваться информацией без межпакетных контрактов.

ArchiMate‑экспорт, ArchiMate‑импорт и генерация кода из диаграмм — это внешние инструменты. Они зависят от координатора модели, но модель о них не знает. Они будут реализованы в отдельных проектах, вне текущего плана. Координатор предоставляет API (`get_all_entities()`, `get_relations_graph()`, `get_flow()`), которого достаточно для любого внешнего потребителя.

### Почему не ORM

ORM привязывает модель к хранилищу. Наша модель — это спецификация, а не маппинг таблиц. Один и тот же `Order` может читаться из PostgreSQL, MongoDB, REST API или мока — это забота ресурсного менеджера (адаптера), а не модели. Модель не содержит ни одного импорта, связанного с хранилищем.

### Почему не отдельный язык (TLA+, Alloy, DSL)

Потому что модель — это одновременно и спецификация, и исполняемый Python‑код. Разработчик не переключается между двумя мирами. IDE подсказывает поля, mypy проверяет типы, координатор проверяет целостность графа — всё в одном месте. Диаграмму можно построить из кода, а код — сгенерировать из диаграммы (внешними инструментами).

### Почему frozen‑объекты без двусторонних ссылок в рантайме

Модель декларирует связи (Order → Customer, Customer → Orders), но в рантайме объекты не держат живых циклических ссылок друг на друга. `Order` хранит `customer_id` и опционально загруженный объект `Customer`. Обратная связь `Customer.orders` — это метаданные для координатора и ArchiMate, а не список в памяти. Это исключает циклические зависимости, проблемы с сериализацией и скрытые побочные эффекты. Если action нужен клиент заказа — он просит менеджер явно.

### Почему описания обязательны

Каждое поле и каждая связь обязаны иметь текстовое описание. Это не комментарий — это структурное метаданное, которое попадёт в ArchiMate‑диаграмму, в OCEL‑схему, в автогенерированную документацию. Модель без описаний — это код, а не спецификация. Поэтому отсутствие описания — ошибка сборки, а не предупреждение.

### Почему свой автомат, а не python‑statemachine

Рассматривались два варианта: обёртка над библиотекой `python-statemachine` и собственная минимальная реализация. Выбран второй. Наш автомат — это декларативная метаданная для координатора, а не runtime‑движок с колбэками, observers и async‑переходами. Нам нужны только граф состояний, шесть проверок целостности при старте и статический метод `validate_transition`. Это 100–150 строк кода без внешних зависимостей, которые мы полностью контролируем. Библиотека даёт много лишнего и создаёт зависимость, изменение API которой придётся чинить.

### Почему gate‑host паттерн

Каждый декоратор (`@entity`, `@meta`) требует, чтобы класс наследовал соответствующий gate‑host — маркерный миксин, подтверждающий, что разработчик осознанно подключил функциональность. Это тот же паттерн, что в ActionMachine: не магия, а явное согласие. `BaseEntity` предоставляет `EntityGateHost` автоматически, поэтому разработчику не нужно думать об этом — но если кто‑то попытается повесить `@entity` на голый класс, он получит понятную ошибку.

### Почему типы владения, а не просто «связь»

В ArchiMate существует три структурных отношения: Composition, Aggregation, Association. Они несут разную семантику: при удалении родителя композитные дети удаляются, агрегатные — отвязываются, ассоциированные — не затрагиваются. Если модель не различает эти типы, экспортёр не может построить корректную диаграмму, а архитектор не видит реальных зависимостей. Поэтому шесть контейнеров (три типа × две кардинальности) — не перебор, а минимально необходимый набор.

### Почему Inverse обязателен

Автоматический поиск обратных связей ломается при дублировании типов. Если у сущности `Customer` есть два поля `orders` и `invoices`, оба типа `AssociationMany[Order]`, то координатор не сможет определить, какое из них является парой для `Order.customer`. Явный `Inverse` — одна строка, которая делает связь однозначной, читаемой и безопасной при рефакторинге. Поэтому каждая связь обязана иметь либо `Inverse(TargetClass, "field_name")`, либо `NoInverse()` (явное указание, что обратной связи нет). Иначе ошибка сборки.

### Контейнер связи: id всегда, объект — опционально

Контейнер `One[T]` всегда хранит `id` связанной сущности — строку. Полный объект `entity: T | None` может быть загружен менеджером или нет. Обращение к атрибуту контейнера проксируется на `entity`. Если менеджер загрузил клиента — `order.customer.name` вернёт имя. Если загрузил только id — `order.customer.id` вернёт id, а `order.customer.name` выбросит понятную ошибку. Это не lazy‑loading — никаких скрытых запросов к базе. Объект либо есть, либо нет.

### Как модель используется в ресурсных менеджерах

Менеджер — адаптер. Его работа: взять данные из внешнего мира (SQL row, JSON API, MongoDB), преобразовать в типизированные frozen‑сущности и вернуть. Вся грязь (маппинг полей, префиксы, glom‑спецификации) живёт внутри менеджера. Наружу выходят только чистые объекты модели. Action не знает, откуда данные. Модель не знает про форматы. Для упрощения работы менеджера предоставляются утилиты: `EntityDedup` для дедупликации в рамках одного вызова, `hydrate` для сборки из плоских данных, `hydrate_nested` для вложенного JSON.

### Строгий режим hydrate

`hydrate` собирает сущность из плоского словаря по префиксам. Если в словаре нет ключа `order_amount`, а поле `amount` обязательное — Pydantic выбросит `ValidationError` про поле `amount`. Но разработчик не сразу поймёт, что проблема в отсутствующем ключе с префиксом, а не в логике модели.

Строгий режим (`strict=True`) добавляет раннюю проверку перед вызовом Pydantic: для каждого обязательного поля сущности в словаре должен быть ключ `{prefix}_{field_name}`. Если нет — сразу понятная ошибка: «Обязательное поле 'amount' не найдено. Ожидался ключ 'order_amount'.» Лишние ключи не проверяются — они игнорируются.

По умолчанию строгий режим выключен, чтобы не ломать сценарии с частичными данными. Включается явно при разработке и отладке — особенно когда SQL‑запрос или JSON‑схема часто меняются.

### Deprecated‑поля для плавной миграции

Бизнес меняется, поля переезжают между сущностями. Первым шагом старое поле остаётся на месте с пометкой `deprecated`, создаётся новое. Выходит релиз с обратной совместимостью. Когда все перешли — старое удаляется. Для простых полей используется `Field(deprecated=True)` (нативная поддержка Pydantic v2 и IDE). Для связей — `Annotated[..., deprecated("причина")]` по PEP 702 через `typing_extensions`. IDE зачёркивает имя, mypy выдаёт предупреждение.

### Утилиты гидрации — не ORM, а помощники менеджера

`hydrate` собирает frozen‑сущность из плоского словаря по префиксам ключей. `hydrate_nested` делает то же из вложенного JSON, рекурсивно создавая дочерние сущности. Ни та, ни другая функция не знает про хранилище — они работают с готовым словарём. Если внешний JSON не совпадает со структурой модели, менеджер сначала трансформирует его (например, через `glom`), а потом вызывает `hydrate_nested`. Трансформация — ответственность адаптера, не модели.

## Этап 1 — Конечные автоматы (v0.1)

**Зачем:** у большинства бизнес‑сущностей есть жизненный цикл. Если он не описан явно, каждый разработчик изобретает его заново, допускает невалидные переходы, и никто не замечает это до продакшена. Автомат, описанный прямо внутри сущности, служит живой документацией, проверяется на целостность при старте приложения и даёт менеджеру записи инструмент для рантайм‑валидации переходов.

**Архитектурное решение:**  
Проверки целостности автомата выполняются **координатором сущностей (`EntityCoordinator`)** при старте приложения, в момент сборки метаданных. Это соответствует общему принципу ActionMachine: все проверки конфигурации выполняются в координаторе до обработки первого запроса. Автомат остаётся простым контейнером данных, без метаклассов.

**Как устроен:**

Внутри сущности объявляется поле, инициализируемое вызовом `BaseState(description)`, где `description` — текстовое описание самого автомата. Далее через fluent‑цепочку добавляются состояния:

- `.state(key, display_name)` — создаёт новое состояние, возвращает builder.
- `.to(*target_keys)` — задаёт переходы, возвращает тот же builder.
- `.initial()`, `.intermediate()`, `.final()` — помечают состояние, завершают его описание и возвращают исходный `BaseState`.

**Пример:**

```python
class OrderEntity(BaseEntity):
    _state = (
        BaseState("Жизненный цикл заказа")
        .state("new", "Новый заказ").to("confirmed", "cancelled").initial()
        .state("imported", "Импортирован").to("confirmed").initial()
        .state("confirmed", "Подтверждён").to("shipped", "cancelled").intermediate()
        .state("shipped", "Отправлен").to("delivered").intermediate()
        .state("delivered", "Доставлен").final()
        .state("cancelled", "Отменён").final()
    )
```

**Рантайм API** (доступны через поле сущности, например `OrderEntity._state`):

- `get_states() -> dict[str, StateInfo]`
- `get_initial_keys() -> set[str]`
- `get_final_keys() -> set[str]`
- `get_transitions() -> dict[str, set[str]]`
- `validate_transition(old_key, new_key) -> bool`

**Проверки целостности (координатор при старте):**

1. Каждое состояние завершено флагом (нет незавершённых builder’ов).
2. Есть хотя бы одно начальное состояние (`.initial()`).
3. Есть хотя бы одно финальное состояние (`.final()`).
4. Финальные состояния не имеют переходов (`.to()` пуст).
5. Все цели переходов существуют (ссылаться можно только на уже объявленные состояния; порядок объявления важен).
6. Каждое не‑финальное состояние имеет хотя бы один переход.
7. Из каждого начального состояния достижимо хотя бы одно финальное (обход графа).

Нарушение любого правила — исключение, приложение не запускается.

**Размещение:** `src/action_machine/domain/state.py`

## Этап 2 — Базовая сущность (v0.2)

**Зачем:** единый стандарт для всех сущностей домена. Иммутабельность, обязательные описания, поддержка автоматов.

**Родитель:** `BaseEntity` наследует `BaseModel` (Pydantic) с `frozen=True`, `extra="forbid"`, а также `ABC` (абстрактный) и `ReadableMixin`.

**Поля состояний:** любое количество полей типа `BaseState`.

**Обязательные декораторы:** `@entity` (проверяет `EntityGateHost`, регистрирует в координаторе), `@meta(description, domain)`.

**Обязательные описания полей:** каждое простое поле — `Field(description="...")`. Отсутствие — ошибка сборки.

**Deprecated:** `Field(deprecated=True)`.

**Валидация состояний при создании экземпляра:** значение каждого поля-автомата должно быть одним из ключей, объявленных в соответствующем `BaseState`. Иначе `ValidationError`.

**Пример:**

```python
@entity
@meta(description="Заказ клиента", domain=ShopDomain)
class OrderEntity(BaseEntity):
    _state = BaseState(...)   # из этапа 1
    _payment_state = BaseState("Статус оплаты")...
    id: str = Field(description="Идентификатор заказа")
    amount: float = Field(description="Сумма заказа", ge=0)
```

**Размещение:** `src/action_machine/domain/entity.py`

## Этап 3 — Связи между сущностями (v0.3)

**Три типа владения:** Composition, Aggregation, Association.  
**Шесть контейнеров:** `CompositeOne[T]`, `CompositeMany[T]`, `AggregateOne[T]`, `AggregateMany[T]`, `AssociationOne[T]`, `AssociationMany[T]`.

**Объявление связи:**

```python
customer: Annotated[
    AssociationOne[Customer],
    Inverse(Customer, "orders"),
] = Rel(description="Клиент, оформивший заказ")
```

- `Rel(description)` — обязательное описание.
- `Inverse(TargetClass, "field_name")` или `NoInverse()` — обязательны. Если ни то ни другое — ошибка сборки.
- Контейнер хранит `id` (строку) и опционально `entity: T | None`. Атрибуты проксируются на `entity`, если он загружен.

**Два поля одного типа:** разные `Inverse` решают двусмысленность.  
**Union/XOR:** каждый тип в Union должен иметь свой `Inverse`.  
**Deprecated:** `Annotated[..., deprecated("причина")]`.  
**Матрица совместимости:** допустимые пары: Composite ↔ Association, Aggregate ↔ Association, Association ↔ Association. Composite ↔ Composite, Composite ↔ Aggregate — ошибка.

**Размещение:** `src/action_machine/domain/relations.py`

## Этап 4 — Метаданные и координатор сущностей (v0.4)

**Зачем:** координатор — единственный источник правды о структуре модели. Выполняет межсущностные проверки и предоставляет API для внешних инструментов.

**Метаданные:** frozen датаклассы `EntityFieldMeta`, `EntityRelationMeta`, `EntityFlowMeta`, `EntityMetadata`.

**`EntityCoordinator`** — ленивый синглтон. При первом обращении:

- Обходит все классы, декорированные `@entity`.
- Собирает метаданные (поля, связи, автоматы).
- Строит граф.
- Выполняет проверки: существование целевых сущностей, Inverse‑пары, совместимость типов владения, Union‑покрытие, полнота Inverse/NoInverse, проверки автоматов (этап 1), логирует deprecated.

Если хотя бы одна проверка не пройдена — исключение, приложение не запускается. Ошибки выводятся списком.

**API координатора:**

- `get_metadata(cls) -> EntityMetadata`
- `get_all_entities() -> list[EntityMetadata]`
- `get_relations_graph() -> dict`
- `get_flow(cls) -> EntityFlowMeta | None`

**Связь с координатором ActionMachine:** `GateCoordinator` получит ссылку на `EntityCoordinator` (этап 6). Обратной зависимости нет.

**Размещение:** `src/action_machine/domain/coordinator.py`

## Этап 5 — Утилиты создания сущностей (v0.5)

**`EntityDedup`** — дедупликация в рамках одного вызова.  
`get_or_create(cls, key, factory)` возвращает существующий объект или создаёт новый через фабрику.

**`hydrate`** — сборка из плоского словаря с префиксами:

```python
order = hydrate(Order, row, "order").rel("customer", "customer").build()
```

- `.rel("field_name", prefix)` — для связей `One`.
- Для `Many` — именованный аргумент: `items=[hydrate(...).build()]`.
- Вложенность через точку: `"customer.company"`.
- `strict=True` — проверяет наличие ключей для обязательных полей. По умолчанию `False`.

**`hydrate_nested`** — сборка из вложенного JSON, рекурсивно, используя метаданные координатора.

**Размещение:** `src/action_machine/domain/hydration.py`

## Этап 6 — Интеграция с ActionMachine (v0.6)

**Связка координаторов:** `GateCoordinator` получает ссылку на `EntityCoordinator`. Обратной зависимости нет.

**Менеджеры возвращают сущности модели** — типизированные frozen‑объекты. Action работает с ними через менеджера.

**Валидация переходов в менеджере записи:**

```python
if not OrderEntity._state.validate_transition(current, new):
    raise InvalidTransition(...)
```

**Кросс‑проверки (опционально для v1.0):** построение карты «action → используемые сущности».

**Размещение:** расширение `GateCoordinator` в `src/action_machine/core/`
