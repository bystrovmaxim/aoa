## Как связаны намерения и инварианты

**Намерение** — это декларация того, что должно быть: кто имеет доступ, от чего зависит операция, в каком домене она находится. Вы выражаете намерение через декораторы: `@meta`, `@check_roles`, `@depends`.

**Инвариант** — это правило, которое система берёт под свой контроль и проверяет автоматически.

Связь между ними двухуровневая:

**1. У системы есть свои инварианты — они требуют, чтобы вы объявили намерения.**

AOA не даёт написать «голую» операцию без архитектурного контекста. Это не рекомендация — это грамматика фреймворка:

| Системный инвариант | Что вы обязаны объявить |
|---------------------|-------------------------|
| Каждая операция должна иметь роли | `@check_roles(...)` — нельзя пропустить |
| Каждая операция должна иметь домен | `@meta(domain=...)` — нельзя пропустить |
| Каждая операция должна иметь summary | `@summary_aspect` — нельзя пропустить |
| Зависимости не могут образовывать цикл | `@depends` проверяется на старте |

Нельзя «забыть» `@check_roles` и надеяться, что никто не заметит — система не запустит операцию без явной декларации ролей. Даже если операция открыта для всех, нужно написать `@check_roles(NoneRole)` — это осознанное намерение, а не молчаливое отсутствие проверки.

**2. Объявленное намерение порождает инвариант рантайма.**

Когда вы выполнили требование системы и объявили намерение — система берёт его под контроль:

| Вы объявили | Система гарантирует |
|-------------|---------------------|
| `@check_roles(AdminRole)` | Роль проверяется при каждом вызове |
| `@depends(PaymentAction, mode=UseCase.include)` | `PaymentAction` реально выполнится в этой сессии |
| `@result_string("validated_id", required=True)` | Следующий шаг получит `state.validated_id` нужного типа |
| `@compensate("charge_aspect", ...)` | При ошибке откат выполнится автоматически |

**Итого:** системные инварианты заставляют вас думать об архитектуре. Объявленные намерения превращаются в правила, которые система исполняет без вашего участия.

## Когда проверяются инварианты

| Момент | Что проверяется |
|--------|-----------------|
| **Старт приложения** | Структура графа, именование, наличие обязательных деклараций, отсутствие циклов |
| **Рантайм (каждый вызов)** | Роли, контракты аспектов |
| **После завершения root-сессии** | Выполнение всех `UseCase.include` зависимостей |
| **При сборке (build time)** | Lifecycle FSM, корректность переходов состояний |
| **CI (статический анализ)** | Границы пакетов, именование классов |

## Глоссарий

---

### Инвариант именования

**Что это.** Классы операций должны заканчиваться на `Action`, методы регулярных аспектов — на `_aspect`, методы компенсаторов — на `_compensate`, методы summary-аспектов — на `_summary`.

**Какую проблему решает.** Код становится предсказуемым: любой разработчик, открыв файл, сразу понимает роль каждого метода без чтения его тела. Снижает когнитивную нагрузку при онбординге и при возврате к старому коду.

**Когда проверяется.** При старте приложения и в статических тестах (`check_action_class_naming`).

**Связь с другими инвариантами.** Именование методов связано с инвариантом компенсатора — `@compensate` обязан ссылаться на существующий `_aspect` метод.

**Аналоги.** Convention over configuration в Rails; naming rules в Angular (`*.component.ts`, `*.service.ts`).

**Как нарушить и что получить:**
```python
# ✗ Нарушение — метод назван произвольно
@regular_aspect("Валидация")
async def do_validation(self, ...):  # должно быть validate_aspect
    ...

# ✓ Корректно
@regular_aspect("Валидация")
async def validate_aspect(self, ...):
    ...
```

### Инвариант DAG (ацикличность зависимостей)

**Что это.** Граф зависимостей между операциями (`@depends`) должен быть ациклическим. Операция A не может прямо или косвенно зависеть от себя самой.

**Какую проблему решает.** Исключает бесконечные рекурсии и нераскрываемые цепочки вызовов. Делает порядок выполнения предсказуемым и анализируемым.

**Когда проверяется.** При старте приложения, во время построения координатора графа.

**Ошибка.** `CyclicDependencyError` с указанием цепочки цикла.

**Связь с другими инвариантами.** Связан с инвариантом Include-контракта — цикл сделал бы include-контракт неразрешимым.

**Аналоги.** Проверка циклических импортов в Python; topological sort в системах сборки (Make, Gradle); DAG в Airflow.

---

### Инвариант обязательных ролей

**Что это.** Каждая публичная операция обязана иметь декларацию `@check_roles`. Операция без явного указания ролей не пройдёт валидацию.

**Какую проблему решает.** Исключает случай, когда разработчик «забыл» добавить проверку прав доступа. Права доступа становятся частью контракта, а не опциональным дополнением.

**Когда проверяется.** При старте приложения; в статических тестах структуры.

**Важное уточнение.** `@check_roles` не заменяет бизнес-логику авторизации — он гарантирует, что **декларация о ролях существует** и проверяется при каждом вызове. Если операция доступна всем, используется специальная роль `NoneRole`.

**Аналоги.** `@login_required` в Django (но там это опционально); `@PreAuthorize` в Spring Security.

```python
# ✗ Нарушение — нет @check_roles
@meta(description="Удалить заказ", domain=StoreDomain)
class DeleteOrderAction(BaseAction[...]):
    ...

# ✓ Если операция открыта для всех — указываем явно
@meta(description="Получить статус", domain=StoreDomain)
@check_roles(NoneRole)
class GetOrderStatusAction(BaseAction[...]):
    ...
```

### Инвариант обязательного домена

**Что это.** Каждая операция и доменная сущность должна быть привязана к домену через `@meta(domain=...)` или `@entity(domain=...)`.

**Какую проблему решает.** Обеспечивает полноту архитектурной карты системы. Операция без домена — это «потерянный» компонент, который невидим для визуализации и анализа.

**Когда проверяется.** При построении координатора графа на старте.

**Связь с другими инвариантами.** Является основой для инварианта границ пакетов — домены задают логические границы.

---

### Инвариант Summary-аспекта

**Что это.** Каждая операция должна иметь ровно один метод, помеченный `@summary_aspect`. Этот метод обязан вернуть типизированный `Result`.

**Какую проблему решает.** Гарантирует, что у операции всегда есть явная «точка выхода» с результатом. Исключает ситуацию, когда результат формируется неявно или в нескольких местах.

**Когда проверяется.** При старте приложения.

**Ошибка.** `MissingSummaryAspectError`.

---

### Инвариант компенсатора

**Что это.** Метод `@compensate(target, description)` обязан ссылаться на существующий `@regular_aspect` метод в том же классе. Компенсатор без соответствующего аспекта недопустим.

**Какую проблему решает.** Гарантирует целостность саги: каждый шаг, для которого объявлен откат, реально существует. Исключает «мёртвые» компенсаторы.

**Когда проверяется.** При старте приложения.

**Связь с другими инвариантами.** Именование: имя метода компенсатора должно заканчиваться на `_compensate`.

```python
# ✓ charge_aspect существует → charge_compensate валиден
@regular_aspect("Списание средств")
async def charge_aspect(self, ...): ...

@compensate("charge_aspect", "Возврат средств")
async def charge_compensate(self, ...): ...
```

### Include-контракт (инвариант выполнения зависимостей)

**Что это.** Если операция объявляет зависимость с `UseCase.include`, зависимая операция **обязана фактически войти в исполнение** (`_run_internal`) в рамках той же root-сессии. Недостаточно просто объявить зависимость — она должна быть реально вызвана через `await box.run(...)`.

**Какую проблему решает.** Гарантирует, что архитектурный контракт соответствует реальному поведению. Операция не может «обещать» выполнение зависимости и не выполнить её.

**Когда проверяется.** После завершения root-сессии, перед `emit_global_finish`.

**Ошибка.** `IncludeContractViolationError` с указанием невыполненных зависимостей в `missing_include_types`.

**Связь с другими инвариантами.** Связан с инвариантом DAG — Include требует ацикличного графа.

**Отличие от `UseCase.extend`.** `extend` — зависимость может выполняться или нет в зависимости от логики операции. `include` — обязательно при каждом успешном выполнении root-операции.

### Инвариант State-контракта (чекеры)

**Что это.** Декораторы-чекеры (`@result_string`, `@result_instance`, `@result_bool` и др.) объявляют контракт результата аспекта: какой ключ должен появиться в `state` и какого типа. Если аспект не вернул ожидаемое значение — ошибка.

**Какую проблему решает.** Устраняет класс ошибок «следующий шаг сломался, потому что предыдущий не положил данные в state». Данные в `state` гарантированы типом и наличием, а не надеждой на предыдущий метод.

**Когда проверяется.** В рантайме, сразу после выполнения аспекта.

**Аналоги.** Post-conditions в Design by Contract (Eiffel, Kotlin contracts).

```python
@regular_aspect("Валидация")
@result_string("validated_id", required=True)   # гарантирует: state.validated_id — строка, не None
async def validate_aspect(self, params, state, box, connections):
    return {"validated_id": params.order_id}

@summary_aspect("Создание")
async def create_summary(self, params, state, box, connections):
    state.validated_id  # можно использовать без проверки — контракт гарантирован
```

### Инвариант Lifecycle FSM

**Что это.** Переходы между состояниями сущности должны соответствовать объявленному конечному автомату (FSM). Недопустимые переходы и недостижимые состояния обнаруживаются при сборке.

**Какую проблему решает.** Исключает ситуацию, когда сущность переходит в состояние, которое «не должно быть достижимо» — например, заказ переходит из `cancelled` в `shipped`. Логика жизненного цикла задаётся один раз и не может быть нарушена случайным кодом.

**Когда проверяется.** При вызове `build()` на lifecycle-объекте; визуализируется в Maxitor как FSM-диаграмма.

**Аналоги.** XState (JS); statecharts в авиации и медицинском ПО.

---

### Инвариант границ пакетов

**Что это.** Зависимости между пакетами монорепо строго однонаправленные. `aoa-examples` не может импортировать из `aoa-maxitor`; graph primitives живут в `aoa.action_machine.graph.core` и не выносятся в отдельный wheel.

**Какую проблему решает.** Предотвращает архитектурную деградацию — «спагетти» зависимостей между пакетами, которая накапливается незаметно и делает систему нетестируемой по частям.

**Когда проверяется.** Скриптом `scripts/check_package_boundaries.py` в CI и при локальном запуске `uv run task check`.

**Аналоги.** `ArchUnit` (Java); `dependency-cruiser` (JS/TS); модульные границы в Nx.
