## 🧠 Архитектура сообщений (Action-фреймворк)

В основе лежит **командно-ориентированный подход**, где каждая операция оформлена как класс-наследник `BaseSimpleAction` (или его транзакционных вариантов). Эти классы называют **действиями**.

### 🔹 Базовые классы

- **`BaseSimpleAction`** – действие без управления транзакцией.  
  Реализует **конвейер из четырёх аспектов**, которые вызываются в строгом порядке:

  1. `_permissionAuthorizationAspect` – проверка прав (ролей).
  2. `_validationAspect` – валидация входных параметров.
  3. `_handleAspect` – основная бизнес-логика (обязателен к переопределению).
  4. `_postHandleAspect` – пост-обработка (логирование, уведомления).

  Каждый аспект получает на вход текущий результат (словарь) и возвращает (возможно, модифицированный) словарь. После выполнения аспекта к результату применяются все чекеры, привязанные к этому методу (декораторы вида `@StringFieldChecker`, `@IntFieldChecker` и т.д.).

- **`BaseTransactionAction`** – действие, выполняемое **внутри уже открытой транзакции**.  
  Оно ожидает, что переданный в `run()` контекст является экземпляром `TransactionContext` и содержит поле `connection`.  
  Проверяет тип соединения (если задан декоратор `@requires_connection_type`) и делегирует выполнение родительскому `run()`.  
  Управление жизненным циклом транзакции (открытие, коммит, откат) полностью лежит на внешнем координаторе, использующем `BaseConnectionManager`.

- **`BaseConnectionManager`** – абстрактный менеджер соединений/транзакций.  
  Предоставляет методы `open()`, `commit()`, `rollback()` и свойство `connection`.  
  Конкретные наследники реализуют логику для конкретного типа БД или ресурса (например, PostgreSQL, Kafka).  
  Все ошибки управления соединением выражаются через доменные исключения (`ConnectionAlreadyOpenError`, `ConnectionNotOpenError`).

### 🔹 Декларативная валидация (чекеры)

Классы-наследники `BaseFieldChecker` (например, `StringFieldChecker`, `IntFieldChecker`) могут применяться как декораторы:

- **На класс** – добавляют проверку входных параметров. Чекеры накапливаются в атрибуте класса `_field_checkers` и проверяются в аспекте `_validationAspect`.
- **На метод аспекта** – добавляют проверку результата, возвращаемого этим аспектом. Чекеры накапливаются в атрибуте метода `_result_checkers` и применяются после выполнения аспекта.

Пример:

```python
@StringFieldChecker("base_url")                # проверка входного параметра base_url
@StringFieldChecker("token")                    # проверка входного параметра token
class FetchIssuesToCsvAction(BaseSimpleAction):
    @IntFieldChecker("count", min_value=0)       # проверка результата _handleAspect: поле count должно быть >=0
    @StringFieldChecker("output_file", not_empty=True)  # поле output_file должно быть непустой строкой
    def _handleAspect(self, ctx, params, result):
        # ... логика ...
        result["count"] = total
        result["output_file"] = output_file
        return result
```

### 🔹 Проверка ролей

Декоратор `CheckRoles` задаёт требования к ролям пользователя:

- `CheckRoles.NONE` – доступ без роли (гость).
- `CheckRoles.ANY` – требуется любая роль (аутентифицированный).
- `CheckRoles(["admin", "manager"])` – список допустимых ролей.

Если пользователь не удовлетворяет требованию, выбрасывается `AuthorizationException`.

### 🔹 Контекст выполнения

- **`Context`** – базовый контекст, содержит `user_id` и `roles`.
- **`TransactionContext`** – расширенный контекст для транзакционных действий.  
  Конструктор принимает базовый контекст и соединение, копируя все поля из базового и добавляя `connection`.

Контекст передаётся во все аспекты, но не содержит результат – результат передаётся отдельным параметром (аккумулируется между аспектами).

### 🔹 Обработка ошибок

Все бизнес-исключения наследуются от `Exception` и имеют русскоязычные сообщения:

- `AuthorizationException`
- `ValidationFieldException`
- `HandleException`
- `TransactionException` (базовое для ошибок транзакций)
  - `ConnectionAlreadyOpenError`
  - `ConnectionNotOpenError`

Исключения **не перехватываются внутри действия** – они всплывают наружу. На верхнем уровне (в фасаде) они преобразуются в единый формат ответа.

## 🌐 Концепция MCP – фасад для внешнего мира

Для изоляции внешних систем (n8n, тестовых скриптов) от внутренней архитектуры действий создан **фасад** – класс `YouTrackMCPServer`. Его задачи:

- Предоставлять **статические методы**, соответствующие каждому бизнес-действию.
- Принимать все параметры **явно** (включая контекст).
- Создавать экземпляр нужного действия и вызывать его `run()`.
- Перехватывать любые исключения, логировать их и возвращать **единообразный JSON-ответ**:

```json
{
  "success": true,
  "result": { ... },
  "errors": []
}
```

или при ошибке:

```json
{
  "success": false,
  "result": null,
  "errors": ["Текст ошибки"]
}
```

Этот подход гарантирует, что вызывающий код всегда получит предсказуемую структуру и сможет легко обработать результат.

## 🚀 Реализация для YouTrack

В качестве примера реализовано действие **`FetchIssuesToCsvAction`**, которое выгружает задачи из YouTrack в CSV-файл.

### 📌 Параметры действия

- `base_url` (str) – адрес экземпляра YouTrack.
- `token` (str) – перманентный токен доступа.
- `output_file` (str) – путь для сохранения CSV.
- `page_size` (int, 1‑500) – размер страницы при пагинации.
- `project_id` (str, опционально) – идентификатор проекта; если не указан, выгружаются задачи из всех проектов.

### 📌 Логика работы

1. Запрос к API YouTrack с пагинацией (используется `page_size`).
2. Для каждой полученной страницы формируется плоский словарь (кастомные поля преобразуются в читаемый вид) и дописывается в CSV-файл. Файл создаётся с заголовками на первой странице, последующие страницы дописываются без заголовков.
3. В конце возвращается результат: `{"count": число_задач, "output_file": путь_к_файлу}`.

### 📌 Фасадный метод

```python
@staticmethod
def fetch_issues_to_csv(
    ctx: Context,
    base_url: str,
    token: str,
    output_file: str,
    page_size: int,
    project_id: Optional[str] = None
) -> Dict[str, Any]:
    # формирует params, создаёт действие, вызывает run, обрабатывает исключения
    ...
```

### 📌 Вызов из n8n (через Docker)

После сборки образа и запуска контейнера `python-runner` можно выполнить команду:

```bash
docker exec python-runner python -c "
from YouTrackMCP import YouTrackMCPServer
from ActionEngine.Context import Context
ctx = Context(user_id='n8n', roles=['user'])
result = YouTrackMCPServer.fetch_issues_to_csv(
    ctx=ctx,
    base_url='https://youtrack.brusnika.tech',
    token='ваш_токен',
    output_file='/tmp/issues.csv',
    page_size=100,
    project_id='OPD_IPPM'
)
print(result)
"
```

Для удобства можно создать скрипт-обёртку `call_mcp.py`, который принимает JSON с параметрами и вызывает нужный метод фасада – это упрощает интеграцию с n8n.

## 🧩 Заключение

Проект предоставляет **гибкую и расширяемую архитектуру** для создания бизнес-действий с единообразным интерфейсом. Основные преимущества:

- ✅ **Декларативная валидация** – требования к параметрам видны прямо у класса/метода.
- ✅ **Stateless-действия** – результат передаётся по цепочке аспектов, что упрощает тестирование и исключает ошибки состояния.
- ✅ **Поддержка транзакций** – чёткое разделение управления соединением (через `BaseConnectionManager`) и бизнес-логики (через `BaseTransactionAction`).
- ✅ **Единый фасад** – внешние системы получают стандартизированный JSON-ответ.
- ✅ **Русскоязычные ошибки** – все сообщения об исключениях на русском для удобства команды.
- ✅ **Лёгкость расширения** – новые действия добавляются созданием класса в `YouTrackMCP/` и одного метода в фасаде.

Вся инфраструктура (сборка образа, CI/CD, запуск в Docker) уже готова. Проект можно развивать, реализуя действия для PostgreSQL, отправки уведомлений, интеграции с другими системами – всё будет работать по тем же принципам.
