# ActionMachine

Системы усложняются. Код теперь пишут и люди, и AI. Причём AI генерирует
код быстрее, чем люди успевают его читать. Разработчики в команде приходят
и уходят. Каждое изменение — решение, основанное на контексте, который
неизбежно теряется.

И люди, и AI накапливают технический долг — но AI усугубляет: он не видит
глубинных связей и добавляет код, не понимая замысла. Реализация фич
замедляется. Последствия изменений становятся непредсказуемыми.

Новый человек не может разобраться в системе за разумное время. Код фиксирует
*что* система делает, но молчит о том, *что было задумано и почему*.
Документация не спасает — она начинает устаревать в момент написания.

Это ежедневная реальность любого проекта старше года.

Наш ответ — строить системы так, чтобы они выражали бизнес-намерения как код.
Благодаря этому код становится исполняемой спецификацией, которая не даёт ему
забыть контекст. Это даёт и человеку, и AI чёткую карту системы — границы
дозволенного, связи между частями, замысел каждого решения.

Система становится антихрупкой: чем больше в ней выраженных намерений, тем
плотнее сетка проверок и тем труднее сломать что-то незаметно.

## Намерение как код

**Бизнес-намерение** — это замысел. Не реализация, не алгоритм, не строчка кода.
Это ответ на вопрос «что бизнес хотел и зачем». То, что автор вкладывает в код,
воплощая этот замысел. В обычном коде это остаётся только в голове разработчика
и в «голове» AI — и впоследствии теряется.

А соблюсти согласованность замысла не в рамках одной функции, а в рамках
системы — не может ни человек, ни AI. Потому что намерения не выражены явно.
Их нельзя прочитать машинно, проверить, сопоставить.

**ActionMachine заставляет выражать бизнес-намерения явно** — прямо в коде,
как его неотъемлемую часть.

### Ключевые концепции библиотеки: Intent

В коде **action-machine** участие типа в грамматике фреймворка задаётся **маркерными миксинами Intent** в MRO (например, `RoleIntent`, `DependencyIntent`, `AspectIntent`). Класс **заявляет** фрагмент грамматики и **обязуется** иметь согласованные декларации (`@check_roles`, `@depends`, аспекты и т.д.). Сборка графа метаданных (`GateCoordinator.build()`) проверяет полноту и согласованность; невыполненное намерение даёт ошибку на этапе сборки.

**Устаревший термин GateHost** описывал тот же паттерн как «разрешение» декоратора. **Intent** подчёркивает **обязательство** и **проверяемость**; имя `GateCoordinator` при этом не меняется.

## Быстрый старт

```python
from action_machine import Machine

# Machine сканирует все классы с @meta, @check_roles, BaseEntity и т.д.,
# строит цифровой двойник — машиночитаемую модель всех намерений системы —
# и проверяет их согласованность. Если что-то не сходится — приложение
# не запустится.
machine = Machine()

result = await machine.run(
    "PlaceOrderAction",
    params={
        "customer_id": "c-42",
        "items": [{"product_name": "Книга", "quantity": 1, "price": 500.0}],
        "total": 500.0,
    },
    context={"user_id": "c-42", "roles": ["customer"]},
)

# результат типизирован — это PlaceOrderResult
print(result.order_id)
print(result.payment_id)
```

Если хотя бы одно намерение не выражено или не согласовано — машина не
запустится и скажет почему. Если запустилась — всё, что объявлено, проверено.

## Какие бизнес-намерения можно выразить

### Модель предметной области

#### Бизнес-домены

Домен — это область ответственности системы. Каждый домен — отдельный класс
с обязательным описанием.

```python
@meta(description="Управление заказами: оформление, позиции, оплата")
class OrdersDomain(BaseDomain):
    name = "orders"

@meta(description="Клиентская база: профили, контакты, история")
class CustomersDomain(BaseDomain):
    name = "customers"

@meta(description="Доставка: отгрузка, отслеживание, вручение")
class ShipmentsDomain(BaseDomain):
    name = "shipments"
```

Координатор при старте проверяет: имя домена уникально, описание присутствует,
класс наследуется от BaseDomain. Домен — это класс, а не строка: опечатка
в имени приводит к ImportError, а не к молчаливому багу.

#### Жизненный цикл сущностей

Жизненный цикл сущности — это намерение, зафиксированное в коде. Прежде
чем описывать сущности, нужно явно перечислить их возможные состояния.

```python
class OrderStatus(BaseStatus):
    """Жизненный цикл заказа"""
    DRAFT = "draft"                  # создан, но не оформлен
    PLACED = "placed"                # оформлен
    PAID = "paid"                    # оплачен
    SHIPPED = "shipped"              # передан в доставку
    DELIVERED = "delivered"          # доставлен — финальный
    CANCELLED = "cancelled"          # отменён — финальный

class ShipmentStatus(BaseStatus):
    """Жизненный цикл доставки"""
    CREATED = "created"              # создана
    DISPATCHED = "dispatched"        # отправлена
    IN_TRANSIT = "in_transit"        # в пути
    DELIVERED = "delivered"          # вручена — финальный
    RETURNED = "returned"            # возвращена — финальный
```

Координатор при старте проверяет: каждый статус описан, финальные состояния
помечены явно, переходы между состояниями не содержат циклов. Если сущность
ссылается на статус — класс статуса обязан существовать.

#### Сущности и связи

Сущности описывают что существует в системе. Связи между ними описывают
кто кому принадлежит и как. Каждая связь двусторонняя — обе стороны объявлены
явно.

```python
@meta(description="Заказ клиента", domain=OrdersDomain)
class OrderEntity(BaseEntity):
    status: OrderStatus

    customer: AssociationOne[CustomerEntity] = Rel(
        description="Клиент, оформивший заказ",
        inverse=Inverse(CustomerEntity, "orders"),
    )
    items: CompositeMany[OrderItemEntity] = Rel(
        description="Позиции заказа",
        inverse=Inverse(OrderItemEntity, "order"),
    )
    shipment: AggregateOne[ShipmentEntity] = Rel(
        description="Доставка заказа",
        inverse=Inverse(ShipmentEntity, "order"),
    )


@meta(description="Позиция заказа", domain=OrdersDomain)
class OrderItemEntity(BaseEntity):
    product_name: str = Field(description="Название товара")
    quantity: int = Field(description="Количество", gt=0)
    price: float = Field(description="Цена за единицу", gt=0)

    order: CompositeOne[OrderEntity] = Rel(
        description="Заказ, которому принадлежит позиция",
        inverse=Inverse(OrderEntity, "items"),
    )


@meta(description="Клиент", domain=CustomersDomain)
class CustomerEntity(BaseEntity):
    name: str = Field(description="Имя клиента")
    email: str = Field(description="Email клиента")

    orders: AssociationMany[OrderEntity] = Rel(
        description="Заказы клиента",
        inverse=Inverse(OrderEntity, "customer"),
    )


@meta(description="Доставка заказа", domain=ShipmentsDomain)
class ShipmentEntity(BaseEntity):
    status: ShipmentStatus
    address: str = Field(description="Адрес доставки")
    tracking_number: str = Field(description="Номер отслеживания")

    order: AggregateOne[OrderEntity] = Rel(
        description="Заказ, для которого создана доставка",
        inverse=Inverse(OrderEntity, "shipment"),
    )
```

Здесь нет ни одной строчки реализации, но замысел читается полностью.
Позиции не существуют без заказа (композиция). Доставка связана с заказом,
но живёт в своём домене и может существовать независимо (агрегация).
Клиент и заказ ссылаются друг на друга (ассоциация).

Координатор при старте проверяет: каждая связь имеет Inverse, обе стороны
указывают друг на друга, описание присутствует у каждого поля и каждой
сущности. Несогласованная связь, пропущенный Inverse, отсутствующее
описание — ошибка сборки, приложение не запустится.

Эта модель — не декоративная схема. Менеджер ресурсов работает с ней
напрямую:

```python
# внутри аспекта — загрузка через box
order = await box.resources.load(OrderEntity, order_id)
customer = await order.customer.resolve()
items = await order.items.resolve_all()
```

Загружает сущности, проходит по связям, возвращает результат операции
в терминах модели. Модель — рабочий инструмент, а не документация на стене.

### Поведение

Какие действия меняют состояние системы, кто имеет право их выполнять,
из каких шагов состоит процесс и чем он завершается.

```python
@meta(description="Оформление заказа", domain=OrdersDomain)
@check_roles("customer")
@depends(PaymentService, description="Сервис обработки платежей")
@depends(WarehouseService, description="Сервис управления складом")
@connection(PostgresConnectionManager, key="db", description="Основная БД")
class PlaceOrderAction(BaseAction[PlaceOrderParams, PlaceOrderResult]):

    @regular_aspect("Валидация корзины")
    @result_bool("cart_valid")
    async def validate_cart_aspect(self, params, state, box, connections):
        ...

    @regular_aspect("Резервирование товаров")
    @result_string("reservation_id")
    async def reserve_stock_aspect(self, params, state, box, connections):
        ...

    @regular_aspect("Списание средств")
    @result_string("payment_id")
    async def charge_payment_aspect(self, params, state, box, connections):
        ...

    @summary_aspect("Создание заказа")
    async def create_order_summary(self, params, state, box, connections):
        ...
```

Без чтения тела методов видно: оформить заказ может только покупатель.
Действию нужны два сервиса и соединение с БД. Процесс состоит из четырёх
шагов — валидация, резервирование, оплата, создание. Каждый шаг именован
и типизирован по результату.

Координатор при старте проверяет: у каждого действия есть описание, роли
и хотя бы один аспект. Каждый аспект именован. Если аспект объявляет
результат через `@result_*` — соответствующее поле обязано пройти валидацию.
Имя класса заканчивается на `Action`, имя метода — на `_aspect` или
`_summary`. Нарушение любого из этих правил — ошибка при старте.

### Ограничения

Какие данные обязательны, какие контракты должны соблюдаться на входе
и выходе, какие поля чувствительны.

```python
class PlaceOrderParams(BaseParams):
    customer_id: str = Field(description="ID клиента", min_length=1)
    items: list[OrderItemInput] = Field(description="Товары в корзине", min_length=1)
    total: float = Field(description="Итоговая сумма в рублях", gt=0)


class PlaceOrderResult(BaseResult):
    order_id: str = Field(description="ID созданного заказа")
    payment_id: str = Field(description="ID транзакции платежа")

    @property
    @sensitive(max_chars=4)
    def card_last4(self) -> str:
        return self._card_last4
```

Контракт действия зафиксирован: без клиента, без товаров или с нулевой
суммой действие не запустится. Результат типизирован. Данные карты помечены
как чувствительные — при сериализации и логировании фреймворк автоматически
заменяет значение на маску `***1234`.

Координатор при старте проверяет: параметры и результат указаны в сигнатуре
действия, все поля имеют описание, типы параметров совместимы с объявленными
ограничениями. BaseParams неизменяем (frozen) — параметры нельзя модифицировать
после создания.

## Как это работает

Код на ActionMachine не позволяет забыть намерение — ни человеку, ни AI.
Без явно выраженного замысла код не соберётся.

При старте приложения из этих намерений автоматически строится **цифровой
двойник** — машиночитаемая модель того, что есть, как связано, что может
происходить и кто за что отвечает. Код порождает спецификацию самого себя.

Фреймворк проверяет согласованность всех выраженных намерений между собой.
**Если приложение запустилось — намерения согласованы структурно.** Связи
не противоречат друг другу, роли объявлены, контракты совместимы. Фреймворк
гарантирует целостность структуры, но не корректность бизнес-логики — можно
согласованно описать неправильный процесс. Правильность замысла остаётся
ответственностью автора. ActionMachine гарантирует, что этот замысел выражен
явно, читаем и проверяем — а значит, ошибку в нём можно найти.

## С ActionMachine и без

Обычный код — оформление заказа:

```python
async def place_order(user, cart, db):
    if not cart.items:
        raise ValueError("empty cart")
    reservation = await reserve(cart)
    payment = await charge(user, cart.total)
    order = await db.orders.create(user_id=user.id, items=cart.items, payment_id=payment.id)
    await notify(user, order)
    return order
```

Код работает. Но кто может вызывать эту функцию? Какие сервисы ей нужны?
Что происходит между резервированием и оплатой? Какие поля обязательны
в результате? Какие данные чувствительны? Ничего из этого в коде не выражено.
Чтобы ответить — нужно читать всю реализацию, искать вызовы, проверять тесты.

Тот же процесс на ActionMachine:

```python
@meta(description="Оформление заказа", domain=OrdersDomain)
@check_roles("customer")

    @regular_aspect("Списание средств")
    @result_string("payment_id")
    async def charge_payment_aspect(self, params, state, box, connections): ...

```

Те же четыре шага. Но теперь роли, зависимости, последовательность, контракты
входа и выхода — всё выражено до первой строки реализации. Не нужно читать
тело, чтобы понять что происходит и почему.

## Ключевые принципы

### Грамматика принуждает, а не рекомендует

ActionMachine не предлагает «лучшие практики». Он физически не позволяет
написать код без выраженного намерения. Нет описания у действия — TypeError
при старте. Нет ролей — TypeError. Аспект вернул поле без объявленного
чекера — ValidationFieldError. Имя класса не заканчивается на `Action` —
NamingSuffixError. Связь без Inverse — ошибка сборки.

Это не линтер и не code review. Это грамматика языка: неправильное
высказывание невозможно произнести. Разница между «рекомендуем описывать»
и «без описания не запустится» — принципиальная.

В обычном проекте структура — это конвенция, договорённость между
разработчиками, которую можно нарушить. В ActionMachine структура —
требование рантайма.

ActionMachine — инструмент, который не прощает лени. И в этом его ценность.
Архитектура не предполагается — она принудительно поддерживается кодом.

### Одна точка входа, одна точка выхода

Каждая операция запускается через `machine.run()` и возвращает типизированный
результат. Внутри — линейный конвейер именованных шагов, всегда сверху вниз.
Вложенные действия через `box.run()` формируют дерево — циклы запрещены
структурно и обнаруживаются при старте.

При одинаковых входных данных система всегда проходит одним путём.

### Бизнес-логика отделена от инфраструктуры

Аспект получает параметры, состояние, инструменты и соединения — и ничего
больше. Контекст пользователя доступен только через явную декларацию
`@context_requires`. Логирование — через `box.info()`. Зависимости —
через `box.resolve()`.

Разработчик пишет только бизнес-логику. Инфраструктура невидима для него —
но работает.

### Одно действие — любой транспорт

Действие не знает, как его вызвали. HTTP, MCP, CLI, очередь сообщений —
это забота адаптера, а не бизнес-логики. Одна и та же кодовая база
обслуживает REST API и AI-агентов без дублирования:

```python
# FastAPI — HTTP для людей и сервисов
app = FastApiAdapter(machine=machine, auth_coordinator=auth) \
    .post("/api/v1/orders", PlaceOrderAction) \
    .build()

# MCP — tools для AI-агентов
server = McpAdapter(machine=machine, auth_coordinator=auth) \
    .tool("orders.place", PlaceOrderAction) \
    .build()
```

MCP-адаптер превращает каждое действие в MCP tool. Описания, параметры,
ограничения — всё берётся из кода без дублирования. Resource
`system://graph` экспортирует граф системы: какие действия существуют,
к каким доменам принадлежат, от чего зависят. AI-агент работает с системой
не по догадкам, а по карте: видит архитектуру, а не угадывает контекст
из неявного кода.

## Стоимость и окупаемость

Да, на одну операцию приходится несколько классов: домен, статус, сущности,
параметры, результат, действие с аспектами. Это больше кода, чем написать
функцию и вызвать её.

Но каждый из этих классов — это не бойлерплейт. Это зафиксированное решение.
В обычном коде те же решения принимаются — но не записываются. Стоимость
записи — минуты. Стоимость восстановления утраченного контекста — часы и дни.

На проекте с десятком операций затраты на описание намерений заметны. На
проекте с сотнями — они окупаются кратно. Ошибки согласованности ловятся
при старте, а не на продакшене. Рефакторинг перестаёт быть лотереей — если
приложение запустилось после изменений, структура не нарушена.

Система, построенная на явных намерениях, устойчива к изменениям — потому
что каждое изменение проверяется относительно всех остальных выраженных
намерений. ActionMachine окупается там, где цена потерянного контекста
превышает стоимость следования грамматике.

---

## Что из этого следует

Система прозрачна на любом уровне детализации. Открываешь список действий —
видишь что система умеет. Открываешь действие — видишь шаги, роли,
зависимости. Открываешь шаг — видишь бизнес-логику без инфраструктурного
шума. Открываешь сущность — видишь связи, типы владения и домен.

Система становится предсказуемой. Последствия изменения видны до того,
как оно сделано — потому что все связи, роли и контракты выражены явно.

Новый человек читает систему, а не расследует её. Структура операции —
это её документация. Модель данных — это её схема.

## FAQ

**Можно внедрить в существующий проект или только в новый?**

Можно в существующий. ActionMachine не требует переписывать систему целиком.
Новые операции пишутся через грамматику, старые продолжают работать как
раньше. Миграция идёт постепенно — действие за действием. Чем больше
операций переведено, тем полнее цифровой двойник и тем сильнее гарантии
согласованности.

**Какой оверхед в рантайме?**

Сборка модели, валидация намерений и построение цифрового двойника происходят
один раз при запуске приложения. После старта в рантайме оверхед минимален —
это вызов метода через конвейер аспектов с проверкой ролей и типов.
Для подавляющего большинства бизнес-операций разница с прямым вызовом
функции неизмерима.

**Не слишком ли это тяжело для MVP?**

Для прототипа на салфетке — да, это избыточно. Но MVP, написанные
«быстро и просто», имеют свойство уходить в продакшен как есть —
и мучить создателей годами. Церемония описания намерений — это час
сейчас вместо недель на восстановление контекста потом. Если проект
точно одноразовый — ActionMachine не нужен. Если есть шанс, что он
проживёт дольше полугода — инвестиция окупится.

**Как это соотносится с DDD?**

ActionMachine разделяет с DDD идею явного моделирования предметной области:
домены, сущности, агрегаты, жизненные циклы. Но DDD — это набор принципов,
которые остаются в голове архитектора. ActionMachine — это грамматика,
которая принуждает эти принципы выразить в коде и проверяет их при старте.
Если вы практикуете DDD — ActionMachine формализует то, что вы и так
делаете. Если нет — он заставит вас думать в этих терминах.

**Работает ли это с AI-агентами?**

Да. MCP-адаптер экспортирует действия как MCP tools. Одна и та же
кодовая база обслуживает HTTP-клиентов через FastAPI-адаптер
и AI-агентов через MCP-адаптер — без дублирования кода.

**Это попытка сделать из Python Java?**

Нет. Python остаётся гибким внутри методов — пишите как хотите.
Но приложение в целом заперто в каркас явных намерений. Гибкость
внутри, строгость снаружи. Это не статическая типизация — это
семантическая строгость поверх динамического языка.

## Глоссарий (термины библиотеки action-machine)

Краткий сводный словарь для чтения кода и документации пакета (см. также
`docs/CHANGELOG.md`, раздел *Terminology*).

| Термин | Смысл |
|--------|--------|
| **Intent** | Маркерный миксин в MRO класса: тип **заявляет** участие в фрагменте грамматики (`RoleIntent` → `@check_roles`, `DependencyIntent` → `@depends`, …). Фреймворк проверяет согласованность деклараций при **`GateCoordinator.build()`**. |
| **IntentInspector** | Подкласс `BaseIntentInspector`, регистрируемый в координаторе; обходит носителей соответствующего Intent и строит узлы и рёбра графа метаданных. |
| **GateCoordinator** | После `register(...).build()` фиксирует граф фасетов (`rustworkx`: на узле только `node_type`, `name`, `class_ref`) и карту типизированных **снимков** (`get_snapshot(cls, facet_key)`). Тело фасета в снимках; `get_node` / `hydrate_graph_node` подмешивают `meta`. Сырой `get_graph()` — низкоуровневый обход без `meta`. |
| **Ключ снимка / гидратация** | Координатор при `build()` связывает узел графа (`node_type:name`) с ключом `facet_snapshot_storage_key()` инспектора; при конфликте (слитый `action`: `depends` и `connections`) единого `meta` на узле нет. Заглушки без снимка — fallback на `get_snapshot(cls, node_type)` (кроме `action`). |
| **Декоратор** | Конструкция уровня класса или метода, которая при импорте записывает **scratch** и проверяет `issubclass(..., нужный Intent)`. |
| **Scratch** | Атрибуты на классе или методе (`_meta_info`, `_role_info`, `_new_aspect_meta`, …), которые заполняет декоратор и которые читает инспектор при сборке графа. |

**Бизнес-намерение** (разделы выше в этом README) — это замысел предметной
области в терминах продукта. **Intent** в таблице — технический термин
фреймворка для маркеров в коде; смысл рядом, но уровни разные.

## Бенчмарки и нагрузочные тесты

Корректность и «тяжёлые» проверки производительности **разводим**: обычный
прогон остаётся быстрым; регрессионные якоря по времени/масштабу живут рядом с
кодом, но помечаются так, чтобы их можно было отключать выборочно.

**Где лежит что (канон):**

| Место | Назначение |
|--------|------------|
| `tests/bench/` | Сквозной контур **`TestBench`**: одна и та же операция на async/sync машинах, сравнение результатов. Общие действия — в `tests/scenarios/domain_model/`, фикстуры — в `tests/conftest.py`. |
| `tests/<слой>/test_*_performance.py` | Узкий SUT (один модуль): `tests/model/test_navigation_performance.py` (`resolve` / `substitute`), `tests/graph/test_gate_coordinator_performance.py` (холодный `GateCoordinator.register().build()`). Пороги по `time.perf_counter()` с запасом к медиане локально (в CI — ×2 или отдельный job). |
| `tests/scenarios/**/test_bench_*.py` | Сценарии уровня рантайма/графа, где нужен полный координатор; не плодить третье место — либо сюда, либо в `tests/bench/`, по смыслу. |

**Как запускать:**

```bash
# весь набор тестов (как в CI по умолчанию)
uv run pytest tests/

# только пакет TestBench
uv run pytest tests/bench/

# только тесты с маркером benchmark (навигация по state, сборка графа, …)
uv run pytest tests/ -m benchmark

# когда появятся тесты с маркером slow — быстрый цикл без них
uv run pytest tests/ -m "not slow"
```

**Маркеры** (см. `[tool.pytest.ini_options]` в `pyproject.toml`):

- **`slow`** — долгие или чувствительные к окружению прогоны; в дневном цикле
  можно снимать через `-m "not slow"`. Новые тяжёлые тесты помечайте им.
- **`integration`** — внешние сервисы; по необходимости отдельный job.
- **`benchmark`** — регрессионные пороги по времени/объёму; выборочный запуск:
  `pytest -m benchmark` (см. `tests/model/test_navigation_performance.py`,
  `tests/graph/test_gate_coordinator_performance.py`).

Пороги в миллисекундах в репозитории **не должны быть хрупкими**: закладывайте
запас относительно медианы на типовой машине разработчика; для ночных прогонов
допустимы более мягкие или относительные критерии.
