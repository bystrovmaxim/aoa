# Проект kanban_assistant

Данный проект представляет собой фреймворк для создания бизнес-действий (сообщений) с декларативной валидацией, проверкой прав и поддержкой транзакций. В качестве примера реализован модуль для работы с YouTrack: выгрузка задач в CSV. Вся логика упакована в Docker-образ, автоматически собирается через CI/CD и может вызываться из n8n.

## 🧠 Архитектура сообщений (Action-фреймворк)

В основе лежит командно-ориентированный подход, где каждая операция оформлена как класс-наследник `BaseSimpleAction` (или его транзакционных вариантов). Эти классы называют **действиями**.

### 🔹 Базовые классы

- **`BaseSimpleAction`** – действие без управления транзакцией. Реализует конвейер из четырёх аспектов, которые вызываются в строгом порядке:
  1. `_permissionAuthorizationAspect` – проверка прав (ролей).
  2. `_validationAspect` – валидация входных параметров через декларативные чекеры.
  3. `_handleAspect` – основная бизнес-логика (переопределяется в наследнике).
  4. `_postHandleAspect` – пост-обработка (логирование, уведомления).

  Каждый аспект принимает текущий результат (`result`) и возвращает новый. Таким образом, действие не хранит состояние – результат путешествует по аспектам и возвращается из метода `run()`.

- **`BaseTransactionAction`** – действие, управляющее транзакцией. Открывает соединение, выполняет бизнес-логику в `_doInTransaction`, затем фиксирует или откатывает. Предоставляет публичные методы `openTransaction()`, `commit()`, `rollback()` для ручного управления.

- **`BaseInnerTransactionAction`** – действие, выполняемое внутри уже открытой транзакции. Получает готовое соединение через конструктор и в `run()` создаёт расширенный контекст с ним, вызывая родительский `run()`. Это позволяет компоновать сложные операции из нескольких маленьких действий.

### 🔹 Декларативная валидация

Чекеры (наследники `BaseFieldChecker`) задаются декораторами над классом действия. Например:

```python
@StringFieldChecker("base_url")
@StringFieldChecker("token")
@IntFieldChecker("page_size", min_value=1, max_value=500)
class FetchIssuesToCsvAction(BaseSimpleAction):
    ...
```

Чекеры автоматически проверяют наличие, тип и дополнительные ограничения (длина, диапазон, формат даты). При ошибке выбрасывается `ValidationFieldException` с сообщением на русском.

### 🔹 Проверка ролей

Декоратор `CheckRoles` задаёт требования к ролям пользователя:

- `CheckRoles.NONE` – доступ без роли (гость).
- `CheckRoles.ANY` – требуется любая роль (аутентифицированный).
- `CheckRoles(["admin", "manager"])` – список допустимых ролей.

Если пользователь не удовлетворяет требованию, выбрасывается `AuthorizationException`.

### 🔹 Контекст выполнения

Класс `Context` содержит `user_id` и `roles`. В транзакционных действиях используется `TransactionContext`, который добавляет поле `connection` (открытое соединение). Контекст передаётся во все аспекты, но не содержит результат – результат передаётся отдельным параметром.

### 🔹 Обработка ошибок

Все бизнес-исключения наследуются от `Exception` и имеют русскоязычные сообщения. Они не перехватываются внутри действия – всплывают наружу, где фасад преобразует их в единый формат ответа.

## 🌐 Концепция MCP (Model-Controller-Provider) – фасад для внешнего мира

Для изоляции внешних систем (n8n, тестовых скриптов) от внутренней архитектуры действий создан **фасад** – класс `YouTrackMCPServer`. Его задачи:

- Предоставлять статические методы, соответствующие каждому бизнес-действию.
- Принимать все параметры явно (включая контекст).
- Создавать экземпляр нужного действия и вызывать его `run()`.
- Перехватывать любые исключения, логировать их и возвращать единообразный JSON-ответ:
  ```json
  {
    "success": true,
    "result": { ... },
    "errors": []
  }
  ```
  или при ошибке:
  ```json
  {
    "success": false,
    "result": null,
    "errors": ["Текст ошибки"]
  }
  ```

Этот подход гарантирует, что вызывающий код всегда получит предсказуемую структуру и сможет легко обработать результат.

## 🚀 Реализация для YouTrack

В качестве примера реализовано действие **`FetchIssuesToCsvAction`**, которое выгружает задачи из YouTrack в CSV-файл.

### 📌 Параметры действия

- `base_url` (строка) – адрес экземпляра YouTrack.
- `token` (строка) – перманентный токен доступа.
- `output_file` (строка) – путь для сохранения CSV.
- `page_size` (целое, 1‑500) – размер страницы при пагинации.
- `project_id` (строка, опционально) – идентификатор проекта; если не указан, выгружаются задачи из всех проектов.

### 📌 Логика работы

1. Запрос к API YouTrack с пагинацией (используется `page_size`).
2. Для каждой полученной страницы формируется плоский словарь (кастомные поля преобразуются в читаемый вид) и дописывается в CSV-файл. Файл создаётся с заголовками на первой странице, последующие страницы дописываются без заголовков.
3. В конце возвращается результат: `{"count": число_задач, "output_file": путь_к_файлу}`.

### 📌 Фасадный метод

```python

    # формирует params, создаёт действие, вызывает run, обрабатывает исключения
    ...
```

### 📌 Вызов из n8n (через Docker)

После того как образ собран и контейнер `python-runner` запущен, можно выполнить команду:

```bash
docker exec python-runner python -c "
from YouTrackMCP import YouTrackMCPServer
from ActionEngine.Context import Context
ctx = Context(user_id='n8n', roles=['user'])
result = YouTrackMCPServer.fetch_issues_to_csv(
    ctx=ctx,
    base_url='https://youtrack.brusnika.tech',
    token='ваш_токен',
    output_file='/tmp/issues.csv',
    page_size=100,
    project_id='OPD_IPPM'
)
print(result)
"
```

Либо использовать скрипт-обёртку `call_mcp.py`, который принимает JSON с параметрами и вызывает нужный метод фасада – это удобно для n8n.

## 🧩 Заключение

Проект предоставляет гибкую и расширяемую архитектуру для создания бизнес-действий с единообразным интерфейсом. Основные преимущества:

- **Декларативная валидация** – требования к параметрам видны прямо у класса.
- **Stateless-действия** – результат передаётся по цепочке аспектов, что упрощает тестирование.
- **Поддержка транзакций** – действия могут управлять соединением или быть частью большей транзакции.
- **Единый фасад** – внешние системы получают стандартизированный ответ.
- **Русскоязычные ошибки** – все сообщения об исключениях на русском для удобства команды.

Новые действия добавляются просто: создаётся класс в `YouTrackMCP/`, описываются чекеры и роли, реализуется `_handleAspect`, и в фасад добавляется соответствующий статический метод. Вся инфраструктура (сборка образа, CI/CD, запуск в Docker) уже готова.

Для дальнейшего развития можно реализовать действия по записи в PostgreSQL, отправке уведомлений, интеграции с другими системами – всё будет работать по тем же принципам.
