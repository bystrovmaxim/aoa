# Hexagonal Architecture (Порты и Адаптеры)

## Суть идеи

Hexagonal Architecture (она же «Порты и Адаптеры») — это архитектурный паттерн, предложенный Алистером Кокбёрном в 2005 году. Основная идея: **бизнес-логика приложения не должна зависеть от внешнего мира** — ни от HTTP-фреймворка, ни от базы данных, ни от очередей сообщений, ни от UI.

Приложение представляется как ядро (гексагон), окружённое портами (интерфейсами) и адаптерами (реализациями).

---

## Три слоя

### 1. Ядро (Domain / Business Logic)
Чистая бизнес-логика. Не знает ни о каком фреймворке, протоколе или хранилище. Определяет **что** делать, но не **как** взаимодействовать с внешним миром.

### 2. Порты (Interfaces)
Абстрактные интерфейсы, через которые ядро общается с внешним миром. Делятся на два типа:

- **Входные порты (Driving)** — как внешний мир вызывает ядро (например, «запустить действие», «обработать команду»)
- **Выходные порты (Driven)** — как ядро запрашивает внешние ресурсы (например, «сохранить в БД», «отправить email»)

### 3. Адаптеры (Implementations)
Конкретные реализации портов для конкретной инфраструктуры:

- **Входные адаптеры**: HTTP-контроллер FastAPI, CLI-команда, обработчик MCP-инструмента, Telegram-бот
- **Выходные адаптеры**: PostgreSQL-репозиторий, SMTP-сервис, файловая система

---

## Как это выглядит в ActionMachine

Ваш фреймворк **уже реализует** Hexagonal Architecture, причём весьма последовательно [1]:

### Ядро — это Action

```python
@depends(EmailService, description="Сервис email")
@CheckRoles(CheckRoles.ANY, desc="Доступно любому")
class NotificationAction(BaseAction['NotificationAction.Params', 'NotificationAction.Result']):
    @aspect("Выбор канала")
    def choose_channel(self, params, state, deps):
        state['service'] = deps.get(EmailService)
        return state

    @summary_aspect("Отправка")
    def send(self, params, state, deps):
        state['service'].send(params.recipient, params.message)
        return NotificationAction.Result(success=True)
```

Это чистая бизнес-логика. Она не знает, откуда пришёл запрос (HTTP? CLI? MCP?) и какой именно `EmailService` будет использоваться [1].

### Порты — это абстрактные классы

| Порт | Тип | Что абстрагирует |
|------|-----|------------------|
| `BaseActionMachine` | Входной | Как запускается действие [1] |
| `Authenticator` | Выходной | Откуда берётся информация о пользователе [1] |
| `CredentialExtractor` | Выходной | Как извлекаются учётные данные из запроса [1] |
| `ContextAssembler` | Выходной | Как собираются метаданные запроса [1] |
| `BaseConnectionManager` | Выходной | Как работает соединение с хранилищем [1] |
| `Plugin` | Выходной | Как расширяется поведение без изменения логики [1] |

### Адаптеры — это конкретные реализации

| Адаптер | Какой порт реализует |
|---------|---------------------|
| `ActionProductMachine` | `BaseActionMachine` — продуктовый запуск с плагинами [1] |
| `ActionTestMachine` | `BaseActionMachine` — тестовый запуск с моками [1] |
| `PostgresConnectionManager` | `BaseConnectionManager` — для PostgreSQL [1] |
| `CsvConnectionManager` | `BaseConnectionManager` — для CSV-файлов [1] |
| `ConsoleLoggingPlugin` | `Plugin` — логирование в консоль [1] |

---

## Главное правило: зависимости направлены внутрь

```
[FastAPI / MCP / CLI / Telegram]     ← Входные адаптеры
         ↓
[BaseActionMachine.run()]            ← Входной порт
         ↓
[NotificationAction: аспекты]        ← Ядро (бизнес-логика)
         ↓
[deps.get(EmailService)]             ← Выходной порт
         ↓
[SmtpEmailService / MockEmailService] ← Выходные адаптеры
```

Ядро **никогда** не импортирует конкретный адаптер. Оно работает только через `DependencyFactory`, которая подставляет нужную реализацию [1].

---

## Почему это ценно

### 1. Одно действие — много точек входа

Одно и то же `NotificationAction` можно вызвать из:

```
FastAPI-эндпоинт  →  ActionProductMachine  →  NotificationAction
MCP-инструмент    →  ActionProductMachine  →  NotificationAction
CLI-команда       →  ActionProductMachine  →  NotificationAction
pytest            →  ActionTestMachine     →  NotificationAction
```

Бизнес-логика написана **один раз** [1].

### 2. Полная тестируемость без инфраструктуры

```python
machine = ActionTestMachine({
    EmailService: fake_email,
    SmsService: fake_sms
}, context=context)
result = machine.run(action, params)
```

Тест не поднимает SMTP-сервер, не подключается к БД, не запускает HTTP-сервер. Вместо выходных адаптеров подставляются моки через `ActionTestMachine` [1].

### 3. Замена инфраструктуры без изменения логики

Переход с PostgreSQL на MongoDB, с SMTP на SendGrid, с REST на GraphQL — всё это замена **адаптеров**, а не переписывание бизнес-логики.

## Сравнение с классической трёхслойной архитектурой

| | Трёхслойная | Hexagonal (ActionMachine) |
|--|------------|--------------------------|
| Бизнес-логика зависит от БД? | Часто да (ORM внутри сервисов) | Нет — через порт `BaseConnectionManager` [1] |
| Можно тестировать без HTTP? | Сложно | Да — `ActionTestMachine` [1] |
| Можно добавить CLI без дублирования? | Обычно нет | Да — новый входной адаптер [1] |
| Плагины/middleware? | Глобальные | Событийная система с regex-подписками [1] |

---

## Итог

Hexagonal Architecture — это не абстрактная теория, а конкретный способ организации кода, при котором бизнес-логика живёт в изолированном ядре, а всё остальное (протоколы, хранилища, фреймворки) подключается через порты и адаптеры. ActionMachine реализует этот паттерн через `BaseAction` (ядро), абстрактные классы вроде `Authenticator` и `BaseConnectionManager` (порты), и конкретные реализации вроде `PostgresConnectionManager` и `ActionTestMachine` (адаптеры) [1]. Результат — бизнес-логика, которая не зависит от протокола вызова и тестируется без инфраструктуры [1].
