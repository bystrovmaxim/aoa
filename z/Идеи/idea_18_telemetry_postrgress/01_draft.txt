Создать «тонкий» прокси для PostgreSQL, который просто пересылает трафик, но при этом позволяет запретить прямое подключение, технически несложно для базового прототипа. Однако создание надежного, безопасного и поддерживаемого решения — это задача средней и высокой сложности.

Главная проблема заключается не в проксировании как таковом, а в том, что PostgreSQL использует собственный бинарный протокол (message-based), а не человекочитаемый HTTP.

Вот анализ того, с чем вы столкнетесь, от самого простого пути к более сложному.

1. Самый простой способ: Туннелирование TCP (Уровень 4)
Если ваша задача — просто запретить прямой доступ к порту 5432, но разрешить через прокси, проще всего использовать готовые решения (HAProxy, Nginx stream module) или написать простой TCP-прокси.

Как это выглядит:
Клиент подключается к вашему прокси, прокси открывает соединение к реальному PostgreSQL и просто копирует байты туда-сюда.

Сложность: Низкая.

На Python это делается в ~50 строк кода через socket и select.

Но у такого подхода есть минусы: вы не можете контролировать, кто именно подключился (логин/пароль), если не будете разбирать пакеты.

2. Продвинутый уровень: Анализ протокола (Уровень 7)
Чтобы запрещать доступ на основе логики (например, пускать только пользователя app_user и блокировать admin_user), вам придется реализовать часть протокола PostgreSQL (PG Wire Protocol). Вот тут и начинаются сложности.

Основные сложности создания "тонкого" прокси:
Стартовый Handshake (SSL/TLS):

Клиент сначала спрашивает: "Поддерживаешь ли ты SSL?".

Если ваш прокси сам не умеет в SSL, а PostgreSQL требует — все сломается.

Вам нужно либо термировать SSL на прокси (прокси расшифровывает трафик), либо пробрасывать его на лету. Проброс сложнее.

Аутентификация:

Клиент отправляет пакет StartupMessage, где содержит имя пользователя и базу данных.

Хорошая новость: Пароль в этом пакете не передается (если только это не протокол простой аутентификации). Пароль идет позже, в зашифрованном виде (или открыто, если без SSL).

Вам нужно распарсить этот пакет, чтобы понять, кто пытается подключиться (user = 'evil_hacker').

Состояние соединения (Statefulness):

PostgreSQL протокол подразумевает сессию. После аутентификации идет обмен сообщениями (подготовка запросов, Bind, Execute).

Ваш прокси должен понимать, когда сессия началась и закончилась, и корректно пересылать ошибки.

Поддержка Prepared Statements и Extended Query:

Если ваше приложение использует подготовленные запросы (как делают Java (JDBC) и некоторые ORM), пакеты имеют бинарный формат. Просто резать строку по \n не получится, нужно читать заголовки (int32 length).

Пример структуры минимального "умного" прокси (Python asyncio)
Технически, вот как выглядит каркас такого прокси (опуская детали парсинга пакетов):

python
import asyncio
import struct

# Упрощенный парсер для чтения имени пользователя из StartupMessage
def parse_startup_message(data):
    # Пропускаем длину (int32) и версию протокола (int32)
    offset = 8
    params = {}
    while offset < len(data):
        # Читаем параметр (ключ и значение разделены \0)
        key_end = data.find(b'\x00', offset)
        key = data[offset:key_end].decode()
        offset = key_end + 1
        
        if offset >= len(data):
            break
            
        value_end = data.find(b'\x00', offset)
        value = data[offset:value_end].decode()
        offset = value_end + 1
        
        params[key] = value
        
        if key == 'user':
            # Нашли пользователя
            print(f"Попытка подключения пользователя: {value}")
            # Здесь можно принять решение: разорвать соединение или пустить
            # if value == 'bad_guy':
            #    return False
    return params

async def handle_client(reader, writer):
    # 1. Читаем первый пакет от клиента (StartupMessage)
    # Сначала читаем длину пакета (первые 4 байта)
    header = await reader.readexactly(4)
    msg_len = struct.unpack('!I', header)[0] # Длина включает сами эти 4 байта
    
    # Читаем тело сообщения
    body = await reader.readexactly(msg_len - 4)
    data = header + body
    
    # 2. Проверяем, кто хочет подключиться
    user_info = parse_startup_message(data)
    
    # 3. Решение о блокировке
    if user_info and user_info.get('user') == 'hacker':
        print("Блокируем хакера")
        writer.close()
        return
    
    # 4. Если всё ок - коннектимся к реальному PostgreSQL
    pg_reader, pg_writer = await asyncio.open_connection('localhost', 5432)
    
    # 5. Отправляем стартовый пакет в PostgreSQL
    pg_writer.write(data)
    await pg_writer.drain()
    
    # 6. Запускаем двустороннюю пересылку данных (туннель)
    async def forward(src, dst):
        try:
            while True:
                data = await src.read(4096)
                if not data:
                    break
                dst.write(data)
                await dst.drain()
        except:
            pass
        finally:
            dst.close()
    
    # Запускаем две задачи: клиент->PG и PG->клиент
    task1 = asyncio.create_task(forward(reader, pg_writer))
    task2 = asyncio.create_task(forward(pg_reader, writer))
    
    await asyncio.gather(task1, task2, return_exceptions=True)

async def main():
    server = await asyncio.start_server(handle_client, '0.0.0.0', 6432)
    async with server:
        await server.serve_forever()

# asyncio.run(main())
Реалистичные альтернативы (Не изобретайте велосипед)
Если вам нужно именно "запретить всем работать напрямую", возможно, вам не нужен свой прокси:

SSH Tunneling (Проще всего):

Запретите внешние подключения к 5432 ( firewall).

Пусть разработчики подключаются через ssh -L 5432:localhost:5432 user@proxy-server, а уже с proxy-server идут в базу.

Прокси-сервер в этом случае — просто шлюз.

PgBouncer (Промышленный стандарт):

Это пул соединений для PostgreSQL.

Он работает как прокси. Вы можете настроить его в режиме auth_user, чтобы он требовал пароль или перенаправлял трафик.

Он понимает протокол PostgreSQL из коробки. Вы просто ставите его перед базой и говорите приложению стучаться в PgBouncer.

Минус: Он спроектирован для производительности, а не для сложной логики "запретить Васе".

Envoy / HAProxy с фильтрацией:

Современные балансировщики умеют смотреть в SNI или даже в первые пакеты протокола (но для PostgreSQL это сложно). Обычно они работают на L4, блокируя по IP, но не по логину.

Резюме
Просто переслать трафик (TCP proxy): Очень просто (10 минут).

Переслать трафик и заблокировать по логину: Средне (нужно парсить протокол, день-два на отладку).

Сделать надежный продакшн-прокси с поддержкой SSL, подготовленных запросов и т.д.: Сложно (лучше взять готовое).

Совет: Если ваша цель — именно запретить прямое подключение, проще всего закрыть порт 5432 на фаерволе и настроить PgBouncer в режиме прокси (или использовать SSH-туннели). Если же это учебная задача — реализация парсера протокола на Python/asyncio вполне реальна.

моя цель делать перехват изменений в бд и отправлять сообщение об обновлении таблицы ролей централизрва5г 
Мы получили сообщение: "моя цель делать перехват изменений в бд и отправлять сообщение об обновлении таблицы ролей централизрва5г". Видимо, опечатки: "централизрва5г" = "централизованно". Пользователь уточняет свою цель: он хочет перехватывать изменения в базе данных (вероятно, PostgreSQL) и отправлять сообщение об обновлении таблицы ролей в централизованную систему.

Изначальный вопрос был о создании прокси для блокировки прямого доступа. Теперь он поясняет, что его настоящая цель — мониторить изменения таблицы ролей и оповещать о них. Возможно, он думал, что для этого нужно перехватывать трафик через прокси. Но есть более подходящие методы.

Нужно объяснить, как лучше решить эту задачу: через триггеры + уведомления (LISTEN/NOTIFY), логическую репликацию, триггеры с вызовом внешней функции (pg_notify), или использование таких инструментов как Debezium. И также пояснить, что прокси для этого не нужен.

Ответ должен быть информативным, практичным и предложить несколько вариантов с их плюсами и минусами.

Понял, ваша цель — отслеживать изменения в таблице ролей (например, INSERT, UPDATE, DELETE) и отправлять уведомления об этих изменениях в централизованную систему (очередь сообщений, другой сервис и т.п.). Прокси-сервер для базы данных здесь не нужен, это не тот инструмент. Задача решается средствами самой PostgreSQL или дополнительными инструментами для Change Data Capture (CDC).

Вот несколько подходов от простого к сложному.

1. Встроенный механизм NOTIFY / LISTEN (Самый простой)
PostgreSQL имеет встроенную систему асинхронных уведомлений. Вы можете создать триггеры на таблице ролей, которые при каждом изменении будут отправлять уведомление через pg_notify.

Как это работает:

Создаёте функцию на PL/pgSQL, которая вызывает pg_notify('channel_name', payload).

Создаёте триггеры (AFTER INSERT OR UPDATE OR DELETE) на таблице ролей, вызывающие эту функцию.

Внешний слушатель (ваше приложение на Python/Java/Go) подключается к PostgreSQL и выполняет LISTEN channel_name. Приложение получает уведомления в реальном времени.

Пример:

sql
-- Функция, которая будет вызываться триггером
CREATE OR REPLACE FUNCTION notify_role_change()
RETURNS trigger AS $$
DECLARE
    payload json;
BEGIN
    payload = json_build_object(
        'operation', TG_OP,
        'table', TG_TABLE_NAME,
        'schema', TG_TABLE_SCHEMA,
        'new_row', row_to_json(NEW),
        'old_row', row_to_json(OLD)
    );
    PERFORM pg_notify('role_changes', payload::text);
    RETURN NEW; -- для AFTER триггера можно RETURN NULL
END;
$$ LANGUAGE plpgsql;

-- Триггер на вставку
CREATE TRIGGER role_insert_trigger
AFTER INSERT ON roles
FOR EACH ROW EXECUTE FUNCTION notify_role_change();

-- Триггер на обновление
CREATE TRIGGER role_update_trigger
AFTER UPDATE ON roles
FOR EACH ROW EXECUTE FUNCTION notify_role_change();

-- Триггер на удаление
CREATE TRIGGER role_delete_trigger
AFTER DELETE ON roles
FOR EACH ROW EXECUTE FUNCTION notify_role_change();
Плюсы:

Не требует дополнительных сервисов.

Минимальная задержка.

Простота реализации.

Минусы:

Уведомления живут только в рамках сессии базы данных. Если слушатель отключится — уведомления будут потеряны.

Размер полезной нагрузки ограничен (по умолчанию 8000 байт, но можно увеличить параметром max_notify_payload_size).

Нет гарантии exactly-once (может быть at-most-once).

2. Логическая репликация и слоты репликации (Более надёжно, поток изменений)
Начиная с PostgreSQL 9.4, поддерживается логическая репликация. Вы можете создать публикацию на таблице ролей и подключиться к слоту репликации из внешнего приложения, которое будет получать все изменения в структурированном виде (протокол pgoutput или test_decoding).

Как это работает:

Включите wal_level = logical в postgresql.conf и перезапустите БД.

Создайте публикацию: CREATE PUBLICATION role_pub FOR TABLE roles;.

Ваше приложение подключается по протоколу репликации (используя, например, библиотеку pglogical или драйвер с поддержкой репликации) и читает поток изменений.

Плюсы:

Надёжность: изменения сохраняются в WAL, можно восстановиться с последней подтверждённой позиции.

Подходит для высоких нагрузок.

Можно обрабатывать изменения даже если приложение временно недоступно (лаги компенсируются).

Минусы:

Сложнее в реализации (требуется понимание протокола репликации).

Каждая подписка потребляет ресурсы на сервере БД (отдельный слот репликации).

Не все драйверы языков программирования поддерживают протокол репликации "из коробки" (часто нужно использовать специализированные библиотеки).

3. Использование Debezium (Промышленный CDC)
Debezium — это платформа Change Data Capture, построенная на Apache Kafka. Она подключается к PostgreSQL через логическую репликацию и публикует все изменения в Kafka в структурированном формате (например, JSON или Avro). Затем другие сервисы могут подписаться на топики Kafka.

Как это работает:

Разворачивается Kafka и Debezium Connector для PostgreSQL.

Коннектор читает изменения из слота репликации и отправляет их в Kafka.

Ваше приложение или микросервис подписывается на топик public.roles и получает события.

Плюсы:

Масштабируемость, отказоустойчивость.

Интеграция с экосистемой Kafka (Kafka Streams, ksqlDB).

Отличное сообщество и документация.

Минусы:

Требуется развернуть и администрировать Kafka (или использовать облачный Managed Kafka).

Избыточно, если нужно просто оповещать одно приложение.

4. Хранимые процедуры с внешними вызовами (HTTP из БД)
Можно через хранимые процедуры отправлять HTTP-запросы в ваш центральный сервис. Для этого понадобится установить расширение, например, pg_http или pgsql-http, либо использовать pl/python или pl/perl для выполнения HTTP-вызовов.

Пример с pl/python:

sql
CREATE OR REPLACE FUNCTION notify_role_change()
RETURNS trigger AS $$
import requests
import json

data = {
    'operation': TD['event'],
    'new': TD['new'] if 'new' in TD else None,
    'old': TD['old'] if 'old' in TD else None
}
requests.post('http://central-service/api/role-update', json=data, timeout=1)
return None
$$ LANGUAGE plpython3u;
Плюсы:

Прямая отправка в центральный сервис.

Минусы:

HTTP-вызовы из БД могут сильно замедлить транзакцию и увеличить нагрузку.

Риск потери данных при сбое сети.

Сложно гарантировать доставку.

Не рекомендуется для production с высокими требованиями к надёжности.

5. Логирование в отдельную таблицу + опрос
Самый простой, но не самый эффективный способ: создать триггеры, которые будут дублировать изменения в отдельную таблицу role_changes_log. Затем внешний процесс периодически опрашивает эту таблицу (например, раз в секунду) и отправляет изменения в центр.

Плюсы:

Простота реализации.

Гарантия сохранности (изменения хранятся в той же БД).

Минусы:

Задержки из-за опроса (не real-time).

Дополнительная нагрузка на БД при опросе и очистке лога.

Нужно следить за разрастанием таблицы логов.

Какой подход выбрать?
Если вам нужно просто и быстро оповещать одно приложение о нечастых изменениях, при этом потеря нескольких уведомлений некритична — используйте NOTIFY/LISTEN (вариант 1).

Если нужна высокая надёжность и возможность восстановления после сбоев (не терять изменения) — используйте логическую репликацию напрямую (вариант 2) или Debezium (вариант 3).

Если изменения редки и вы готовы мириться с задержкой — подойдёт логирование с опросом (вариант 5).

Важно: прокси для базы данных здесь действительно не нужен — это только добавит сложности и не даст преимуществ.

Да, Debezium умеет собирать и отдавать телеметрию (метрики) о своей работе. Это одна из ключевых функций для производственной эксплуатации, позволяющая отслеживать задержки, количество обработанных изменений и общее состояние коннекторов .

Возможности телеметрии можно разделить на два направления: классические метрики производительности и состояния (JMX) и современная распределенная трассировка (OpenTelemetry).

1. Метрики производительности и состояния (JMX)
Это основной и самый зрелый способ получения телеметрии из Debezium. Все коннекторы Debezium публикуют метрики через JMX (Java Management Extensions) .

Категории метрик:
Debezium предоставляет три основные категории метрик :

Snapshot metrics: Информация о процессе выполнения начального снимка таблицы (сколько строк просканировано, сколько таблиц осталось).

Streaming metrics: Метрики работы коннектора в режиме реального времени (чтение лога базы данных). Сюда входят задержка репликации, размер очереди событий, и, что важно для вашей задачи — счетчики событий.

Schema history metrics: Статус истории схемы таблиц.

Важно для вашей задачи (мониторинг таблицы ролей):
Начиная с версии Debezium 3.0, появились экспериментальные метрики, которые считают события для каждой таблицы :

NumberOfCreateEventsSeen (количество вставок)

NumberOfUpdateEventsSeen (количество обновлений)

NumberOfDeleteEventsSeen (количество удалений)

NumberOfTruncateEventsSeen

Чтобы их включить, нужно установить свойство internal.advanced.metrics.enable в true .

Эти метрики публикуются с лейблом (тегом) table, что позволяет в Prometheus/Grafana строить графики именно по таблице roles .

Как их забрать:
JMX — это не протокол для хранения или визуализации, а "вентиль", из которого метрики нужно вытягивать. Стандартный путь:

Включить JMX в коннекторе Debezium (Debezium Server или Kafka Connect) .

Запустить JMX Exporter (например, для Prometheus), который будет читать JMX-метрики и трансформировать их в формат, понятный для систем сбора .

Настроить Prometheus на сбор метрик с экспортера.

Визуализировать в Grafana .

2. Распределенная трассировка (OpenTelemetry / OpenTracing)
Если вам нужно отслеживать не только факт изменения, но и путь события от базы данных до конечного потребителя (например, до вашего микросервиса, обрабатывающего изменения), то нужна трассировка.

Debezium Server: Умеет отдавать телеметрию (метрики JVM, кастомные метрики) и трейсы через OpenTelemetry Java Agent. Это позволяет видеть, сколько времени занимает чтение из WAL и отправка в PubSub/Kafka, и отправлять это в CloudWatch, Jaeger и т.д. .

Debezium для Kafka Connect: Поддерживает интеграцию с OpenTracing (и есть планы по миграции на OpenTelemetry) . Существует SMT (Single Message Transform) ActivateTracingSpan, который может создавать спаны (единицы работы) и передавать контекст трассировки дальше в Kafka . Это позволяет связать воедино операцию в исходном микросервисе, запись в БД, чтение Debezium и обработку в микросервисе-получателе.

Ограничения: В Debezium Server пока нет поддержки распределенной трассировки с передачей контекста между компонентами (например, от источника к приемнику) .

3. Инструменты визуализации
Prometheus + Grafana: "Золотой стандарт" для сбора и визуализации JMX-метрик Debezium. В репозитории примеров Debezium есть готовые дашборды для Grafana .

CloudWatch (AWS): Если вы используете Debezium Server в AWS, можно отправлять OpenTelemetry метрики через AWS Distro for OpenTelemetry Collector прямо в CloudWatch .

Jaeger: Используется для просмотра распределенных трейсов .

Резюме для вашей задачи
Чтобы централизованно мониторить обновления таблицы roles, вам подойдет подход с JMX-метриками:

Настройте Debezium коннектор (для PostgreSQL) версии 3.0 или новее.

Включите в конфигурации internal.advanced.metrics.enable=true.

Настройте JMX-экспортер, чтобы он забирал метрики NumberOfUpdateEventsSeen (и другие) с фильтром по лейблу table = "public.roles" (или как ваша таблица называется).

Отправляйте их в Prometheus и стройте дашборды в Grafana или отправляйте алерты.

Если же вам когда-нибудь понадобится отследить полный жизненный цикл конкретного изменения (например, чтобы понять, где возникает задержка), потребуется распределенная трассировка (OpenTelemetry).

Что значит "собирать пакеты изменений и отправлять их как один unit of work"? Скорее всего, речь о группировке нескольких DML-операций (INSERT/UPDATE/DELETE) в одну транзакцию или в один пакет для отправки в БД. Это типично для паттерна Unit of Work (UoW), где изменения накапливаются в памяти и потом коммитятся одним коммитом.

Телеметрия выполнения пакета: нужно собирать метрики времени выполнения, потребления ресурсов на стороне БД. Как это сделать? Есть несколько подходов:

Использовать расширения PostgreSQL, такие как pg_stat_statements, которые собирают статистику по выполненным запросам. Можно ассоциировать запросы с конкретным пакетом, вставив traceid в комментарий SQL, и потом извлечь статистику из pg_stat_statements, отфильтровав по комментарию. Но pg_stat_statements агрегирует статистику по нормализованному запросу, а не по отдельным выполнениям. Поэтому не подходит для per-packet телеметрии.

Использовать логирование с расширением auto_explain. Можно включить логирование планов запросов с определенным порогом времени, но это не даст детальной per-query метрики.

Использовать мониторинг на стороне клиента: засекать время выполнения каждого запроса и суммировать по пакету, но это не даст информации о потреблении ресурсов на стороне БД.

Использовать системные представления, такие как pg_stat_activity, pg_stat_database, чтобы в реальном времени получать метрики по соединению. Но они дают текущие значения, а не агрегаты за выполненный пакет.

Использовать протокол pg_stat_statements с тегированием запросов через комментарии и затем сопоставлять с execution plan? Но pg_stat_statements не хранит traceid.

Возможно, нужно собирать телеметрию не для каждого пакета, а агрегированную по traceid, используя логи или расширения типа pg_stat_kcache, которое собирает информацию о потреблении CPU и IO по каждому запросу. Но опять же, нужна связка с traceid.

Другой подход: использовать расширение для мониторинга, например, pg_mon, и через него получать данные, но это сложно.

Или можно использовать протокол pg_stat_activity, но для каждого запроса он показывает текущее состояние, а не историю.

Ограничение производительности: не более 5% падения. Это означает, что сбор метрик должен быть очень легковесным. Например, встроенные счетчики PostgreSQL уже собираются с малыми накладными расходами.

Возможно, нужно предложить решение, основанное на pg_stat_statements + добавление traceid в комментарий, а затем после выполнения пакета запрашивать из pg_stat_statements данные для запросов, выполненных с этим комментарием. Но pg_stat_statements обновляется после завершения запроса, и мы можем его опросить, но нужно быть уверенным, что данные еще не сброшены. Это возможно, но тогда потребуется дополнительный запрос к pg_stat_statements после коммита, что добавит нагрузку.

Но pg_stat_statements хранит агрегированную статистику с момента последнего сброса. Если у нас много пакетов с уникальными traceid, статистика будет агрегирована по одинаковым запросам (но разные traceid будут отличаться только комментарием, так что запросы с разными traceid будут разными из-за комментария? Если комментарий разный, то и запросы будут разными, и pg_stat_statements будет хранить отдельную статистику для каждого уникального текста запроса. Это может привести к раздуванию pg_stat_statements. Но если количество уникальных traceid большое, это не scale. Поэтому не подходит.

Другой вариант: использовать логирование на уровне БД через расширение, которое пишет в лог информацию о запросах с пометкой traceid, и потом парсить логи. Но это создает большую нагрузку на ввод-вывод и может сильно влиять на производительность.

Или использовать внешний инструмент мониторинга, который через eBPF или другие средства собирает метрики, но это выходит за рамки библиотеки.

Это можно сделать, используя команду EXPLAIN (ANALYZE, BUFFERS) для каждого запроса, но выполнение EXPLAIN ANALYZE для каждого запроса замедлит его в разы. Не подходит.

Есть возможность включить параметр track_io_timing = on и затем использовать представление pg_stat_statements, чтобы увидеть время ввода-вывода. Но опять же, это агрегированная статистика.

Но пользователь хочет получить телеметрию для конкретного пакета. Реалистично ли это без значительного замедления? Возможно, можно использовать протокол extended query и в ответе от сервера получать статистику? В протоколе PostgreSQL нет такой возможности. Стандартный протокол не возвращает информацию о потребленных ресурсах.

Единственный способ — это включить логирование и парсить логи, или использовать расширения, такие как pg_stat_statements, но тогда нужно сопоставлять traceid с запросом. Можно передавать traceid в комментарии SQL, и затем, когда запрос выполнен, обратиться к pg_stat_statements и найти запись с этим комментарием, но как мы узнаем, какая именно запись соответствует нашему запросу? Если запрос уникален, то да, можно. Но тогда при каждом выполнении мы добавляем уникальный комментарий, что приводит к росту числа уникальных запросов в pg_stat_statements. Это может негативно сказаться на производительности (замедление сбора статистики, увеличение памяти). И не масштабируется.

Альтернатива: использовать сессионные переменные. Например, установить переменную сеанса (custom GUC) с traceid, и тогда все запросы в этой сессии будут выполняться с этим traceid. Затем можно включить логирование с параметром log_line_prefix, включающим эту переменную. Но это опять логи.

Или можно использовать расширение pg_stats_temp, которое хранит статистику за текущую сессию. Но такого расширения нет.

Возможно, есть решения на уровне ядра PostgreSQL: прослушивать события через уведомления или использовать механизмы аудита, например pgAudit, который пишет в лог все запросы с параметрами. Но опять же, парсинг логов.

Условие "производительность базы не должна упасть больше чем на 5 процентов" — это очень жесткое ограничение. Включение детального логирования или частых опросов pg_stat_statements может превысить это ограничение.

Другой подход: использовать сторонние инструменты мониторинга, которые работают на уровне ОС и БД и собирают метрики независимо, а затем связывать их по traceid, передаваемому через комментарий. Например, использовать Prometheus exporter для PostgreSQL, который собирает данные из pg_stat_statements, и затем алертить или собирать дашборды. Но это не даст связки traceid с конкретным пакетом.

Или можно использовать инструменты трассировки, такие как OpenTelemetry, которые позволяют инструментировать приложение и получать спаны выполнения запросов. На стороне БД можно получать метрики через JMX, если БД запущена на JVM? Но PostgreSQL не на JVM.

Возможно, нужно рассматривать решение на основе расширения pg_stat_kcache + pg_stat_statements, и сопоставлять по queryid. Если мы передаем traceid в комментарии, то queryid будет разным для каждого traceid? Нет, queryid вычисляется по нормализованному запросу, обычно комментарии игнорируются. Так что запросы с разными traceid будут иметь одинаковый queryid. Поэтому через queryid не различить.

В PostgreSQL 14+ появилась поддержка параметра compute_query_id, который позволяет вычислять queryid. И pg_stat_statements может игнорировать комментарии? По умолчанию нормализация удаляет комментарии. То есть все запросы с разными комментариями будут считаться одним запросом. Это хорошо для агрегации, но плохо для различения.

Таким образом, сложно получить per-packet телеметрию без потери производительности.

Может быть, пользователь имеет в виду что-то другое: библиотека собирает изменения в пакеты, и для каждого пакета нужно получить метрики выполнения со стороны БД: время выполнения, количество прочитанных/записанных строк, загрузку CPU и т.п. Но это можно получить на стороне клиента, измерив время и используя информацию о количестве затронутых строк (rowcount). А нагрузка на БД видна только через системные представления, но она общая.

Допустим, мы готовы мириться с некоторой задержкой и дополнительным запросом после каждого пакета к pg_stat_statements, чтобы получить статистику по запросам, которые выполнялись в рамках этого пакета, но для этого нужно знать идентификаторы запросов. Можно было бы использовать расширение pg_stat_statements с включенным параметром pg_stat_statements.track = 'all', и после выполнения пакета сделать запрос к pg_stat_statements, фильтруя по временному промежутку. Но нет временной метки в pg_stat_statements.

В общем, это сложно.

Предлагаю рассмотреть вариант: использовать логирование на уровне сессии через параметр log_statement = 'all' и включать в логи traceid через log_line_prefix, который может включать переменную сеанса. Тогда мы можем асинхронно обрабатывать логи и извлекать метрики, но это повлияет на производительность ввода-вывода и может превысить 5%.

Или использовать внешний агент, который через eBPF собирает информацию о каждом запросе, но это вне библиотеки.

Возможно, правильный ответ: в рамках библиотеки можно реализовать сбор телеметрии на стороне клиента (время выполнения, количество строк, возможно, оценка стоимости через EXPLAIN без ANALYZE), а для получения более глубоких метрик со стороны БД рекомендовать использовать специализированные инструменты (например, расширение pg_stat_monitor от Percona, которое позволяет группировать запросы по тегам). pg_stat_monitor — это форк pg_stat_statements, который добавляет возможность агрегации по тегам (например, по application_name или по комментарию). Оно позволяет собирать статистику с группировкой по пользовательским тегам, и может быть менее нагруженным. Но его нужно устанавливать отдельно.

Вот что говорит документация pg_stat_monitor: оно собирает статистику выполнения запросов с возможностью группировки по различным атрибутам, включая application_name, user, database, и также может включать информацию о времени выполнения, буферах, и т.д. И позволяет получать данные за определенные промежутки времени. Возможно, это решение.

Но вопрос: "какими средствами я могу получить телеметрию выполнения пакета со стороны базы (процессор, память, чтение, запись) и самое главное, например в комментарии этого пакета я положу traceid и хочу чтобы он сохранился рядом с записью телеметрии". Это наводит на мысль, что нужно, чтобы сам PostgreSQL сохранял телеметрию, ассоциированную с traceid.

Тогда нужно использовать расширение, которое сохраняет статистику выполнения запросов и позволяет добавлять произвольные теги. pg_stat_monitor поддерживает теги (через параметр pg_stat_monitor.tags). Можно задать тег, который будет извлекаться из комментария. Но как это работает? Возможно, нужно передавать тег через application_name или через специальный синтаксис комментария. Изучим документацию pg_stat_monitor: в версии 2.0 появилась поддержка тегов. Вы можете задать список тегов, которые будут извлекаться из SQL-комментария с помощью регулярных выражений. Например, вы можете определить тег "traceid" и он будет извлекаться из комментария вида /* traceid=123 */ и сохраняться в статистике. Затем можно запрашивать статистику, сгруппированную по traceid. Это идеально подходит под задачу.

Теперь о производительности: pg_stat_monitor, как и pg_stat_statements, имеет небольшие накладные расходы (обычно менее 5%). Если правильно настроить, можно уложиться в лимит.

Таким образом, реалистичный вариант:

Установить расширение pg_stat_monitor в PostgreSQL (доступно в некоторых дистрибутивах, можно собрать из исходников).

Включить его и настроить параметры: pg_stat_monitor.pgsm_query_max_len, pg_stat_monitor.pgsm_enable_query_plan и т.д.

Настроить извлечение тегов: например, задать параметр pg_stat_monitor.pgsm_extract_comments = 'traceid=([^,]+)' или что-то подобное. Это позволит извлекать traceid из комментариев SQL.

В библиотеке Python, при выполнении пакета, добавлять в каждый SQL-запрос комментарий с traceid (например, /* traceid=abc123 */ SELECT ...). Это можно делать автоматически для всех запросов в рамках одного unit of work.

После выполнения пакета, библиотека может (опционально) выполнить запрос к представлению pg_stat_monitor, чтобы получить агрегированную статистику по запросам с данным traceid. Например, SELECT * FROM pg_stat_monitor WHERE tags->>'traceid' = 'abc123'; (синтаксис может отличаться). Это позволит получить время выполнения, количество прочитанных буферов, время CPU и т.д. для каждого запроса в пакете.

Важно, что сбор статистики в pg_stat_monitor происходит асинхронно и не замедляет сами запросы (только небольшая дополнительная запись в shared memory). Дополнительный запрос к pg_stat_monitor после пакета может добавить небольшую нагрузку, но он выполняется один раз на пакет, что приемлемо, если пакеты не слишком частые.

Ограничения: pg_stat_monitor хранит данные в кольцевом буфере, поэтому старые данные могут быть перезаписаны. Но если мы запрашиваем статистику сразу после выполнения пакета, то она еще должна быть доступна.

Также нужно учесть, что pg_stat_monitor не входит в стандартную поставку PostgreSQL, и его установка может потребовать дополнительных привилегий.

Альтернатива: использовать pg_stat_statements + ручное сопоставление, но это менее точно.

Еще вариант: использовать логирование и последующий парсинг, но это больше нагрузка и сложность.

Таким образом, я предлагаю решение на основе pg_stat_monitor. Теперь нужно сформулировать ответ пользователю: описать этот подход, его преимущества, требования к установке, настройке и пример кода библиотеки.

Также уточним вопросы: возможно, пользователь ожидает решения, не требующего установки расширения. Может быть, можно использовать что-то встроенное? Но встроенного способа связать traceid с телеметрией нет.

Другой вариант: использовать функции мониторинга производительности, доступные в коммерческих решениях (например, RDS Performance Insights), но это не универсально.

Поэтому предложу pg_stat_monitor как наиболее реалистичный.

Реалистичный подход к сбору телеметрии пакетов с привязкой к traceid
Для решения задачи — получение телеметрии выполнения пакета изменений (CPU, память, чтение, запись) со стороны PostgreSQL и привязка её к переданному traceid с падением производительности не более 5% — оптимальным решением будет использование расширения pg_stat_monitor. Это расширение является форком и эволюцией pg_stat_statements, специально разработанным для сбора детальной статистики запросов с возможностью группировки по произвольным тегам, извлекаемым из комментариев SQL.

Почему pg_stat_monitor?
Требование	Решение pg_stat_monitor
Получение телеметрии выполнения пакета (CPU, буферы, чтение/запись)	Собирает время выполнения, количество прочитанных/изменённых строк, буферы shared/hit/read/dirtied, время CPU, планы запросов и многое другое
Привязка к traceid через комментарий	Поддерживает извлечение тегов из SQL-комментариев по заданным регулярным выражениям
Низкое влияние на производительность (<5%)	Работает аналогично pg_stat_statements — накладные расходы составляют доли процента при правильной настройке
Асинхронный сбор и доступ через SQL	Статистика накапливается в разделяемой памяти и доступна через системное представление
Совместимость с асинхронным Python	Любая библиотека (asyncpg, psycopg 3) может выполнять обычные SQL-запросы для получения данных
Архитектура решения
Настройка PostgreSQL

Установить расширение pg_stat_monitor (доступно в PGDG-репозиториях или сборка из исходников).

Включить в postgresql.conf:

ini
shared_preload_libraries = 'pg_stat_monitor'
pg_stat_monitor.pgsm_enable = on
pg_stat_monitor.pgsm_extract_comments = 'traceid=([^, ]+)'   # извлечение traceid
pg_stat_monitor.pgsm_query_max_len = 1024                    # опционально
pg_stat_monitor.pgsm_max = 5000                               # размер кольцевого буфера
track_io_timing = on                                           # для сбора времени I/O
Перезапустить кластер.

Библиотека на Python

Для каждого «пакета изменений» (Unit of Work) генерируется уникальный traceid (например, UUID).

Все SQL-запросы внутри пакета автоматически дополняются комментарием:

sql
/* traceid=abc-123 */ INSERT INTO roles (name) VALUES ('admin');
/* traceid=abc-123 */ UPDATE roles SET ... ;
После завершения пакета (коммита) библиотека выполняет запрос к pg_stat_monitor для получения статистики по данному traceid:

sql
SELECT 
   query, 
   calls, 
   total_time, 
   min_time, 
   max_time, 
   shared_blks_hit, 
   shared_blks_read, 
   shared_blks_dirtied,
   local_blks_written,
   temp_blks_read,
   temp_blks_written,
   blk_read_time,
   blk_write_time,
   cpu_user_time,
   cpu_system_time
FROM pg_stat_monitor 
WHERE tags->>'traceid' = 'abc-123';
Обработанную статистику можно сохранять в центральном хранилище или логе.

Управление производительностью

Запрос к pg_stat_monitor выполняется после коммита и не блокирует основной поток.

Для снижения нагрузки можно:

опрашивать статистику не для каждого пакета, а агрегировать по traceid в памяти и отправлять пачками;

использовать асинхронные задачи (например, через asyncio.create_task) для фонового сбора метрик;

настроить размер кольцевого буфера так, чтобы статистика для пакета гарантированно не вытеснялась до её сбора.

Что можно получить из pg_stat_monitor?
Расширение предоставляет десятки метрик, включая:

Временны́е показатели: общее время, минимальное, максимальное, среднеквадратичное отклонение.

Буферный кэш: чтения из shared buffers (hit), с диска (read), грязные страницы (dirtied), локальные и временные буферы.

I/O время: время чтения/записи блоков (если включено track_io_timing).

CPU время: пользовательское и системное время (требуется pg_stat_statements + pg_stat_kcache, но pg_stat_monitor может интегрироваться с ним).

Планы запросов: если включено pgsm_enable_query_plan, можно получить план выполнения.

Подробный список колонок см. в документации pg_stat_monitor.

Пример реализации на Python (asyncpg)
python
import asyncio
import uuid
import asyncpg

class PgUoWBatch:
    def __init__(self, conn):
        self.conn = conn
        self.traceid = str(uuid.uuid4())
        self.queries = []

    async def execute(self, query, *args):
        # Добавляем комментарий с traceid в начало запроса
        tagged_query = f"/* traceid={self.traceid} */ {query}"
        self.queries.append(tagged_query)
        return await self.conn.execute(tagged_query, *args)

    async def commit(self):
        async with self.conn.transaction():
            for q in self.queries:
                await self.conn.execute(q)  # здесь уже с тегами
        # После коммита собираем телеметрию
        await self.collect_metrics()

    async def collect_metrics(self):
        # Используем отдельное соединение или фоновую задачу
        row = await self.conn.fetchrow("""
            SELECT 
                sum(total_time) as total_time,
                sum(shared_blks_hit) as shared_blks_hit,
                sum(shared_blks_read) as shared_blks_read,
                sum(cpu_user_time) as cpu_user_time,
                sum(cpu_system_time) as cpu_system_time
            FROM pg_stat_monitor 
            WHERE tags->>'traceid' = $1
        """, self.traceid)
        # Отправляем метрику в центральную систему (например, через логи или метрики)
        print(f"Metrics for {self.traceid}: {row}")

# Использование
async def main():
    conn = await asyncpg.connect(...)
    batch = PgUoWBatch(conn)
    await batch.execute("INSERT INTO roles(name) VALUES($1)", 'editor')
    await batch.execute("UPDATE roles SET name=$1 WHERE id=$2", 'admin', 1)
    await batch.commit()
Влияние на производительность
Добавление комментария: незначительно увеличивает длину запроса, но современные парсеры обрабатывают это быстро.

Извлечение тегов в pg_stat_monitor: выполняется один раз при завершении запроса и использует очень лёгкое регулярное выражение. В тестах Percona накладные расходы pg_stat_monitor находятся в пределах 2–3% по сравнению с pg_stat_statements.

Дополнительный запрос к pg_stat_monitor: один короткий запрос на пакет. При частоте 1000 пакетов/сек это даст 1000 запросов, что обычно укладывается в 5% нагрузки, особенно если выполнять его асинхронно и с небольшим приоритетом.

Рекомендации: для высоконагруженных систем стоит настроить буфер pg_stat_monitor так, чтобы он вмещал статистику за всё время жизни пакета, и, возможно, использовать пул соединений для сбора метрик, чтобы не задерживать основные запросы.

Альтернативные варианты (менее предпочтительные)
Вариант	Плюсы	Минусы
pg_stat_statements + ручной парсинг логов	Не требует расширений	Невозможно связать traceid с запросом без уникальных комментариев (раздувает статистику), сложность агрегации
Логирование с log_line_prefix и парсинг	Можно получить детальные данные	Высокое влияние на I/O (>5%), задержки, сложность синхронизации
Использование eBPF-агентов (например, Pixie)	Очень детально, не требует изменений в БД	Выходит за рамки библиотеки, сложность развертывания, не всегда доступно
Итог
Связка pg_stat_monitor + библиотека Python с добавлением traceid в комментарии даёт реалистичный и эффективный способ получать телеметрию выполнения пакетов изменений с привязкой к конкретному traceid при падении производительности не более 5%. Это решение требует однократной настройки PostgreSQL и небольшой доработки кода библиотеки, но в остальном полностью отвечает поставленным требованиям.

Если у вас есть возможность установки расширений на стороне БД — этот путь наиболее надёжен и масштабируем.

# Исследование: телеметрия PostgreSQL для Unit of Work

## По твоим ответам

### 1. Про 100 пакетов/сек

100 UoW/сек — это **средняя** нагрузка для бизнес-приложения. Не низкая, но и не экстремальная. Для контекста:

- **Низкая**: 1-10 UoW/сек — внутренний CRM, бухгалтерия
- **Средняя**: 50-200 UoW/сек — e-commerce, SaaS-платформа, логистика
- **Высокая**: 1000+ UoW/сек — платёжные системы, ad-tech
- **Экстремальная**: 10000+ UoW/сек — соцсети, телеком

При 100 UoW/сек у тебя есть **10мс бюджета на телеметрию** одного пакета прежде чем она станет заметной. Это очень комфортно — можно позволить себе довольно подробный сбор.

---

### 2. Стратегии телеметрии — от лёгкой к тяжёлой

#### Уровень 0 — Только клиентская сторона (overhead ~0%)

**Что получаем:**
- Wall clock time (замер в Python)
- Количество запросов в пакете
- Rows affected (из ответа PostgreSQL на каждую команду)

**Что НЕ получаем:**
- Ничего со стороны базы

**Как:** просто `time.monotonic()` вокруг транзакции

---

#### Уровень 1 — pg_stat_statements diff (overhead < 1%)

**Что получаем:**
- Всё из уровня 0
- shared/local/temp blocks hit/read/write
- blk_read_time, blk_write_time (если `track_io_timing = on`)
- WAL bytes generated (PG13+)
- total_exec_time со стороны сервера (точнее чем wall clock)

**Что НЕ получаем:**
- CPU отдельно от IO
- Потребление памяти

**Как работает:**
1. Перед транзакцией — SELECT из `pg_stat_statements` агрегат по текущему user+database
2. Выполняем пакет
3. После — ещё один SELECT, считаем diff

**Проблема точности:** pg_stat_statements — глобальная статистика. Если параллельно работают другие соединения от того же пользователя, diff будет "загрязнён" их метриками. При 100 UoW/сек это **реальная проблема**.

**Решение проблемы:** фильтровать по `queryid`. Каждый нормализованный запрос имеет уникальный queryid. Если ты знаешь свои запросы — можешь брать diff только по ним. Но всё равно не идеально если один и тот же запрос выполняется параллельно.

---

#### Уровень 2 — EXPLAIN (ANALYZE, BUFFERS, WAL) (overhead 2-5%)

**Что получаем:**
- Всё из уровня 1
- **Точные** данные по каждому запросу, без загрязнения
- Shared/local/temp buffers hit/read/dirtied/written — **для конкретного выполнения**
- WAL records и bytes
- Planning time vs Execution time отдельно
- Полный план выполнения (для диагностики)

**Что НЕ получаем:**
- CPU (отдельно)
- Память

**Как работает:** каждый запрос в пакете оборачивается в `EXPLAIN (ANALYZE, BUFFERS, WAL, FORMAT JSON) <sql>`. Запрос **реально выполняется** (это не dry-run), но к нему добавляется инструментирование.

**Overhead:** ~2-5% потому что:
- Планировщик делает дополнительную работу по сбору runtime-статистики
- JSON-сериализация плана
- Парсинг JSON на стороне Python

**Критический нюанс:** EXPLAIN ANALYZE для DML (INSERT/UPDATE/DELETE) **реально выполняет** изменения. Это нормально внутри транзакции — они либо коммитятся, либо откатываются. Но нужно аккуратно обрабатывать RETURNING и побочные эффекты.

---

#### Уровень 3 — pg_stat_kcache (overhead 2-3%, но нужно расширение)

**Что получаем:**
- Всё из уровня 1
- **CPU user time и system time** (из getrusage)
- **Page faults** (minor/major) — прокси для использования памяти
- Физические reads/writes на уровне ОС (не только buffer pool)

**Что НЕ получаем:**
- RSS / heap использование конкретного бэкенда (этого PostgreSQL просто не умеет отдавать per-query)

**Как работает:** расширение `pg_stat_kcache` хукается в те же точки что и `pg_stat_statements` и вызывает `getrusage()` до и после каждого запроса. Данные доступны через view `pg_stat_kcache`.

**Важно:** это расширение от Dalibo, не входит в core. Нужно компилировать или ставить из пакетов. На managed PostgreSQL (RDS, Cloud SQL) — **недоступно**.

---

#### Уровень 4 — Комбинированный (мой рекомендуемый для тебя)

**EXPLAIN ANALYZE + pg_stat_kcache** если доступен, иначе **EXPLAIN ANALYZE + pg_stat_statements diff**

**Что получаем:**
- Точные buffer-метрики на запрос (из EXPLAIN)
- CPU и page faults (из pg_stat_kcache, если есть)
- WAL bytes
- Planning vs Execution time
- trace_id привязан через SQL-комментарий

### 3. Кто собирает и куда отправляет

#### Кто собирает: сравнение

```
                    Сбор в Python          Сбор в PostgreSQL
                    (наш модуль)           (расширения/триггеры)
                    ─────────────          ─────────────────────
Гибкость           Полная                 Ограничена SQL/C
trace_id            Легко передать         Через SET/комментарии
Overhead на БД      Доп. запросы           Минимальный
Overhead на app     Парсинг/отправка       Нулевой
Контроль            Полный                 Нужен DBA
Managed PG          Работает               Ограничения
```

**Мой вердикт:** собирать в Python-модуле. Причины:
- trace_id уже там
- гибкость в обработке
- не нужен доступ к серверу PostgreSQL для установки расширений
- managed PostgreSQL не всегда позволяет ставить расширения

#### Куда отправлять: по скорости

```
Вариант             Latency        Throughput       Сложность
────────            ───────        ──────────       ─────────
In-process buffer   ~0             Бесконечный      Низкая
 → batch flush

UDP socket          ~0.01ms        Очень высокий    Низкая
(StatsD/OTel)

Unix domain socket  ~0.02ms        Очень высокий    Низкая

Redis LPUSH         ~0.1-0.3ms     50-100k/sec      Средняя

Kafka produce       ~0.5-2ms       100k+/sec        Высокая
(async batch)

HTTP POST           ~1-5ms         10-20k/sec       Средняя
(OTel collector)
```

**Мой рекомендуемый подход:**

```
Python UoW
    │
    ▼
In-process ring buffer (asyncio.Queue, bounded)
    │
    ▼ (отдельная asyncio task, batch flush каждые 100ms или 50 записей)
    │
    ├──▶ OpenTelemetry Collector (OTLP/gRPC) — если уже есть OTel
    │
    ├──▶ UDP → StatsD/Telegraf → InfluxDB/Prometheus — самый простой
    │
    └──▶ Redis Stream (XADD) — если нужна гарантия доставки
              без сложности Kafka
```

**Почему не напрямую в Kafka:** для 100 UoW/сек Kafka — overkill. Kafka оправдана при 10k+ msg/sec или если она уже в инфраструктуре. Latency Kafka producer с `acks=1` — это 1-5ms, и нужен отдельный клиент (aiokafka), который добавляет сложности.

**Самый быстрый реалистичный путь:**
1. Складываем в `asyncio.Queue(maxsize=10000)` — **не блокирует** UoW
2. Background task читает батчами и отправляет по UDP в OTel Collector или Telegraf
3. Если очередь переполнена — дропаем (телеметрия не стоит того чтобы тормозить бизнес-логику)

### 4. asyncpg vs psycopg3 (async)

#### asyncpg

**Архитектура:** написан на Cython, использует нативный PostgreSQL binary protocol напрямую, без libpq.

**Плюсы для нашей задачи:**
- Быстрее на ~2-3x для простых запросов
- `conn.get_server_pid()` — мгновенно, без запроса
- `prepared statements` кешируются автоматически — EXPLAIN ANALYZE для каждого запроса не нуждается в повторном planning
- Бинарный протокол — меньше парсинга

**Минусы для нашей задачи:**
- `EXPLAIN ANALYZE` возвращает текст, парсить нужно самим
- Нет встроенного хука "до/после каждого запроса" — нужно оборачивать вручную
- Пул соединений (`asyncpg.Pool`) — свой, не совместим с SQLAlchemy напрямую
- `copy` протокол — свой API

**Нюанс точности:** asyncpg кеширует prepared statements. Когда ты отправляешь `/* trace_id=abc */ SELECT * FROM users WHERE id = $1`, pg_stat_statements увидит **нормализованную** версию. С `track_comments = on` (PG14+) комментарий сохранится в `pg_stat_statements.query`, но queryid будет одинаковым для разных trace_id — это **хорошо** для агрегации, но нужно иметь в виду.

#### psycopg3 (async)

**Архитектура:** обёртка над libpq, async через `psycopg.AsyncConnection`, использует текстовый или бинарный протокол.

**Плюсы для нашей задачи:**
- Pipeline mode (PG14+) — можно отправить все запросы пакетом без ожидания ответа на каждый, потом собрать все результаты
- `cursor.statusmessage` — rows affected из коробки
- `conn.execute()` поддерживает `prepare=True/False` — контроль над кешированием
- Лучше совместимость с SQLAlchemy 2.0
- `conn.notifies()` — удобно для получения уведомлений от БД

**Минусы для нашей задачи:**
- Медленнее на ~30-50% для мелких запросов (из-за libpq overhead)
- Пул (`psycopg_pool.AsyncConnectionPool`) чуть менее зрелый

**Критическая разница для телеметрии: Pipeline mode**

```
asyncpg (обычный режим):
  Query 1 → Response 1 → Query 2 → Response 2 → Query 3 → Response 3
  [         Round Trip          ][         Round Trip          ][...]

psycopg3 pipeline:
  Query 1 → Query 2 → Query 3 → Response 1 → Response 2 → Response 3
  [     Одна пачка отправки    ][     Одна пачка получения    ]
```

Pipeline экономит round-trips. Для пакета из 10 запросов при latency 1ms к БД — это **9ms экономии**. Но: **в pipeline нельзя использовать EXPLAIN ANALYZE** (потому что каждый ответ меняет контекст следующего запроса).

Значит:
- **Для уровня 0-1 телеметрии** → psycopg3 с pipeline будет **быстрее**
- **Для уровня 2-3 телеметрии** → asyncpg и psycopg3 примерно одинаковы (pipeline невозможен)

#### Итоговое сравнение

```
                        asyncpg          psycopg3 async
                        ───────          ──────────────
Скорость (мелкие)       ★★★★★            ★★★
Pipeline mode           Нет (есть PR)    Да (PG14+)
EXPLAIN ANALYZE         Работает         Работает
pg_stat_statements      Одинаково        Одинаково
Prepared stmt cache     Авто             Ручной контроль
Overhead телеметрии     Одинаковый       Одинаковый
SQLAlchemy 2.0          Через greenlet   Нативно
Зрелость                8 лет            3 года
Managed PG              Ок               Ок
```

Каждая стратегия даёт разный набор метрик и имеет разное влияние на производительность. Выбирать можно динамически в зависимости от текущей нагрузки или режима отладки.

Уровень 0 — только клиентские замеры
Метрики: общее время выполнения пакета (wall‑clock), количество запросов, число затронутых строк (берётся из ответа БД).

Плюсы: абсолютно безопасно для БД, не требует доступа к внутренностям PostgreSQL.
Минусы: нет данных о работе самой СУБД (буферы, ввод‑вывод, CPU).

Нагрузка: <1 % (один дополнительный короткий запрос до и после транзакции).
Особенность: данные агрегируются по всем запросам одного пользователя/базы. При параллельной работе нескольких UoW одного пользователя показатели «зашумляются». Метрики: точные показатели для каждого запроса в пакете:

буферы (shared/local/temp hit/read/dirtied/written),

время планирования и выполнения,

количество строк на каждом узле плана,

объём WAL (запись).
Нагрузка: 2–5 % (дополнительная работа планировщика по сбору статистики и сериализации плана).
Плюсы: абсолютная точность, привязка к конкретному trace_id через комментарий, возможность получить план запроса для анализа узких мест.
Минусы: EXPLAIN ANALYZE для DML действительно выполняет изменения – это нормально внутри транзакции, но требует аккуратности с операторами, имеющими побочные эффекты (например, функции с volatile).

Плюсы: даёт то, что не получить иными способами – отдельно процессорное время и намёки на использование памяти.

# Исследование: телеметрия PostgreSQL для Unit of Work

## По твоим ответам

### 1. Про 100 пакетов/сек

100 UoW/сек — это **средняя** нагрузка для бизнес-приложения. Не низкая, но и не экстремальная. Для контекста:

- **Низкая**: 1-10 UoW/сек — внутренний CRM, бухгалтерия
- **Средняя**: 50-200 UoW/сек — e-commerce, SaaS-платформа, логистика
- **Высокая**: 1000+ UoW/сек — платёжные системы, ad-tech
- **Экстремальная**: 10000+ UoW/сек — соцсети, телеком

При 100 UoW/сек у тебя есть **10мс бюджета на телеметрию** одного пакета прежде чем она станет заметной. Это очень комфортно — можно позволить себе довольно подробный сбор.

---

### 2. Стратегии телеметрии — от лёгкой к тяжёлой

#### Уровень 0 — Только клиентская сторона (overhead ~0%)

**Что получаем:**
- Wall clock time (замер в Python)
- Количество запросов в пакете
- Rows affected (из ответа PostgreSQL на каждую команду)

**Что НЕ получаем:**
- Ничего со стороны базы

**Как:** просто `time.monotonic()` вокруг транзакции

---

#### Уровень 1 — pg_stat_statements diff (overhead < 1%)

**Что получаем:**
- Всё из уровня 0
- shared/local/temp blocks hit/read/write
- blk_read_time, blk_write_time (если `track_io_timing = on`)
- WAL bytes generated (PG13+)
- total_exec_time со стороны сервера (точнее чем wall clock)

**Что НЕ получаем:**
- CPU отдельно от IO
- Потребление памяти

**Как работает:**
1. Перед транзакцией — SELECT из `pg_stat_statements` агрегат по текущему user+database
2. Выполняем пакет
3. После — ещё один SELECT, считаем diff

**Проблема точности:** pg_stat_statements — глобальная статистика. Если параллельно работают другие соединения от того же пользователя, diff будет "загрязнён" их метриками. При 100 UoW/сек это **реальная проблема**.

**Решение проблемы:** фильтровать по `queryid`. Каждый нормализованный запрос имеет уникальный queryid. Если ты знаешь свои запросы — можешь брать diff только по ним. Но всё равно не идеально если один и тот же запрос выполняется параллельно.

---

#### Уровень 2 — EXPLAIN (ANALYZE, BUFFERS, WAL) (overhead 2-5%)

**Что получаем:**
- Всё из уровня 1
- **Точные** данные по каждому запросу, без загрязнения
- Shared/local/temp buffers hit/read/dirtied/written — **для конкретного выполнения**
- WAL records и bytes
- Planning time vs Execution time отдельно
- Полный план выполнения (для диагностики)

**Что НЕ получаем:**
- CPU (отдельно)
- Память

**Как работает:** каждый запрос в пакете оборачивается в `EXPLAIN (ANALYZE, BUFFERS, WAL, FORMAT JSON) <sql>`. Запрос **реально выполняется** (это не dry-run), но к нему добавляется инструментирование.

**Overhead:** ~2-5% потому что:
- Планировщик делает дополнительную работу по сбору runtime-статистики
- JSON-сериализация плана
- Парсинг JSON на стороне Python

**Критический нюанс:** EXPLAIN ANALYZE для DML (INSERT/UPDATE/DELETE) **реально выполняет** изменения. Это нормально внутри транзакции — они либо коммитятся, либо откатываются. Но нужно аккуратно обрабатывать RETURNING и побочные эффекты.

---

#### Уровень 3 — pg_stat_kcache (overhead 2-3%, но нужно расширение)

**Что получаем:**
- Всё из уровня 1
- **CPU user time и system time** (из getrusage)
- **Page faults** (minor/major) — прокси для использования памяти
- Физические reads/writes на уровне ОС (не только buffer pool)

**Что НЕ получаем:**
- RSS / heap использование конкретного бэкенда (этого PostgreSQL просто не умеет отдавать per-query)

**Как работает:** расширение `pg_stat_kcache` хукается в те же точки что и `pg_stat_statements` и вызывает `getrusage()` до и после каждого запроса. Данные доступны через view `pg_stat_kcache`.

**Важно:** это расширение от Dalibo, не входит в core. Нужно компилировать или ставить из пакетов. На managed PostgreSQL (RDS, Cloud SQL) — **недоступно**.

---

#### Уровень 4 — Комбинированный (мой рекомендуемый для тебя)

**EXPLAIN ANALYZE + pg_stat_kcache** если доступен, иначе **EXPLAIN ANALYZE + pg_stat_statements diff**

**Что получаем:**
- Точные buffer-метрики на запрос (из EXPLAIN)
- CPU и page faults (из pg_stat_kcache, если есть)
- WAL bytes
- Planning vs Execution time
- trace_id привязан через SQL-комментарий

### 3. Кто собирает и куда отправляет

#### Кто собирает: сравнение

```
                    Сбор в Python          Сбор в PostgreSQL
                    (наш модуль)           (расширения/триггеры)
                    ─────────────          ─────────────────────
Гибкость           Полная                 Ограничена SQL/C
trace_id            Легко передать         Через SET/комментарии
Overhead на БД      Доп. запросы           Минимальный
Overhead на app     Парсинг/отправка       Нулевой
Контроль            Полный                 Нужен DBA
Managed PG          Работает               Ограничения
```

**Мой вердикт:** собирать в Python-модуле. Причины:
- trace_id уже там
- гибкость в обработке
- не нужен доступ к серверу PostgreSQL для установки расширений
- managed PostgreSQL не всегда позволяет ставить расширения

#### Куда отправлять: по скорости

```
Вариант             Latency        Throughput       Сложность
────────            ───────        ──────────       ─────────
In-process buffer   ~0             Бесконечный      Низкая
 → batch flush

UDP socket          ~0.01ms        Очень высокий    Низкая
(StatsD/OTel)

Unix domain socket  ~0.02ms        Очень высокий    Низкая

Redis LPUSH         ~0.1-0.3ms     50-100k/sec      Средняя

Kafka produce       ~0.5-2ms       100k+/sec        Высокая
(async batch)

HTTP POST           ~1-5ms         10-20k/sec       Средняя
(OTel collector)
```

**Мой рекомендуемый подход:**

```
Python UoW
    │
    ▼
In-process ring buffer (asyncio.Queue, bounded)
    │
    ▼ (отдельная asyncio task, batch flush каждые 100ms или 50 записей)
    │
    ├──▶ OpenTelemetry Collector (OTLP/gRPC) — если уже есть OTel
    │
    ├──▶ UDP → StatsD/Telegraf → InfluxDB/Prometheus — самый простой
    │
    └──▶ Redis Stream (XADD) — если нужна гарантия доставки
              без сложности Kafka
```

**Почему не напрямую в Kafka:** для 100 UoW/сек Kafka — overkill. Kafka оправдана при 10k+ msg/sec или если она уже в инфраструктуре. Latency Kafka producer с `acks=1` — это 1-5ms, и нужен отдельный клиент (aiokafka), который добавляет сложности.

**Самый быстрый реалистичный путь:**
1. Складываем в `asyncio.Queue(maxsize=10000)` — **не блокирует** UoW
2. Background task читает батчами и отправляет по UDP в OTel Collector или Telegraf
3. Если очередь переполнена — дропаем (телеметрия не стоит того чтобы тормозить бизнес-логику)

### 4. asyncpg vs psycopg3 (async)

#### asyncpg

**Архитектура:** написан на Cython, использует нативный PostgreSQL binary protocol напрямую, без libpq.

**Плюсы для нашей задачи:**
- Быстрее на ~2-3x для простых запросов
- `conn.get_server_pid()` — мгновенно, без запроса
- `prepared statements` кешируются автоматически — EXPLAIN ANALYZE для каждого запроса не нуждается в повторном planning
- Бинарный протокол — меньше парсинга

**Минусы для нашей задачи:**
- `EXPLAIN ANALYZE` возвращает текст, парсить нужно самим
- Нет встроенного хука "до/после каждого запроса" — нужно оборачивать вручную
- Пул соединений (`asyncpg.Pool`) — свой, не совместим с SQLAlchemy напрямую
- `copy` протокол — свой API

**Нюанс точности:** asyncpg кеширует prepared statements. Когда ты отправляешь `/* trace_id=abc */ SELECT * FROM users WHERE id = $1`, pg_stat_statements увидит **нормализованную** версию. С `track_comments = on` (PG14+) комментарий сохранится в `pg_stat_statements.query`, но queryid будет одинаковым для разных trace_id — это **хорошо** для агрегации, но нужно иметь в виду.

#### psycopg3 (async)

**Архитектура:** обёртка над libpq, async через `psycopg.AsyncConnection`, использует текстовый или бинарный протокол.

**Плюсы для нашей задачи:**
- Pipeline mode (PG14+) — можно отправить все запросы пакетом без ожидания ответа на каждый, потом собрать все результаты
- `cursor.statusmessage` — rows affected из коробки
- `conn.execute()` поддерживает `prepare=True/False` — контроль над кешированием
- Лучше совместимость с SQLAlchemy 2.0
- `conn.notifies()` — удобно для получения уведомлений от БД

**Минусы для нашей задачи:**
- Медленнее на ~30-50% для мелких запросов (из-за libpq overhead)
- Пул (`psycopg_pool.AsyncConnectionPool`) чуть менее зрелый

**Критическая разница для телеметрии: Pipeline mode**

```
asyncpg (обычный режим):
  Query 1 → Response 1 → Query 2 → Response 2 → Query 3 → Response 3
  [         Round Trip          ][         Round Trip          ][...]

psycopg3 pipeline:
  Query 1 → Query 2 → Query 3 → Response 1 → Response 2 → Response 3
  [     Одна пачка отправки    ][     Одна пачка получения    ]
```

Pipeline экономит round-trips. Для пакета из 10 запросов при latency 1ms к БД — это **9ms экономии**. Но: **в pipeline нельзя использовать EXPLAIN ANALYZE** (потому что каждый ответ меняет контекст следующего запроса).

Значит:
- **Для уровня 0-1 телеметрии** → psycopg3 с pipeline будет **быстрее**
- **Для уровня 2-3 телеметрии** → asyncpg и psycopg3 примерно одинаковы (pipeline невозможен)

Полный список параметров телеметрии (Уровень 4)
Используя комбинацию EXPLAIN (ANALYZE, BUFFERS, WAL, SERIALIZE) и, если есть доступ, pg_stat_kcache, вы получите исчерпывающий набор метрик.

1. От EXPLAIN ANALYZE (Точность выполнения)
Planning Time: Время, которое планировщик PostgreSQL потратил на создание плана запроса.

Execution Time: Время, затраченное на непосредственное выполнение запроса.

Total Time (Planning + Execution): Общее время обработки запроса на стороне сервера.

Actual Rows: Реальное количество строк, обработанных на каждом этапе плана.

Loops: Количество выполнений узла плана (важно для Nested Loops).

2. От BUFFERS (Работа с памятью и диском)
Shared Buffers (hit/read/dirtied/written): Сколько данных было взято из кэша (hit) или прочитано с диска (read). Показывает эффективность кэширования и нагрузку на диск.

Local Buffers (hit/read/dirtied/written): Аналогично для временных таблиц в рамках сессии.

Temp Buffers (read/written): Объем данных, записанных во временные файлы на диск (обычно из-за сортировок или хеш-таблиц, не поместившихся в work_mem). Критический показатель нехватки памяти.

I/O Timings (blk_read_time / blk_write_time): Время, затраченное на фактические операции ввода-вывода (требует включения track_io_timing).

3. От WAL (Нагрузка на запись)
WAL Records: Количество записей в журнал предзаписи (WAL), сгенерированных запросом.

WAL Bytes: Объем данных, записанных в WAL. Прямой показатель нагрузки на систему репликации и восстановления.

4. От SERIALIZE (Сетевая нагрузка) — Новое в PostgreSQL 17
Serialization Time: Время, затраченное на преобразование данных из внутреннего формата PostgreSQL в формат для отправки клиенту (например, в текст).

Serialization Output Size: Реальный размер данных, который был бы отправлен по сети. Это позволяет оценить сетевую стоимость запроса .

5. От pg_stat_kcache (Системные ресурсы)
CPU Time (user/system): Время CPU в пользовательском режиме и режиме ядра, потраченное на выполнение запроса. Позволяет отличить CPU-bound запросы от I/O-bound .

Page Faults (minor/major): Количество обращений к страницам памяти. Может служить косвенным индикатором потребления памяти процессом.

Оценка оверхеда для Уровня 4 (asyncpg + EXPLAIN...)
Оценим дополнительную нагрузку на систему при использовании этого подхода.

1. Оверхед на стороне БД
CPU: Увеличивается на 3-7%. Это связано с дополнительной работой планировщика по сбору статистики (ANALYZE) и сериализации данных (BUFFERS, WAL). pg_stat_kcache добавляет еще ~2-3% .

Память: Практически не изменяется. EXPLAIN ANALYZE не потребляет дополнительной памяти, кроме как для хранения собираемой статистики. Основной расход памяти — на буферные кэши, которые мы как раз и измеряем.

Время выполнения (Latency): Добавляется от 0.5 до 3 мс на каждый запрос, в зависимости от его сложности. Это время на сбор и вывод статистики. Для 1000 запросов это может дать дополнительные 0.5-3 секунды, но это время ожидания, а не блокировки.

2. Оверхед на стороне Python (asyncpg)
Это самое важное. Весь этот оверхед — это время, которое ваше приложение тратит на обработку телеметрии. Оно складывается из:

Передачи данных от PostgreSQL.

Парсинга JSON-плана, возвращаемого EXPLAIN (FORMAT JSON).

Формирования финального объекта с метриками.

Помещения этого объекта в asyncio.Queue (что практически бесплатно).

Оценка дополнительного времени на стороне Python:

100 команд в пакете: +5-15 мс

200 команд в пакете: +10-30 мс

500 команд в пакете: +25-75 мс

1000 команд в пакете: +50-150 мс

Почему такой разброс?

5 мс: Для простых команд (INSERT INTO table (id) VALUES (1)), где план простой, а JSON — маленький.

15 мс и выше: Для сложных запросов с большими планами (много джойнов, агрегаций), где JSON может быть объемным, и его парсинг в Python занимает время.

Суть решения и итоговая оценка
Суть: Вы получаете "черный ящик" для каждого пакета изменений. Вы знаете не только то, что он выполнился, но и какой ценой: сколько ресурсов CPU, памяти, дискового ввода-вывода и сети это стоило вашей базе данных. Привязка через trace_id в комментарии позволяет связать эти метрики с конкретным бизнес-процессом или запросом пользователя.

Итоговая оценка для 100 UoW/сек с 200 командами в каждом:

Нагрузка на БД: Дополнительные ~5% CPU и до +2 секунд на выполнение пакета (из-за накладных расходов EXPLAIN).

Нагрузка на Python: Дополнительные 10-30 мс на обработку телеметрии для каждого пакета. Это время тратится в асинхронной задаче, которая не блокирует основной поток выполнения вашего UoW. Вы просто кладете данные в очередь и идете дальше.

Отправка в Kafka: Это происходит в фоновом режиме и не влияет на задержку выполнения самого UoW. При 100 UoW/сек и среднем размере метрик в 2 КБ, вы генерируете поток данных в 200 КБ/сек, что Kafka "съест" даже не заметив.

Вердикт: Предложенная архитектура (asyncpg + EXPLAIN (ANALYZE, BUFFERS, WAL, SERIALIZE) + фоновая отправка в Kafka) является реалистичной и эффективной. Она дает вам хирургическую точность в понимании работы базы данных с минимальным влиянием на производительность, укладываясь в заявленные 5% при нагрузке до 500-1000 команд в пакете. Вы получаете бесценные данные для профилирования, оптимизации и биллинга (если это требуется).

# Уровень 4: EXPLAIN ANALYZE + pg_stat_kcache через asyncpg

## Суть решения в одном абзаце

Каждый SQL-запрос внутри пакета (Unit of Work) оборачивается в `EXPLAIN (ANALYZE, BUFFERS, WAL, FORMAT JSON)`. Это заставляет PostgreSQL **реально выполнить** запрос, но при этом собрать детальную статистику по буферам, WAL и времени. Параллельно расширение `pg_stat_kcache` автоматически вызывает системный `getrusage()` до и после каждого запроса, что даёт CPU time и page faults на уровне ОС. В SQL-комментарий вшивается trace_id. Всё это собирается в Python, складывается в asyncio.Queue, и background-таск батчами отправляет в Kafka.

---

## Почему asyncpg, а не psycopg3

Для уровня 4 pipeline mode psycopg3 **невозможен** — EXPLAIN ANALYZE требует последовательного выполнения (каждый запрос меняет состояние данных, следующий может от этого зависеть). Раз pipeline отпадает, asyncpg выигрывает за счёт:

- **Cython + нативный binary protocol** — парсинг JSON-планов быстрее
- **Автоматический кеш prepared statements** — планировщик PostgreSQL не тратит время на повторное планирование одинаковых запросов
- **Меньше overhead на каждый round-trip** — нет прослойки libpq

## Оценка overhead

### Методология оценки

EXPLAIN ANALYZE добавляет инструментирование к каждому **узлу** плана выполнения. Для каждого узла PostgreSQL вызывает `InstrStartNode()` / `InstrStopNode()` — это `gettimeofday()` или `clock_gettime()`. pg_stat_kcache добавляет `getrusage()` на каждый **запрос** (не узел — это важно).

### Overhead на стороне PostgreSQL

#### Фиксированный overhead на один запрос

| Компонент | Время | Память | CPU |
|---|---|---|---|
| EXPLAIN ANALYZE runtime instrumentation | ~0.05-0.1мс | ~2-5KB на узел плана | ~0.05мс |
| EXPLAIN BUFFERS (счётчики блоков) | ~0.01мс | ~200 байт | пренебрежимо |
| EXPLAIN WAL (счётчики WAL) | ~0.01мс | ~100 байт | пренебрежимо |
| JSON-сериализация плана | ~0.05-0.5мс | ~1-50KB (зависит от плана) | ~0.05-0.3мс |
| pg_stat_kcache getrusage() x2 | ~0.002мс | ~0 | пренебрежимо |
| **Итого на один простой запрос** | **~0.1-0.7мс** | **~5-55KB** | **~0.1-0.4мс** |

#### Для сложного запроса (20+ узлов плана, джойны)

| Компонент | Время |
|---|---|
| EXPLAIN ANALYZE instrumentation | ~0.3-1мс |
| JSON-сериализация | ~0.5-2мс |
| **Итого на сложный запрос** | **~1-3мс** |

### Overhead на стороне Python (asyncpg)

#### На один запрос

| Компонент | Время |
|---|---|
| Получение JSON из asyncpg | ~0.01мс (бинарный протокол) |
| `json.loads()` плана | ~0.02-0.2мс (зависит от размера) |
| Извлечение метрик из dict | ~0.005мс |
| Формирование SQL-комментария с trace_id | ~0.001мс |
| **Итого на один запрос** | **~0.04-0.2мс** |

#### Фиксированный overhead на пакет

| Компонент | Время |
|---|---|
| pg_stat_kcache snapshot до (SELECT) | ~0.3мс |
| pg_stat_kcache snapshot после (SELECT) | ~0.3мс |

## Проблема и решение

При 100 команд/сек overhead ~3.6% — **укладываемся в 5%**.

При 500+ команд/сек — **не укладываемся**.

### Решение: адаптивный sampling

| Команд/сек | Стратегия | Эффективный overhead |
|---|---|---|
| < 200 | 100% запросов через EXPLAIN ANALYZE | < 5% |
| 200-500 | 50% запросов через EXPLAIN ANALYZE, остальные — только wall clock + rows affected | < 5% |
| 500-1000 | 20% запросов | < 5% |
| 1000+ | 10% запросов + pg_stat_statements агрегат | < 5% |

Sampling решается тривиально — для каждого пакета кидаем кубик, и если не попал в выборку, выполняем запросы обычным `conn.execute()` без EXPLAIN-обёртки. trace_id всё равно остаётся в SQL-комментарии.

Суть решения (одним абзацем)
Каждый SQL-запрос внутри пакета (Unit of Work) оборачивается в EXPLAIN (ANALYZE, BUFFERS, WAL, FORMAT JSON), что заставляет PostgreSQL выполнить запрос и вернуть детальную статистику по буферам, WAL и времени выполнения. Параллельно расширение pg_stat_kcache собирает системные метрики (CPU time, page faults) через вызов getrusage(). Все данные обогащаются trace_id из SQL-комментария, собираются в Python, асинхронно складируются в очередь и фоновым процессом батчами отправляются в Kafka. Это даёт полную картину потребления ресурсов базы данных для каждого бизнес-пакета.

Почему asyncpg – лучший выбор для Уровня 4
Скорость – прямой бинарный протокол без прослойки libpq даёт выигрыш в производительности до 5 раз по сравнению с psycopg3.

Невозможность pipeline – EXPLAIN ANALYZE требует последовательного выполнения (каждый запрос может изменить состояние данных), поэтому преимущество psycopg3 в pipeline mode здесь не работает.

Кеш prepared statements – asyncpg автоматически кеширует подготовленные запросы, экономя время планировщика.

Лёгкость парсинга JSON – бинарный протокол уменьшает накладные расходы на передачу данных.

Финальная архитектура (ещё раз коротко)
Приложение формирует пакет SQL-запросов, каждый с комментарием /* trace_id=... */.

Исполнитель (на asyncpg):

Открывает транзакцию.

Устанавливает SET LOCAL application_name = trace_id (для связи с логами).

Делает снимок pg_stat_kcache.

Для каждого запроса: если сэмплинг разрешает – выполняет EXPLAIN (ANALYZE, BUFFERS, WAL, FORMAT JSON), иначе – простой execute().

После коммита делает второй снимок pg_stat_kcache и вычисляет разницу.

Формирует объект PacketTelemetry со всеми метриками.

Асинхронная очередь (asyncio.Queue) принимает метрики без блокировки основного потока.

Фоновая задача батчами (каждые 100 мс или 100 записей) отправляет данные в Kafka (топик uow_telemetry, ключ = trace_id).

Kafka выступает надёжным буфером и точкой входа для централизованного сбора.

### Источники CPU-метрик в PostgreSQL

#### Источник А: pg_stat_kcache (прямое измерение через getrusage)

| Параметр | Что именно измеряет | Точность |
|---|---|---|
| `user_time_sec` | Время, которое ядра CPU потратили на код PostgreSQL в user space — парсинг SQL, построение плана, сортировки в памяти, вычисление выражений, обход индексов, сериализация результатов | Высокая, но это суммарно по процессу за период, а не per-query. Мы берём diff — получаем per-UoW |
| `system_time_sec` | Время CPU в kernel space — системные вызовы read()/write()/mmap()/brk(). Высокое значение означает что запросы много взаимодействуют с ОС: дисковый IO, выделение памяти, сетевые буферы | Аналогично |
| `user_time + system_time` | **Полное CPU time** данного пакета. Это прямой аналог того, что показывает `time` в bash или profiler | — |

**Нюанс точности:** getrusage() возвращает CPU time **процесса** (backend). Поскольку один backend обслуживает одно соединение, а наш UoW выполняется в одной транзакции на одном соединении — diff между двумя вызовами getrusage() даёт **именно CPU нашего пакета**. Загрязнения от других сессий **нет** — это per-process метрика.

#### Источник Б: EXPLAIN ANALYZE (косвенное через timing)

EXPLAIN ANALYZE даёт `actual_time` на каждом узле плана. Это **wall clock time**, не CPU time. Но:

- Если `blk_read_time` и `blk_write_time` доступны (track_io_timing=on), то:

```
CPU time ≈ actual_time − blk_read_time − blk_write_time
```

Это **приблизительная** оценка CPU per-query. Неточная потому что не учитывает ожидание локов, сетевые задержки внутри сервера и прочее. Но для большинства запросов — разумное приближение.

#### Источник В: pg_stat_activity (моментальный снимок)

| Параметр | Что даёт |
|---|---|
| `wait_event_type` | Если бэкенд ждёт — на что: CPU, IO, Lock, Network, IPC |
| `wait_event` | Конкретное событие ожидания |

Это **не** накопительная метрика, а снимок текущего момента. Не подходит для нашего случая напрямую, но полезно для диагностики.

### Итого по CPU — что реально получаем

| Метрика | Источник | Гранулярность | Точность |
|---|---|---|---|
| `cpu_user_time_sec` | pg_stat_kcache diff | per-UoW (весь пакет) | ★★★★★ точная |
| `cpu_system_time_sec` | pg_stat_kcache diff | per-UoW (весь пакет) | ★★★★★ точная |
| `cpu_total_time_sec` | user + system | per-UoW | ★★★★★ |
| `estimated_cpu_per_query_ms` | actual_time − io_time | per-query (каждый SQL) | ★★★☆☆ приблизительная |

## 2. Полные данные из плана EXPLAIN ANALYZE

Я перечислил основные метрики. Вот **всё**, что PostgreSQL возвращает в JSON-плане, чего я не упомянул.

### Метрики корневого уровня (top-level)

| Параметр | Описание | Когда полезно |
|---|---|---|
| `Planning Time` | Время работы оптимизатора запросов в миллисекундах. Включает анализ статистики таблиц, перебор вариантов джойнов, выбор индексов | Высокое значение — слишком много джойнов или partitions |
| `Execution Time` | Чистое время выполнения без передачи данных клиенту | Основная метрика |
| `Triggers` | Массив триггеров, которые сработали, с временем каждого | Если есть триггеры — тут видно их стоимость |
| `JIT` | Объект с метриками Just-In-Time компиляции (PG11+) | см. ниже |

### JIT-компиляция (если включена, PG11+)

Когда запрос достаточно сложный, PostgreSQL может компилировать выражения в машинный код через LLVM.

| Параметр | Описание |
|---|---|
| `jit.Functions` | Количество функций, скомпилированных JIT |
| `jit.Options.Inlining` | Было ли применено инлайнирование функций |
| `jit.Options.Optimization` | Был ли применён LLVM-оптимизатор |
| `jit.Options.Expressions` | Были ли скомпилированы выражения (WHERE, вычисляемые столбцы) |
| `jit.Options.Deforming` | Была ли скомпилирована распаковка строк из heap-формата |
| `jit.Timing.Generation` | Время генерации LLVM IR кода | 
| `jit.Timing.Inlining` | Время инлайнирования |
| `jit.Timing.Optimization` | Время оптимизации LLVM |
| `jit.Timing.Emission` | Время генерации машинного кода из IR |
| `jit.Timing.Total` | Суммарное время JIT. Может быть значительным — десятки мс на первый вызов |

### Метрики каждого узла плана (per-node)

Каждый узел (Seq Scan, Index Scan, Hash Join, Sort и т.д.) содержит:

#### Базовые метрики узла

| Параметр | Описание |
|---|---|
| `Node Type` | Тип операции: Seq Scan, Index Scan, Index Only Scan, Bitmap Heap Scan, Bitmap Index Scan, Hash Join, Merge Join, Nested Loop, Sort, HashAggregate, GroupAggregate, Limit, Materialize, CTE Scan, Subquery Scan, Append, Merge Append, Gather, Gather Merge |
| `Parallel Aware` | Узел поддерживает параллельное выполнение |
| `Async Capable` | Узел поддерживает асинхронное выполнение (PG14+, для Append с foreign tables) |
| `Relation Name` | Имя таблицы (для Scan-узлов) |
| `Schema` | Схема таблицы |
| `Alias` | Алиас из SQL-запроса |
| `Index Name` | Имя использованного индекса |
| `Scan Direction` | Forward / Backward (для Index Scan) |
| `Index Cond` | Условие, применённое на уровне индекса (то, что индекс умеет фильтровать) |
| `Filter` | Условие, применённое после чтения строки (то, что индекс **не** умеет фильтровать — строка прочитана, но отброшена) |
| `Join Type` | Inner, Left, Right, Full, Semi, Anti |
| `Join Filter` | Условие фильтрации при джойне |
| `Hash Cond` | Условие хеш-джойна |
| `Merge Cond` | Условие merge-джойна |

#### Оценки планировщика vs реальность

| Параметр | Описание |
|---|---|
| `Plan Rows` | Сколько строк **ожидал** планировщик на этом узле |
| `Plan Width` | Ожидаемая средняя ширина строки в байтах |
| `Actual Rows` | Сколько строк **реально** прошло через узел |
| `Actual Loops` | Сколько раз узел был выполнен (для Nested Loop внутренний узел выполняется многократно) |
| `Startup Cost` | Оценочная стоимость до получения первой строки (в условных единицах планировщика) |
| `Total Cost` | Оценочная полная стоимость |
| `Actual Startup Time` | Реальное время до первой строки в мс |
| `Actual Total Time` | Реальное полное время в мс. **Важно**: это время **одного** loop. Полное время = Actual Total Time × Actual Loops |

**Ключевой диагностический приём**: если `Plan Rows` сильно отличается от `Actual Rows` — статистика таблицы устарела (нужен ANALYZE) или планировщик ошибается. Это причина плохих планов.

#### Метрики буферов (per-node, при BUFFERS)

Каждый узел содержит свои счётчики:

| Параметр | Описание |
|---|---|
| `Shared Hit Blocks` | Блоки из shared_buffers на этом узле |
| `Shared Read Blocks` | Блоки, прочитанные с диска на этом узле |
| `Shared Dirtied Blocks` | Блоки, изменённые на этом узле |
| `Shared Written Blocks` | Блоки, записанные на этом узле |
| `Local Hit Blocks` | Аналогично для временных таблиц |
| `Local Read Blocks` | — |
| `Local Dirtied Blocks` | — |
| `Local Written Blocks` | — |
| `Temp Read Blocks` | Временные файлы для сортировки / хешей |
| `Temp Written Blocks` | — |
| `I/O Read Time` | Время чтения на этом узле (track_io_timing) |
| `I/O Write Time` | Время записи на этом узле |

Это позволяет **точно определить**, какой именно узел плана создаёт нагрузку на диск.

#### Метрики сортировки (для Sort-узлов)

| Параметр | Описание |

| Параметр | Описание |
|---|---|
| `Workers Planned` | Сколько параллельных worker-ов планировщик хотел запустить |
| `Workers Launched` | Сколько реально запустилось (может быть меньше — если max_parallel_workers исчерпан) |

## Что **невозможно** получить

Для полноты картины — чего в PostgreSQL **нет** и не будет:

| Хотелось бы | Почему нет |
|---|---|
| RSS / heap memory per-query | PostgreSQL выделяет память через palloc в контекстах, но не трекает суммарный RSS per-statement. Только Hash Peak Memory для хеш-узлов |
| GPU utilization | PostgreSQL не использует GPU (есть PG-Strom, но это отдельная история) |
| Network bytes per-query | Протокол не считает это на стороне сервера |
| Lock wait time per-query | `wait_event` — снимок, не накопительная метрика. В PG17 обещают `pg_stat_lock_waits` |
| Точный CPU per-query (не per-UoW) | getrusage() слишком дорого вызывать на каждый запрос. pg_stat_kcache даёт per-statement в pg_stat_statements, но при diff по всему backend — только per-UoW |

### 1. Оптимизация SQL-запросов (не просто «медленный запрос»)

LLM увидит не только время выполнения, но и:

*   Какие индексы использовались (Index Scan vs Seq Scan).
*   Сколько блоков прочитано с диска (`shared_read_blocks`) и сколько попало в кэш (`shared_hit_blocks`).
*   Объём временных файлов (`temp_written_blocks`) — сигнал, что `work_mem` мал.
*   План запроса целиком — LLM может найти проблемные узлы (Nested Loop без индекса, неправильный порядок джойнов).

**Пример инсайта:**  
«В аспекте `load_order_items` запрос выполняется за 2 секунды, потому что используется Seq Scan по таблице `order_items`. Причина — отсутствие индекса на поле `order_id`. Рекомендуется создать индекс. Ожидаемое ускорение — 10 раз.»

---

### 2. Оптимизация бизнес-логики (избыточные вызовы)

LLM видит последовательность аспектов и вложенные вызовы. Сопоставляя с логами, можно найти:

*   N+1 проблему — когда в цикле выполняются одинаковые запросы.
*   Дублирующиеся запросы в разных аспектах одного Action.
*   Возможность объединения нескольких запросов в один (batch).

**Пример инсайта:**  
«Action `ProcessOrder` вызывает `ValidateCardAction` дважды: первый раз в аспекте `check`, второй — в аспекте `charge`. Оба раза выполняется один и тот же SQL-запрос. Рекомендуется сохранить результат валидации в state и переиспользовать.»

---

### 3. Проблемы производительности (узкие места и аномалии)

Сравнивая метрики по разным запускам одного Action, LLM может выявить:

*   Выбросы по времени выполнения — например, аспект `save_to_db` иногда выполняется 5 секунд вместо 50 мс. Это указывает на блокировки или конфликты.
*   Корреляцию между параметрами и нагрузкой — например, запросы с `user_id` из определённого сегмента потребляют больше ресурсов.
*   Неэффективное использование ресурсов — большое количество WAL при малом количестве изменённых строк.

**Пример инсайта:**  
«В 5% случаев аспект `update_inventory` выполняется в 10 раз дольше среднего. Анализ плана показывает, что в этих случаях возникает конфликт блокировок на таблице `products`. Рекомендуется изменить уровень изоляции или порядок обновления.»

---

### 4. Проблемы корректности (логические ошибки)

LLM может сопоставить бизнес-логику (код аспекта) с фактическими изменениями в БД:

*   Запрос обновляет не те строки (проверка WHERE).
*   Вставленные данные не соответствуют ожидаемым (типы, форматы).
*   Нарушение ссылочной целостности (из-за отсутствия транзакций).

**Пример инсайта:**  
«Аспект `apply_discount` выполняет UPDATE, устанавливая скидку 10% для всех пользователей, но по бизнес-правилам скидка должна применяться только к премиум-пользователям. В логах видно, что обновляются и обычные пользователи. Рекомендуется исправить условие WHERE.»

---

### 5. Инсайты по использованию ресурсов (кто и сколько потребляет)

Благодаря `trace_id` и контексту (пользователь, роли, A/B тег), можно построить профили нагрузки:

*   Какие роли (`admin`, `user`) генерируют наибольшую нагрузку на диск?
*   Какие версии API создают больше всего WAL?
*   Как A/B тест влияет на потребление памяти (page faults)?

**Пример инсайта:**  
«Пользователи с ролью `premium` выполняют в среднем в 3 раза больше запросов к таблице `analytics`, что приводит к повышенной нагрузке на диск. Рекомендуется кэшировать эти данные или изменить тарифную политику.»

---

### 6. Инсайты по архитектуре (рефакторинг и выделение новых действий)

LLM может предложить:

*   Разбить слишком сложное действие на несколько меньших.
*   Объединить часто используемую группу аспектов в отдельное действие.
*   Выявить циклы в зависимостях (по графу вызовов).

**Пример инсайта:**  
«Action `CreateOrderAction` содержит 8 аспектов и 3 вложенных вызова. Из них блок «проверка платежа» повторяется в двух других местах системы. Рекомендуется вынести его в отдельное действие `ValidatePaymentAction` и использовать повторно.»

---

### 7. Предсказание влияния изменений (симуляция)

Собрав статистику по всем выполнениям, можно построить модель «типичного» выполнения. Затем, изменив код Action (например, добавив новый запрос), LLM может оценить:

*   Как изменится общая нагрузка на БД.
*   Насколько вырастет время ответа.
*   Какие узкие места могут возникнуть.

**Пример инсайта:**  
«Если добавить в аспект `finish` дополнительный запрос к таблице `audit`, то медианное время выполнения Action увеличится на 15%, а нагрузка на диск возрастёт на 20%. Рекомендуется использовать асинхронную запись или фоновый процесс.»

---

### 8. Генерация тестов (регрессионных и нагрузочных)

На основе реальных логов LLM может автоматически генерировать тесты, которые воспроизводят проблемные сценарии или граничные случаи. Это уже обсуждалось в идее `production_test_generation`.

**Пример инсайта:**  
«На основе production-лога с медленным выполнением сгенерирован тест, проверяющий, что аспект `load_user` выполняется не более 100 мс. При нарушении тест упадёт, сигнализируя о регрессии.»

---

### Итоговый инсайт (самый важный)

Ты построил систему, где **каждая бизнес-операция является одновременно исполняемым кодом и машиночитаемой моделью своего поведения в production**. LLM, получая эту модель вместе с детальными логами, может выступать в роли:

*   **DBA-аналитика** — оптимизация SQL и индексов.
*   **Архитектора** — рефакторинг и декомпозиция.
*   **Бизнес-аналитика** — связь нагрузки с пользовательскими сегментами и A/B тестами.
*   **QA-инженера** — генерация тестов и выявление регрессий.
*   **Сайт-релиабилити инженера** — предсказание узких мест до того, как они станут проблемой.

Это и есть **наблюдаемость, встроенная в архитектуру**, а не добавленная сверху. Ты создал платформу, на которой можно не просто мониторить, а **диалогически исследовать поведение системы**, задавая вопросы на естественном языке и получая ответы, подкреплённые реальными данными. Это меняет правила игры.
