idea_16_rate_limiting_actions.md

Ограничение частоты вызовов на уровне бизнес-операций (Rate Limiting for Actions)

Проблема

В традиционных системах rate limiting реализуется на уровне HTTP-эндпоинтов, IP-адресов или API-ключей. Это работает для защиты инфраструктуры, но не для защиты бизнес-логики. Один и тот же эндпоинт может обслуживать разные бизнес-операции, а одна бизнес-операция может вызываться через разные транспорты (HTTP, MCP, CLI, внутренний вызов из другого действия).

Примеры проблем, которые не решает традиционный rate limiting:

Пользователь вызывает CreateOrderAction 100 раз в минуту через API — эндпоинт-лимит на /orders сработает, но если тот же пользователь вызывает то же действие через MCP-инструмент — лимит не применяется.

Внутренний сервис вызывает SendNotificationAction в цикле, создавая спам — нет никакого лимита, потому что вызов не проходит через HTTP.

Действие ChargeCardAction вызывается из пяти разных родительских действий — лимит на каждый эндпоинт считается отдельно, хотя бизнес-операция одна.

В AOA бизнес-операции уже являются атомарными единицами с чёткими границами — Actions [2][5]. Это создаёт естественную точку для ограничения частоты: не на уровне транспорта, а на уровне самой бизнес-операции, независимо от способа её вызова.

Решение

Декоратор @rate_limit на уровне класса Action, который задаёт максимальное количество вызовов за временной интервал. ActionMachine проверяет лимит перед выполнением действия — до проверки ролей, до запуска аспектов, до вызова плагинов. Если лимит превышен, машина выбрасывает RateLimitExceededError или возвращает альтернативный результат.

Ключевое отличие от традиционного подхода: лимит привязан не к транспорту (HTTP-эндпоинт), а к бизнес-операции (Action). Вызов через FastAPI, MCP, CLI или deps.run_action() — всё считается одинаково.

Как это работает

Разработчик помечает Action декоратором @rate_limit:

```python
@rate_limit(
    max_calls=10,
    period=60.0,
    scope="user",
    fallback_result=None
)
@depends(PaymentGateway, description="Платёжный шлюз")
@CheckRoles(["user"], desc="Доступно пользователям")
class ChargeCardAction(BaseAction):

    @aspect("Валидация карты")
    @BoolFieldChecker("valid", desc="Карта валидна")
    async def validate(self, params, state, deps, connections):
        gateway = deps.get(PaymentGateway)
        result = await gateway.validate(params.card_token)
        return {"valid": result}

    @summary_aspect("Списание")
    async def charge(self, params, state, deps, connections):
        gateway = deps.get(PaymentGateway)
        txn = await gateway.charge(params.card_token, params.amount)
        return ChargeCardAction.Result(txn_id=txn, success=True)
```

ActionMachine при вызове run() выполняет следующую последовательность:

1. Проверка rate limit — если лимит превышен, действие не выполняется.
2. Проверка circuit breaker (idea_15) — если предохранитель разомкнут, действие не выполняется.
3. Проверка feature flag (idea_14) — если фича выключена, действие не выполняется.
4. Проверка ролей — @CheckRoles [2].
5. Проверка connections — @connection [2].
6. Событие global_start для плагинов.
7. Выполнение аспектов.

Области действия лимита (scope)

Декоратор @rate_limit принимает параметр scope, который определяет, как группируются вызовы для подсчёта:

scope="global" — общий лимит на всё приложение. Все вызовы этого действия, от всех пользователей, суммируются. Полезно для защиты внешних API с жёстким лимитом.

scope="user" — лимит на одного пользователя. Используется user_id из Context для идентификации. Каждый пользователь имеет свой отдельный счётчик. Полезно для защиты от спама и абьюза.

scope="role" — лимит на роль. Все пользователи с ролью "operator" делят один счётчик. Полезно для ограничения команды операторов.

scope="action" — лимит на класс действия без привязки к пользователю. Эквивалентно "global", но семантически отличается — подчёркивает, что ограничение относится к самой операции.

scope — callable — функция, принимающая Context и возвращающая строку-ключ для группировки. Позволяет создавать произвольные области: по IP, по организации, по подписке.

```python
# Лимит по организации пользователя
@rate_limit(
    max_calls=100,
    period=3600.0,
    scope=lambda ctx: f"org:{ctx.get_extra('org_id')}"
)
class BulkExportAction(BaseAction):
    ...
```

Поведение при превышении лимита

Разработчик выбирает один из двух режимов при объявлении @rate_limit:

Режим исключения (по умолчанию). Если fallback_result не указан, машина выбрасывает RateLimitExceededError с именем действия, текущим значением счётчика и временем до сброса. Транспортный слой (FastAPI, MCP) может перехватить это исключение и вернуть HTTP 429 Too Many Requests.

Режим альтернативного результата. Если указан fallback_result, машина возвращает его как Result действия. Аспекты не выполняются. Плагины получают событие rate_limit_exceeded.

```python
# Режим исключения — клиент получает 429
@rate_limit(max_calls=5, period=60.0, scope="user")
class SendSMSAction(BaseAction):
    ...

# Режим альтернативного результата — тихое ограничение
@rate_limit(
    max_calls=100,
    period=3600.0,
    scope="global",
    fallback_result=ExportAction.Result(status="rate_limited", data=None)
)
class ExportAction(BaseAction):
    ...
```

Хранение состояния счётчиков

Состояние счётчиков (количество вызовов за период) хранится в RateLimitProvider — ресурсном менеджере, который регистрируется на уровне машины.

InMemoryRateLimitProvider — простейшая реализация с хранением в словаре. Подходит для одного инстанса приложения. При перезапуске счётчики сбрасываются.

RedisRateLimitProvider — реализация на основе Redis с атомарными инкрементами и TTL. Подходит для распределённых систем с несколькими инстансами.

SlidingWindowRateLimitProvider — реализация со скользящим окном вместо фиксированных интервалов. Даёт более плавное ограничение.

RateLimitProvider определяется как интерфейс:

```python
class RateLimitProvider(ABC):
    @abstractmethod
    async def check_and_increment(
        self,
        key: str,
        max_calls: int,
        period: float
    ) -> RateLimitResult:
        """
        Проверяет лимит и увеличивает счётчик.

        Возвращает RateLimitResult с полями:
        - allowed: bool — разрешён ли вызов
        - current_count: int — текущее количество вызовов
        - reset_at: float — timestamp сброса счётчика
        """
        pass
```

Машина получает RateLimitProvider при инициализации, а не через @depends на каждом действии — потому что rate limiting является инфраструктурой машины, а не зависимостью бизнес-логики.

События для плагинов

ActionMachine генерирует событие rate_limit_exceeded, которое содержит:

- action_name — полное имя действия
- scope_key — ключ области (например, "user:42")
- current_count — текущее количество вызовов
- max_calls — установленный лимит
- period — период в секундах
- reset_at — время сброса счётчика
- context — контекст выполнения
- fallback_used — True/False

```python
@on('rate_limit_exceeded', '.*', ignore_exceptions=True)
async def on_rate_limited(self, state_plugin, event):
    alert(f"⚠️ {event.action_name} rate limited for {event.scope_key} "
          f"({event.current_count}/{event.max_calls} in {event.period}s)")
    return state_plugin
```

Интеграция с интроспектором

Интроспектор бизнес-процессов (idea_12) может извлекать параметры rate limit из атрибута _rate_limit_spec, добавляемого декоратором @rate_limit [5]. Это позволяет автоматически генерировать:

Карту лимитов — какие Actions имеют ограничения, с какими параметрами, в каких областях.

Отчёт по текущему использованию — какие лимиты близки к исчерпанию.

Предупреждения — действие зависит от внешнего API (@depends), но не имеет @rate_limit, хотя внешний API имеет собственные ограничения.

Тестирование

В ActionTestMachine rate limiting можно контролировать явно:

```python
# Тест нормальной работы — лимит не превышен
machine = ActionTestMachine(
    mocks={PaymentGateway: FakePaymentGateway()},
    rate_limit_states={"ChargeCardAction:user:42": {"count": 0}}
)
result = await machine.run(ChargeCardAction(), params)
assert result.success is True

# Тест превышения лимита
machine = ActionTestMachine(
    mocks={PaymentGateway: FakePaymentGateway()},
    rate_limit_states={"ChargeCardAction:user:42": {"count": 10}}
)
with pytest.raises(RateLimitExceededError):
    await machine.run(ChargeCardAction(), params)
```

Альтернативно, можно полностью отключить rate limiting в тестовой машине — по умолчанию ActionTestMachine игнорирует лимиты, чтобы не усложнять тесты бизнес-логики.

Влияние

Защита бизнес-операций, а не транспорта. Лимит применяется независимо от того, как вызвано действие — через HTTP, MCP, CLI или из другого действия. Один пользователь не может обойти лимит, переключившись на другой канал.

Гранулярность. Можно установить разные лимиты для разных операций: SendSMSAction — 5 в минуту, CreateOrderAction — 100 в час, ExportReportAction — 3 в день. В традиционных системах для этого нужно настраивать каждый эндпоинт отдельно.

Прозрачность. Через плагинную систему [2] видно, какие лимиты срабатывают, для каких пользователей и как часто. Это данные для принятия решений о масштабировании или изменении тарифов.

Защита внешних API. Если ChargeCardAction вызывает PaymentGateway с лимитом 100 запросов в минуту, @rate_limit на действии гарантирует, что лимит не будет превышен — независимо от количества пользователей и транспортов.

Уникальность

Существующие решения для rate limiting (nginx, Kong, FastAPI Limiter, django-ratelimit) работают на уровне транспорта — HTTP-запросов, IP-адресов, API-ключей. Они не знают о бизнес-операциях, потому что в обычных фреймворках бизнес-операции не формализованы.

В AOA rate limiting работает на уровне Action — атомарной бизнес-операции [5]. Это означает:

Лимит применяется ко всем путям вызова одновременно — HTTP, MCP, CLI, deps.run_action().

Интроспектор может построить полную карту лимитов из метаданных [5].

Плагины видят срабатывания лимитов с полным бизнес-контекстом [2].

Декларативная генерация транспорта (idea_04) может автоматически добавлять заголовки X-RateLimit-Limit, X-RateLimit-Remaining и X-RateLimit-Reset в HTTP-ответы на основе метаданных @rate_limit.

Feature Flags (idea_14), Circuit Breaker (idea_15) и Rate Limiting работают вместе как три уровня защиты: ручная (флаги), автоматическая при сбоях (предохранитель), автоматическая при нагрузке (лимит). Все три проверяются в одном месте — перед выполнением действия — через единый механизм.

Связь с другими идеями

Circuit Breaker (idea_15) — защищает от сбоев, rate limiting — от перегрузки. Оба проверяются до выполнения аспектов. Если circuit breaker разомкнут, rate limit не увеличивает счётчик — действие не было выполнено.

Feature Flags (idea_14) — если фича выключена, rate limit не применяется. Три уровня защиты: ручная (флаги), автоматическая при сбоях (предохранитель), автоматическая при нагрузке (лимит).

Перехватчики исключений (idea_01) — @catch обработчики не вызываются при превышении лимита, потому что аспекты не выполняются.

Таймауты действий (idea_07) — таймаут и rate limit — комплементарные механизмы. Таймаут ограничивает время одного вызова, rate limit — количество вызовов. Оба применяются на уровне Action.

Координатор логеров (idea_02) — отдельный шаблон для событий rate limiting с контекстом (scope_key, current_count, max_calls).

Интроспектор бизнес-процессов (idea_12) — автоматическая карта лимитов всех Actions.

Декларативная генерация транспорта (idea_04) — автоматическое добавление заголовков rate limit в HTTP-ответы и информации о лимитах в OpenAPI.

Востребованность

Очень высокая для любых систем с пользователями: SaaS-платформы (лимиты по тарифам), финтех (защита платёжных API), e-commerce (защита от ботов), API-сервисы (монетизация через тарифы). Особенно ценно для систем, доступных через несколько транспортов — где традиционный HTTP-level rate limiting создаёт дыры в защите.
