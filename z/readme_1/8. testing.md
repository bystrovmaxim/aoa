# Testing

## Обзор

ActionEngine был спроектирован по принципу **Unit‑Test‑First Architecture**:  
каждый слой системы изолирован и полностью мокаем.

Архитектура действий, аспектов, фабрик зависимостей и плагинов позволяет:

- тестировать любое действие в изоляции  
- тестировать аспекты по одному  
- мокать зависимости без DI‑контейнеров  
- тестировать вложенные действия  
- проверять вызовы MockAction  
- тестировать цепочки аспектов  
- проверять логику плагинов через фиктивные обработки  
- полностью исключать инфраструктуру из тестов  

Качество кода тестовой машины (`ActionTestMachine`) подтверждено радион‑метриками:  
файл получил рейтинг **A (72.83 MI)** [1].

---

# 1. Основные принципы тестирования в ActionEngine

### 1. Actions — чистые бизнес‑юниты
Все данные приходят через `params`, а инфраструктура через `deps`.  
Поэтому каждое действие можно протестировать как функцию.

### 2. Аспекты тестируются без машины
Вы можете вызывать аспект напрямую:

```python
factory = machine.build_factory(MyAction)
state = action.my_aspect(params, {}, factory)
```

### 3. DependencyFactory можно подменить моками
TestMachine автоматически заменяет зависимости:

```python
machine = ActionTestMachine({EmailService: fake_email})
```

### 4. Плагины можно отключать или подключать вручную
TestMachine не использует плагины по умолчанию → тесты детерминированы.

### 5. Nested Actions тестируются "как есть"
Вложенные вызовы автоматически проходят через мок‑фабрику.

# 2. ActionTestMachine

Основной инструмент тестирования — **ActionTestMachine**.

```python
machine = ActionTestMachine(
    mocks={ChildAction: ChildAction.Result(20)}
)
```

Возможности:

- заменяет зависимости на моки  
- может оборачивать результаты в MockAction  
- может использовать side_effect функции  
- повторяет pipeline настоящей машины  
- управляет вложенными действиями  
- полностью синхронна

# 3. Пример: тестирование одного аспекта

```python
def test_choose_channel_aspect():
    fake_email = object()
    fake_sms = object()

    machine = ActionTestMachine({
        EmailService: fake_email,
        SmsService: fake_sms
    })
    
    factory = machine.build_factory(NotificationAction)
    action = NotificationAction()

    params = NotificationAction.Params(channel='email', message='hi', recipient='a@b.c')

    result_state = action.choose_channel(params, {}, factory)

    assert result_state['service'] is fake_email
    assert result_state['selected_channel'] == 'email'
```

Этот тест полностью изолирован:

- нет плагинов  
- нет контекста  
- нет настоящей инфраструктуры

# 4. Пример: тестирование полного действия

```python
def test_notification_action_with_mock_services():
    fake_email = FakeEmailService()
    fake_sms = FakeSmsService()

    machine = ActionTestMachine({
        EmailService: fake_email,
        SmsService: fake_sms
    })

    action = NotificationAction()

    params = NotificationAction.Params(
        channel='email',
        message='Hello',
        recipient='user@test.com'
    )

    result = machine.run(action, params)

    assert result.success is True
    assert fake_email.sent == [('user@test.com', 'Hello')]
```

Выполняются:

- роли  
- аспекты  
- summary  
- зависимость EmailService  
- DI фабрика  

Но всё полностью мокировано.

# 5. Тестирование вложенных действий

ActionEngine позволяет вызывать действия из других действий:

```python
child_result = deps.run_action(ChildAction, ChildAction.Params(value))
```

TestMachine корректно перехватывает такие вызовы:

```python
machine = ActionTestMachine({ChildAction: ChildAction.Result(20)})
result = machine.run(ParentAction(), ParentAction.Params(5))
assert result.result == 30
```

### Пример из tests.py (реальный):

```python
child_result = ChildAction.Result(20)
machine = ActionTestMachine({ChildAction: child_result})
```

Тест → **A (2)** [1].

---

# 6. Side‑effects для вложенных действий

Если нужно протестировать динамическое поведение:

```python
machine = ActionTestMachine({
    ChildAction: lambda params: ChildAction.Result(params.value * 3)
})
```

TestMachine создаёт MockAction со `side_effect`.

# 7. MockAction

MockAction — это лёгкий мок действия:

```python
mock = MockAction(result=ChildAction.Result(100))

machine = ActionTestMachine({ChildAction: mock})
```

MockAction:

- считает количество вызовов  
- запоминает последние параметры  
- может работать с side_effect  

Метрики MockAction:

- класс — A (3)  
- `run` — A (3)  

Источник: [1].

### Пример проверки:

```python
assert mock.call_count == 2
assert mock.last_params.value == 7
```

# 8. Тестирование чекеров

checker = IntFieldChecker("age", desc="Возраст", min_value=1)
