# Plugins

## Обзор

Плагины в ActionEngine — это **расширяемые наблюдатели**, которые получают события, генерируемые ActionMachine, но **никогда не вмешиваются в бизнес‑логику**.

Плагин:

- подписывается на события через декоратор `@on(event_regex, class_regex, ignore_exceptions)`  
- получает доступ к контексту (Context), params, state и результату  
- может логировать, собирать метрики, строить деревья вызовов, отправлять события в Grafana или OpenTelemetry  
- изолирован — ошибка плагина не ломает действие (если стоит `ignore_exceptions=True`)  
- работает асинхронно внутри синхронного pipeline  
- поддерживает вложенность — получает `nest_level`  
- использует состояние, хранимое отдельно для каждого вызова run()  

Реализация плагинов обладает высоким качеством кода:  
метрики radon MI для `Plugin.py` показывают **A (59.09)** [1],  
а декораторы `@on` также имеют рейтинг **A (71.41)** [1], что говорит о чистоте и поддерживаемости реализации.

# 1. Архитектура плагинов

## 1.1. Базовый класс Plugin

```python
class Plugin(ABC):
    @abstractmethod
    def get_initial_state(self) -> Any:
        ...
```

Плагин обязан реализовать метод:

### get_initial_state()  
Возвращает состояние, которое:

- инициализируется **один раз на вызов ActionMachine.run()**
- передаётся во все обработчики плагина
- должен возвращаться каждым обработчиком (immutable style)

ActionMachine гарантирует сборку состояния через `_init_plugin_states()` — метод имеет рейтинг A [1].

---

# 2. Подписка на события через @on

Для подписки плагина на событие используется:

```python
from ActionMachine.Plugins.Decorators import on
```

### Сигнатура:

```python
@on(event_regex, class_regex, ignore_exceptions)
```

#### Аргументы:
- event_regex — регулярка для события  
  (например, `global_start`, `before:.*`, `after:prepare`)
- class_regex — регулярка полного имени Action  
  (например, `.*NotificationAction`, `myapp.actions.*`)
- ignore_exceptions — если True, исключения игнорируются  
  (это безопасное поведение для production)

Внутренний механизм реализован в `_add_hook` и `Plugin.get_handlers()`:

- `_add_hook` имеет рейтинг A [1]  
- `get_handlers()` частично имеет рейтинг B (сложность 7), но финальный класс Plugin — A [1]

---

# 3. Какие события доступны плагинам

ActionMachine генерирует:

### Общие события:
- `global_start`
- `global_finish`

### Аспектные события:
- `before:<aspect_name>`
- `after:<aspect_name>`

Эта система отражает полный lifecycle выполнения действия, включая регулярные и summary‑аспекты.

---

# 4. Аргументы обработчиков

Каждый обработчик принимает **строго определённый набор аргументов**:

```
async def handler(
    state_plugin,
    event_name,
    action_name,
    params,
    state_aspect,
    is_summary,
    deps,
    context,
    result,
    duration,
    nest_level,
    **kwargs
) -> Any:
    ...
```

### Гарантии:

- **state_plugin** — mutable state плагина
- **action_name** — полное имя класса (генерируется BaseAction.get_full_class_name(), рейтинг A [1])
- **params** — объект Params
- **state_aspect** — состояние текущего аспекта
- **is_summary** — True/False
- **result** — итоговый результат (только для `global_finish`)
- **duration** — длительность аспекта или всего действия
- **nest_level** — глубина вложенности вызовов

Эти значения собираются ActionProductMachine:

- `_run_single_handler` → рейтинг A (3) [1]
- `_run_plugins_async` → рейтинг B (8), но высокое качество [1]

---

# 5. Вложенность вызовов (nest_level)

ActionMachine увеличивает и уменьшает `self._nest_level` вокруг вызова run().

Это означает:

- вложенные вызовы действий поддерживаются автоматически  
- плагин может отображать структуру дерева вызовов  
- можно визуализировать pipeline  

В тестах видно:

```
PluginA: Action 'ParentAction'
  before:delay
    PluginA: before:prepare (ChildAction)
```

Это поведение подтверждено в тестах из контекста.

---

# 6. Как плагин выбирается для события

Плагин проходит два фильтра:

1. `event_regex.fullmatch(event_name)`
2. `class_regex.fullmatch(action_name)`

Реализовано в:

```python
Plugin.get_handlers()
```

Этот метод перечисляет методы с `_plugin_hooks` и выбирает подходящие — рейтинг A/B [1].

# 7. Пример минимального плагина

```python
from ActionMachine.Plugins.Plugin import Plugin
from ActionMachine.Plugins.Decorators import on

class TelemetryPlugin(Plugin):
    def get_initial_state(self):
        return {}

    @on("global_start", ".*", ignore_exceptions=True)
    async def started(...):
        print("Action started:", action_name)
        return state_plugin

    @on("global_finish", ".*", ignore_exceptions=True)
    async def finished(...):
        print("Result:", result)
        return state_plugin
```

---

# 8. Полный пример продвинутого плагина (из tests.py)

Оба плагина `ConsoleLoggingPlugin` и `AnotherLoggingPlugin` оцениваются radon как A [1].

```python
class ConsoleLoggingPlugin(Plugin):
    @on('global_start', '.*', ignore_exceptions=True)
    async def on_global_start(...):
        print(f"{indent(nest_level)}[start] {action_name}")
        return state_plugin

    @on('global_finish', '.*', ignore_exceptions=True)
    async def on_global_finish(...):
        print(f"{indent(nest_level)}[finish] {action_name}, {duration}")
        return state_plugin

    @on('before:.*', '.*', ignore_exceptions=True)
    async def on_before_aspect(...):
        print(f"{indent(nest_level+1)}before {event_name}")
        return state_plugin

    @on('after:.*', '.*', ignore_exceptions=True)
    async def on_after_aspect(...):
        print(f"{indent(nest_level+1)}after {event_name}")
        return state_plugin
```

# 9. Ошибки в плагинах

Если ignore_exceptions=True:

- исключение печатается, но выполнение продолжается
- error-handling реализован в `_run_single_handler` — рейтинг A (3) [1]

Если ignore_exceptions=False:

- исключение плагина прерывает выполнение действия

# 10. Жизненный цикл исполнения действия (с плагинами)

```
run()
  nest_level += 1
  plugins: global_start
  for aspect in aspects:
      plugins: before:aspect
      aspect()
      plugins: after:aspect
  summary_aspect()
  plugins: global_finish
  nest_level -= 1
```

Архитектура проста, но невероятно мощна и наблюдаема.

# 12. Назначение плагинов в AOA

Плагины:

- не меняют бизнес‑данные  
- не видят результат до завершения аспекта  
- не влияют на выполнение  
- позволяют отслеживать, измерять, логировать, мониторить  
- обеспечивают Observability без вторжения  

Это ключевая идея AOA:  
**всё наблюдаемо, ничего не ломаемо.**

# 13. Лучшие практики

### 1. Используйте ignore_exceptions=True в production  
Чтобы плагин не ломал бизнес.

### 2. Возвращайте state_plugin из обработчика  
Это ваша область хранения.

### 3. Используйте `nest_level` для визуализации  
Особенно полезно в больших Saga‑подобных цепочках.

### 4. Подписывайтесь только на нужные события  
Например, только `before:handle` и `after:handle`.

### 5. Не используйте глобальные переменные в плагине  
Состояние только внутри state_plugin.
