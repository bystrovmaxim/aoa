# ActionEngine Documentation

Добро пожаловать в официальную документацию **ActionEngine** — фреймворка нового поколения для построения прозрачной, тестируемой и безопасной бизнес‑логики.

ActionEngine предлагает новую парадигму разработки:  
**AOA — Action‑Oriented Architecture**, где каждая бизнес‑операция представлена как независимое действие (Action), а инфраструктура полностью вынесена в плагины.

---

## Что такое ActionEngine?

ActionEngine — это фреймворк для разработки backend‑приложений, в котором:

- бизнес‑логика организована в виде *действий*  
- каждый шаг выполнения описывается *аспектами*  
- инфраструктура расширяется через *плагины*  
- тестирование встроено в архитектуру  
- вызовы действий могут быть вложенными  
- параметры и результаты строго типизированы  
- транспортный слой можно автоматически генерировать

Этот подход избавляет от:

- смешивания инфраструктуры и логики  
- неявных зависимостей  
- хрупких сервисов  
- громоздкого DI  
- трудных для тестирования контроллеров  
- непредсказуемых middleware  

---

## Основные особенности

### 1. Чистая бизнес‑логика  
Действия ничего не знают о контексте HTTP/MCP/CLI, плагинах, логировании и инфраструктуре.

### 2. Аспекты (aspects)
Каждое действие описывает не только основную логику, но и этапы подготовки:

- `@aspect` — обычные этапы (валидация, подготовка, предвычисления)  
- `@summary_aspect` — итоговый этап, возвращающий Result

### 3. Plugins = инфраструктура "в тени"  
Плагины:

- получают события исполнения  
- могут логировать, собирать метрики, трассировать  
- изолированы от бизнес-логики  
- read‑only  
- могут работать асинхронно  
- не могут сломать действие

### 4. ActionMachine
Машина управления действиями:

- собирает аспекты  
- проверяет роли  
- вызывает плагины  
- ведёт вложенность  
- обеспечивает единообразный pipeline выполнения

### 5. DependencyFactory (DI)
Минималистичный DI, который:

- берёт зависимости из списка `@depends`  
- создаёт и кеширует их  
- позволяет вызывать вложенные действия

### 6. Unit‑test‑first архитектура
Каждый слой изолирован и полностью мокаемый:

- можно тестировать отдельный аспект  
- можно тестировать действие целиком  
- можно мокать зависимости  
- можно заменять вложенные Action на MockAction

### 7. Строгая типизация params/result  
Params и Result — dataclass‑структуры, полностью покрытые типами.

## Быстрый старт

### Установка
```
pip install actionengine
```

### Определим действие
```python
from ActionMachine import BaseAction, BaseParams, BaseResult
from ActionMachine import aspect, summary_aspect

class CreateOrder(BaseAction):
    class Params(BaseParams):
        user_id: str
        product_id: int
        quantity: int

    class Result(BaseResult):
        order_id: str

    @aspect("Проверка товара")
    def check_product(self, params, state, deps):
        ...
        state["price"] = 100
        return state

    @summary_aspect("Создание заказа")
    def handle(self, params, state, deps):
        order_id = create_order_in_db(...)
        return CreateOrder.Result(order_id)
```

### Запуск через машину
```python
from ActionMachine import ActionProductMachine, Context

machine = ActionProductMachine(context=Context())
result = machine.run(CreateOrder(), CreateOrder.Params(...))
```
