# Actions

## Обзор

**Action** — это атомарная бизнес‑операция в ActionEngine.  
Каждое действие представляет собой самостоятельный, детерминированный и тестируемый блок бизнес‑логики.

Action:

- принимает строго типизированные параметры (`Params`)  
- выполняет этапы, описанные через *аспекты*  
- получает инфраструктурные зависимости через `@depends`  
- не видит контекста (Context доступен только плагинам)  
- возвращает строго типизированный `Result`  
- не содержит состояния  

Такой подход создаёт идеальные условия для тестирования и чистой архитектуры.  
Качество реализации базового класса Actions (BaseAction) имеет высокий рейтинг по radon (A) [1].

---

# 1. Структура действия

Типичное действие состоит из трёх элементов:

1. Внутренний dataclass `Params`
2. Внутренний dataclass `Result`
3. Набор аспектов (`@aspect`, `@summary_aspect`)

Пример:

```python
from dataclasses import dataclass
from ActionMachine.Core.BaseAction import BaseAction
from ActionMachine.Core.BaseParams import BaseParams
from ActionMachine.Core.BaseResult import BaseResult
from ActionMachine.Core.AspectMethod import aspect, summary_aspect

class ExampleAction(BaseAction):

    @dataclass(frozen=True)
    class Params(BaseParams):
        value: int

    @dataclass(frozen=True)
    class Result(BaseResult):
        doubled: int

    @aspect("Подготовка данных")
    def prepare(self, params, state, deps):
        state["prepared"] = params.value * 2
        return state

    @summary_aspect("Основной шаг")
    def handle(self, params, state, deps):
        return ExampleAction.Result(doubled=state["prepared"])
```

---

# 2. Params — входные данные

`Params` — неизменяемая структура данных, которая:

- должна наследовать `BaseParams`
- обязана быть dataclass  
- содержит только бизнес‑данные  
- используется как единый вход для всех аспектов  

Пример:

```python
@dataclass(frozen=True)
class Params(BaseParams):
    user_id: str
    product_id: int
```

`BaseParams` имеет оценку A по метрикам качества кода [1].

---

# 3. Result — выходные данные

`Result` — структура результата выполнения действия.

- наследует `BaseResult`
- обязана быть dataclass
- immutable
- строго типизирована

Пример:

```python
@dataclass(frozen=True)
class Result(BaseResult):
    success: bool
```

---

# 4. Аспекты

Аспект — это метод в Action, который выполняет часть обработки.

Есть два типа:

### Обычный аспект (`@aspect`)
Исполняется **до** summary‑аспекта.  
Должен возвращать dict — состояние (state), накапливаемое между аспектами.  

### Summary‑аспект (`@summary_aspect`)
Должен быть **один**.  
Возвращает итоговый объект Result.

В ActionMachine это обрабатывается через `_collect_aspects`, который получает рейтинг A [1].

---

# 5. Параметры аспектов

Каждый аспект принимает:

```
(self, params, state, deps)
```

Где:

- **params**: объект Params  
- **state**: словарь, передаваемый между аспектами  
- **deps**: DependencyFactory  

State:

- создаётся пустым `{}`  
- передаётся между аспектами копированием  
- summary‑аспект получает финальное состояние  

---

# 6. Зависимости — @depends

Action может объявлять зависимости через декоратор:

```python
@depends(EmailService)
@depends(SmsService)
class NotificationAction(BaseAction):
    ...
```

DependencyFactory:

- создаёт экземпляры сервисов  
- кеширует их  
- поддерживает `run_action()` для вложенных действий  

DependencyFactory имеет рейтинг A (71.46) [1].

# 7. Nested Actions — вызов других действий

Любое действие может вызвать другое через DependencyFactory:

```python
child_result = deps.run_action(ChildAction, ChildAction.Params(5))
```

Особенности:

- работает через новую машину run()
- корректно учитывает плагины  
- полностью поддерживает вложенность и отступы  
- тестируется в файле tests.py (все тесты имеют рейтинг A) [1]

# 8. Пример полного действия из tests.py

Этот пример подтверждён тестами и имеет рейтинг A по radon [1].

```python
@depends(EmailService)
@depends(SmsService)
class NotificationAction(BaseAction):

    @dataclass(frozen=True)
    class Params(BaseParams):
        channel: str
        message: str
        recipient: str

    @dataclass(frozen=True)
    class Result(BaseResult):
        success: bool

    @aspect("Выбор канала")
    def choose_channel(self, params, state, deps):
        if params.channel == 'email':
            state['service'] = deps.get(EmailService)
            state['selected_channel'] = 'email'
        elif params.channel == 'sms':
            state['service'] = deps.get(SmsService)
            state['selected_channel'] = 'sms'
        else:
            raise ValueError("Unknown channel")
        return state

    @summary_aspect("Отправка")
    def send(self, params, state, deps):
        service = state['service']
        service.send(params.recipient, params.message)
        return NotificationAction.Result(success=True)
```

# 9. Правила написания Action

### 1. Action не хранит состояние  
Все данные приходят через params/state/deps.

### 2. Action не должен обращаться к контексту  
Контекст доступен только плагинам.

### 3. Аспекты должны быть чистыми  
Минимум side‑effects.

### 4. Summary‑аспект единственный  
И всегда последняя стадия.

### 5. State — только словарь  
И только то, что понадобится следущему аспекту.

### 6. Result должен быть dataclass  
Никаких нестабильных структур.

# 10. Ролевой доступ

Roles определяются декоратором:

```python
@CheckRoles(CheckRoles.ANY, desc="Доступно аутентифицированным пользователям")
class MyAction(BaseAction):
    ...
```

Проверка ролей выполняется в ActionProductMachine, в методах:

- `_check_action_roles`  
- `_check_any_role`  
- `_check_list_role`  
- `_check_single_role`

Все они имеют высокий рейтинг качества A [1].
