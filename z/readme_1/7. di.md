# Dependency Injection (DI)  
## Минималистичный и предсказуемый DI в ActionEngine

ActionEngine использует максимально простой, прозрачный и детерминированный механизм внедрения зависимостей (DI), основанный на:

- декларативном указании зависимостей через `@depends`
- автоматической сборке фабрик зависимостей
- кешировании экземпляров
- полном контроле со стороны ActionMachine
- отсутствии магии, глобальных контейнеров и сервис-локаторов  

Система DI ActionEngine получила высокие оценки качества кода по radon:  
`DependencyFactory` — A (71.46 MI) [1].

---

# 1. Основная идея DI

Каждое действие может объявить зависимости через декоратор:

```python
from ActionMachine.Core.AspectMethod import depends

@depends(EmailService)
@depends(SmsService)
class NotificationAction(BaseAction):
    ...
```

**Важные свойства:**

- зависимости объявляются *на уровне класса*  
- порядок не имеет значения  
- зависимости не передаются через конструктор  
- фабрика создаётся автоматически машиной  
- фабрика создаёт и кеширует объекты

---

# 2. Декоратор @depends

Сигнатура:

```python
@depends(klass, description="", factory=None)
```

Где:

- **klass** — зависимость (класс сервиса, репозитория или Action)
- **description** — текстовое описание (для документации)
- **factory** — пользовательская фабрика (если нужно кастомное создание)

Исходник:

```python
def depends(klass: Type, *, description: str = "", factory: Optional[Callable[[], Any]] = None):
    def decorator(cls):
        deps = list(getattr(cls, '_dependencies', []))
        deps.append({
            'class': klass,
            'description': description,
            'factory': factory
        })
        cls._dependencies = deps
        return cls
    return decorator
```

**Особенности:**

- список зависимостей создаётся заново (наследование не поломает DI)
- описание хранится и может использоваться генераторами документации
- если factory не указана → используется `klass()`

# 3. DependencyFactory — сердце DI

`DependencyFactory` создаётся ActionMachine для каждого класса действия и кешируется:

```python
self._factory_cache[action_class] = DependencyFactory(self, deps_info)
```

Метрики качества: **A (71.46)** [1].

### 3.1. Основные возможности фабрики

Конструктор:

```python
def __init__(self, machine, deps_info):
    self._machine = machine
    self._deps = {info['class']: info for info in deps_info}
    self._instances = {}
```

Фабрика:

- хранит карту `{класс → {описание, фабрика, данные}}`
- кеширует созданные экземпляры
- поднимает ошибки, если зависимость не объявлена

### 3.2. Получение зависимости

```python
instance = deps.get(EmailService)
```

Метод `get`:

```python
def get(self, klass):
    if klass in self._instances:
        return self._instances[klass]
    if klass not in self._deps:
        raise ValueError("Dependency ... not declared in @depends")
    info = self._deps[klass]
    instance = info['factory']() if info['factory'] else klass()
    self._instances[klass] = instance
    return instance
```

Гарантии:

- зависимость всегда создаётся только один раз
- у каждой Action есть собственный DI‑слой
- зависимости никогда не смешиваются между действиями разных типов

# 4. Вызов вложенных действий — run_action

Ключевая особенность:

**Action может вызывать другое Action через ту же фабрику.**

```python
child_result = deps.run_action(ChildAction, ChildAction.Params(5))
```

Реализация:

```python
def run_action(self, action_class, params):
    instance = self.get(action_class)
    return self._machine.run(instance, params)
```

Возможности:

- полная поддержка вложенности
- плагины корректно отражают nested‑структуру
- ChildAction вызывает свои аспекты, плагины и summary
- ParentAction получает результат в типизированном виде

Тесты подтверждают корректность (например, `test_parent_action_with_real_child`) [1].

# 5. Совместимость DI с ActionTestMachine

В тестах DI автоматически заменяется на мок‑объекты:

```python
machine = ActionTestMachine({
    ChildAction: ChildAction.Result(20)
})
```

TestMachine создаёт свою версию DependencyFactory:

```python
class TestDependencyFactory(DependencyFactory):
    def get(self, klass):
        if klass in mock_map:
            return mock_map[klass]
        return super().get(klass)
```

Поэтому:

- можно возвращать динамические значения (side_effect)
- можно подменять только одну зависимость
- можно подменять вложенные Action
- поведение полностью идентично production-машине

Метрики TestMachine: **A (72.83)** [1].

# 6. Примеры DI

## 6.1. Простое действие

```python
@depends(Database)
class LoadUser(BaseAction):

    @summary_aspect("Load")
    def handle(self, params, state, deps):
        db = deps.get(Database)
        return LoadUser.Result(db.load(params.user_id))
```

## 6.2. Кастомная фабрика

```python
@depends(
    EmailService,
    factory=lambda: EmailService(api_key=os.getenv("EMAIL_KEY"))
)
class Notify(BaseAction):
    ...
```

## 6.3. Вложенные действия

```python
@depends(PaymentAction)
class Checkout(BaseAction):

    @summary_aspect("Checkout")
    def checkout(self, params, state, deps):
        payment = deps.run_action(PaymentAction, PaymentAction.Params(...))
        return Checkout.Result(...)
```

# 7. Гарантии DI‑модели

| Гарантия | Что означает |
|---------|--------------|
| Изоляция | Каждый Action получает свой набор зависимостей |
| Безопасность | Нельзя получить зависимость, не объявленную через @depends |
| Предсказуемость | Нет скрытых DI‑контейнеров |
| Кеширование | Одна зависимость — один объект внутри одного run() |
| Тестируемость | Лёгкая подмена зависимостей в ActionTestMachine |
| Прозрачность | Зависящий код виден прямо в классе Action |
| Минимализм | DI без регистрации, модулей, providers и magic wiring |

# 8. Почему этот DI считается образцовым

Потому что он:

- **минималистичен**  
- **не магический**  
- **объявляется там же, где используется** (в Action)  
- **не требует конфигураций, yaml, providers и container graph**  
- **заточен под бизнес‑логику**  
- **использует строгие ошибки при нарушении контракта**  
- **имитирован в тестах абсолютно прозрачно**  

И как видно из метрик radon, механика DI (DependencyFactory и ActionProductMachine._get_factory) относится к наиболее чистым частям фреймворка — все они имеют рейтинг **A** [1].
