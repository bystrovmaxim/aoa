# Core Concepts

## Обзор

ActionEngine основан на новой парадигме **Action‑Oriented Architecture (AOA)**, которая переносит фокус разработчика с инфраструктуры на бизнес‑логику.  
AOA вводит простую модель:

- **Actions** — атомарные бизнес‑операции  
- **Aspects** — декларативные шаги  
- **Params / Result** — строго типизированные данные  
- **ActionMachine** — оркестрация исполнения  
- **Plugins** — инфраструктурные наблюдатели  
- **DependencyFactory** — простой и предсказуемый DI  
- **Context** — read‑only контейнер метаданных запроса  

Все эти компоненты проектировались с целью максимальной чистоты, тестируемости и предсказуемости. Метрики кода подтверждают высокое качество концепций:  
например, ActionProductMachine имеет рейтинг A по читаемости и сложности [1].

---

# 1. Actions

## Что такое действие?

**Action** — это класс, представляющий одну бизнес‑операцию.

- принимает `Params`  
- отдаёт `Result`  
- объявляет аспекты через декораторы  
- получает зависимости через `@depends`  
- не хранит состояние  

Пример:

```python
class CreateOrder(BaseAction):
    @dataclass(frozen=True)
    class Params(BaseParams):
        user_id: str
        product_id: int

    @dataclass(frozen=True)
    class Result(BaseResult):
        order_id: str

    @aspect("Проверка данных")
    def check(self, params, state, deps):
        ...
        return state

    @summary_aspect("Создание")
    def handle(self, params, state, deps):
        return CreateOrder.Result("order-42")
```

### Принципы

- **чистота** — не видит контекст  
- **детерминизм** — всегда возвращает один и тот же результат при равных данных  
- **расширяемость** — аспекты + плагины позволяют изменять поведение без изменения кода действия

# 2. Params и Result

## Params

Параметры действия — неизменяемая структура данных.  
В ActionEngine реализованы как dataclass‑объекты:

```python
@dataclass(frozen=True)
class Params(BaseParams):
    user_id: str
    quantity: int
```

BaseParams имеет рейтинг A (100.00) [1], что подтверждает минимализм и качество реализации.

## Result

Результат действия — тоже dataclass:

```python
@dataclass(frozen=True)
class Result(BaseResult):
    id: str
    success: bool
```

# 3. Aspects

Аспекты — сердце ActionEngine.

Это функции, которые описывают этапы выполнения действия.

## Обычные аспекты (`@aspect`)

Выполняются последовательно, перед summary.

- принимают (self, params, state, deps)  
- возвращают новый state (dict)

## Summary‑аспект (`@summary_aspect`)

Должен быть один.  
Возвращает итоговый Result.

## Пример

```python
@aspect("Проверка прав")
def check(self, params, state, deps):
    ...
    return state

@summary_aspect("Основная логика")
def handle(self, params, state, deps):
    return Result(...)
```

ActionProductMachine корректно собирает аспекты и сортирует их по номеру строки [1].

---

# 4. State (данные аспектов)

Каждый аспект получает `state: Dict[str, Any]`, куда можно положить промежуточные данные:

- чекеры контролируют допустимые поля
- плагины видят state, но не могут изменять params/result
- данные передаются только вперёд

Если аспект вернёт что‑то кроме dict — будет ошибка (типовая гарантия).

# 5. DependencyFactory

## Простое DI без магии

Каждое действие может объявить зависимости:

```python
@depends(EmailService)
@depends(LogService)
class MyAction(BaseAction):
    ...
```

DependencyFactory:

- создаёт экземпляры зависимостей  
- кеширует их  
- умеет вызывать вложенные действия:

```python
child_result = deps.run_action(ChildAction, ChildAction.Params(...))
```

Метрика DependencyFactory — A (71.46) [1].

# 6. Context

## Read‑only контейнер сведений о запросе

Context хранит:

- user_info  
- request_info  
- environment_info  
- дополнительные поля через set_extra  

Имеет рейтинг A (53.79–100.00 для всех составляющих) [1].

Ключевой принцип:

**Действие не видит контекст.  
Плагины видят.**

Это обеспечивает безопасность и детерминизм.

# 7. Plugins

Плагины — наблюдатели, которые получают события от машины.

- подписываются через `@on(event_regex, class_regex)`  
- работают асинхронно  
- могут быть десятки  
- их ошибки изолированы  
- не могут менять данные  
- могут собирать метрики, логировать, строить графы  
- поддерживают вложенность (nest_level)

Пример:

```python
class Telemetry(Plugin):

    @on("before:.*", ".*", ignore_exceptions=True)
    async def before(self, ..., nest_level):
        print(f"{nest_level}> starting aspect")
```

Plugin и декораторы имеют рейтинги A/B [1].

# 9. Checkers

Чекеры — декларативная валидация:

- String  
- Int  
- Float  
- Bool  
- Date  
- InstanceOf  

Все реализованы через BaseFieldChecker.

Например:

```python
@StringFieldChecker("name", desc="Имя", min_length=3)
class Params(BaseParams):
    ...
```

Почти все чекеры имеют рейтинг A (70–98) [1].

---

# 10. Roles

Декоратор CheckRoles контролирует доступ:

```python
@CheckRoles(CheckRoles.ANY, desc="Требуется авторизация")
class MyAction(BaseAction):
    ...
```

ActionProductMachine.run вызывает:

- `_check_action_roles`
- `_check_any_role`
- `_check_single_role`

Все эти методы имеют рейтинг A [1].

---

# 11. Nested Actions

Любое действие может вызвать другое:

```python
result = deps.run_action(ChildAction, ChildAction.Params(5))
```

Плагины при этом корректно отражают вложенность (nest_level).

Тесты демонстрируют это поведение.
