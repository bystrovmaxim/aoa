# ActionMachine — Машина действий

ActionMachine — это ядро фреймворка ActionEngine, управляющее выполнением действий, аспектов, зависимостей, ролевой моделью и плагинами.  
Это полностью синхронная модель выполнения, в которой асинхронность используется только внутри плагинов.

Машина определяет:

- порядок выполнения аспектов  
- проверки ролей  
- применение чекеров  
- вложенные вызовы действий  
- выполнение before/after событий  
- сборку зависимостей  
- обработку ошибок  
- вложенность вызовов (nest level)  

ActionMachine — **мозг AOA (Action-Oriented Architecture)**.

---

# 1. Основная идея

Машина — это **оркестратор**, который:

1. принимает действие  
2. принимает params  
3. проверяет роли  
4. вызывает плагины (global_start)  
5. выполняет регулярные аспекты  
6. вызывает плагины (before/after каждого аспекта)  
7. вызывает summary‑аспект  
8. вызывает плагины (global_finish)  

Машина **не видит бизнес-логики** и **не содержит инфраструктуры** — она лишь управляет потоком выполнения.

# 5. Сборка аспектов

Аспект собираются методом:

```python
_get_aspects()
```

Который:

- кеширует аспекты  
- сортирует их по номеру строки  
- гарантирует наличие summary‑аспекта  
- отбрасывает унаследованные методы  

Метод `_collect_aspects` имеет рейтинг **A (4)** [1].

# 6. DependencyFactory и DI

DI реализован просто и прозрачно:

- действие объявляет зависимости через `@depends`
- машина создаёт фабрику
- фабрика создаёт объекты по требованию
- зависимости кешируются
- вложенные действия вызываются через:

```python
deps.run_action(ChildAction, ChildAction.Params(...))
```

DependencyFactory имеет рейтинг **A (71.46)** [1].

---

# 7. Ролевая модель

ActionProductMachine поддерживает 4 типа ролевых спецификаций:

- `CheckRoles.NONE` — доступ без роли  
- `CheckRoles.ANY` — требуется любая роль  
- `['manager', 'admin']` — хотя бы одна  
- `'admin'` — конкретная роль  

Каждая проверка реализована отдельным методом:

- `_check_none_role`  
- `_check_any_role`  
- `_check_list_role`  
- `_check_single_role`  

Все методы имеют рейтинг **A (1–3)** [1].

---

# 8. Поддержка плагинов

Plugins — это асинхронные наблюдатели, вызываемые машиной.  
Ключевые методы:

- `_run_plugins_sync` — синхронная обёртка  
- `_run_plugins_async` — асинхронный запуск  
- `_init_plugin_states` — состояние плагинов  
- `_run_single_handler` — вызов одного обработчика  

Метрики:

- `_run_plugins_async` — B (8)  
- `_run_single_handler` — A (3)  
- `_run_plugins_sync` — A (3)  

что подтверждает хорошее качество кода [1].

# 9. Вложенность вызовов (nest_level)

Каждый вызов run увеличивает уровень:

```
nest_level += 1
```

Плагины получают уровень вложенности:

```python
nest_level: int
```

Это позволяет:

- визуализировать вложенные вызовы  
- строить иерархию в логах  
- упрощает трассировку

# 12. Важные архитектурные свойства

## 1. Чистота  
Машина не изменяет бизнес-данные.  
Params immutable.  
State всегда создаётся заново.

## 2. Предсказуемость  
Выполнение аспектов строго линейное.  
Никакого динамического вмешательства.

## 3. Безопасность  
Плагины read‑only.  
Ошибки в плагинах управляемы.

## 4. Тестируемость  
Машина поддерживает мокинг зависимостей и действий.

## 5. Расширяемость  
Можно добавлять:

- новые плагины  
- новые типы аспектов  
- кастомные роли  
- транзакционные менеджеры
