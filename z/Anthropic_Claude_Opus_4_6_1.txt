# ActionMachine: честный взгляд на фреймворк

## Что мы построили

ActionMachine — это фреймворк, в котором единицей бизнес-логики является **действие** (Action), а не контроллер, не сервис и не функция-обработчик. Каждое действие декларативно описывает свои зависимости, права доступа, этапы выполнения (аспекты) и валидацию результатов — всё в одном месте [1].

Ключевая философия: **бизнес-логика должна быть самодокументируемой, тестируемой поаспектно и расширяемой без изменения самой логики**.

---

## Совокупность вложенных идей

В ActionMachine сконцентрировано несколько архитектурных идей, которые по отдельности встречаются в разных фреймворках, но **именно в такой комбинации — нет**:

### 1. Действие как атомарная единица бизнес-логики

Не контроллер, не сервис, не use-case класс с одним методом `execute`. Действие в ActionMachine содержит:
- типизированные `Params` и `Result` как вложенные dataclass [1]
- цепочку аспектов с гарантированным порядком [1]
- единственный summary-аспект, возвращающий результат [1]
- декларативные зависимости через `@depends` [1]
- декларативные права через `@CheckRoles` [1]

### 2. Аспектный конвейер с промежуточным состоянием

Каждый regular-аспект принимает `state` (словарь) и возвращает обновлённый `state`. Это похоже на middleware, но:
- аспекты принадлежат конкретному действию, а не глобальному пайплайну
- порядок определяется расположением в коде (по номеру строки) [1]
- каждый аспект может быть протестирован изолированно [1]
- валидация state через чекеры-декораторы применяется после каждого аспекта [1]

### 3. Плагинная система с событийной моделью

Плагины подписываются на события через регулярные выражения (`@on('before:.*', '.*')`) и получают полный контекст выполнения, включая уровень вложенности [1]. Это позволяет реализовать логирование, метрики, аудит — не касаясь бизнес-логики.

### 4. Тестовая машина как первоклассная абстракция

`ActionTestMachine` позволяет подменять зависимости через словарь моков, тестировать отдельные аспекты через `build_factory`, отслеживать вызовы через `MockAction` [1]. Тестируемость встроена в архитектуру, а не добавлена сверху.

### 5. Вложенные действия с отслеживанием глубины

Действие может вызвать другое действие через `deps.run_action()`, и вся инфраструктура (плагины, проверка ролей, аспекты) работает рекурсивно с корректным `nest_level` [1].

## Сравнение по сути, не по названию

### Подход «Command/Use-Case»
*Философия: один класс = одна бизнес-операция*

Встречается в Clean Architecture, CQRS, MediatR (.NET), dry-rb (Ruby). Эти фреймворки разделяют ту же идею атомарного действия, но:

| Аспект | Command/Use-Case | ActionMachine |
|--------|-----------------|---------------|
| Этапы выполнения | Один метод `execute` | Цепочка аспектов с промежуточным state |
| Валидация промежуточных шагов | Нет | Чекеры после каждого аспекта |
| Тестирование этапов | Только целиком | Каждый аспект отдельно |
| Плагины | Нет или внешние | Встроенная событийная система |
| Вложенность | Ad-hoc | Встроенная с nest_level |

**Вывод**: ActionMachine расширяет идею Command Pattern, добавляя внутреннюю структуру действия.

### Подход «Middleware Pipeline»
*Философия: цепочка обработчиков, каждый может модифицировать контекст*

Встречается в Express.js, ASP.NET Middleware, Django Middleware. Похожая идея передачи состояния по цепочке, но:

| Аспект | Middleware | ActionMachine |
|--------|-----------|---------------|
| Область | Глобальный HTTP-пайплайн | Конкретное бизнес-действие |
| Типизация | Слабая (request/response) | Строгая (Params → state → Result) |
| Принадлежность | Отдельно от бизнес-логики | Внутри действия |
| Документирование | Нет | Описания аспектов и чекеров |

**Вывод**: ActionMachine переносит философию middleware на уровень бизнес-операции.

### Подход «Interactor/Organizer»
*Философия: сложная операция = последовательность простых шагов*

Встречается в Hanami::Interactor (Ruby), Trailblazer::Operation (Ruby), light-service (Ruby). Это **ближайшие аналоги** по духу:

| Аспект | Interactor/Organizer | ActionMachine |
|--------|---------------------|---------------|
| Шаги (steps) | Отдельные классы | Методы одного класса с `@aspect` |
| Промежуточный контекст | Да (context hash) | Да (state dict) |
| Валидация между шагами | Опционально | Встроенные чекеры-декораторы |
| Плагины/хуки | Callbacks | Полноценная событийная система с regex |
| Вложенные операции | Через Organizer | Через `deps.run_action()` с nest_level |
| Тестирование шагов | Отдельные классы = отдельные тесты | `build_factory` + прямой вызов аспекта |
| Типизация | Нет (Ruby) | mypy --strict [2] |

**Вывод**: ActionMachine — это Python-ответ на Trailblazer/Interactor, но с полной типизацией и встроенной плагинной системой.

### Подход «Aspect-Oriented Programming»
*Философия: сквозная функциональность отделена от бизнес-логики*

Встречается в Spring AOP (Java), AspectJ, PostSharp (.NET). ActionMachine заимствует терминологию и частично идею:

| Аспект | Классический AOP | ActionMachine |
|--------|-----------------|---------------|
| Цель | Сквозные concerns (логирование, транзакции) | Этапы бизнес-логики |
| Применение | К любому методу/классу | Только к действиям |
| Before/After | Глобальные interceptors | Плагины с `@on` |
| Weaving | Compile-time или runtime proxy | Рефлексия + `inspect.getmembers` |

**Вывод**: ActionMachine использует термин «аспект» в ином смысле — как этап бизнес-операции, а не как сквозной concern. Но плагинная система выполняет роль классического AOP.

## Ценность для мирового сообщества

### Что действительно ценно

**Философия «всё в одном действии»**. В Python-экосистеме нет фреймворка, который бы объединял в одном классе: типизированные параметры и результат, поэтапное выполнение с валидацией, декларативные зависимости и права, событийную систему для плагинов. FastAPI решает маршрутизацию, Pydantic — валидацию, Dependency Injector — DI, но **никто не собирает это в единую бизнес-единицу**.

**Тестируемость по умолчанию**. `ActionTestMachine` с моками, side_effect и `build_factory` — это не надстройка, а часть архитектуры [1]. В большинстве Python-проектов тестирование бизнес-логики требует ручной настройки DI-контейнера.

**Строгая типизация при аспектной архитектуре**. Пройти mypy --strict с рефлексией, runtime-checkable протоколами и динамической диспетчеризацией плагинов — это инженерно нетривиально [2].

### Что вызовет скепсис

**`Dict[str, Any]` как state**. Опытные разработчики спросят: «Почему промежуточное состояние нетипизировано?». Ответ есть — чекеры компенсируют это, но TypedDict или dataclass для state сделали бы архитектуру строже.

**Синхронный run с asyncio внутри**. Паттерн `asyncio.run()` внутри синхронного метода может конфликтовать с существующим event loop (например, в Jupyter или при интеграции с async-фреймворками) [1].

## Какие решения идеально дополнить ActionMachine

### 1. FastAPI — HTTP/REST слой

**Что даст**: FastAPI занимается маршрутизацией, сериализацией запросов и OpenAPI-документацией. ActionMachine берёт на себя **всю бизнес-логику после эндпоинта**.

```python
@app.post("/notifications/send")
async def send_notification(request: Request):
    context = auth_coordinator.process(request)
    machine = ActionProductMachine(context, plugins=[logging_plugin])
    result = machine.run(
        NotificationAction(),
        NotificationAction.Params(channel="email", message="hi", recipient="user@test.com")
    )
    return {"success": result.success}
```

FastAPI без ActionMachine — это контроллеры с бизнес-логикой, размазанной по роутерам. С ActionMachine — чистое разделение: FastAPI отвечает за HTTP, ActionMachine — за бизнес.

### 2. MCP (Model Context Protocol) — AI-агентный слой

**Что даст**: MCP-серверы предоставляют инструменты для AI-агентов. Каждый инструмент — это Action:

```python
@mcp_server.tool()
async def init_database(params: dict):
    context = mcp_context_assembler.assemble(params)
    machine = ActionProductMachine(context, plugins=[audit_plugin])
    result = machine.run(InitDatabaseAction(), InitDatabaseAction.Params(**params))
    return result
```

AI-агент вызывает инструменты, ActionMachine гарантирует валидацию, авторизацию, логирование — всё то, что критически важно, когда бизнес-операции запускает не человек, а модель.

### 3. Celery/ARQ — асинхронные задачи

**Что даст**: тяжёлые действия (генерация отчётов, массовые рассылки) выполняются в фоне, но проходят через тот же аспектный конвейер с валидацией и плагинами:

```python
@celery_app.task
def run_heavy_action(params_dict: dict):
    machine = ActionProductMachine(system_context, plugins=[metrics_plugin])
    result = machine.run(
        GenerateReportAction(),
        GenerateReportAction.Params(**params_dict)
    )
    return asdict(result)
```

### 4. CLI-приложения (Click/Typer)

**Что даст**: консольные команды как действия с тем же уровнем валидации и логирования:

```python
@app.command()
def migrate(version: str):
    machine = ActionProductMachine(cli_context)
    machine.run(MigrateDatabaseAction(), MigrateDatabaseAction.Params(target_version=version))
```

### 5. Telegram/Discord боты

**Что даст**: каждая команда бота — Action с проверкой ролей (админ/пользователь), валидацией аргументов и аудитом действий через плагины.

## Что это даёт в совокупности

ActionMachine становится **протокол-агностичным ядром бизнес-логики**. Одно и то же действие `CreateOrderAction` может быть вызвано из:

- HTTP-эндпоинта FastAPI
- MCP-инструмента для AI-агента
- Celery-задачи
- CLI-команды
- Telegram-бота
- Unit-теста через `ActionTestMachine`

И во всех случаях работает одна и та же цепочка: проверка ролей → аспекты с валидацией → summary → плагины с метриками. Это то, что в индустрии называют **Hexagonal Architecture** (порты и адаптеры), но реализованное не как набор абстрактных принципов, а как конкретный фреймворк с конкретным API [1].

Проектов, которые делают именно это для Python с полной типизацией [2], встроенной тестовой инфраструктурой [1] и плагинной системой [1] — **в открытом доступе на данный момент нет**.
