PostgresUnitOfWork — интеллектуальная буферизация команд

Проблема

В типичных приложениях при сохранении множества связанных записей (например, заказ + позиции заказа) разработчики пишут циклы с отдельными INSERT, что приводит к N+1 запросам к базе данных. Даже если используется executemany, он требует, чтобы все данные были известны заранее. В сложных сценариях, где команды генерируются в разных частях бизнес-логики (разные аспекты или даже разные действия), собрать их в один batch без дополнительной инфраструктуры невозможно [6].

В mainstream-фреймворках (Django ORM, SQLAlchemy) буферизация либо отсутствует, либо работает только на уровне сессии и скрыта от разработчика. Разработчик часто не контролирует, когда именно данные уходят в базу данных [6].

Решение

PostgresUnitOfWork — это ресурсный менеджер, реализующий паттерн Unit of Work с автоматической буферизацией и batch-выполнением. Он предоставляет два метода [6]:

execute(query, params) — добавляет команду во внутренний буфер, но не отправляет её в базу данных немедленно. Команды накапливаются до достижения лимита batch-размера или до явного вызова commit().

select(query, params) — выполняет запрос немедленно, но перед этим автоматически сбрасывает буфер (flush), чтобы гарантировать актуальность данных.

При вызове commit() все накопленные команды выполняются как один составной запрос (или в рамках одной транзакции), что радикально снижает количество сетевых вызовов и повышает производительность [6].

Как это работает

Аспект действия вызывает execute() для каждой строки, которую нужно вставить или обновить. Команда попадает в буфер, но в базу не отправляется. Если следующий аспект вызывает select(), UnitOfWork автоматически сбрасывает буфер (flush) перед выполнением запроса — это гарантирует, что SELECT увидит все предыдущие INSERT/UPDATE. При вызове commit() все оставшиеся команды в буфере отправляются как один batch [6].

Пример использования в аспектах:

```python
@aspect("Создать позиции заказа")
async def create_items(self, params, state, deps, connections):
    uow = connections["connection"]
    for item in params.items:
        await uow.execute(
            "INSERT INTO order_items (order_id, product_id, qty) VALUES ($1, $2, $3)",
            (state["order_id"], item.product_id, item.quantity)
        )
    # Ничего не ушло в БД — всё в буфере
    return state

@aspect("Проверить наличие товаров")
async def check_stock(self, params, state, deps, connections):
    uow = connections["connection"]
    # Автоматический flush перед SELECT — все INSERT из предыдущего аспекта
    # будут отправлены в БД перед выполнением запроса
    rows = await uow.select(
        "SELECT product_id, stock FROM products WHERE product_id = ANY($1)",
        ([item.product_id for item in params.items],)
    )
    state["stock"] = {r["product_id"]: r["stock"] for r in rows}
    return state

@summary_aspect("Завершить заказ")
async def finish(self, params, state, deps, connections):
    uow = connections["connection"]
    await uow.commit()  # Все оставшиеся команды уходят batch-ом
    return CreateOrderAction.Result(order_id=state["order_id"])
```

Политики автоматического сброса

UnitOfWork поддерживает несколько политик автоматического flush [6]:

По размеру буфера — когда накопилось N команд (например, 100), буфер сбрасывается автоматически. Это предотвращает переполнение памяти при большом количестве операций.

Перед SELECT — как описано выше, любой запрос на чтение автоматически вызывает flush. Это гарантирует консистентность данных без ручного управления.

По времени — опциональная политика для высоконагруженных систем, где буфер сбрасывается каждые N миллисекунд.

Интеграция с системой транзакций AOA

PostgresUnitOfWork интегрирован с иерархией ресурсов AOA [2][6]:

Корневое действие создаёт UnitOfWork и получает полный доступ к execute(), select(), commit(), rollback().

При передаче в дочернее действие через deps.run_action() UnitOfWork автоматически оборачивается в прокси через get_wrapper_class() [2]. Прокси разрешает execute() и select(), но запрещает commit() и rollback() — точно так же, как для обычных connection managers.

Дочернее действие добавляет команды в тот же буфер, но не может зафиксировать транзакцию. Все команды — и родительские, и дочерние — будут зафиксированы одним commit() в корневом действии.

Влияние

Производительность — вставка тысяч строк выполняется за один сетевой вызов вместо тысяч. Это особенно ценно для систем с высокими требованиями к пропускной способности [6].

Атомарность — все накопленные команды либо применяются, либо откатываются как единое целое. Это обеспечивается транзакционной семантикой PostgreSQL [6].

Прозрачность — разработчик видит, где команда попадает в буфер (execute), а где выполняется немедленно (select). Автоматический flush перед SELECT предотвращает неконсистентные чтения [6].

Тестируемость — буфер можно проверить на содержимое до отправки, не трогая реальную базу данных. В ActionTestMachine UnitOfWork подменяется моком, который накапливает команды в памяти [6].

Уникальность

В mainstream-фреймворках буферизация скрыта от разработчика (SQLAlchemy Session) или отсутствует (Django ORM). Ни один Python-фреймворк не предлагает явную, контролируемую буферизацию с автоматическим flush перед SELECT и интеграцией с системой прокси-обёрток для безопасной передачи между вложенными действиями [6].

Подход executemany требует собрать все параметры заранее, что невозможно, если команды порождаются динамически в разных частях кода (разные аспекты, разные вложенные действия). PostgresUnitOfWork решает эту проблему — команды накапливаются из любых точек конвейера и группируются автоматически [6].

Что можно добавить

Поддержка вложенных Unit of Work — чтобы дочерние действия могли добавлять команды в общий буфер верхнего уровня (уже частично реализовано через прокси).

Стратегии слияния команд — например, объединение нескольких UPDATE одной таблицы по одному ключу в один запрос.

Интеграция с трассировкой — чтобы через плагины видеть, какие команды были сгруппированы и сколько времени сэкономили.

Метрики буфера — размер буфера, количество flush-операций, среднее количество команд в batch — доступны через плагинную систему для мониторинга.

Связь с другими идеями

PostgresUnitOfWork усиливает иерархию ресурсов с прокси-безопасностью [2]: прокси-обёртка запрещает commit/rollback, но разрешает execute — и команды дочернего действия попадают в общий буфер. В сочетании с перехватчиками исключений (idea_01) обработчик ошибок в корневом действии получает доступ к реальному UnitOfWork и может выполнить rollback, даже если ошибка произошла глубоко во вложенном вызове.
