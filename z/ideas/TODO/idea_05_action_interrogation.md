idea_05_action_interrogation.md

Action Interrogation — допрос действия (мультиассерт)

Проблема

В классическом тестировании комплексный тест с пятью assert-проверками падает на первом неуспешном — и остальные четыре проверки не выполняются. Разработчик видит одну ошибку, чинит её, запускает снова — видит следующую. Цикл «починил → запустил → увидел следующую» повторяется столько раз, сколько сломанных проверок. Это убивает продуктивность [5].

В AOA эта проблема особенно болезненна, потому что setup теста — это описание бизнес-сценария: создание моков, конфигурация DI, выполнение конвейера аспектов через machine.run [2]. Дублировать этот setup в пяти отдельных тестах — значит пять раз описывать одну и ту же реальность.

Решение

Один запуск действия — все проверки независимы. Хелпер multi_assert принимает результат выполнения действия и список проверок (лямбд или callable), выполняет каждую в try/except, собирает все ошибки и в конце выбрасывает агрегированное исключение с полным отчётом. Ни одна проверка не блокирует выполнение остальных [5].

Результат — Action Interrogation: допрос действия. Опиши реальность один раз, а потом задай все вопросы к ней сразу.

Пример

```python
@pytest.mark.asyncio
async def test_process_payment_full():
    fake_gateway = FakePaymentGateway()
    fake_email = FakeEmailService()

    machine = ActionTestMachine({
        UserRepository: FakeUserRepository(),
        PaymentGateway: fake_gateway,
        EmailService: fake_email,
    })

    result = await machine.run(
        ProcessPaymentAction(),
        ProcessPaymentAction.Params(
            user_id=1, card_token="tok_test", amount=100.0
        )
    )

    multi_assert(
        lambda: assert_eq(result.transaction_id, "txn_fake_123",
                          "ID транзакции"),
        lambda: assert_eq(result.amount, 102.0,
                          "Сумма с комиссией 2%"),
        lambda: assert_eq(fake_gateway.charged_amount, 102.0,
                          "Сумма списания с карты"),
        lambda: assert_eq(fake_email.sent_count, 1,
                          "Отправлено одно письмо"),
        lambda: assert_eq(fake_email.last_recipient, "user@example.com",
                          "Получатель письма"),
    )
```

Если проваливаются второй и четвёртый assert, разработчик видит:

```
MultiAssertError: 2 из 5 проверок провалены:

  ❌ [2/5] Сумма с комиссией 2%:
     Ожидалось: 102.0
     Получено:  100.0

  ❌ [4/5] Отправлено одно письмо:
     Ожидалось: 1
     Получено:  0

  ✅ [1/5] ID транзакции
  ✅ [3/5] Сумма списания с карты
  ✅ [5/5] Получатель письма
```

Все пять проверок выполнены. Все ошибки видны сразу. Разработчик чинит оба бага за один цикл.

Как реализуется

Хелпер multi_assert — это примерно 30 строк кода:

```python
class MultiAssertError(AssertionError):
    def __init__(self, failures, total):
        report = f"{len(failures)} из {total} проверок провалены:\n"
        for idx, desc, err in failures:
            report += f"\n  ❌ [{idx}/{total}] {desc}:\n     {err}\n"
        super().__init__(report)

def multi_assert(*checks):
    failures = []
    for i, check in enumerate(checks, 1):
        try:
            check()
        except AssertionError as e:
            desc = getattr(check, '__doc__', f'check {i}')
            failures.append((i, desc, e))
    if failures:
        raise MultiAssertError(failures, len(checks))
```

Альтернативно, можно использовать существующие open-source решения:

pytest-check (pip install pytest-check) — плагин pytest с check.equal(), который собирает все провалы и выводит в конце.

softest — аналогичная библиотека с «мягкими» assert-ами.

Интеграция с pytest-check:

```python
from pytest_check import check

@pytest.mark.asyncio
async def test_process_payment_full():
    # ... setup и run ...

    with check:
        assert result.transaction_id == "txn_fake_123"
    with check:
        assert result.amount == 102.0
    with check:
        assert fake_gateway.charged_amount == 102.0
    with check:
        assert fake_email.sent_count == 1
    with check:
        assert fake_email.last_recipient == "user@example.com"
```

Все проверки выполнятся, все ошибки попадут в отчёт.

Почему это особенно ценно в AOA

За цену одного теста (один setup, один machine.run) разработчик получает 5-10 независимых проверок [2]. Экономия на setup-фазе, которая может быть дорогой: создание моков, конфигурация DI, выполнение конвейера аспектов с чекерами и авторизацией [2].

Цикл «починить → запустить → увидеть следующую» сокращается с N итераций до одной. Все провалы видны сразу.

Покрытие тестами растёт без пропорционального роста количества тестов. Один мультиассерт-тест покрывает столько же, сколько 5 отдельных тестов с одинаковым setup.

Читаемость: тест остаётся единым сценарием — «выполнить платёж → проверить всё» — а не разбитым на пять функций с дублированным setup.

Каждый assert проверяет отдельный бизнес-инвариант — не просто значение переменной, а «комиссия начислена правильно», «письмо отправлено», «получатель верный». Описания assert-ов (desc) автоматически становятся частью документации теста — в духе самодокументируемости AOA [5].

Связь с другими идеями

В сочетании с генерацией тестов из production-логов (idea_08) LLM-агент, генерирующий тесты из дерева выполнения, может автоматически создавать мультиассерт-блоки — по одной проверке на каждое поле Result и на каждый побочный эффект (вызов мока) [5][6].

Обнаружение мёртвых тестов (idea_13) упрощается, когда мультиассерт снижает количество тестовых функций — меньше «шума», проще выявлять холостые тесты [6].

Постпроверки эффектов (idea_10) — мультиассерт естественный формат для постпроверок, где каждый побочный эффект проверяется независимо [6].

Влияние

Очень высокая востребованность — каждый разработчик, который когда-либо писал тест с тремя assert-ами и тратил три цикла на починку, оценит мгновенную выгоду. Это не требует изменения мышления — только замена assert на multi_assert или with check. Порог входа — нулевой.

Уникальность

Идея мягких assert-ов существует в мире тестирования (pytest-check, softest, NUnit Assert.Multiple), но ни один фреймворк не интегрирует её с семантикой бизнес-операций. В AOA это приобретает особую ценность: ActionTestMachine уже обеспечивает единую точку запуска с полным конвейером [2], а каждый assert проверяет отдельный бизнес-инвариант с человекочитаемым описанием [5]. Тест превращается из «проверки одного факта» в допрос бизнес-операции — один запуск, десять вопросов, все ответы сразу.
