# 📘 Математическая модель AOA  
## Формальная спецификация

AOA можно описать как **детерминированную композиционную модель вычислений**, в которой действие (Action) представляет собой вычислительную единицу вида:

```
Action : P → (S × R)
```

где  
- **P** — неизменяемое множество входных данных (*Params*),  
- **S** — промежуточное состояние (*state*) между аспектами,  
- **R** — неизменяемый результат (*Result*).

Действие не имеет побочных эффектов в своей чистой форме: вся внешняя активность вынесена в плагины и DI‑зависимости.

---

# 1. Объекты модели

## 1.1. Входные параметры  
Пусть:

```
Params ⊂ Struct
```

где Struct — множество алгебраических типов, доступных через dataclass.  
`Params` immutable [1], что соответствует коду BaseParams в <source id="2">.

## 1.2. Состояние аспектов  
Состояние — это частичная функция:

```
state : Key ⇀ Value
```

State формируется аспектами и передаётся линейно:

```
state₀ = {}
stateₙ = fₙ(...f₂(f₁(state₀))...)
```

State проверяется чекерами, что формализуется как:

```
∀ checkerᵢ : state ∈ Dom(checkerᵢ) → checkerᵢ(state) = True
```  
Все чекеры (BaseFieldChecker и наследники) имеют рейтинг A [1].

## 1.3. Результат  
```
Result ⊂ Struct
```

immutable (BaseResult — A по MI [1]).

---

# 2. Аспектная модель

Каждое действие A декларирует список аспектов:

```
Aspects(A) = [a₁, a₂, ..., aₙ, summary]
```

Согласно коду <source id="2">:

- все non‑summary аспекты имеют тип `regular`  
- summary единственный (проверка в `_collect_aspects`, рейтинг A [1])  
- порядок определяется `co_firstlineno`

Каждый аспект:

```
aᵢ : (P × S × D) → S
```

где D — множество зависимостей, возвращаемых фабрикой DI.

Summary‑аспект:

```
summary : (P × S × D) → R
```

---

# 3. Модель зависимости (DI)

Пусть:

```
Deps(A) = {d₁, d₂, ..., dₖ}
```

Каждая зависимость определяется через декоратор `@depends` (<source id="2">).  
DependencyFactory гарантирует:

```
get : Class → Instance
run_action : (Action, Params) → Result
```

и кеширование:

```
∀ c ∈ Deps(A), get(c) возвращает один и тот же экземпляр в рамках одного run().
```

DependencyFactory — A (71.46 MI) [1].

---

# 4. Пайплайн выполнения

Формально выполнение действия — это композиция функций:

```
run(A, p) =
    let
        s₀ = {}
        sᵢ = aᵢ(p, sᵢ₋₁, deps)
        r  = summary(p, sₙ, deps)
    in
        r
```

При этом:

- все aᵢ и summary детерминированы относительно входов  
- deps создаёт внешние сервисы, но не меняет p или s  
- вложенные вызовы action реализуют рекурсивный вызов `run()` через `run_action`  

ActionProductMachine реализует эту композицию строго линейно (см. `_execute_regular_aspects` и `run`), всё — рейтинг A/B [1].

---

# 5. Модель ролей

AOA включает встроенную ролевую модель:

```
role_spec(A) ∈ { NONE, ANY, Role, Set(Role) }
```

Машина проверяет:

```
authorize(A, context.user.roles)
```

где логика задана функциями `_check_any_role`, `_check_list_role`, `_check_single_role` (<source id="2">), все — рейтинг A [1].

---

# 6. Модель плагинов

Плагины — это наблюдаемые функции:

```
plugin_handler : (event × action × params × state × result × context × nest) → state_plugin
```

Пространство событий:

```
Events = {
   global_start,
   before:aᵢ,
   after:aᵢ,
   global_finish
}
```

Плагинная система определяет отображение:

```
Plugins × Events × Actions → (PluginState → PluginState)
```

Согласно реализации <source id="2">:

- обработчики асинхронны  
- ошибки подавляются если ignore_exceptions=True  
- состояние плагина — отдельное отображение  
- вложенность (nest_level) — целое число ℕ  

Плагины формально не могут влиять на выполнение:

```
∀ plugin, handler: NO mutation of (params, state, result)
```

Plugin и Decorators — A [1].

---

# 7. Вложенные действия

Механизм:

```
deps.run_action(A₂, p₂)
```

математически — рекурсивный вызов:

```
run(A₁, p₁) calls run(A₂, p₂)
```

Вложенность фиксируется:

```
nest_level ∈ ℕ
nest_level(parent) < nest_level(child)
```

---

# 8. Глобальная формула выполнения

Полная модель:

```
run(A, p) =

    check_roles(A, roles(context)) ;
    emit(global_start, A, p) ;

    s₀ = {} ;
    for i=1..n:
        emit(before:aᵢ) ;
        sᵢ = aᵢ(p, sᵢ₋₁, deps) ;
        emit(after:aᵢ) ;
    
    r = summary(p, sₙ, deps) ;

    emit(global_finish, A, p, r) ;

    return r
```

Где:

- `emit` — вызов обработчиков плагинов  
- проверка типов и полей осуществляется чекерами (`_apply_checkers`)  
- контекст read‑only  
- deps обеспечивает композицию между действиями  

Эта модель полностью соответствует коду ActionProductMachine.run() (<source id="2">), рейтинг A [1].

---

# 9. Ключевые свойства модели

### Детерминизм
При одинаковых `(A, Params, Deps)` результат всегда одинаков.

### Изоляция
`Params`, `Result`, `State` immutable относительно внешнего мира.

### Линейность
Аспекты образуют строго линейный pipeline.

### Безопасность
Плагины следуют:

```
Plugins ⊂ Observers
```

### Композиционность
```
run(A₁, Params₁) может включать run(A₂, Params₂)
```

### Проверяемость
Все части модели тестируются автоматически и получили высокий MI [1].
