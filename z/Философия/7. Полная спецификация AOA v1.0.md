# 📘 AOA v1.0 — Полная спецификация  
*(Action‑Oriented Architecture — формальная спецификация версии 1.0)*

## 1. Терминология

### Action  
Атомарная бизнес‑операция, представленная классом, наследующим BaseAction.  
Action не имеет состояния, а логика разделена на аспекты.

### Params  
Неизменяемые входные данные действия. Наследуют BaseParams.  
Action может принимать только один объект Params.

### Result  
Неизменяемый итог выполнения действия. Наследует BaseResult.

### Aspect  
Метод Action, помеченный декоратором @aspect или @summary_aspect.  
Аспекты определяют этапы выполнения действия.

### DependencyFactory  
Объект, создающий и кеширующий зависимости, объявленные через @depends.  
Используется action для получения сервисов и вызова других actions.

### ActionMachine  
Оркестратор выполнения действий, отвечающий за аспекты, плагины, DI и роли.

### Plugin  
Наблюдатель событий действия. Не влияет на бизнес‑логику.

### Context  
Контейнер метаданных запроса. Action не имеет доступа к контексту.

---

## 2. Цели архитектуры

AOA решает следующие задачи:

- изоляция бизнес‑логики от инфраструктуры  
- предсказуемое выполнение  
- неизменяемые данные  
- отсутствие скрытых зависимостей  
- единая схема DI без контейнеров  
- тестируемость из коробки  
- наблюдаемость без вторжения в бизнес‑код  
- строгая линейная модель исполнения  

---

## 3. Основные аксиомы AOA

### А1. Действие — атом бизнес‑логики  
Action содержит только бизнес‑логику.  
Не допускается хранение состояния.  
Action всегда stateless.

Это гарантируется дизайном BaseAction, который не содержит полей для состояния [1].

### А2. Params и Result неизменяемы  
Params и Result должны быть dataclass(frozen=True) или аналогичной immutable‑структурой.  
BaseParams и BaseResult являются пустыми абстрактными типами [2].

### А3. Аспекты — единственный механизм описания логики  
Все этапы выполнения должны оформляться как аспекты.  
В Action запрещены другие методы, выполняющие бизнес‑логику напрямую.

### А4. Summary‑аспект — единственная точка возврата результата  
Должен быть ровно один summary‑аспект.  
ActionProductMachine выбрасывает ошибку при отсутствии или дублировании [1].

### А5. Линейный pipeline  
Все аспекты выполняются строго в порядке определения в исходном коде.  
ActionProductMachine сортирует аспекты по строкам [1].

---

## 4. Структура Action

Action обязан содержать:

- внутренний Params‑класс  
- внутренний Result‑класс  
- ноль или больше обычных аспектов  
- один summary‑аспект  
- ноль или больше объявлений зависимостей через @depends  

Action не может обращаться к контексту — контекст доступен только плагинам.

---

## 5. Формальная модель аспектов

Пусть:

- P — Params  
- S — состояние аспектов (dict)  
- D — зависимости  
- R — Result

Тогда:

Обычный аспект:

```
aᵢ: (P, S, D) → S
```

Summary‑аспект:

```
summary: (P, S, D) → R
```

Аспект обязан вернуть dict, иначе ошибка [1].

---

## 6. Модель исполнения (ActionMachine)

ActionMachine.run выполняет следующие шаги (см. код метода run [2]):

1. Увеличить уровень вложенности (_nest_level).  
2. Проверить роли действия через _check_action_roles [1].  
3. Передать событие global_start плагинам.  
4. Выполнить регулярные аспекты:  
   - before:aspect  
   - вызов аспекта  
   - after:aspect  
5. Выполнить summary‑аспект.  
6. Передать событие global_finish плагинам.  
7. Уменьшить уровень вложенности.

Все вызовы плагинов выполняются через асинхронные обработчики [2].

---

## 7. Модель состояния аспектов

- начальное состояние: пустой dict  
- каждый аспект обязан вернуть новый dict  
- результат передаётся следующему аспекту  
- summary‑аспект получает всё накопленное состояние  

Если аспект возвращает поля, не прошедшие чекеры — ValidationFieldException [1].

---

## 8. Чекеры (валидация)

Каждый аспект может иметь result‑чекеры, прикреплённые к нему декораторами.  
Каждый Params‑класс может иметь входные чекеры.

BaseFieldChecker реализует строгую модель проверки полей [2].

Чекеры могут проверять:

- тип  
- диапазон  
- формат  
- обязательность  
- структуру  

Все чекеры в проекте имеют рейтинг A [1].

---

## 9. Ролевая модель

Action обязан иметь @CheckRoles.  
Если декоратор отсутствует — ошибка [2].

CheckRoles поддерживает:

- NONE  
- ANY  
- роль  
- список ролей  

ActionMachine проверяет роли перед началом выполнения [1].

---

## 10. Dependency Injection (DI)

DI основан на принципах:

- Action объявляет зависимости через @depends  
- DependencyFactory создаётся для каждого Action‑класса  
- фабрика создаёт и кеширует экземпляры сервисов  
- Action не знает о фабрике и не управляет ею  
- вложенные actions вызываются через deps.run_action  

DependencyFactory.get гарантирует, что зависимости доступны только если они явно объявлены [2].

---

## 11. Вложенные действия

Любое действие может вызвать другое:

```
child_result = deps.run_action(ChildAction, ChildAction.Params(...))
```

Вложенный вызов:

- полностью проходит pipeline  
- вызывает плагины  
- увеличивает уровень вложенности  

ActionMachine.run поддерживает вложенность через _nest_level [2].

---

## 12. Плагины как Observers

Плагины:

- подписываются на события через @on  
- являются наблюдателями  
- не могут модифицировать Params или Result  
- могут иметь своё состояние  
- ошибки могут игнорироваться (ignore_exceptions=True)  
- вызываются асинхронно через _run_plugins_async [1]

Плагины получают:

- event_name  
- action_name  
- params  
- state_aspect  
- is_summary  
- deps  
- context  
- result (только global_finish)  
- duration  
- nest_level  

---

## 13. Context

Context содержит:

- UserInfo  
- RequestInfo  
- EnvironmentInfo  

Action не видит контекст.  
Context передаётся только плагинам.

Context.as_dict собирает полную структуру данных о запросе [2].

---

## 14. Тестируемость (режим TestMachine)

ActionTestMachine:

- подменяет зависимости  
- мокает вложенные действия  
- синхронна  
- поддерживает MockAction  
- использует TestDependencyFactory для DI [2]

ActionTestMachine.run пропускает pipeline, если вызывается MockAction напрямую [2].

---

## 15. Ошибки

AoA определяет следующие категории ошибок:

- ValidationFieldException — нарушение vaildation правил [2]  
- AuthorizationException — роль не соответствует требованиям [2]  
- HandleException — ошибка бизнес‑логики  
- TransactionException — ошибки менеджеров ресурсов (БД, CSV и т.д.)  

---

## 16. Инварианты AOA

Инварианты — гарантируются исполнением ActionMachine.

1. Action stateless.  
2. Params immutable.  
3. Result immutable.  
4. Context не доступен Actions.  
5. Ровно один summary‑аспект.  
6. Аспекты выполняются линейно.  
7. DI создаёт объекты единожды на действие.  
8. Plugins read‑only.  
9. Nested actions полностью изолированы.  
10. Все ошибки либо обрабатываются, либо поднимаются наверх.

---

## 17. Наблюдаемость (Observability)

AOA обеспечивает следующие свойства:

- полная трассировка исполнения через плагины  
- измерение длительности аспектов  
- метрики всех фаз действия  
- визуализация вложенности действий через nest_level  

Эти свойства встроены в ActionProductMachine (см. вызовы before/after и global_* [2]).

---

## 18. Версионность

Каждое Action может быть версионировано именованием класса или модуля.  
Поскольку Actions автономны, их версии не конфликтуют.

---

## 19. Расширяемость

AOA позволяет вводить новые:

- типы чекеров  
- плагины  
- роли  
- виды зависимостей  
- обвязки транспорта (HTTP/MCP/CLI)  
- Connection менеджеры  

Не нарушая базовую архитектуру.

---

# 📌 Итог

AOA v1.0 — это строгая, формализованная архитектура, в которой:

- Action — единица логики  
- Params/Result — неизменяемы  
- Аспекты — структурный pipeline  
- DependencyFactory — прозрачный DI  
- ActionMachine — формальный исполнитель  
- Plugins — безопасные наблюдатели  
- Context — изолирован  
- Тестирование встроено в дизайн  

Все эти свойства подтверждены реализацией ActionEngine, тестами и метриками качества кода [1].
