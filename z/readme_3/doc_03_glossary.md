# Глоссарий терминов AOA

Этот документ содержит все ключевые термины архитектуры AOA и ActionEngine. Он служит справочником, к которому можно возвращаться в процессе изучения фреймворка.

---

## Action

Атомарная бизнес-операция. Stateless. Состоит из аспектов. Принимает Params, возвращает Result. Не знает об HTTP, БД, контексте запроса. Получает зависимости только через DI. Единственная точка входа — summary-аспект. Одновременно является машиночитаемой спецификацией самого себя — список зависимостей, ролевая модель, упорядоченные шаги и контракт каждого шага доступны через атрибуты класса без запуска кода.

---

## Aspect

Шаг выполнения Action. Объявляется декораторами @aspect или @summary_aspect. Принимает params, state, deps. Вызывается строго в порядке определения в классе — сверху вниз.

---

## Summary-aspect

Финальный аспект Action. Обязан быть ровно один. Возвращает Result. Если summary-аспект отсутствует или их больше одного, ActionMachine выбросит ошибку.

---

## Params

Входные данные действия. Неизменяемый dataclass. Содержит только бизнес-данные. Не содержит данных контекста, транспорта или инфраструктуры. Проверяется чекерами.

---

## Result

Выходные данные действия. Неизменяемый dataclass. Возвращается из summary-аспекта. Содержит только итог выполнения операции.

---

## State

Временное состояние конвейера. Обычный Python dict. Передаётся от аспекта к аспекту. Существует только внутри одного вызова run. Не сохраняется между вызовами. Является единственным mutable-объектом в AOA. Не накапливается автоматически — каждый аспект явно возвращает только нужные поля.

---

## DependencyFactory

Фабрика зависимостей. Создаётся ActionMachine автоматически. Создаёт и кеширует зависимости, объявленные через @depends, в рамках одного вызова run. Запускает вложенные действия через run_action. Обеспечивает детерминированность DI.

---

## Resource Manager

Адаптер внешней системы: БД, HTTP-API, файла, очереди. Управляет долгоживущим состоянием — соединениями, сессиями, кешами. Не содержит бизнес-логики. Реализует порт. Передаётся действиям через DI.

---

## Port

Абстрактный интерфейс, описывающий контракт взаимодействия с внешней системой. Action зависит только от порта, а не от конкретной реализации адаптера. Это позволяет подменять адаптеры без изменения бизнес-логики.

---

## Adapter

Конкретная реализация порта. Инкапсулирует работу с внешней системой. Преобразует форматы данных. Ловит инфраструктурные исключения и приводит их к единому виду.

---

## Plugin

Наблюдатель за выполнением Actions. Подписывается на события через @on. Read-only: не может изменять params, state или result. Работает асинхронно. Не влияет на ход выполнения действия. Используется для логирования, метрик, аудита, трассировки и построения семантического дерева выполнения.

---

## Context

Объединяет данные о пользователе, запросе и окружении. Создаётся AuthCoordinator до начала выполнения действия. Передаётся ActionMachine. Не доступен Actions напрямую. Виден только плагинам и машине.

---

## UserInfo

Часть Context. Содержит user_id, список ролей и дополнительные поля. Используется для ролевой модели и плагинов.

---

## RequestInfo

Часть Context. Содержит trace_id, timestamp, path, method, full_url, client_ip, protocol, user_agent и дополнительные поля. Формируется ContextAssembler.

---

## EnvironmentInfo

Часть Context. Содержит hostname, service_name, service_version, environment, container_id и дополнительные поля. Обычно создаётся один раз при старте приложения.

---

## AuthCoordinator

Оркестратор аутентификации. Последовательно использует CredentialExtractor, Authenticator, ContextAssembler для формирования полного Context. Если аутентификация не прошла, возвращает None.

---

## CredentialExtractor

Компонент аутентификации. Извлекает сырые учётные данные из объекта запроса. Протокол-агностичен: работает с HTTP, MCP, CLI.

---

## Authenticator

Компонент аутентификации. Принимает учётные данные от CredentialExtractor. Проверяет их. Создаёт UserInfo. Возвращает None если аутентификация не прошла.

---

## ContextAssembler

Компонент аутентификации. Извлекает метаданные запроса — trace_id, IP, метод, путь, user_agent. Возвращает словарь для создания RequestInfo.

---

## CheckRoles

Декоратор авторизации. Обязателен для каждого Action. ActionMachine проверяет роли до запуска аспектов. Нарушение приводит к AuthorizationException. Принимает значения NONE, ANY, список конкретных ролей. Ролевая политика хранится в атрибуте _role_spec и доступна программно без запуска кода.

---

## ActionMachine

Исполнитель действий. Единственная точка запуска любой бизнес-операции через machine.run(action, params). Проверяет роли, запускает аспекты, вызывает плагины, применяет чекеры, управляет вложенностью, измеряет время выполнения. Одновременно является хранилищем метамодели каждого действия — кеш аспектов, зависимости, ролевые спецификации доступны через getattr в любой момент.

---

## ActionProductMachine

Production-реализация ActionMachine. Полный конвейер с плагинами, ролями, чекерами, вложенностью.

---

## ActionTestMachine

Тестовая реализация ActionMachine. Позволяет подменять зависимости через словарь моков. Автоматически оборачивает BaseResult и функции в MockAction. Полностью повторяет логику Production-машины.

---

## MockAction

Тестовый Action. Запоминает параметры последнего вызова. Считает количество вызовов. Возвращает фиксированный результат или вычисляет его через side_effect.

---

## @depends

Декоратор объявления зависимости Action. Принимает класс зависимости и опциональную фабрику через параметр factory. DependencyFactory создаёт объект по требованию и кеширует его. Нельзя использовать зависимость не объявленную через @depends. Список зависимостей с описаниями хранится в атрибуте _dependencies класса.

---

## @aspect

Декоратор обычного аспекта. Аспект принимает params, state, deps. Обязан вернуть dict.

---

## @summary_aspect

Декоратор финального аспекта. Обязан вернуть Result. Должен быть ровно один в каждом Action.

---

## @connection

Декоратор объявления connections-зависимости Action. Указывает имя ключа в словаре connections и тип интерфейса. ActionMachine проверяет соответствие объявлений и переданных connections до запуска аспектов. Объявленные соединения с описаниями хранятся в атрибуте _connections класса.

---

## connections

Словарь ресурсных менеджеров, явно передаваемых в дочернее действие через deps.run_action(). При передаче каждый менеджер автоматически оборачивается в прокси, запрещающий управление транзакцией.

---

## WrapperConnectionManager

Прокси-обёртка ресурсного менеджера. Создаётся автоматически при передаче соединения в дочернее действие. Блокирует open, commit, rollback. Разрешает execute. Гарантирует, что только действие-владелец управляет транзакцией. При глубокой вложенности оборачивается снова — запрет сохраняется на любом уровне.

---

## TypedDict

Опциональный способ типизации state. Описывает пространство возможных ключей state для конкретного Action. Используется с total=False, так как каждый аспект работает с подмножеством полей. Улучшает подсказки IDE и ловит опечатки через mypy.

---

## Checker

Декоратор или объект, проверяющий поля state после выполнения аспекта. Проверяет наличие, тип, допустимые значения. При нарушении выбрасывает ValidationFieldException. Контракт каждого аспекта хранится в атрибуте _result_checkers и доступен программно.

---

## nest_level

Уровень вложенности вызовов run. Увеличивается при каждом вложенном вызове через deps.run_action(). Передаётся плагинам и используется для построения иерархических логов и семантического дерева выполнения.

---

## run_action

Метод DependencyFactory для вызова вложенных действий. Создаёт экземпляр Action, запускает полный конвейер, корректно обрабатывает вложенность и плагины. Основной механизм композиции бизнес-операций в AOA.

---

## CoreHelper

Вспомогательный модуль для работы с asyncio. Содержит run_in_thread — метод для безопасного выполнения синхронного кода в отдельном потоке без блокировки event loop.

---

## HandleException

Стандартное инфраструктурное исключение ActionEngine. Используется Resource Manager-ами при ошибках внешних систем.

---

## ValidationFieldException

Исключение, выбрасываемое при нарушении контракта аспекта — лишние поля в state, неверный тип, нарушение чекера.

---

## AuthorizationException

Исключение, выбрасываемое при несоответствии ролей пользователя требованиям Action.

---

## TransactionProhibitedError

Исключение, выбрасываемое прокси-обёрткой при попытке дочернего действия вызвать open, commit или rollback на переданном соединении.

---

## Strangler-path

Паттерн постепенной миграции легаси. Старый код изолируется за интерфейсом, создаётся Action-обёртка, логика последовательно переносится в аспекты, легаси-код удаляется по мере переноса.

---

## Anti-corruption layer

Слой, защищающий новый код от прямой зависимости от легаси. В AOA реализуется через порт и адаптер: новый код работает с интерфейсом, адаптер делегирует вызовы легаси-классу.

---

## Метамодель AOA

Машиночитаемое представление каждого Action, доступное через атрибуты класса без запуска кода. Включает список зависимостей через _dependencies, ролевую политику через _role_spec, маркеры аспектов через _is_aspect, контракты аспектов через _result_checkers и объявленные соединения через _connections. Метаданные не расходятся с кодом потому что без них машина не запустит действие. Является основой для статического анализа бизнес-архитектуры, автогенерации документации и матрицы доступа.

---

## Семантическое дерево выполнения

Структурированный JSON-граф одного вызова machine.run. Содержит params, state на каждом шаге конвейера, result и duration каждого аспекта с полной вложенностью вызовов. Строится плагином на 50 строк без изменения бизнес-логики. Может передаваться языковой модели для объяснения что именно произошло в конкретном вызове.
