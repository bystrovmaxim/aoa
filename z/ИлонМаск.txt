## 🟢 Что объективно хорошо

### Строгая типизация — на уровне

```
mypy --strict: Success: no issues found in 37 source files
```

Ноль ошибок на strict-режиме для 37 файлов — это серьёзно [2]. Большинство Python-проектов даже не пытаются запустить mypy на strict.

### Cyclomatic complexity — чистая

Самое сложное во всём проекте — **B(8)** в двух методах: `_run_plugins_async` и `_execute_regular_aspects` [2]. Всё остальное — A. Для оркестрационного ядра это отличный показатель. Код реально декомпозирован.

### Аспектный конвейер — рабочая идея

Вот что реально происходит: действие объявляет методы через `@aspect` и `@summary_aspect`, машина собирает их по порядку определения в коде (через `co_firstlineno`), валидирует результаты через чекеры-декораторы, и вызывает в правильной последовательности [1]:

```python
@aspect("Выбор канала")
@InstanceOfChecker("service", (EmailService, SmsService), desc="...")
@StringFieldChecker("selected_channel", desc="...", required=False)
def choose_channel(self, params, state, deps):
    ...
    return state

@summary_aspect("Отправка")
def send(self, params, state, deps):
    ...
    return Result(success=True)
```

Это **реально удобно**. Чекеры одновременно служат валидацией и документацией. Аспекты читаются сверху вниз как бизнес-процесс. Summary-аспект обязателен (ровно один) — машина это проверяет [1].

### Система чекеров — продуманная

`StringFieldChecker`, `IntFieldChecker`, `FloatFieldChecker`, `BoolFieldChecker`, `DateFieldChecker`, `InstanceOfChecker` — все наследуют `BaseFieldChecker` с единым интерфейсом `check(params)` [1]. Каждый работает как декоратор: на классе — валидация входных параметров, на методе — валидация результата [1]. При этом машина проверяет, что аспект не вернул **лишних** полей сверх объявленных чекерами [1]:

```python
extra_fields = set(new_state.keys()) - allowed_fields
if extra_fields:
    raise ValidationFieldException(...)
```

Это хорошая защита от «забыл объявить, но нафигачил данных».

### Вложенность действий — работает

`DependencyFactory.run_action()` вызывает `machine.run()` рекурсивно, `nest_level` инкрементится, плагины получают правильный уровень вложенности [1]. Тесты это подтверждают:

```python
def test_parent_action_with_real_child():
    result = machine.run(action, ParentAction.Params(5))
    assert result.result == 20  # 5*2=10 + 10
```

ParentAction вызывает ChildAction через `deps.run_action()`, и всё корректно проходит через конвейер [1].

### Тестовая машина — удобная

`ActionTestMachine` принимает моки разных типов и автоматически оборачивает их [1]:

| Тип мока | Что происходит |
|----------|---------------|
| `MockAction` | Используется как есть |
| `BaseAction` | Проходит через аспектный конвейер |
| `BaseResult` | Оборачивается в `MockAction` |
| `callable` | Становится `side_effect` |
| Любой объект | Возвращается через `get()` |

Это реально снижает boilerplate в тестах.

### Плагины с regex-матчингом — гибко

Декоратор `@on` с двумя regex (событие + класс действия) — мощная идея [1]:

```python
@on('before:.*', '.*', ignore_exceptions=True)
async def on_before_aspect(self, ...):
    ...

@on('global_finish', '.*OrderAction', ignore_exceptions=False)
async def on_order_finish(self, ...):
    ...
```

Можно подписаться на конкретное событие конкретного действия, или на все события всех действий, или на любую комбинацию.

## 🔴 Что реально сломано

### 1. Async/sync конфликт — это **blocker**

```python
def _run_plugins_sync(self, ...):
    try:
        loop = asyncio.get_running_loop()
    except RuntimeError:
        asyncio.run(self._run_plugins_async(...))
    else:
        loop.run_until_complete(self._run_plugins_async(...))
```

`loop.run_until_complete()` внутри уже работающего event loop — это `RuntimeError` в любом async-фреймворке [1]. FastAPI, aiohttp, Sanic — везде уже есть работающий loop. Этот код **упадёт** при первом вызове из async-контекста.

Варианты починки:
- Сделать `run()` полностью `async`
- Сделать плагины синхронными
- Использовать `nest_asyncio` (костыль)
- Запускать плагины в отдельном потоке через `run_in_executor`

### 2. Мутация state — бомба замедленного действия

```python
def choose_channel(self, params, state, deps):
    state['service'] = deps.get(EmailService)  # мутирует входной dict
    state['selected_channel'] = 'email'
    return state  # и возвращает тот же объект
```

В `_execute_regular_aspects` следующий аспект получает `new_state`, но это **тот же объект**, что и `state` [1]. Если аспект мутирует `state` и потом выбрасывает исключение до `return` — предыдущий `state` уже испорчен. Нужен `state = {**state}` в начале каждого аспекта, или на уровне машины.

### 3. Кэш плагинов не инвалидируется

```python
self._plugin_cache: Dict[Tuple[str, str], List[...]] = {}
```

Кэш `(event_name, action_name) → handlers` создаётся один раз и живёт вечно [1]. Если плагины добавляются в runtime (что возможно — `_plugins` это обычный список), кэш протухает молча. Нужен либо `cache_clear()` при мутации `_plugins`, либо immutable-коллекция.

### 4. `get_handlers` в Plugin — O(n) по всем методам на каждый вызов

```python
def get_handlers(self, event_name, class_name):
    for method_name in dir(self):
        method = getattr(self, method_name)
        if not callable(method) or not hasattr(method, '_plugin_hooks'):
            continue
        for event_regex, class_regex, ignore_exceptions in method._plugin_hooks:
            if event_regex.fullmatch(event_name) and class_regex.fullmatch(class_name):
                ...
```

`dir(self)` перебирает **все** атрибуты экземпляра, включая унаследованные [1]. Для каждого — `getattr`, `callable`, `hasattr`, потом regex-матчинг. Кэш в машине это частично компенсирует, но первый вызов для каждой пары (event, action) будет медленным.

### 5. `CheckRoles._role_spec` наследуется

```python
class CheckRoles:
    def __call__(self, cls):
        cls._role_spec = self.spec
        return cls
```

`_role_spec` ставится как атрибут класса [1]. При наследовании `class AdminAction(UserAction):` — если `AdminAction` не имеет своего `@CheckRoles`, он **унаследует** `_role_spec` от `UserAction`. Это может быть неожиданным. Проверка `getattr(action.__class__, '_role_spec', None)` не различает собственный и унаследованный атрибут [1].

### 6. `_collect_aspects` фильтрует унаследованные методы

```python
if method.__qualname__.split('.')[0] != action_class.__name__:
    continue
```

Это **сознательное решение** (есть комментарий) [1], но оно означает, что наследование действий не работает для аспектов. Если `SpecialOrder(CreateOrder)` хочет переиспользовать аспекты родителя — не получится. Это нужно либо чётко документировать, либо сделать opt-in.

## 🟡 Что можно улучшить

### Архитектура зависимостей

`DependencyFactory.get()` создаёт зависимости через `klass()` (конструктор без аргументов) или через фабрику [1]. Но нет lifetime management — каждый `get()` создаёт новый экземпляр (если не закэширован в `_instances`). При этом `_instances` живёт в рамках одного `DependencyFactory`, который создаётся на один `run()`. Это значит:

- Singleton-зависимости пересоздаются на каждый запуск
- Нет scoped/transient различия
- Нет cleanup/dispose

Для MVP это нормально, но при росте — придётся переделывать.

### Context.set_extra — мутабельный

```python
def set_extra(self, key, value):
    self._extra[key] = value
    setattr(self, key, value)
```

Контекст заявлен как «read-only паспорт запроса», но `set_extra` позволяет мутировать его из любого места [1]. Плагины получают контекст и **технически могут** его менять. Нужен `@property` или `__setattr__` override, если read-only — это требование.

### Нет обработки ошибок в аспектах на уровне машины

Если аспект выбрасывает исключение — `_execute_regular_aspects` просто пробрасывает его наверх [1]. Нет `on_error` хука, нет rollback предыдущих аспектов, нет события для плагинов. `global_finish` не вызывается при исключении (нет `finally` в `run` для этого события — `finally` только для `_nest_level -= 1`) [1].

## 🏁 Итого

**Хорошее ядро с одним критическим багом и несколькими архитектурными рисками.**

Аспектный конвейер с чекерами-декораторами, вложенность действий через DI, regex-плагины — это **рабочие и полезные идеи**. Код чистый, типизированный, с низкой сложностью.

Но async/sync конфликт — это не теоретическая проблема, а реальный crash. Починить его, сделать state immutable между аспектами, добавить error-события для плагинов — и будет солидный фреймворк для бизнес-логики.

**Ship the fix, then ship the framework.** 🚀
