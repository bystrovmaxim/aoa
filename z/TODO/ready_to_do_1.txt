## 1. Явные транзакции без магии

То, что ты описываешь, — это **эволюция Dependency Injection в сторону ресурсов**. В mainstream-фреймворках (Spring, Django ORM) транзакции часто либо автоматические (и тогда непонятно, где начинаются и заканчиваются), либо требуют магических декораторов (`@transactional`), которые скрывают реальный поток данных.

Твой подход:
- **Явная передача connection** в параметрах аспектов — делает транзакцию частью бизнес-логики, а не скрытой магией.
- **Возможность группировать несколько Action в одну транзакцию** простым пробросом одного и того же connection.
- **Нет неявного поведения** — разработчик всегда видит, где начинается и заканчивается транзакция.

**Уникальность:**  
Большинство фреймворков либо навязывают транзакции на уровне HTTP-запроса (одна транзакция на запрос), либо требуют сложных паттернов (Unit of Work). Ты даёшь разработчику **полный контроль**, но при этом не заставляешь писать тонны boilerplate-кода. Это особенно ценно для систем с распределёнными транзакциями или сложными сценариями согласованности данных.

**Что можно добавить:**  
- Контекстный менеджер для безопасного управления транзакциями внутри Action.  
- Автоматический rollback при исключении, но с возможностью переопределить поведение.  
- Поддержка вложенных транзакций с savepoints (если БД позволяет).  
- Интеграция с паттерном Saga для микросервисных сценариев.

## 5. Иерархия ресурсов и безопасная передача транзакций между действиями

Пятая идея — это **архитектура ресурсных менеджеров, которая позволяет явно, без магии, управлять транзакциями и при этом гарантировать, 
что вложенные действия не смогут случайно закрыть или закоммитить транзакцию**. Она основана на двух простых принципах:

1. **Явная передача ресурса через `deps.run_action(action, params, resources=...)`.**  
   Разработчик сам решает, какой экземпляр ресурса передать дочернему действию. Никакого автоматического подхватывания «текущей транзакции».

2. **Автоматическое создание прокси-обёртки для переданного ресурса.**  
   Когда ресурс передаётся в `run_action`, машина вызывает специальный метод `get_wrapper()`, определённый в иерархии классов ресурсов. 
   Этот метод возвращает класс-обёртку, который:
   - Блокирует вызовы «опасных» методов (begin, commit, rollback) на вложенных уровнях.
   - Пропускает все остальные методы (execute, select и т.д.) к реальному ресурсу.

**Структура иерархии ресурсов:**

```
BaseResourceManager (1-й уровень)
    ↑
BaseDatabaseConnectionManager (2-й уровень)
    ↑
PostgresConnectionManager (3-й уровень, конкретная реализация)
```

- **BaseResourceManager** — маркерный класс, позволяющий определить любой менеджер ресурса через `isinstance`. 
Содержит абстрактный метод `get_wrapper()`, который должен вернуть класс-обёртку для данного ресурса.
- **BaseDatabaseResourceManager** — наследует `BaseResourceManager` и добавляет специфичные для БД методы: `open()`, `close()`, `begin()`, `commit()`, `rollback()`, `execute()`. 
В нём же реализована логика прокси его конструктор получает менеджер соединений у которого тип BaseDatabaseResourceManager, его опысные методы  то методы _doCommit _doOpenConnection _doRollback и выбрасывают исключение, а `execute` делегируется внутреннему объекту который он получил в конструкторе.
- **PostgresConnectionManager** — конкретная реализация. Она реализует `get_wrapper()`, возвращая `BaseDatabaseConnectionManager` (или специализированный класс-обёртку). Также она транслирует методы все методы _doCommit _doOpenConnection _doRollback execute в вызовы к реальному соединению.

**Гибкая передача параметров при создании ресурса через `deps.get()`**

Чтобы передавать параметры (например, строку подключения) в момент вызова `deps.get()`, нужно, чтобы фабрика, указанная в `@depends`, умела их принимать. Стандартная фабрика (конструктор класса) не принимает дополнительные аргументы, поэтому для такой гибкости необходимо определить фабрику с поддержкой `**kwargs`.

Пример:

```python
@depends(PostgresConnectionManager,
         factory=lambda **kwargs: PostgresConnectionManager(kwargs.get('dsn')))
class RootAction(BaseAction):
    @aspect("Старт")
    def start(self, params, state, deps):
        # создаём экземпляр с конкретным dsn
        db = deps.get(PostgresConnectionManager, dsn="postgresql://user:pass@localhost/db")
        db.open()
        db.begin()
        ...
```

Здесь лямбда принимает любые именованные аргументы и передаёт `dsn` в конструктор. Если `dsn` не передан, в конструктор уйдёт `None` (конструктор должен это обрабатывать). Можно сделать проверку:

```python
factory=lambda **kwargs: PostgresConnectionManager(kwargs['dsn']) if 'dsn' in kwargs else PostgresConnectionManager()
```

Но тогда при отсутствии `dsn` будет вызван конструктор без аргументов. Это работает, если у класса есть конструктор по умолчанию.

**Важно:** Если в `@depends` не указана `factory`, то `deps.get()` не сможет принять дополнительные аргументы, так как будет вызван конструктор класса без параметров. Попытка передать аргументы приведёт к ошибке. Поэтому для ресурсов, которым нужны параметры при создании, обязательно указывать `factory`.

**Явная передача ресурса дочернему действию:**

```python
# Корневое действие
@depends(PostgresConnectionManager,
         factory=lambda **kwargs: PostgresConnectionManager(kwargs.get('dsn')))
class RootAction(BaseAction):
    @aspect("Старт")
    def start(self, params, state, deps):
        db = deps.get(PostgresConnectionManager, dsn="postgresql://...")
        db.open()
        db.begin()

        child_result = deps.run_action(
            ChildAction(),
            params,
            resources={PostgresConnectionManager: db}   # явная передача
        )

        db.commit()
        return Result()

# Дочернее действие
@depends(PostgresConnectionManager)   # фабрика не нужна, получит прокси
class ChildAction(BaseAction):
    @aspect("Шаг")
    def step(self, params, state, deps):
        db = deps.get(PostgresConnectionManager)   # получит прокси
        db.execute("INSERT ...")   # работает
        # db.commit()   # вызовет исключение (запрещено на вложенном уровне)
        return state
```

Что происходит при вызове `run_action` с параметром `resources`:
1. Машина проверяет, что переданный объект является экземпляром `BaseResourceManager`.
2. Вызывает у него метод `get_wrapper()`, получает класс-обёртку.
3. Создаёт экземпляр этого класса, передавая в конструктор оригинальный ресурс.
4. Помещает созданную обёртку в фабрику зависимостей для дочернего действия, переопределяя стандартное создание.

Таким образом, дочернее действие получает именно прокси, который знает, что оно вложенное, и блокирует управляющие методы.

**Уникальность:**  
- Нет неявного «текущего контекста транзакции». Передача ресурса явна, что делает код прозрачным и легко тестируемым.
- Блокировка опасных методов на вложенных уровнях реализована через простую иерархию классов и прокси, без использования сложных паттернов типа ThreadLocal или contextvars.
- Разработчик может создавать свои ресурсы, просто унаследовавшись от `BaseResourceManager` и реализовав `get_wrapper()` — механизм автоматически обеспечит безопасность вложенных вызовов.
- Идеально сочетается с предыдущим пунктом (перехватчики исключений): обработчик ошибок в корневом действии получит доступ к реальному ресурсу и сможет выполнить rollback и close, даже если ошибка произошла глубоко внутри.

**Что это даёт разработчикам:**
- **Полный контроль над транзакциями** — явное открытие, коммит, роллбек.
- **Безопасность вложенных вызовов** — никакой случайный commit внутри дочернего действия не сломает транзакцию.
- **Прозрачность** — всегда видно, какой ресурс куда передаётся.
- **Гибкость** — можно передавать разные экземпляры одного и того же ресурса в разные ветки (например, для параллельной работы с разными БД).

**Что можно добавить (опционально):**
- Поддержка нескольких ресурсов в одном вызове `run_action` (словарь ресурсов).
- Автоматическая передача ресурсов, если они объявлены в `@depends` и есть только один кандидат — но это уже необязательный «синтаксический сахар», который может быть включён по желанию.
- Интеграция с координатором логеров — логирование всех вызовов методов ресурса (особенно полезно для отладки транзакций).
