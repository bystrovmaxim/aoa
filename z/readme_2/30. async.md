# Асинхронность в ActionEngine

ActionEngine построен на основе asyncio — современной модели асинхронного программирования в Python. Все аспекты действий объявляются как `async def`, и ActionMachine также полностью асинхронна. Это позволяет эффективно обрабатывать большое количество конкурентных запросов без лишних потоков.

Однако работа с asyncio требует понимания нескольких ключевых моментов: как правильно запускать машину, как вызывать другие действия, как работать с ресурсами и как не заблокировать event loop при выполнении тяжёлых операций. В этой главе мы разберём все эти вопросы.

---

## 1. Запуск ActionMachine

ActionMachine (класс `ActionProductMachine`) предоставляет асинхронный метод `run`:

```python
async def run(self, action, params, resources=None) -> Result:
    ...
```

### 1.1 Асинхронный запуск (из корутины)

Если вы находитесь внутри асинхронного контекста (например, в FastAPI-эндпоинте), достаточно просто `await`:

```python
from ActionMachine.Core import ActionProductMachine
from ActionMachine.Context import Context

machine = ActionProductMachine(context=Context())
params = MyAction.Params(...)
result = await machine.run(MyAction(), params)
```

### 1.2 Синхронный запуск (из обычной функции)

Если ваше приложение синхронное (например, консольный скрипт или тесты без asyncio), вы можете запустить машину с помощью `asyncio.run()`:

```python
import asyncio

def sync_run():
    params = MyAction.Params(...)
    result = asyncio.run(machine.run(MyAction(), params))
    return result
```

При этом `asyncio.run()` создаёт новый event loop, выполняет корутину и закрывает loop. Это удобно для простых скриптов, но для высоконагруженных сервисов предпочтительнее использовать асинхронный подход.

## 2. Вызов других действий (композиция)

Внутри аспекта вы можете вызвать другое действие через `deps.run_action()`. Этот метод **асинхронный** и должен использоваться с `await`:

```python
@aspect("Вызов дочернего действия")
async def call_child(self, params, state, deps):
    child_result = await deps.run_action(
        ChildAction,
        ChildAction.Params(value=params.num)
    )
    state["child_value"] = child_result.value
    return state
```

Метод `run_action`:
- создаёт экземпляр указанного действия (через DI),
- запускает его через ту же ActionMachine,
- возвращает результат.

Это позволяет строить сложные бизнес-процессы из маленьких, переиспользуемых действий.

## 3. Работа с ресурсами

Ресурсные менеджеры (адаптеры внешних систем) предоставляют свои методы для работы. Получить ресурс можно через `deps.get(класс_интерфейса)`. Сам вызов `get` — синхронный, но методы ресурса могут быть как синхронными, так и асинхронными.

### 3.1 Синхронный ресурс

Пример синхронного ресурса — `CsvConnectionManager`:

```python
@aspect("Запись в CSV")
async def write_csv(self, params, state, deps):
    csv_mgr = deps.get(CsvConnectionManager)   # ← синхронно
    csv_mgr.open()                              # синхронный метод
    csv_mgr.write_rows(headers, rows)           # синхронный метод
    csv_mgr.commit()                             # синхронный метод
    return state
```

### 3.2 Асинхронный ресурс

Если ресурс использует асинхронный драйвер (например, `asyncpg` для PostgreSQL или `httpx.AsyncClient`), его методы будут корутинами, и их нужно вызывать с `await`:

```python
@aspect("Запрос к API")
async def fetch_data(self, params, state, deps):
    http_client = deps.get(HttpClient)           # ← синхронно
    response = await http_client.get("/api/data") # асинхронный вызов
    state["data"] = response.json()
    return state
```

Таким образом, работа с ресурсами в асинхронных аспектах естественна: если ресурс асинхронный — используем `await`, если синхронный — вызываем как обычно.

## 4. Тяжёлые вычисления и блокировки event loop

Основная проблема асинхронного кода — случайная блокировка event loop. Если внутри корутины выполняется длительная операция (CPU-intensive вызов или синхронный I/O), весь цикл событий останавливается, и другие запросы перестают обрабатываться.

В ActionEngine эта проблема решается с помощью специального модуля-хелпера `CoreHelper`, который предоставляет утилиты для безопасного выполнения синхронного кода в потоках.

### 4.1 Использование правильных асинхронных аналогов

Вместо `time.sleep` используйте `await asyncio.sleep`.  
Вместо синхронных HTTP-библиотек (например, `requests`) используйте асинхронные (`httpx.AsyncClient`, `aiohttp`).

Пример **неправильно**:

```python
@aspect("Ой, блокировка")
async def bad_example(self, params, state, deps):
    time.sleep(5)          # блокирует event loop!
    return state
```

Правильно:

```python
@aspect("Хорошо")
async def good_example(self, params, state, deps):
    await asyncio.sleep(5)   # не блокирует
    return state
```

### 4.2 Вынос тяжёлых вычислений в поток

Если вам нужно выполнить CPU-интенсивную операцию (например, обработка изображения, сложный расчёт) или вызвать синхронную библиотеку, которую нельзя заменить асинхронной, используйте функцию `run_in_thread` из модуля `ActionMachine.Core.CoreHelper`.

#### Модуль CoreHelper

```python
# ActionMachine/Core/CoreHelper.py
import asyncio
from typing import TypeVar, Callable

T = TypeVar('T')

class CoreHelper:
    @staticmethod
    async def run_in_thread(func: Callable[..., T], *args) -> T:
        """Запускает синхронную функцию в отдельном потоке."""
        loop = asyncio.get_running_loop()
        return await loop.run_in_executor(None, func, *args)
```

Использование в аспекте:

```python
from ActionMachine.Core.CoreHelper import CoreHelper

@aspect("Тяжёлый расчёт")
async def heavy_computation(self, params, state, deps):
    def calculate(data):
        # долгая CPU-работа
        return sum(x * x for x in range(data))

    result = await CoreHelper.run_in_thread(calculate, params.big_number)
    state["result"] = result
    return state
```

Этот подход позволяет не блокировать event loop, даже если синхронная функция выполняется секунды.

### 4.3 Библиотеки, которые ломают asyncio (asyncio.run внутри)

Некоторые синхронные библиотеки (например, старые версии некоторых клиентов) могут внутри себя вызывать `asyncio.run()` или `loop.run_until_complete()`. Если такая библиотека будет вызвана внутри уже работающего event loop, возникнет ошибка `RuntimeError: This event loop is already running`.  
Решение — также вынести вызов такой библиотеки в поток через `CoreHelper.run_in_thread`:

```python
from ActionMachine.Core.CoreHelper import CoreHelper

@aspect("Вызов кривой библиотеки")
async def call_legacy_lib(self, params, state, deps):
    def legacy_call():
        return legacy_module.do_something(params.data)   # внутри может быть asyncio.run

    result = await CoreHelper.run_in_thread(legacy_call)
    state["legacy"] = result
    return state
```

В отдельном потоке нет запущенного event loop, поэтому `asyncio.run()` отработает корректно.

## 5. Три проблемы и их обход — кратко

| Проблема                                   | Пример                                      | Решение                                    |
|--------------------------------------------|---------------------------------------------|--------------------------------------------|
| Блокирующий вызов (time.sleep)             | `time.sleep(5)`                             | `await asyncio.sleep(5)`                   |
| Тяжёлые CPU-вычисления                      | сложный цикл, обработка данных              | `await CoreHelper.run_in_thread(heavy_func, ...)` |
| Синхронная библиотека с asyncio.run внутри | вызов старого клиента                        | `await CoreHelper.run_in_thread(legacy_call, ...)` |

Эти три паттерна покрывают практически все случаи, когда асинхронный код может «зависнуть» или упасть. Благодаря простому хелперу `CoreHelper.run_in_thread` вы всегда можете безопасно выполнить синхронный код, не жертвуя производительностью.

---

## 6. Почему хелпер вынесен в отдельный модуль

Все вспомогательные утилиты собраны в классе `CoreHelper` в модуле `ActionMachine.Core.CoreHelper`. Это позволяет:

- не перегружать `DependencyFactory` дополнительными методами;
- легко расширять набор утилит (например, добавить `run_in_process` для особо тяжёлых задач);
- импортировать только нужные функции там, где они необходимы;
- тестировать утилиты независимо от основного DI.

---

## 7. Заключение

ActionEngine с самого начала спроектирован для асинхронного выполнения, но при этом предоставляет гибкие механизмы для интеграции с синхронным кодом и тяжёлыми вычислениями.

- Все аспекты — `async def`, машина — `async`, поэтому разработчик пишет единообразный код.
- Вызов дочерних действий через `await deps.run_action(...)` интуитивно понятен.
- Ресурсы могут быть как синхронными, так и асинхронными; их методы вызываются обычным образом (с `await` при необходимости).
- Для тяжёлых операций используйте `CoreHelper.run_in_thread` — он надёжно изолирует синхронный код от event loop.

Понимание этих принципов позволит вам создавать эффективные, отзывчивые и масштабируемые приложения на ActionEngine.
