# Формальная спецификация AOA v1.0

Этот документ формализует архитектурные правила, модели и гарантии Action-Oriented Architecture. Спецификация определяет как должны быть устроены действия, аспекты, ресурсы, плагины, DI и машина действий, чтобы система оставалась детерминированной, тестируемой и чистой [1].

---

## Назначение AOA

AOA — это архитектурная модель в которой любая бизнес-операция оформляется как действие, состоящее из линейной последовательности аспектов, выполняемой под контролем ActionMachine.

Архитектура гарантирует линейность исполнения, предсказуемость, тестируемость, изоляцию инфраструктуры, отсутствие скрытого состояния, неизменяемость данных и безопасность конвейера [1].

---

## Аксиомы AOA

Первая аксиома. Действие stateless. Action не содержит состояния в атрибутах экземпляра. Всё состояние конвейера живёт только в объекте state внутри одного вызова run.

Вторая аксиома. Конвейер детерминирован. Аспекты выполняются строго в порядке определения в файле. Порядок определяется по co_firstlineno и не зависит ни от каких внешних факторов [1].

Третья аксиома. Данные неизменяемы. Params и Result — frozen dataclass. Python запрещает их изменение после создания.

Четвёртая аксиома. Context невидим Actions. ActionMachine передаёт Context только в механизм проверки ролей и в плагины. Actions физически не имеют к нему доступа [1].

Пятая аксиома. Плагины read-only. Плагины получают события и могут читать данные. Механизм их вызова не предоставляет возможности изменить params, state или result [1].

Шестая аксиома. DI декларативен. Зависимость существует только если объявлена через @depends. Попытка получить необъявленную зависимость вызывает ошибку.

Седьмая аксиома. Summary-аспект ровно один. ActionMachine выбрасывает ошибку если summary-аспект отсутствует или их больше одного [1].

Восьмая аксиома. Единственная точка входа. Любая бизнес-операция запускается только через machine.run(action, params).

---

## Аксиома живой документации

Девятая аксиома. Каждый Action является полностью машиночитаемой спецификацией самого себя.

Это аксиома которой нет ни в одном другом Python-фреймворке.

Список зависимостей с описаниями доступен через атрибут _dependencies. Ролевая модель доступна через _role_spec. Упорядоченный список шагов — через _is_aspect и co_firstlineno. Контракт каждого шага — через _result_checkers. Объявленные соединения — через _connections.

Эти данные не расходятся с кодом и не устаревают потому что без них код не запустится. Машина физически не выполнит Action у которого нет CheckRoles, нет summary-аспекта или нарушен контракт чекера.

Это означает что метаданные верифицированы самим фактом работающей системы. Из этого вытекают возможности недоступные в других архитектурах: матрица доступа Action × Role собираемая из кода автоматически, граф зависимостей между операциями, статический анализ архитектурных проблем, diff бизнес-логики между версиями [1].

## Формальная модель исполнения

Выполнение действия описывается следующей формулой:

```
state₀ = {}
state₁ = a₁(params, state₀, deps)
state₂ = a₂(params, state₁, deps)
...
stateₙ = aₙ(params, stateₙ₋₁, deps)
result = summary(params, stateₙ, deps)
```

Каждый аспект принимает текущее состояние, обрабатывает его и возвращает новое. ActionMachine полностью заменяет state на результат каждого аспекта. Накопление не происходит автоматически [1].

---

## Спецификация Action

Action — атом бизнес-логики.

Требования:

Не имеет состояния внутри экземпляра. Оперирует только params, state и deps. Определяет аспекты выполнения. Имеет ровно один summary-аспект. Объявляет зависимости через @depends. Имеет обязательный декоратор CheckRoles. Не имеет доступа к Context. Не вызывает инфраструктуру напрямую.

Action описывает что должно быть сделано, но не как именно. И одновременно является машиночитаемой спецификацией самого себя — всё что известно о нём до первого запуска доступно через атрибуты класса [1].

---

## Спецификация Params

Params — неизменяемый dataclass представляющий вход бизнес-логики.

Требования:

Типизирован. Объявлен с frozen=True. Описывает только бизнес-данные. Не содержит данных контекста, транспорта или инфраструктуры. Валидация выполняется чекерами.

---

## Спецификация Result

Result — неизменяемый dataclass возвращаемый summary-аспектом.

Требования:

Объявлен с frozen=True. Содержит только итог выполнения операции. Не содержит инфраструктурных или временных данных. Является публичным контрактом действия.

---

## Спецификация Aspect

Аспект — этап выполнения бизнес-операции.

Regular-аспект:

Помечается декоратором @aspect. Принимает params, state, deps. Обязан вернуть dict. Вызывается строго в порядке определения.

Summary-аспект:

Помечается декоратором @summary_aspect. Обязан вернуть экземпляр Result. Обязан быть ровно один в каждом Action.

Оба типа:

Не должны изменять params. Не должны обращаться к глобальному состоянию. Должны быть чистыми по отношению к контексту.

---

## Спецификация State

State — временное состояние конвейера.

Правила:

Является обычным Python dict. Передаётся от аспекта к аспекту. Существует только внутри одного вызова run. Не сохраняется между вызовами. Не накапливается автоматически — каждый аспект явно возвращает только нужные поля. Является единственным mutable-объектом в AOA.

---

## Спецификация Resource Manager

Ресурс — адаптер внешней системы с долгоживущим состоянием.

Правила:

Может хранить долгоживущее состояние — соединения, кэши, сессии. Не содержит бизнес-логики. Реализует интерфейс-порт. Используется действиями только через DI. Создаётся DependencyFactory. Безопасно заменяется в тестах [1].

Главный критерий: если объект управляет соединением или долгоживущим состоянием — он ресурс. Если принимает решения — он действие.

---

## Спецификация Plugin

Плагин — механизм наблюдения.

Правила:

Подписывается на события через @on. Работает асинхронно. Может читать params, state, result, context. Не может изменять бизнес-данные. Не может отменить выполнение действия. Ошибки подавляются при ignore_exceptions=True. Хранит собственное изолированное состояние которое не смешивается между вызовами [1].

Плагины являются механизмом построения семантического дерева выполнения. Каждый вызов machine.run может быть записан как структурированный граф с state на каждом узле, duration каждого аспекта и полной вложенностью вызовов. Такое дерево можно передать языковой модели, загрузить в систему мониторинга или использовать для автоматической генерации тестов [1].

---

## Спецификация DependencyFactory

DependencyFactory управляет зависимостями в рамках одного вызова run.

Правила:

Создаётся ActionMachine автоматически. Создаёт зависимости объявленные через @depends. Кеширует зависимости в рамках одного вызова. Если указана factory — вызывает её. Если factory не указана — вызывает конструктор klass(). Запрещает использование необъявленных зависимостей. Поддерживает вложенные действия через run_action [1].

---

## Спецификация ActionMachine

ActionMachine — единственная точка исполнения любого действия.

Отвечает за:

Проверку ролей через CheckRoles до запуска первого аспекта. Событие global_start. Выполнение regular-аспектов строго по порядку. Событие before до каждого аспекта. Применение чекеров после каждого аспекта. Событие after после каждого аспекта. Выполнение summary-аспекта. Событие global_finish. Создание DependencyFactory. Управление nest_level при вложенных вызовах. Измерение времени выполнения [1].

ActionMachine не содержит бизнес-логики — только правила конвейера. И одновременно является хранилищем метамодели каждого действия. Кеш аспектов, зависимости, ролевые спецификации — всё это живёт в атрибутах класса и доступно через getattr в любой момент, не только во время выполнения [1].

---

## Инварианты AOA

Следующие условия должны выполняться всегда:

Первый инвариант. params immutable — после создания Params не изменяется никогда.

Второй инвариант. result immutable — после создания Result не изменяется никогда.

Третий инвариант. state exists only inside run — state не существует вне одного вызова run.

Четвёртый инвариант. context not visible to actions — Actions не имеют доступа к Context.

Пятый инвариант. exactly one summary-aspect — в каждом Action ровно один summary-аспект.

Шестой инвариант. linear pipeline — аспекты выполняются строго линейно без ветвлений на уровне машины.

Седьмой инвариант. DI deterministic — одинаковый набор @depends при одинаковых factories дают одинаковые зависимости.

Восьмой инвариант. plugins read-only — плагины не изменяют бизнес-данные.

Девятый инвариант. no global side-effects in Actions — Actions не обращаются к глобальному состоянию.

Десятый инвариант. nested actions safe — вложенные вызовы полностью изолированы и не разрушают состояние родителя.

Одиннадцатый инвариант. metadata is verified — метаданные каждого Action верифицированы самим фактом работающей системы и не расходятся с кодом.

---

## Правила ролей

Для каждого Action обязателен декоратор CheckRoles. ActionMachine проверяет роли до запуска аспектов. Нарушение вызывает AuthorizationException.

CheckRoles.NONE — аутентификация не требуется. CheckRoles.ANY — пользователь должен быть аутентифицирован. Список ролей — пользователь должен иметь хотя бы одну из указанных ролей.

---

## Правила тестируемости

AOA гарантирует следующие свойства тестируемости:

Каждый аспект можно тестировать отдельно как обычный метод. ActionTestMachine подменяет любые зависимости через словарь моков. MockAction подменяет вложенные действия и запоминает параметры вызовов. Поведение тестовой машины полностью совпадает с production-машиной. Тесты детерминированы и не зависят от окружения [1].

---

## Спецификация транзакций

Управление транзакциями в AOA явное и контролируемое.

Правило первое. Действие-владелец само открывает соединение, управляет транзакцией и закрывает её. Никакого автоматического begin.

Правило второе. При передаче соединения в дочернее действие через connections оно автоматически оборачивается в прокси-обёртку. Прокси разрешает execute и запрещает open, commit, rollback [1].

Правило третье. ActionMachine проверяет соответствие объявленных через @connection зависимостей и переданных connections до запуска аспектов.

---

## Спецификация метамодели

Метамодель AOA — это машиночитаемое представление каждого Action доступное через атрибуты класса.

Атрибут _dependencies содержит список зависимостей с описаниями. Атрибут _role_spec содержит ролевую политику. Атрибут _is_aspect маркирует методы-аспекты. Атрибут _result_checkers содержит контракт каждого аспекта. Атрибут _connections содержит объявленные соединения с описаниями.

Из этих данных возможно построить: матрицу доступа Action × Role, граф зависимостей между операциями, статический анализ архитектурных проблем, автогенерацию OpenAPI-схем, автогенерацию MCP-инструментов, diff бизнес-логики между версиями кодовой базы.

Это возможно потому что разработчик обязан формализовать бизнес-намерения через обязательные декораторы — и машина верифицирует их при каждом запуске [1].

## Спецификация структуры проекта

Рекомендуемая структура:

```
actions/
    create_order.py
    calculate_discount.py

resource_managers/
    domain/
        interface.py
        models.py
    adapters/
        postgres_adapter.py
        legacy_adapter.py

plugins/
    logging.py
    metrics.py
```

Actions содержат бизнес-логику. Resource managers содержат адаптеры. Plugins содержат наблюдение. Смешение запрещено.

---

## Финальная модель AOA

AOA определяет шесть первичных сущностей:

Action — чистая бизнес-логика и живая спецификация самой себя. Aspect — шаг бизнес-логики. Resource — доступ к внешнему миру. Plugin — наблюдение за процессом и построение семантического дерева выполнения. ActionMachine — единый исполнитель и хранилище метамодели. DependencyFactory — связывание домена и инфраструктуры.

Все вместе они образуют строгую, детерминированную и легко тестируемую архитектуру в которой невозможно случайно написать плохой код и в которой каждая бизнес-операция является машиночитаемой спецификацией самой себя [1].
