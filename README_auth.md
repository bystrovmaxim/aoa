# Техническая спецификация: Архитектура подсистемы безопасности и контроля доступа

## 1. Введение и концепция
Система безопасности разработана как независимый слой (ActionEngine.Auth), обеспечивающий защиту инфраструктурных компонентов (API, MCP-сервер) от неавторизованного доступа. Архитектура базируется на принципе **инверсии управления**: бизнес-логика (Action) не занимается аутентификацией, а получает уже сформированный и проверенный контекст (`Context`) от вышестоящих слоев [1].

## 2. Компоненты аутентификации (AuthCoordinator)
Процесс аутентификации управляется координатором `AuthCoordinator`, который оркестрирует взаимодействие трех стратегий [1]:

### 2.1. Извлечение данных (`CredentialExtractor`)
Интерфейс `CredentialExtractor` отвечает за нормализацию входящих данных. В текущей реализации для HTTP-запросов используется `ExtractorCredentialsHTTP`, который извлекает токен доступа из заголовка `X-API-Key` [1]. Этот компонент инкапсулирует логику доступа к сырым HTTP-запросам, позволяя системе работать с любым протоколом, будь то FastAPI или иные интерфейсы [1].

### 2.2. Проверка личности (`Authenticator`)
Интерфейс `Authenticator` преобразует учетные данные в структуру `UserInfo`. Реализация `EnvApiKeyAuthenticator` использует конфигурационные переменные окружения, соответствующие префиксу `API_KEY_`.
- **Механизм парсинга**: Переменная окружения вида `API_KEY_<name>=<secret>:<user_id>:<role1>,<role2>` парсится при старте сервиса.
- **Безопасность**: Данные о пользователях хранятся в оперативной памяти в виде словаря `keys` (Map), что обеспечивает время поиска $O(1)$ для каждого входящего запроса [1].

### 2.3. Сбор контекста (`ContextAssembler`)
Интерфейс `ContextAssembler` собирает метаданные запроса (`RequestInfo`).
- **HTTP-реализация**: `HTTPContextAssembler` извлекает `trace_id` (для обеспечения трассировки), IP-адрес, метод запроса, временную метку (UTC) и кастомные теги (`X-Tags`) [1]. Это критически важно для аудита безопасности и мониторинга попыток несанкционированного доступа.

## 3. Контекст выполнения (`Context`)
Объект `Context` является «транспортным средством» для данных безопасности внутри системы. Он состоит из трех неразрывных частей:
1.  **`UserInfo`**: Идентификатор (`user_id`) и массив ролей. Содержит словарь `extra` для расширяемых метаданных (например, метод аутентификации) [1].
2.  **`RequestInfo`**: Инфраструктурные данные (IP, URL, User-Agent, протокол), используемые для логирования и аналитики [1].
3.  **`EnvironmentInfo`**: Данные о среде выполнения (хост, имя пода, версия сервиса), которые добавляются для диагностики в распределенных системах [1].

## 4. Механизм авторизации и контроля прав
Авторизация реализована декларативно, что минимизирует вероятность ошибок ручного управления доступом.

### 4.1. Декларативный контроль (`CheckRoles`)
Использование декоратора `@CheckRoles` позволяет разработчику явно указывать требования к доступу для каждого класса-действия:
- `CheckRoles.NONE`: Доступ без ограничений.
- `CheckRoles.ANY`: Требует, чтобы пользователь прошел аутентификацию (имел хотя бы одну роль).
- `CheckRoles(["role_name"])`: Требует наличия специфической роли [1].

### 4.2. Исполнение правил (`Aspects`)
Процесс проверки прав встроен в жизненный цикл выполнения действия (`BaseSimpleAction.run`):
1.  **Аспект авторизации (`_permissionAuthorizationAspect`)**: При запуске `run` метод вызывает `_checkRole`, который сравнивает роли из `ctx.user.roles` с тем, что было задано декоратором.
2.  **Обработка исключений**: Если проверка не проходит, выбрасывается `AuthorizationException`, которая прерывает цепочку выполнения до начала обработки бизнес-логики (`_handleAspect`). Это обеспечивает «защиту по умолчанию» [1].

## 5. Транзакционная безопасность и целостность
Для действий, изменяющих состояние БД (`BaseTransactionAction`), используется `TransactionContext`, который расширяет стандартный `Context` полем `connection` [1].
- **Валидация соединения**: Декоратор `@requires_connection_type` гарантирует, что тип объекта соединения (например, `psycopg2.extensions.connection`) соответствует требуемому [1].
- **Атомарность**: Использование `TransactionContext` позволяет менеджеру транзакций (`PostgresConnectionManager` или `CsvConnectionManager`) безопасно выполнить `commit()` при успешном завершении или `rollback()` в аспекте `_onErrorAspect` при возникновении любого сбоя [1].

## 6. Жизненный цикл безопасности (Summary)
Безопасность системы выстроена как конвейер:
1.  **EntryPoint** (FastAPI или MCP) принимает запрос.
2.  **Coordinator** обрабатывает запрос, используя Extractor, Authenticator и Assembler для формирования `Context`.
3.  **Действие** (Action) получает `Context`.
4.  **Aspect-oriented подход**:
    *   `_permissionAuthorizationAspect` блокирует доступ, если роли не подходят.
    *   `_validationAspect` проверяет корректность входных параметров через `FieldCheckers` (String, Int, InstanceOf).
    *   `_handleAspect` выполняет бизнес-логику только после прохождения всех проверок [1].

Такая модульная структура позволяет легко расширять систему новыми способами аутентификации (например, JWT или OAuth) без изменения внутренней логики действий [1].
