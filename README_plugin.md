## Система плагинов ActionMachine

Плагины предназначены для расширения функциональности ActionMachine без изменения бизнес-логики. Они подключаются к событиям жизненного цикла выполнения действия и могут выполнять произвольные действия: логирование, сбор метрик, аудит, дополнительные проверки безопасности, интеграцию с внешними системами и т.п. Плагины **не должны модифицировать** параметры, состояние или результат действия — это ответственность аспектов. Однако они могут читать эти данные и добавлять метаданные (например, в контекст или через собственное состояние).

---

### 1. События

События — это именованные точки, в которых могут срабатывать плагины. Имя события — строка. Система предопределяет следующие события:

* **`global_start`** – вызывается перед началом выполнения действия (до запуска цепочки аспектов) в самом начале метода `run` машины.
* **`global_finish`** – вызывается после успешного завершения действия (после выполнения summary-аспекта, но до возврата результата вызывающему коду) – самая последняя строчка в методе `run` машины.
* **`before:<имя_аспекта>`** – вызывается перед выполнением конкретного регулярного аспекта. Например, `before:choose_channel`.
* **`after:<имя_аспекта>`** – вызывается после выполнения конкретного регулярного аспекта (после применения чекеров к его результату).

Имена событий могут быть расширены: любая часть кода (например, внутри аспекта) может инициировать событие с произвольным именем через вызов `machine.emit(event_name, ...)`. Плагины могут подписываться на эти события с помощью регулярных выражений.

---

### 2. Декоратор `@on`

Методы плагина, которые должны реагировать на события, помечаются декоратором `@on`. Декоратор принимает следующие **обязательные** параметры:

* **`event_regex`** (str) – регулярное выражение, которому должно соответствовать имя события. Примеры: `'global_start'`, `'global_.*'`, `'before:.*'`, `'.*'`.
* **`class_regex`** (str) – регулярное выражение для полного имени класса действия (включая модуль). Примеры: `'myapp.actions.OrderAction'`, `'.*OrderAction'`, `'myapp\\..*'`, `'.*'`.
* **`ignore_exceptions`** (bool) – определяет, как обрабатывать исключения, возникшие в этом обработчике:
  * `False` – исключение прерывает выполнение действия (пробрасывается наружу);
  * `True` – исключение игнорируется (логируется, но не влияет на выполнение).

Один метод может быть помечен **несколькими декораторами `@on`**. Если хотя бы одна из подписок подходит под текущее событие и класс действия, метод будет вызван **один раз** (дубликаты не возникают). Это позволяет гибко настраивать реакцию на разные комбинации.

**Примеры использования масок:**

* Только `global_start` для конкретного класса `OrderAction`: `@on('global_start', 'myapp.actions.OrderAction', ignore_exceptions=True)`
* Все `before:` аспекты для любого действия: `@on('before:.*', '.*', ignore_exceptions=False)`
* Все события для действий, оканчивающихся на `Action`: `@on('.*', '.*Action', ignore_exceptions=True)`
* События `after:` для аспектов из модуля `payment`: `@on('after:.*', 'myapp.payment\\..*', ignore_exceptions=True)`
* Только `global_start` и `global_finish` для всех действий: `@on('global_.*', '.*', ignore_exceptions=True)`
* Конкретный аспект `validate` для любого действия: `@on('before:validate', '.*', ignore_exceptions=False)`
* Комбинация: один метод реагирует на два разных события – два декоратора: 
  `@on('global_start', '.*', ...)` и `@on('before:.*', '.*', ...)`

---

### 3. Плагин и его состояние

Плагин представляет собой класс, наследующий от базового класса `Plugin`. Экземпляр плагина создаётся один раз для машины действий и живёт на протяжении всего времени работы машины (или до его удаления). Плагин **не хранит собственное состояние** в атрибутах экземпляра, потому что состояние должно быть изолировано для каждого отдельного вызова `run`. Вместо этого плагин предоставляет **метод, возвращающий начальное состояние** для каждого запуска.

Для этого в классе плагина должен быть реализован метод `get_initial_state(self) -> Any`, который возвращает объект состояния (может быть любого типа, обычно словарь или пользовательский объект). Машина вызывает этот метод перед первым запуском обработчика для данного выполнения действия и затем передаёт полученное состояние во все обработчики. Обработчик **обязан вернуть** новое состояние (или то же самое), которое будет передано следующему обработчику (в рамках того же события или последующих). Таким образом, состояние живёт только в пределах одного вызова `machine.run` и для каждого плагина своё.

Плагин не должен полагаться на глобальные переменные, так как состояние должно быть изолировано.

---

### 4. Механизм вызова обработчиков

При наступлении события машина действий выполняет следующие шаги:

1. **Формирование ключа события** – определяется имя события (например, `'before:choose_channel'`) и полное имя класса действия (в формате `модуль.класс`, включая наследование). Это имя генерируется базовым классом `BaseAction` и кэшируется в нём для производительности.

2. **Получение списка обработчиков** – для каждого зарегистрированного плагина машина запрашивает список методов-обработчиков, которые соответствуют данному событию и классу действия. Плагин сам отвечает за предоставление этого списка на основе своих декораторов. Для этого у плагина должен быть метод `get_handlers(self, event_name: str, class_name: str) -> List[Tuple[Callable, bool]]`, который:
   * перебирает все методы, помеченные `@on`;
   * для каждой подписки проверяет, соответствует ли `event_regex` переданному имени события, а `class_regex` – полному имени класса действия (используется `re.fullmatch`);
   * возвращает список кортежей `(handler_method, ignore_exceptions)` для всех подходящих подписок (без дубликатов).

   Машина **кэширует** результат этого запроса для пары `(event_name, class_name)`, чтобы при повторных вызовах того же события для того же класса действия не выполнять повторную проверку. Кэш сбрасывается при изменении состава плагинов (например, при динамической регистрации или удалении).

3. **Параллельный асинхронный вызов с ограничением конкурентности** – машина обходит полученный список обработчиков и запускает их **асинхронно и параллельно**, но с ограничением на количество одновременно выполняющихся обработчиков для одного события. Это означает, что обработчики одного события могут выполняться конкурентно, порядок их завершения не гарантирован. Рекомендуемое ограничение – **10 одновременных задач** (значение может быть настроено в конфигурации машины через параметр `max_concurrent_handlers`). Это предотвращает чрезмерную нагрузку и сохраняет отзывчивость системы.

   Для каждого обработчика:
   * перед вызовом машина получает текущее состояние плагина (для данного запуска `run`) из своего внутреннего хранилища;
   * вызывает обработчик с помощью `await handler(state_plugin, event_name, action_name, params, state_aspect, is_summary, deps, context, result, duration)`, передавая актуальные аргументы (см. раздел 5);
   * обработчик возвращает новое состояние, которое машина сохраняет для последующих вызовов в рамках этого же запуска;
   * если обработчик выбрасывает исключение, машина проверяет флаг `ignore_exceptions`:
     * при `False` – исключение пробрасывается наружу, выполнение действия немедленно прерывается;
     * при `True` – исключение логируется (через стандартный логгер) и игнорируется; выполнение продолжается со следующим обработчиком.

   Машина ожидает завершения всех запущенных обработчиков (успешных или с игнорируемыми исключениями) перед тем, как перейти к следующему этапу выполнения действия. Если какой-то обработчик прерывает действие (исключение с `ignore_exceptions=False`), дальнейшие обработчики не запускаются, а исключение пробрасывается наружу.

4. **Продолжение выполнения** – после обработки всех подходящих обработчиков (или их прерывания) машина продолжает нормальный ход выполнения действия.

---

### 5. Аргументы, передаваемые обработчику

Каждый обработчик получает следующие аргументы (все передаются как именованные, чтобы обеспечить гибкость при добавлении новых параметров в будущем):

* **`state_plugin`** (Any) – текущее состояние плагина для данного запуска действия. Изначально равно результату `get_initial_state()`. Обработчик **должен вернуть** новое состояние (или то же самое).
* **`event_name`** (str) – имя текущего события (например, `'before:choose_channel'`).
* **`action_name`** (str) – полное имя класса действия (включая модуль и наследование), сгенерированное базовым классом.
* **`params`** (BaseParams) – входные параметры действия (неизменяемые).
* **`state_aspect`** (Optional[Dict[str, Any]]) – текущее состояние, передаваемое между аспектами. Для событий `before:<аспект>` – состояние **до** выполнения аспекта. Для `after:<аспект>` – состояние **после** выполнения аспекта (уже прошедшее проверку чекерами). Для `global_start` и `global_finish` – `None`.
* **`is_summary`** (bool) – `True`, если событие относится к summary-аспекту (например, `before:process` для аспекта, помеченного `@summary_aspect`), иначе `False`. Для глобальных событий всегда `False`.
* **`deps`** (DependencyFactory) – фабрика зависимостей, позволяющая получать сервисы и другие действия.
* **`context`** (Context) – глобальный контекст выполнения, содержащий информацию о пользователе, запросе и окружении.
* **`result`** (Optional[BaseResult]) – результат выполнения действия. Передаётся **только** для события `global_finish`. Для всех остальных событий – `None`.
* **`duration`** (Optional[float]) – длительность выполнения в секундах. Для событий `after:<аспект>` – время выполнения данного аспекта (без учёта плагинов). Для `global_finish` – общее время выполнения всего действия (от `global_start` до `global_finish`). Для `before:<аспект>` и `global_start` – `None`.

Обработчики **не должны изменять** переданные объекты (кроме `state_plugin`), чтобы не нарушать бизнес-логику. Они могут читать любые данные.

---

### 6. Обработка исключений

Флаг `ignore_exceptions` в декораторе `@on` определяет критичность ошибок в обработчике:

* **`False`** – любое исключение, возникшее в обработчике, прерывает выполнение действия. Исключение пробрасывается наружу (в вызывающий код). Это поведение следует использовать для плагинов, которые выполняют важные проверки безопасности или другие критичные действия.
* **`True`** – исключение логируется (через стандартный механизм логирования) и игнорируется. Выполнение действия продолжается с остальными обработчиками. Это подходит для плагинов, выполняющих некритичные функции (логирование, сбор метрик), сбой в которых не должен влиять на основную бизнес-логику.

Обратите внимание: для события `global_finish` игнорирование исключений может быть особенно полезно, так как действие уже завершено, и ошибки в пост-обработке не должны влиять на результат.

---

### 7. Кэширование подписок

Для повышения производительности машина кэширует результаты запросов к плагинам. Кэш строится на уровне машины и хранит для каждой пары `(event_name, class_name)` список обработчиков, полученный от всех плагинов. Кэш сбрасывается при изменении состава плагинов (добавление/удаление). Это гарантирует, что накладные расходы на проверку регулярных выражений возникают только один раз для каждой комбинации события и класса действия.

---

### 8. Асинхронность и ограничение конкурентности

Все обработчики плагинов являются асинхронными функциями (определяются с `async def`) и вызываются с использованием `await`. Это позволяет эффективно использовать цикл событий asyncio и не блокировать выполнение других задач во время ожидания ввода-вывода (например, при обращении к внешним API или базам данных).

**Порядок вызова обработчиков**

При наступлении события машина действий собирает список подходящих обработчиков от всех зарегистрированных плагинов (с учётом кэширования). Затем она запускает их **параллельно**, но с ограничением на количество одновременно выполняющихся задач. Это означает, что обработчики одного события могут выполняться конкурентно, и порядок их завершения не гарантирован.

Такая модель обеспечивает максимальную производительность, особенно когда обработчики выполняют независимые операции (например, отправка метрик в разные системы, логирование в файл и в Elasticsearch, проверка прав доступа и т.п.). При этом отсутствие гарантий порядка упрощает реализацию и не создаёт проблем, поскольку плагины не должны полагаться на определённую последовательность вызовов.

**Ограничение конкурентности**

Для предотвращения чрезмерной нагрузки на систему и сохранения отзывчивости при большом количестве плагинов машина ограничивает количество одновременно выполняющихся обработчиков для одного события. Рекомендуемое значение по умолчанию – **10**. Это ограничение можно настроить при создании машины через параметр `max_concurrent_handlers` (например, `machine = ActionProductMachine(context, max_concurrent_handlers=20)`).

Если количество подходящих обработчиков превышает лимит, они ставятся в очередь и запускаются по мере освобождения «слотов». Очередь не имеет приоритетов – обработчики запускаются в порядке поступления.

**Длительные операции и фоновые задачи**

Если обработчику необходимо выполнить длительную операцию (например, загрузку большого файла или сложный расчёт), он должен:

* использовать асинхронные библиотеки (например, `aiohttp` вместо `requests`, `asyncpg` вместо `psycopg2`), чтобы не блокировать цикл событий;
* если асинхронной альтернативы нет, можно запустить синхронную операцию в отдельном потоке с помощью `asyncio.to_thread(blocking_func, ...)`. Однако следует помнить, что это создаёт дополнительную нагрузку на пул потоков и может сказаться на производительности при массовом использовании.

В любом случае обработчик должен оставаться отзывчивым и не занимать «слот» конкурентности дольше необходимого. Если операция действительно долгая и не критична к моменту завершения, рассмотрите возможность запуска фоновой задачи через `asyncio.create_task()` и немедленного возврата управления. Однако в этом случае вы теряете возможность обработать исключения из этой задачи в контексте действия – они будут логироваться, но не повлияют на выполнение.

**Взаимодействие с состоянием плагина**

Поскольку обработчики одного события выполняются параллельно, они могут одновременно обращаться к состоянию плагина (`state_plugin`), которое хранится в машине для данного запуска действия. **Гарантии отсутствуют** – если обработчики модифицируют состояние, возможны гонки данных. Поэтому рекомендуется:

* проектировать обработчики так, чтобы они не зависели от промежуточных изменений состояния другими обработчиками;
* если необходимо агрегировать данные от нескольких обработчиков, использовать финальный обработчик на событие `global_finish`, который будет вызван после всех (он выполняется в том же пуле, но после того, как все параллельные обработчики завершатся, потому что машина дожидается их завершения перед переходом к следующему этапу действия).

Более сложные сценарии синхронизации (например, блокировки) не поддерживаются на уровне фреймворка и должны реализовываться разработчиком плагина при необходимости.

### 9. Требования к документированию

Все классы, методы и параметры в системе плагинов должны быть тщательно задокументированы. Документация должна включать:

* **Для базового класса `Plugin`**: описание назначения, метода `get_initial_state`, метода `get_handlers` (если он не переопределяется автоматически).
* **Для декоратора `@on`**: подробное описание каждого параметра, допустимые значения, примеры регулярных выражений, пояснение флага `ignore_exceptions`.
* **Для аргументов обработчика**: полное описание каждого аргумента, включая возможные значения, когда они могут быть `None`, и ограничения на изменение.
* **Примеры использования масок**: множество примеров, демонстрирующих, как с помощью регулярных выражений можно подписаться на различные комбинации событий и классов действий. Примеры должны покрывать:
  * подписку на конкретное событие для конкретного класса;
  * подписку на все события определённого типа (например, все `before:`);
  * подписку на все события для всех действий;
  * подписку на все события для действий из определённого модуля;
  * комбинации с несколькими декораторами на одном методе.
* **Примеры плагинов**: как минимум два полноценных примера (логирование и сбор метрик) с пояснением работы состояния и обработки исключений.

Документация должна быть написана на русском языке (если это требование проекта) и доступна как в коде (docstrings), так и в отдельном руководстве.

### 10. Примеры плагинов с подробными комментариями

#### Пример 1: Плагин логирования

```python
import time
import logging
from actionmachine import Plugin, on

logger = logging.getLogger(__name__)

class LoggingPlugin(Plugin):
    """
    Плагин для логирования времени выполнения действия и аспектов.
    Демонстрирует использование состояния для хранения таймеров.
    """
    def get_initial_state(self):
        """
        Возвращает начальное состояние для одного запуска действия.
        Состояние будет передано во все обработчики и должно быть возвращено обратно.
        Здесь мы используем словарь с ключом 'timers' для хранения времени старта.
        """
        return {"timers": {}}

    @on(
        event_regex='global_start',
        class_regex='.*',
        ignore_exceptions=True
    )
    async def log_start(self, state_plugin, event_name, action_name, params, state_aspect,
                        is_summary, deps, context, result, duration):
        """
        Обработчик события global_start.
        Запоминает время начала действия и логирует факт старта.
        """
        state_plugin["timers"][action_name] = time.time()
        logger.info(f"Action {action_name} started with params {params}")
        return state_plugin

    @on('global_finish', '.*', ignore_exceptions=True)
    async def log_finish(self, state_plugin, event_name, action_name, params, state_aspect,
                         is_summary, deps, context, result, duration):
        """
        Обработчик события global_finish.
        Вычисляет и логирует общее время выполнения действия.
        """
        elapsed = time.time() - state_plugin["timers"].pop(action_name, time.time())
        logger.info(f"Action {action_name} finished in {elapsed:.3f}s with result {result}")
        return state_plugin

    @on('before:.*', '.*', ignore_exceptions=True)
    async def log_before_aspect(self, state_plugin, event_name, action_name, params, state_aspect,
                                 is_summary, deps, context, result, duration):
        """
        Обработчик всех before-аспектов.
        Логирует начало аспекта и текущее состояние.
        """
        aspect_name = event_name.split(':')[1]
        logger.info(f"Aspect {aspect_name} starting with state {state_aspect}")
        return state_plugin

    @on('after:.*', '.*', ignore_exceptions=True)
    async def log_after_aspect(self, state_plugin, event_name, action_name, params, state_aspect,
                                is_summary, deps, context, result, duration):
        """
        Обработчик всех after-аспектов.
        Логирует завершение аспекта, его длительность и новое состояние.
        """
        aspect_name = event_name.split(':')[1]
        logger.info(f"Aspect {aspect_name} finished in {duration:.3f}s with new state {state_aspect}")
        return state_plugin
```

#### Пример 2: Плагин метрик с состоянием

```python
from actionmachine import Plugin, on

class MetricsPlugin(Plugin):
    """
    Плагин для сбора статистики вызовов действий.
    Демонстрирует накопление данных в состоянии плагина.
    """
    def get_initial_state(self):
        return {"call_counts": {}}

    @on(
        event_regex='global_finish',
        class_regex='myapp\\.actions\\..*',
        ignore_exceptions=True
    )
    async def count_action(self, state_plugin, event_name, action_name, params, state_aspect,
                            is_summary, deps, context, result, duration):
        """
        Увеличивает счётчик вызовов для данного действия.
        В реальном проекте здесь можно отправлять метрику в Prometheus/StatsD.
        """
        state_plugin["call_counts"][action_name] = state_plugin["call_counts"].get(action_name, 0) + 1
        print(f"Action {action_name} called {state_plugin['call_counts'][action_name]} times")
        return state_plugin
```

#### Пример 3: Плагин безопасности (критичный)

```python
from actionmachine import Plugin, on

class SecurityPlugin(Plugin):
    """
    Плагин для дополнительной проверки прав доступа перед выполнением чувствительных действий.
    Использует ignore_exceptions=False, чтобы любая ошибка прерывала выполнение.
    """
    def get_initial_state(self):
        return {}

    @on(
        event_regex='before:.*',
        class_regex='.*SensitiveAction',
        ignore_exceptions=False
    )
    async def check_sensitive(self, state_plugin, event_name, action_name, params, state_aspect,
                               is_summary, deps, context, result, duration):
        """
        Проверяет, имеет ли пользователь право выполнять чувствительное действие.
        Если нет — выбрасывает исключение, которое прервёт выполнение.
        """
        if not context.user.has_permission('sensitive'):
            raise PermissionError("User lacks permission for sensitive action")
        return state_plugin
```

#### Пример 4: Плагин с несколькими подписками на одном методе

```python
from actionmachine import Plugin, on

class MultiSubscriptionPlugin(Plugin):
    """
    Плагин, демонстрирующий, как один метод может реагировать на несколько событий.
    """
    def get_initial_state(self):
        return {"counter": 0}

    @on('global_start', '.*', ignore_exceptions=True)
    @on('global_finish', '.*', ignore_exceptions=True)
    async def handle_global(self, state_plugin, event_name, action_name, params, state_aspect,
                             is_summary, deps, context, result, duration):
        """
        Этот метод будет вызван и для global_start, и для global_finish.
        В параметре event_name передаётся конкретное имя события.
        """
        state_plugin["counter"] += 1
        print(f"Global event {event_name} processed, counter={state_plugin['counter']}")
        return state_plugin
```
