Ниже — **формальные правила AOA (Action‑Oriented Architecture)**, сформулированные строго и однозначно.  
Они основаны на архитектуре ActionEngine и подтверждаются качественными метриками кода (например, ActionProductMachine, DependencyFactory, CheckRoles и др. имеют оценки A в radon [1]).

# 📘 Формальные правила AOA

Формальные правила AOA описывают *что допускается*, *что запрещено* и *какие обязательства* накладывает архитектура на разработчика и фреймворк.  
Это — нормативная спецификация парадигмы.

Каждое правило формулируется так, чтобы его нельзя было интерпретировать двусмысленно.

---

# 1. Правила для действий (Actions)

### **1.1. Action обязан быть классом, наследованным от BaseAction.**  
Это гарантирует единообразный интерфейс, включая методы аспектов и доступ к DependencyFactory.

### **1.2. Action обязан иметь внутренние классы Params и Result.**  
Они должны наследоваться от BaseParams и BaseResult.  
Оба класса immutable (dataclass с frozen=True).

BaseParams и BaseResult имеют оценку A по radon [1].

### **1.3. Action не имеет состояния.**  
Внутри экземпляра запрещено хранить изменяемые данные.

### **1.4. Action не имеет доступа к Context.**  
Context передаётся только плагинам.  
Action может оперировать только params, state и deps.

### **1.5. Action обязан иметь ровно один summary‑аспект.**  
ActionProductMachine выбрасывает ошибку при отсутствии или дублировании summary [1].

### **1.6. Регулярные аспекты выполняются строго в порядке определения в классе.**  
Порядок определяется по номеру строки (`co_firstlineno`), что формально реализовано в `_collect_aspects` [1].

# 2. Правила аспектов (Aspects)

### **2.1. Метод является аспектом только если помечен @aspect или @summary_aspect.**  
Проверяется по атрибуту `_is_aspect` [1].

### **2.2. Аспект обязан иметь сигнатуру: (self, params, state, deps).**

### **2.3. Аспект обязан возвращать dict.**  
Иначе машина выбросит TypeError [1].

### **2.4. summary‑аспект обязан возвращать Result.**

### **2.5. Аспекты не обязаны чистыми функциями, но обязаны быть детерминированными.**

---

# 3. Правила параметров (Params) и результата (Result)

### **3.1. Params immutable.**
### **3.2. Result immutable.**

Это подтверждается использованием `dataclass(frozen=True)` в тестах [1].

### **3.3. Params содержат только бизнес‑данные.**  
Контекст и инфраструктурные объекты туда передавать запрещено.

### **3.4. Result содержит только итоговое значение действия.**

### **3.5. Все поля Params и Result строго типизированы.**

### **3.6. Проверка параметров осуществляется через чекеры, а не вручную.**

Чекеры BaseFieldChecker, StringFieldChecker, IntFieldChecker имеют рейтинги A [1].

---

# 4. Правила для state (между аспектами)

### **4.1. State — это dict.**

### **4.2. Каждый аспект обязан возвращать новый state.**

### **4.3. summary‑аспект получает state, но не обязан возвращать dict.**

### **4.4. Поля state обязаны соответствовать чекерам, если они применены.**

Любые "лишние поля" вызывают ValidationFieldException [1].

---

# 5. Правила для Dependency Injection (DI)

### **5.1. Все зависимости объявляются только через @depends.**

### **5.2. Action не может получить зависимость, не объявленную через @depends.**

DependencyFactory выбрасывает ошибку [1].

### **5.3. Экземпляры зависимостей создаются лениво и кешируются.**

### **5.4. Действие может вызывать другие действия только через deps.run_action.**

Это гарантирует единообразную обработку аспектов и плагинов [1].

---

# 6. Правила для Context

### **6.1. Context доступен только плагинам.**

### **6.2. Context неизменяем для Actions.**

### **6.3. Context может включать UserInfo, RequestInfo, EnvironmentInfo.**

Все эти компоненты имеют рейтинг A [1].

---

# 7. Правила плагинов (Plugins)

### **7.1. Плагин обязан наследоваться от Plugin.**

### **7.2. Плагин должен реализовать get_initial_state().**

### **7.3. Методы‑обработчики должны быть async‑функциями.**

### **7.4. Подписка на события — только через @on(event, class_regex).**

Decorators.py (`@on`) имеет рейтинг A [1].

### **7.5. Плагин не имеет права изменять params или result.**

### **7.6. Ошибки в плагинах не должны прерывать действие, если ignore_exceptions=True.**  
Реализовано в `_run_single_handler` [1].

---

# 8. Правила исполнения (Execution Pipeline)

### **8.1. Машина обязана:**

- увеличить nest_level  
- вызвать global_start  
- выполнить аспекты  
- вызвать summary  
- вызвать global_finish  
- уменьшить nest_level  

Полный pipeline описан и реализован в ActionProductMachine.run() — рейтинг A [1].

### **8.2. before/after события вызываются для каждого аспекта.**

### **8.3. Плагины вызываются асинхронно, но run — всегда синхронный.**

---

# 9. Правила для ролевой модели

### **9.1. У каждого Action обязан быть декоратор @CheckRoles.**

Иначе машина выбрасывает ошибку (см. `_check_action_roles`) [1].

### **9.2. Проверка ролей обязана использовать одну из моделей:**

- NONE  
- ANY  
- SINGLE  
- LIST  

### **9.3. Любое несоответствие ролей приводит к AuthorizationException.**

Все методы проверки ролей имеют рейтинг A [1].

---

# 10. Правила тестируемости

### **10.1. Любое действие обязано быть тестируемо без контекста и плагинов.**

### **10.2. Любая зависимость обязана мокаемо заменяться через ActionTestMachine.**

ActionTestMachine — A (72.83 MI) [1].

### **10.3. Аспекты обязаны быть вызываемы напрямую.**
