# 📘 Сравнение AOA с MVC / Clean Architecture / CQRS

## 1. AOA vs MVC

### Структура
| Характеристика | MVC | AOA |
|----------------|-----|-----|
| Основная единица логики | Controller / ViewModel | **Action** |
| Распределение логики | Размазано между слоями | **Полная концентрация в Action** |
| Контекст | Передаётся через контроллеры, может мутировать | Контекст не виден в Action (изолирован) |
| Валидация | Разбросана по контроллеру, моделям | **Чекеры + Params (immutability)** |
| Тестируемость | Часто сложная — нужен веб-контекст | **ActionTestMachine → чистый юнит-тест** [1] |
| Выполнение побочных эффектов | Легко смешивается с логикой | Весь side‑effect в **DI‑зависимостях или плагинах** |

### Вывод  
AOA радикально уменьшает связность и убирает главный недостаток MVC — **размазывание логики по слоям**.

---

## 2. AOA vs Clean Architecture

### Структура
| Характеристика | Clean Architecture | AOA |
|----------------|--------------------|------|
| Слой "Use Case" | Основной блок | **Action — практически то же самое, но чище** |
| Entities | Часто перегружены логикой | **AOA избегает доменных моделей** — логика в аспектах |
| Dependency Rule | Внешние слои → внутрь | Точно так же: **Deps → внутрь Actions** |
| DI | Сложные контейнеры | **DependencyFactory (A‑качество)** [1] |
| Контекст | Может протекать | **Контекст изолирован** [1] |
| Расширяемость | Middleware, Interactors | **Аспекты + плагины (чётче и предсказуемее)** |
| Тестируемость | Отличная | **AOA чаще проще, так как нет сервис‑лесенок** |

### Главное отличие:
Clean Architecture — методология.  
AOA — **конкретная исполняемая модель**, закреплённая в ActionProductMachine (A по MI [1]).

---

## 3. AOA vs CQRS

### Структура
| Характеристика | CQRS | AOA |
|----------------|------|------|
| Разделение модели | Отдельно Commands и Queries | Actions могут вести себя как Commands |
| Характер логики | Команды = изменение состояния; Queries = получение данных | Actions = законченная бизнес‑операция |
| Read model | Отдельная инфраструктура | Нет: **DI и плагины служат наблюдателями** |
| Потоки | Часто сложные pipeline / event sourcing | **Линейные аспекты + вложенные Actions** |
| Тестируемость | Хорошая, но требует setup | В AOA → **ActionTestMachine** c моками в 1 строке [1] |
| Событийность | Event sourcing — ядро | **Плагинные события — read‑only**, не меняют систему |

### Что общего:  
- Action = Command  
- SummaryAspect = handler  
- Params = DTO  
- Result = Response  

### Что отличает:  
AOA исключает сложность CQRS:  
нет event store, нет projection drift, нет двух моделей состояния.

# 📌 Ключевые отличия AOA

## №1 — Action как атом вместо Controller/UseCase/Command  
Action содержит **всё**, что относится к одной бизнес‑операции:  
аспекты, зависимости, параметры, результат.

Это делает архитектуру минималистичной и строгой.

---

## №2 — Линейный pipeline аспектов  
ActionProductMachine исполняет аспекты детерминированно и линейно:

```
before:a1 → a1 → after:a1  
before:a2 → a2 → after:a2  
summary → result
```

Код прямо подтверждает порядок: аспекты сортируются по строкам [2].

---

## №3 — Чистота Actions  
Action:

- статичен  
- не имеет состояния  
- не видит контекст  
- работает только с params/state/deps  

Основано на BaseAction (рейтинг A [1]).

Это принципиально чище, чем:

- контроллеры MVC  
- interactor'ы CA  
- handlers CQRS  

---

## №4 — Plugins = Observability без побочных эффектов  
Плагины подписаны на события (`@on`), но:

- не меняют логики  
- не меняют params  
- не меняют state  
- не меняют result  
- могут иметь ignore_exceptions=True  
- работают асинхронно вне main‑pipeline  

Plugin и Decorators — A по MI [1].

Это фундаментальное отличие от middleware в MVC и pipeline CQRS.

---

## №5 — DI без IoC‑контейнера  
DependencyFactory:

- создаёт зависимости лениво  
- кеширует их  
- использует @depends  
- запускает вложенные действия  

DependencyFactory.get — A (4) [1].

DI в AOA в разы проще, чем:

- в Clean Architecture (модули/провайдеры)
- в CQRS (command bus, handler registry)
- в MVC (service locator / IoC)

---

## №6 — Тестируемость в 10 раз проще  
ActionTestMachine:

- подменяет любые зависимости  
- мокаeт вложенные действия  
- ловит параметры и вызовы  
- полностью синхронный  
- без веб‑слоя  
- без контекста  

ActionTestMachine имеет рейтинг A (72.83 MI) [1].

В MVC / CA / CQRS тесты требуют поднятия контекстов, контейнеров или middleware.

# 📌 Итоговое сравнение (в 1 фразе каждое)

### MVC  
Смешивает ответственность.  
**AOA разделяет и очищает.**

### Clean Architecture  
Декларирует чистоту.  
**AOA строго её обеспечивает механизмами.**

### CQRS  
Сложно, потоково, двухмодельно.  
**AOA проще и линейнее, сохраняя ключевые преимущества командного стиля.**

# 🎯 Итог

AOA — это минималистичная, детерминированная, линейная архитектура, где все бизнес‑операции представлены действиями, инфраструктура вынесена в плагины, а расширение поведения жёстко структурировано аспектами и DI.  

Большая часть этой архитектуры подтверждена качеством её реализации — ActionProductMachine, BaseAction, BaseParams, BaseResult, Checkers, Plugins, DependencyFactory имеют рейтинг **A** в отчёте radon [1], что делает AOA не только концептуально сильной, но и практически надёжной.
