01_idea_catch_error_handlers

Перехватчики исключений с цепочкой обработчиков (@catch)

Проблема

В большинстве фреймворков обработка ошибок либо глобальна (один обработчик на всё приложение), либо размазана по блокам try/except в каждом действии. Ни тот, ни другой подход не позволяют централизованно и при этом контекстно-зависимо реагировать на ошибки. Разработчик вынужден выбирать между «поймать всё в одном месте» и «поймать точечно, но дублировать код».

Решение

Декоратор @catch для методов действия. Метод, помеченный этим декоратором, становится обработчиком исключений для данного действия. Он получает ту же информацию, что и обычный аспект: params, state, deps, connections — а также само исключение. Обработчиков может быть несколько, каждый привязан к конкретному типу исключения, и вызываются они строго в порядке объявления в классе [6].

Как работает цепочка

Каждый обработчик получает текущее исключение и обязан вернуть одно из двух:

- Result — исключение считается полностью обработанным, действие завершается успешно с этим результатом. Цепочка прерывается.
- Исключение (то же или новое) — оно становится текущим и передаётся следующим обработчикам. Уже вызванные обработчики не вызываются повторно.

Если ни один обработчик не вернул Result, итоговое исключение выбрасывается из run() [6].

Пример

```python
class CreateOrderAction(BaseAction):

    @aspect("Сохранение заказа")
    async def save(self, params, state, deps, connections):
        db = connections["connection"]
        await db.execute("INSERT INTO orders ...")
        return state

    @catch(IntegrityError)
    async def on_integrity(self, params, state, deps, exc):
        db = deps.get(PostgresConnectionManager)
        await db.rollback()
        return exc  # передаём дальше по цепочке

    @catch(DatabaseError)
    async def on_db_error(self, params, state, deps, exc):
        db = deps.get(PostgresConnectionManager)
        await db.rollback()
        return BusinessError("Database unavailable") from exc

    @catch(Exception)
    async def on_any_error(self, params, state, deps, exc):
        return CreateOrderAction.Result(success=False, error="Internal error")
```

Важные правила (без магии)

@catch не наследуется автоматически — обработчики действуют только в том классе, где явно определены. Это исключает неожиданное поведение при наследовании [6].

Порядок вызова — строго по порядку объявления в классе. Никакого автоматического ранжирования по «специфичности» исключений. Первый подходящий по типу (isinstance) получает управление [6].

Обработчики не отменяют обычные try/except внутри аспектов. Если аспект поймал исключение сам — до @catch оно не дойдёт. Глобальные обработчики — это защитный механизм последней надежды [6].

События для плагинов

ActionMachine генерирует два типа событий:

- on_error — исключение не было обработано, итоговое исключение будет выброшено.
- on_error_handled — исключение обработано, действие завершилось успешно.

Оба события содержат полный контекст: params, state, action_name, nest_level, duration, exception. Для on_error_handled дополнительно передаётся result [6].

Влияние

Компенсирующие действия (откат транзакции, закрытие соединений, отправка уведомления) выполняются с полным доступом к deps и connections.

Решение «пробросить или заменить исключение» принимается контекстно — с доступом ко всем данным бизнес-процесса.

Плагины видят разницу между обработанными и необработанными ошибками — это критично для мониторинга.

Уникальность

Ни Django, ни FastAPI, ни Spring не дают привязку обработчиков к конкретному действию с доступом ко всем зависимостям и цепочкой передачи между обработчиками. Обычно обработка ошибок либо глобальна, либо локальна — AOA даёт третий путь: контекстная, декларативная, с полной наблюдаемостью через плагины [6].

## idea_02_coordinator_loggers.md — Сквозной логер как окно в машину

### Проблема

В типичных проектах логирование разрозненно: в каждом модуле свой логер, уровни настраиваются глобально, и чтобы понять что происходит в конкретном действии, приходится собирать логи из разных мест и мысленно их склеивать. Разработчик вынужден дублировать вызовы `logger.info()` в каждом аспекте, вручную подставлять контекст (`user_id`, `trace_id`, `action_name`), и при этом менять формат логов можно только через изменение кода [6].

Но проблема глубже. Либо разработчик вызывает `logger.info()` в каждом методе, вручную подставляя контекст — имя сервиса, идентификатор пользователя, трассировку. Либо логирование отсутствует вовсе, потому что добавлять контекст вручную слишком утомительно. В обоих случаях бизнес-логика загрязняется инфраструктурными деталями: разработчик вынужден знать про `request.user.id`, про заголовки, про формат логов. При смене среды — с разработки на production — приходится либо удалять `print()`, либо переписывать вызовы на структурированный логер [1].

В контексте AOA эта проблема имеет дополнительное измерение. ActionMachine владеет полным контекстом выполнения — имя действия, имя аспекта, уровень вложенности, длительность, данные пользователя из Context, `trace_id`, hostname, окружение [1]. Но Action физически не имеет доступа к Context [2]. Значит, разработчик в аспекте не может и не должен вручную подставлять эти данные в лог. При этом именно эти данные делают лог полезным.

### Решение

Логер в AOA — не зависимость через `@depends` и не глобальный синглтон. Это **шестой параметр сигнатуры аспекта**, который ActionMachine создаёт автоматически перед каждым вызовом и наполняет полным контекстом выполнения [3]. Разработчик пишет только бизнес-сообщение. Всё остальное — `action_name`, `aspect_name`, `nest_level`, `duration`, `user_id`, `trace_id`, `hostname`, `environment` — добавляется машиной [3].

Сигнатура аспекта:

```python
# Было (без логера):
async def fetch(self, params, state, deps, connections):
    ...

# Стало (с логером):
async def fetch(self, params, state, deps, connections, log):
    log.info("Загружаю задачи", project=params.project_id)
```

Разработчик пишет одну строку. В тестах это `print`. В production — структурированный JSON в ELK, Telegram-алерт, файл с ротацией. Код Action не меняется никогда [3].

---

### Почему логер — параметр конвейера, а не зависимость

Если логер — зависимость через `@depends`, он загрязняет декларацию бизнес-зависимостей. Интроспектор [4] покажет `ILogger` в графе зависимостей наравне с `PaymentGateway` — но логер не является зависимостью бизнес-логики. Кроме того, логер через DI не знает контекст выполнения: ни `nest_level`, ни `aspect_name`, ни `duration`. Разработчику пришлось бы передавать всё это вручную — а это именно та работа, от которой AOA освобождает [3].

Логер как шестой параметр — это инструмент конвейера, наравне с `params`, `state`, `deps`, `connections`. Пять параметров — полная картина мира аспекта [1]:

- `params` — что приходит в Action
- `state` — что накапливается внутри
- `deps` — что нужно снаружи
- `connections` — какие транзакции открыты
- `log` — где я сейчас и что происходит

Причём `log` — единственный из пяти, кто знает контекст самой машины [1].

---

### Как это реализуется в ActionProductMachine

ActionMachine создаёт `ContextualLogger` перед каждым вызовом аспекта:

```python
async def _call_aspect(self, method, action, params, state, factory, connections):
    log = ContextualLogger(
        coordinator=self._log_coordinator,
        nest_level=self._nest_level,
        action_name=action.get_full_class_name(),
        aspect_name=method.__name__,
        context=self._context,
    )
    return await method(action, params, state, factory, connections, log)
```

`ContextualLogger` обогащает каждое сообщение данными, о которых бизнес-логика не знает и не должна знать [1,3]:

```python
class ContextualLogger:
    def __init__(self, coordinator, nest_level, action_name, aspect_name, context):
        self._coordinator = coordinator
        self._nest_level = nest_level
        self._action_name = action_name
        self._aspect_name = aspect_name
        self._context = context

    def info(self, message, **context):
        self._coordinator.info(
            message,
            nest_level=self._nest_level,
            action=self._action_name,
            aspect=self._aspect_name,
            user_id=self._context.user.user_id,
            trace_id=self._context.request.trace_id,
            **context
        )
```

Разработчик пишет:

```python
log.info("Загружено 150 задач")
```

В production-логе появляется:

```json
{
  "message": "Загружено 150 задач",
  "action": "SyncIssuesAction",
  "aspect": "fetch",
  "nest_level": 2,
  "duration": 0.450,
  "user_id": "agent_1",
  "trace_id": "abc-123",
  "hostname": "pod-xyz-42",
  "service_name": "kanban-assistant",
  "environment": "production"
}
```

Всё это добавлено машиной автоматически, потому что машина владеет полным контекстом [1,3].

### Координатор и множественные логеры

**LogCoordinator** — единая шина логирования. К нему подключается **любое количество независимых логеров**, каждый из которых самостоятельно решает:

- **Что** ему интересно — через фильтр на основе регулярных выражений
- **Как** форматировать — через шаблон с переменными, условной логикой и тегами
- **Куда** писать — консоль, файл, ELK, Telegram, XML, JSON, Syslog

Аспекты и плагины отправляют сообщения в координатор, не зная о конкретных получателях. Координатор рассылает по всем подключённым логерам. Каждый логер применяет свой фильтр и, если сообщение прошло, рендерит его по своему шаблону.

```python
class BaseLogger(ABC):
    @abstractmethod
    def info(self, message: str, **context: Any) -> None: ...
    @abstractmethod
    def warning(self, message: str, **context: Any) -> None: ...
    @abstractmethod
    def error(self, message: str, **context: Any) -> None: ...
    @abstractmethod
    def debug(self, message: str, **context: Any) -> None: ...
```

Адаптеры:

- `PrintLogger` — для тестов и разработки, вывод с отступами по `nest_level`. Подключается по умолчанию в `ActionTestMachine` — ничего не нужно конфигурировать [1].
- `StdLogger` — обёртка над стандартным `logging.getLogger()`.
- `StructuredLogger` — JSON в ELK, Datadog, CloudWatch.
- `TelegramLogger` — алерты критических ошибок.
- `FileLogger` — запись в файл с настраиваемым форматом.
- `SlackLogger`, `XmlLogger`, `SyslogLogger` и т.д.

### Множественность логеров

Один вызов `log.info("Списание средств", amount=150)` в аспекте порождает столько записей, сколько логеров подключено и сколько из них пропустили это сообщение через свой фильтр.

```python
coordinator = LogCoordinator()

coordinator.add_logger(ConsoleLogger(
    format="{indent}[{action}.{aspect}] {message}",
    level="DEBUG"
))

coordinator.add_logger(FileLogger(
    filename="logs/app.log",
    format="{date} {time} | {action} | {aspect} | {user_id} | {message}",
    level="INFO"
))

coordinator.add_logger(ElkLogger(
    format="json",
    level="WARNING"
))

coordinator.add_logger(TelegramLogger(
    chat_id=DIRECTOR_CHAT_ID,
    format="🚨 *{action}* провалился\nПользователь: {user_id}\nОшибка: {exception.message}",
    level="CRITICAL",
    filter_regex=r"exception\.type=(PaymentError|FraudDetected)"
))
```

Логеры работают независимо. Ошибка в одном не блокирует остальные. Добавление нового логера не требует изменения кода — только конфигурации [6].

---

### Шаблоны и переменные

Разработчик при конфигурации логера указывает строку формата с переменными в фигурных скобках. Координатор при рендеринге подставляет реальные значения из полного контекста выполнения.

**Доступные переменные:**

- Из машины и аспекта:
  - `{action_name}`, `{action}` — полное/короткое имя Action
  - `{action_module}` — модуль Action
  - `{aspect_name}`, `{aspect}` — имя аспекта
  - `{nest_level}` — уровень вложенности (0, 1, 2...)
  - `{indent}` — отступ по nest_level (автоматически `"  " * nest_level`)
  - `{duration}` — время выполнения аспекта в секундах
  - `{duration_ms}` — в миллисекундах
  - `{timestamp}`, `{date}`, `{time}` — временны́е метки

- Из Context (пользователь и запрос):
  - `{user_id}`, `{roles}`, `{client_ip}`, `{user_agent}`
  - `{trace_id}`, `{request_path}`, `{request_method}`
  - `{hostname}`, `{service_name}`, `{service_version}`, `{environment}`

- Из бизнес-данных:
  - `{params.field_name}` — любое поле Params через точечную нотацию
  - `{state.field_name}` — любое поле state после аспекта
  - `{result.field_name}` — любое поле Result (только в global_finish)

- Из исключения (если есть):
  - `{exception.message}`, `{exception.type}`, `{exception.traceback}`

### Условная логика — функция `iif`

Для условной подстановки используется встроенная функция `iif`:

```
{iif(условие; значение_если_истина; значение_если_ложь)}
```

Условие — простое выражение с операторами `==`, `!=`, `>`, `<`, `>=`, `<=`, `and`, `or`, `not`. Операнды — переменные или литералы (строки в одинарных кавычках, числа, `true`/`false`). Интерпретатор поддерживает также встроенные функции: `len`, `upper`, `lower`, `format_number`.

**Пример: уровень серьёзности в тексте**

```
{iif(exception.type=='PaymentError'; '💳 ПЛАТЁЖ'; iif(exception.type=='FraudDetected'; '🚨 ФРОД'; '⚠️ ОШИБКА'))}
```

**Пример: пометка крупных транзакций**

```
{iif(params.amount > 100000; '[КРУПНАЯ СДЕЛКА] '; '')}{message}
```

**Пример: статус операции в конце строки**

```
{date} {time} | {action} | {message} | {iif(result.success == true; 'OK'; 'FAILED')}
```

**Создание внутренних тегов через `iif`**

Особенно мощное применение — генерация тегов прямо внутри шаблона. Теги потом используются для поиска и фильтрации в ELK или Grafana:

```
{date} {time} level={iif(exception.type != ''; 'ERROR'; iif(duration > 5.0; 'SLOW'; 'INFO'))} action={action} trace={trace_id} {message}
```

Или для JSON-логера с динамическими полями:

```json
{
  "level": "{iif(exception.type != ''; 'error'; 'info')}",
  "risk": "{iif(params.amount > 1000000; 'high'; iif(params.amount > 100000; 'medium'; 'low'))}",
  "channel": "{iif(params.channel == 'sms'; 'mobile'; 'web')}",
  "message": "{message}"
}
```

Здесь `risk` и `channel` — теги которые разработчик **не пишет в коде**. Они генерируются координатором из данных запроса по правилам конфигурации [6].

### Фильтры через регулярные выражения

Каждый логер может иметь фильтр — регулярное выражение, которое применяется к **сериализованному контексту** сообщения перед рендерингом. Если регулярка не совпала — логер пропускает сообщение.

Фильтр работает по полному контексту в виде строки `"ключ=значение ключ=значение ..."`. Это позволяет фильтровать по любому сочетанию условий:

```python
# Только ошибки платёжной системы
TelegramLogger(filter_regex=r"exception\.type=PaymentError")

# Только для premium-пользователей
FileLogger(filter_regex=r"roles=.*premium.*")

# Только медленные операции (> 3 секунды)
AlertLogger(filter_regex=r"duration=[3-9]\.|duration=\d{2,}\.")

# Только конкретный action и только ошибки
SlackLogger(filter_regex=r"action=.*CheckoutAction.*exception\.type=\w+")

# Исключить технические действия из подробного лога
VerboseLogger(filter_regex=r"^(?!.*action=.*HealthCheck).*$")

# Реагировать на конкретные бизнес-события
DirectorTelegramLogger(filter_regex=r"params\.amount=\d{7,}")  # суммы от 10 млн
```

Фильтр — это не уровень логирования. Логер с `level=INFO` и `filter_regex` пропустит INFO-сообщение только если оно ещё и совпадает с регулярным выражением. Оба условия должны выполняться одновременно.

---

### Форматы вывода

Логеры не ограничены текстом с отступами. Формат определяется конфигурацией и может быть любым.

**Консольный с отступами (для разработки):**

```
→ ProcessOrderAction
  → [validate_user] Проверка пользователя user_id=42
  ← [validate_user] OK 0.003s
  → [charge_card] Списание 1500.0
    → ChargeCardAction
      → [validate] Карта валидна
      ← [validate] OK 0.001s
    ← ChargeCardAction 0.450s
  ← [charge_card] OK 0.451s
← ProcessOrderAction result=order_id=1001 0.512s
```

**JSON (для ELK/Datadog):**

```json
{
  "timestamp": "2025-06-15T14:32:01Z",
  "level": "info",
  "action": "ProcessOrderAction",
  "aspect": "charge_card",
  "nest_level": 1,
  "duration_ms": 451,
  "user_id": "42",
  "trace_id": "abc-123",
  "risk": "low",
  "message": "Списание средств",
  "params.amount": 1500.0
}
```

**XML (для корпоративных систем):**

```xml
<log-entry timestamp="2025-06-15T14:32:01Z" level="INFO">
  <action>ProcessOrderAction</action>
  <aspect>charge_card</aspect>
  <user id="42" trace="abc-123"/>
  <message>Списание средств</message>
  <duration unit="ms">451</duration>
</log-entry>
```

**Syslog RFC 5424:**

```
<134>1 2025-06-15T14:32:01Z hostname service - abc-123 [action="ProcessOrderAction" user="42"] Списание средств
```

**Telegram Markdown:**

```
🔴 *ProcessOrderAction* провалился
👤 Пользователь: `42`
💰 Сумма: `1,500,000 ₽`
❌ Ошибка: `InsufficientFunds`
🔗 Trace: `abc-123`
```

Формат задаётся либо строкой шаблона (`format="..."`) либо именем предустановленного формата (`format="json"`, `format="xml"`, `format="syslog"`). Для нестандартных форматов разработчик создаёт собственный `FormatterAdapter` [6].

### Скоупы

Координатор поддерживает скоупы — возможность разделять логи разных потоков выполнения. Скоуп привязывается к текущему выполнению действия через `trace_id` и автоматически передаётся во все вложенные вызовы через `nest_level`.

Практическое применение: при расследовании инцидента можно включить `level=DEBUG` только для конкретного `trace_id` или `user_id`, не захламляя логи остальных пользователей [6].

---

### Вложенные вызовы и единый поток

Когда корневое действие вызывает дочернее через `deps.run_action()`, машина передаёт тот же координатор с увеличенным `nest_level` [2]. Все сообщения всего дерева выполнения идут в один поток с правильной вложенностью [3]:

```
→ ProcessOrderAction
  ℹ️ Начинаю обработку заказа
    → CalculateDiscountAction
      ℹ️ Скидка VIP: 10%
    ← CalculateDiscountAction
```

Это не лог плагина. Это лог бизнес-логики, написанный разработчиком в аспектах, автоматически выстроенный в дерево машиной [3].

---

### Обратная совместимость

Машина проверяет сигнатуру аспекта через `inspect.signature()`. Если аспект принимает шесть параметров — передаёт `log`. Если пять — вызывает без логера. Старый код не ломается [1].

### Как это выглядит в коде аспекта

Разработчик пишет одну строку. Куда она попадёт — определяет конфигурация:

```python
@aspect("Списание средств")
async def charge(self, params, state, deps, connections, log):
    log.info("Списание средств", amount=params.amount, card=params.card_last4)
    # ↑ Эта строка одновременно:
    # → попадает в консоль с отступом (если DEBUG)
    # → пишется в файл в JSON (если INFO)
    # → уходит в ELK с тегом risk=high (если amount > 100000)
    # → отправляется директору в Telegram (если PaymentError)
    # → сигнализирует в Slack (если выполняется > 5 сек)
```

Код аспекта не знает ни об одном из этих получателей [6].

---

### Почему это архитектурная теорема, а не фича

Логер со сквозным контекстом возможен только если выполнены три условия одновременно [3]:

1. **Единая точка исполнения** — `machine.run()`. Машина гарантированно присутствует при каждом вызове каждого аспекта. Некому создать контекстный логер в Django или FastAPI — нет единой точки.
2. **Формализованный конвейер с метаданными**. Машина знает `action_name`, `aspect_name`, `nest_level`, `duration`, `context` — и всё это гарантированные данные, потому что без них код не запустится [2].
3. **Context невидим для Actions**. Action не может получить контекст самостоятельно [2]. Но логер, созданный машиной, — видит всё.

Если в системе есть единая точка исполнения, формализованный конвейер с метаданными и изоляция контекста от бизнес-логики — то логер со сквозным контекстом следует из архитектуры как математическое следствие [3].

---

### Влияние

- **Бизнес-логика описывает *что* происходит. Машина знает *где, когда и в каком контексте*. Логер соединяет оба мира — без участия бизнес-логики** [3].
- **Принудительная формализация наблюдаемости** — так же как `@CheckRoles` принудительно формализует безопасность. Без `log` в сигнатуре разработчик либо не логирует вообще, либо логирует без контекста. С `log` — контекст бесплатный и всегда правильный [1].
- **Гибкость без изменения кода** — добавить новый канал, поменять формат, включить подробный лог для одного action — только конфигурация.
- **Богатый контекст автоматически** — `user_id`, `trace_id`, `nest_level`, `duration`, `params` подставляются машиной.
- **Условная логика прямо в шаблоне** — теги `risk=high`, `channel=mobile`, `status=FAILED` генерируются из данных без дополнительного кода.
- **Реакция на критические события** — Telegram-логер с `filter_regex` — это не просто лог. Это система раннего предупреждения, встроенная в инфраструктуру без отдельного кода мониторинга.
- **Любой формат** — консоль с отступами, JSON, XML, Syslog, Markdown — один и тот же вызов `log.info()` порождает нужный формат для каждого получателя.

---

### Что можно добавить (из roadmap)

- **Структурированные логи** — чтобы логгеры могли работать не только со строками, но и с JSON-объектами.
- **Автоматическое добавление контекста** — уже есть, но можно расширять.
- **Семплинг** — для высоконагруженных систем настроить вероятностную выборку сообщений.
- **Асинхронная отправка** — чтобы логирование не тормозило основное выполнение [6].

---

### Уникальность

Это не просто «логер с форматированием». Это **конфигурируемая шина логирования с языком шаблонов**, встроенная в инфраструктуру наблюдаемости AOA [6].

Ни один mainstream-фреймворк не предлагает:
- Множество независимых логеров с разными форматами, фильтрами и каналами из одной точки.
- Условную генерацию тегов через `iif` без изменения кода.
- Фильтрацию по произвольному контексту через регулярные выражения.
- Автоматический `nest_level` и `indent` из архитектуры машины.

Особая ценность при расследовании инцидентов: можно быстро добавить Telegram-логер с нужным `filter_regex`, получить мгновенные алерты, расследовать проблему — и убрать логер без единого изменения в коде приложения [6].

---

### Связь с другими идеями

- Координатор логеров усиливает плагинную систему [2]: плагины наблюдают за выполнением, а координатор обеспечивает доставку наблюдений в нужные каналы с нужным форматом.

- В сочетании с телеметрией PostgreSQL (**idea_18**) шаблон может включать `{params.shared_blks_read}` и отправлять алерт в Slack если конкретный action начал читать больше обычного.
- **ExecutionTreePlugin (idea_17)** и логер синхронизированы автоматически — оба получают данные из одного источника: машины. Логи и дерево выполнения всегда согласованы.
- **PostgresUnitOfWork (idea_03)** — логер может писать "Буфер: 47 команд, flush перед SELECT" с привязкой к `trace_id` и `action_name`.
- **Генерация тестов из production-логов (idea_08)** — LLM анализирует логи вместе с деревом выполнения и телеметрией БД для генерации тестов.

PostgresUnitOfWork — интеллектуальная буферизация команд

Проблема

В типичных приложениях при сохранении множества связанных записей (например, заказ + позиции заказа) разработчики пишут циклы с отдельными INSERT, что приводит к N+1 запросам к базе данных. Даже если используется executemany, он требует, чтобы все данные были известны заранее. В сложных сценариях, где команды генерируются в разных частях бизнес-логики (разные аспекты или даже разные действия), собрать их в один batch без дополнительной инфраструктуры невозможно [6].

В mainstream-фреймворках (Django ORM, SQLAlchemy) буферизация либо отсутствует, либо работает только на уровне сессии и скрыта от разработчика. Разработчик часто не контролирует, когда именно данные уходят в базу данных [6].

Решение

PostgresUnitOfWork — это ресурсный менеджер, реализующий паттерн Unit of Work с автоматической буферизацией и batch-выполнением. Он предоставляет два метода [6]:

execute(query, params) — добавляет команду во внутренний буфер, но не отправляет её в базу данных немедленно. Команды накапливаются до достижения лимита batch-размера или до явного вызова commit().

select(query, params) — выполняет запрос немедленно, но перед этим автоматически сбрасывает буфер (flush), чтобы гарантировать актуальность данных.

При вызове commit() все накопленные команды выполняются как один составной запрос (или в рамках одной транзакции), что радикально снижает количество сетевых вызовов и повышает производительность [6].

Как это работает

Аспект действия вызывает execute() для каждой строки, которую нужно вставить или обновить. Команда попадает в буфер, но в базу не отправляется. Если следующий аспект вызывает select(), UnitOfWork автоматически сбрасывает буфер (flush) перед выполнением запроса — это гарантирует, что SELECT увидит все предыдущие INSERT/UPDATE. При вызове commit() все оставшиеся команды в буфере отправляются как один batch [6].

Пример использования в аспектах:

```python
@aspect("Создать позиции заказа")
async def create_items(self, params, state, deps, connections):
    uow = connections["connection"]
    for item in params.items:
        await uow.execute(
            "INSERT INTO order_items (order_id, product_id, qty) VALUES ($1, $2, $3)",
            (state["order_id"], item.product_id, item.quantity)
        )
    # Ничего не ушло в БД — всё в буфере
    return state

@aspect("Проверить наличие товаров")
async def check_stock(self, params, state, deps, connections):
    uow = connections["connection"]
    # Автоматический flush перед SELECT — все INSERT из предыдущего аспекта
    # будут отправлены в БД перед выполнением запроса
    rows = await uow.select(
        "SELECT product_id, stock FROM products WHERE product_id = ANY($1)",
        ([item.product_id for item in params.items],)
    )
    state["stock"] = {r["product_id"]: r["stock"] for r in rows}
    return state

@summary_aspect("Завершить заказ")
async def finish(self, params, state, deps, connections):
    uow = connections["connection"]
    await uow.commit()  # Все оставшиеся команды уходят batch-ом
    return CreateOrderAction.Result(order_id=state["order_id"])
```

Политики автоматического сброса

UnitOfWork поддерживает несколько политик автоматического flush [6]:

По размеру буфера — когда накопилось N команд (например, 100), буфер сбрасывается автоматически. Это предотвращает переполнение памяти при большом количестве операций.

Перед SELECT — как описано выше, любой запрос на чтение автоматически вызывает flush. Это гарантирует консистентность данных без ручного управления.

По времени — опциональная политика для высоконагруженных систем, где буфер сбрасывается каждые N миллисекунд.

Интеграция с системой транзакций AOA

PostgresUnitOfWork интегрирован с иерархией ресурсов AOA [2][6]:

Корневое действие создаёт UnitOfWork и получает полный доступ к execute(), select(), commit(), rollback().

При передаче в дочернее действие через deps.run_action() UnitOfWork автоматически оборачивается в прокси через get_wrapper_class() [2]. Прокси разрешает execute() и select(), но запрещает commit() и rollback() — точно так же, как для обычных connection managers.

Дочернее действие добавляет команды в тот же буфер, но не может зафиксировать транзакцию. Все команды — и родительские, и дочерние — будут зафиксированы одним commit() в корневом действии.

Влияние

Производительность — вставка тысяч строк выполняется за один сетевой вызов вместо тысяч. Это особенно ценно для систем с высокими требованиями к пропускной способности [6].

Атомарность — все накопленные команды либо применяются, либо откатываются как единое целое. Это обеспечивается транзакционной семантикой PostgreSQL [6].

Прозрачность — разработчик видит, где команда попадает в буфер (execute), а где выполняется немедленно (select). Автоматический flush перед SELECT предотвращает неконсистентные чтения [6].

Тестируемость — буфер можно проверить на содержимое до отправки, не трогая реальную базу данных. В ActionTestMachine UnitOfWork подменяется моком, который накапливает команды в памяти [6].

Уникальность

В mainstream-фреймворках буферизация скрыта от разработчика (SQLAlchemy Session) или отсутствует (Django ORM). Ни один Python-фреймворк не предлагает явную, контролируемую буферизацию с автоматическим flush перед SELECT и интеграцией с системой прокси-обёрток для безопасной передачи между вложенными действиями [6].

Подход executemany требует собрать все параметры заранее, что невозможно, если команды порождаются динамически в разных частях кода (разные аспекты, разные вложенные действия). PostgresUnitOfWork решает эту проблему — команды накапливаются из любых точек конвейера и группируются автоматически [6].

Что можно добавить

Поддержка вложенных Unit of Work — чтобы дочерние действия могли добавлять команды в общий буфер верхнего уровня (уже частично реализовано через прокси).

Стратегии слияния команд — например, объединение нескольких UPDATE одной таблицы по одному ключу в один запрос.

Интеграция с трассировкой — чтобы через плагины видеть, какие команды были сгруппированы и сколько времени сэкономили.

Метрики буфера — размер буфера, количество flush-операций, среднее количество команд в batch — доступны через плагинную систему для мониторинга.

Связь с другими идеями

PostgresUnitOfWork усиливает иерархию ресурсов с прокси-безопасностью [2]: прокси-обёртка запрещает commit/rollback, но разрешает execute — и команды дочернего действия попадают в общий буфер. В сочетании с перехватчиками исключений (idea_01) обработчик ошибок в корневом действии получает доступ к реальному UnitOfWork и может выполнить rollback, даже если ошибка произошла глубоко во вложенном вызове.

# Основные концепции AOA и ActionEngine

Этот документ объясняет фундаментальные строительные блоки ActionEngine. Все остальные части архитектуры построены вокруг этих базовых концепций. Понимание этих сущностей позволяет легко читать и писать код в парадигме AOA.

---

## Action — атом бизнес-логики

Action представляет собой отдельную бизнес-операцию. Он не хранит состояния, не знает о контексте запроса, не зависит от транспорта и инфраструктуры.

Action получает входные данные через Params, выполняет серию аспектов, использует зависимости через DI и возвращает результат через Result.

Ключевые свойства Action:

Stateless — никакого состояния внутри класса. Прозрачен — бизнес-логика доступна целиком в виде аспектов. Тестируем — легко покрывается юнит-тестами. Композируем — его можно вызывать из других действий через deps.run_action().

Action — это ядро приложения. Он описывает что должно быть сделано, но не как именно.

---

## Params — входные данные действия

Params — это неизменяемый dataclass, представляющий входные бизнес-данные действия. Это всегда чистые данные, без логики.

Свойства Params:

Immutable — не меняется после создания. Самодокументируем — описывает что нужно действию. Валидируемый чекерами через декораторы. Типобезопасный.

Params отделяет бизнес-данные от инфраструктурных — контекста, ресурсов, транспорта.

```python
from dataclasses import dataclass
from ActionMachine.Core.BaseParams import BaseParams

@dataclass(frozen=True)
class CreateOrderParams(BaseParams):
    user_id: int
    product_id: int
    quantity: int
```

---

## Result — результат выполнения

Result — это неизменяемая структура результата, возвращаемая summary-аспектом.

Свойства Result:

Immutable. Типизирован. Явный контракт — что возвращает действие. Легко сериализуется.

```python
from dataclasses import dataclass
from ActionMachine.Core.BaseResult import BaseResult

@dataclass(frozen=True)
class CreateOrderResult(BaseResult):
    order_id: int
    total: float
```

Result показывает что именно делает данное действие.

---

## Aspect — этап выполнения действия

Аспект — это метод Action, помеченный декоратором @aspect или @summary_aspect.

@aspect — обычный этап выполнения. @summary_aspect — финальный этап, который возвращает Result.

Аспект получает:

params — входные данные. state — временное состояние. deps — зависимости.

Аспекты вызываются строго в порядке определения в классе — сверху вниз. Это делает поведение действия предсказуемым и линейным.

```python
from ActionMachine.Core.AspectMethod import aspect, summary_aspect

@aspect("Проверка наличия товара")
async def check_stock(self, params, state, deps):
    repo = deps.get(ProductRepository)
    product = await repo.get(params.product_id)
    if product.stock < params.quantity:
        raise ValueError("Недостаточно товара на складе")
    state["product"] = product
    return state

@summary_aspect("Создание заказа")
async def create_order(self, params, state, deps):
    return CreateOrderResult(
        order_id=42,
        total=state["product"].price * params.quantity
    )
```

---

## State — передача данных между аспектами

State — это обычный Python dict, передаваемый от аспекта к аспекту.

Он используется для накопления промежуточных данных, передачи результатов подготовки и хранения временных вычислений. State существует только во время одного вызова Action и не сохраняется между вызовами.

Важно понимать: state не накапливается автоматически. Каждый аспект получает предыдущий state, но явно возвращает только нужные поля. ActionMachine полностью заменяет state на то, что вернул аспект.

State — единственный mutable-объект в AOA.

---

## Plugin — наблюдатель за выполнением

Плагины — это чистые наблюдатели. Они получают события global_start, before:aspect, after:aspect, global_finish, могут логировать, собирать метрики, делать аудит, но не могут менять поведение Action и не влияют на результат.

Плагины вызываются ActionMachine автоматически. Работают асинхронно. Безопасны — ошибки в плагине не ломают бизнес-процесс при ignore_exceptions=True.

Плагин — это наблюдатель, а не участник процесса.

---

## DependencyFactory — DI без магии

DependencyFactory создаёт и кеширует зависимости, указанные в Action через @depends.

Особенности:

Нет контейнеров, модулей и сложной конфигурации. Каждый Action явно объявляет свои зависимости. Фабрика создаёт объекты по требованию и кеширует их в рамках одного вызова. Поддерживает вложенные действия через run_action.

```python
from ActionMachine.Core.AspectMethod import depends

@depends(ProductRepository)
@depends(EmailService)
class CreateOrderAction(BaseAction):
    ...
```

DI в ActionEngine — это простой и предсказуемый механизм композиции, а не магический контейнер.

---

## Resource Manager — адаптер к внешним системам

Ресурсный менеджер — это порт или адаптер к внешнему миру: БД, внешнему API, файловой системе, очередям, сервисам.

Ресурсные менеджеры могут хранить состояние — соединения, сессии, кэши. Не содержат бизнес-логики. Предоставляют домену унифицированный интерфейс. Изолируют легаси или внешний сервис.

В AOA ресурсы — это инфраструктура. Actions — это бизнес-логика.

```python
class IssueTracker(ABC):
    @abstractmethod
    def get_issue(self, issue_id: str) -> Issue: ...

    @abstractmethod
    def create_issue(self, title: str, body: str) -> Issue: ...
```

Action работает с портом, а не с конкретным адаптером. Это гарантирует независимость бизнес-логики от реализации ресурса.

---

## Context — информация о запросе

Context создаётся на входе в систему — HTTP, MCP, CLI — и содержит данные пользователя, роли, метаданные запроса, окружение и дополнительные поля.

Action не видит Context. Плагины — видят.

Это разделяет бизнес-логику и транспорт. Действие никогда не зависит от способа доставки запроса.

Context состоит из трёх частей:

UserInfo — данные пользователя: user_id, roles, extra. RequestInfo — данные запроса: trace_id, path, method, client_ip, user_agent. EnvironmentInfo — данные окружения: hostname, service_name, environment.

---

## ActionMachine — исполнитель

Машина запускает действие через единственную точку входа machine.run(action, params).

Машина проверяет роли, вызывает плагины, выполняет аспекты строго по порядку, применяет чекеры, вызывает summary-аспект, управляет вложенностью и измеряет время выполнения.

ActionMachine не содержит бизнес-логики — только правила конвейера.

---

## Четыре типа данных в AOA

AOA разделяет данные по назначению:

Context — данные процесса запроса. Только для плагинов и машины. Params — вход бизнес-логики. Неизменяем. State — временный буфер конвейера. Существует только внутри run. Result — итог работы. Неизменяем.

Это предотвращает хаос типов и смешивание смыслов.

---

## Порт и адаптер

Action работает с интерфейсами — портами, а не с конкретными реализациями. Это позволяет легко заменять адаптеры, мокировать зависимости в тестах, изолировать легаси-монстров и переписывать инфраструктуру без изменения логики.

---

## Композиция действий

Action может вызвать другое Action через:

```python
result = await deps.run_action(OtherAction, OtherAction.Params(...))
```

Это позволяет строить сложные сценарии из маленьких переиспользуемых операций и избегать дублирования логики.

idea_05_action_interrogation.md

Action Interrogation — допрос действия (мультиассерт)

Проблема

В классическом тестировании комплексный тест с пятью assert-проверками падает на первом неуспешном — и остальные четыре проверки не выполняются. Разработчик видит одну ошибку, чинит её, запускает снова — видит следующую. Цикл «починил → запустил → увидел следующую» повторяется столько раз, сколько сломанных проверок. Это убивает продуктивность [5].

В AOA эта проблема особенно болезненна, потому что setup теста — это описание бизнес-сценария: создание моков, конфигурация DI, выполнение конвейера аспектов через machine.run [2]. Дублировать этот setup в пяти отдельных тестах — значит пять раз описывать одну и ту же реальность.

Решение

Один запуск действия — все проверки независимы. Хелпер multi_assert принимает результат выполнения действия и список проверок (лямбд или callable), выполняет каждую в try/except, собирает все ошибки и в конце выбрасывает агрегированное исключение с полным отчётом. Ни одна проверка не блокирует выполнение остальных [5].

Результат — Action Interrogation: допрос действия. Опиши реальность один раз, а потом задай все вопросы к ней сразу.

Пример

```python
@pytest.mark.asyncio
async def test_process_payment_full():
    fake_gateway = FakePaymentGateway()
    fake_email = FakeEmailService()

    machine = ActionTestMachine({
        UserRepository: FakeUserRepository(),
        PaymentGateway: fake_gateway,
        EmailService: fake_email,
    })

    result = await machine.run(
        ProcessPaymentAction(),
        ProcessPaymentAction.Params(
            user_id=1, card_token="tok_test", amount=100.0
        )
    )

    multi_assert(
        lambda: assert_eq(result.transaction_id, "txn_fake_123",
                          "ID транзакции"),
        lambda: assert_eq(result.amount, 102.0,
                          "Сумма с комиссией 2%"),
        lambda: assert_eq(fake_gateway.charged_amount, 102.0,
                          "Сумма списания с карты"),
        lambda: assert_eq(fake_email.sent_count, 1,
                          "Отправлено одно письмо"),
        lambda: assert_eq(fake_email.last_recipient, "user@example.com",
                          "Получатель письма"),
    )
```

Если проваливаются второй и четвёртый assert, разработчик видит:

```
MultiAssertError: 2 из 5 проверок провалены:

  ❌ [2/5] Сумма с комиссией 2%:
     Ожидалось: 102.0
     Получено:  100.0

  ❌ [4/5] Отправлено одно письмо:
     Ожидалось: 1
     Получено:  0

  ✅ [1/5] ID транзакции
  ✅ [3/5] Сумма списания с карты
  ✅ [5/5] Получатель письма
```

Все пять проверок выполнены. Все ошибки видны сразу. Разработчик чинит оба бага за один цикл.

Как реализуется

Хелпер multi_assert — это примерно 30 строк кода:

```python
class MultiAssertError(AssertionError):
    def __init__(self, failures, total):
        report = f"{len(failures)} из {total} проверок провалены:\n"
        for idx, desc, err in failures:
            report += f"\n  ❌ [{idx}/{total}] {desc}:\n     {err}\n"
        super().__init__(report)

def multi_assert(*checks):
    failures = []
    for i, check in enumerate(checks, 1):
        try:
            check()
        except AssertionError as e:
            desc = getattr(check, '__doc__', f'check {i}')
            failures.append((i, desc, e))
    if failures:
        raise MultiAssertError(failures, len(checks))
```

Альтернативно, можно использовать существующие open-source решения:

pytest-check (pip install pytest-check) — плагин pytest с check.equal(), который собирает все провалы и выводит в конце.

softest — аналогичная библиотека с «мягкими» assert-ами.

Интеграция с pytest-check:

```python
from pytest_check import check

@pytest.mark.asyncio
async def test_process_payment_full():
    # ... setup и run ...

    with check:
        assert result.transaction_id == "txn_fake_123"
    with check:
        assert result.amount == 102.0
    with check:
        assert fake_gateway.charged_amount == 102.0
    with check:
        assert fake_email.sent_count == 1
    with check:
        assert fake_email.last_recipient == "user@example.com"
```

Все проверки выполнятся, все ошибки попадут в отчёт.

Почему это особенно ценно в AOA

За цену одного теста (один setup, один machine.run) разработчик получает 5-10 независимых проверок [2]. Экономия на setup-фазе, которая может быть дорогой: создание моков, конфигурация DI, выполнение конвейера аспектов с чекерами и авторизацией [2].

Цикл «починить → запустить → увидеть следующую» сокращается с N итераций до одной. Все провалы видны сразу.

Покрытие тестами растёт без пропорционального роста количества тестов. Один мультиассерт-тест покрывает столько же, сколько 5 отдельных тестов с одинаковым setup.

Читаемость: тест остаётся единым сценарием — «выполнить платёж → проверить всё» — а не разбитым на пять функций с дублированным setup.

Каждый assert проверяет отдельный бизнес-инвариант — не просто значение переменной, а «комиссия начислена правильно», «письмо отправлено», «получатель верный». Описания assert-ов (desc) автоматически становятся частью документации теста — в духе самодокументируемости AOA [5].

Связь с другими идеями

В сочетании с генерацией тестов из production-логов (idea_08) LLM-агент, генерирующий тесты из дерева выполнения, может автоматически создавать мультиассерт-блоки — по одной проверке на каждое поле Result и на каждый побочный эффект (вызов мока) [5][6].

Обнаружение мёртвых тестов (idea_13) упрощается, когда мультиассерт снижает количество тестовых функций — меньше «шума», проще выявлять холостые тесты [6].

Постпроверки эффектов (idea_10) — мультиассерт естественный формат для постпроверок, где каждый побочный эффект проверяется независимо [6].

Влияние

Очень высокая востребованность — каждый разработчик, который когда-либо писал тест с тремя assert-ами и тратил три цикла на починку, оценит мгновенную выгоду. Это не требует изменения мышления — только замена assert на multi_assert или with check. Порог входа — нулевой.

Уникальность

Идея мягких assert-ов существует в мире тестирования (pytest-check, softest, NUnit Assert.Multiple), но ни один фреймворк не интегрирует её с семантикой бизнес-операций. В AOA это приобретает особую ценность: ActionTestMachine уже обеспечивает единую точку запуска с полным конвейером [2], а каждый assert проверяет отдельный бизнес-инвариант с человекочитаемым описанием [5]. Тест превращается из «проверки одного факта» в допрос бизнес-операции — один запуск, десять вопросов, все ответы сразу.

idea_06_env_manager.md

Ресурсный менеджер для переменных окружения (EnvManager)

Проблема

В типичных приложениях доступ к переменным окружения разбросан по всему коду: os.getenv вызывается в разных модулях, сервисах, конфигурационных файлах. Это приводит к дублированию, отсутствию типизации, сложности тестирования и невозможности понять, от каких переменных зависит конкретная бизнес-операция. При смене источника конфигурации (например, переход с переменных окружения на Vault или etcd) приходится менять код во множестве мест [6].

Решение

Создать ресурсный менеджер EnvManager, который инкапсулирует доступ к переменным окружения и предоставляет чистый, типизированный интерфейс для действий. Вместо прямых вызовов os.getenv действие объявляет зависимость от EnvManager через @depends и получает значения через типизированные методы: get_str, get_int, get_bool, get_float [6].

Как это работает

EnvManager реализуется как обычный ресурсный менеджер, доступный через DI [2]. Действие объявляет зависимость:

```python
@depends(EnvManager, description="Доступ к переменным окружения")
@CheckRoles(CheckRoles.ANY, desc="Доступно аутентифицированным пользователям")
class ProcessPaymentAction(BaseAction):

    @aspect("Получить конфигурацию")
    async def load_config(self, params, state, deps, connections):
        env = deps.get(EnvManager)
        state["api_url"] = env.get_str("PAYMENT_API_URL", required=True)
        state["timeout"] = env.get_int("PAYMENT_TIMEOUT", default=30)
        state["debug"] = env.get_bool("PAYMENT_DEBUG", default=False)
        return state

    @summary_aspect("Выполнить платёж")
    async def execute(self, params, state, deps, connections):
        # state["api_url"], state["timeout"], state["debug"] уже доступны
        ...
```

EnvManager предоставляет следующие методы:

get_str(key, default=None, required=False) — возвращает строковое значение. Если required=True и переменная не найдена — выбрасывает исключение.

get_int(key, default=None, required=False) — возвращает целое число. Если значение не может быть преобразовано в int — выбрасывает исключение с понятным сообщением.

get_bool(key, default=None, required=False) — интерпретирует строки "true", "1", "yes" как True, "false", "0", "no" как False. Регистронезависимо.

get_float(key, default=None, required=False) — аналогично get_int, но для чисел с плавающей точкой.

get_list(key, separator=",", default=None) — разбивает строковое значение по разделителю и возвращает список строк.

Каждый метод выбрасывает типизированное исключение (ConfigurationError) с указанием имени переменной и причины ошибки, что упрощает отладку.

Тестирование

В тестах EnvManager подменяется через ActionTestMachine [2] точно так же, как любая другая зависимость:

```python
class MockEnvManager:
    def __init__(self, values):
        self._values = values

    def get_str(self, key, default=None, required=False):
        if key in self._values:
            return str(self._values[key])
        if required:
            raise ConfigurationError(f"Variable {key} not set")
        return default

    def get_int(self, key, default=None, required=False):
        val = self.get_str(key, default=default, required=required)
        return int(val) if val is not None else None

    # ... аналогично для остальных методов


machine = ActionTestMachine({
    EnvManager: MockEnvManager({
        "PAYMENT_API_URL": "https://test.api.com",
        "PAYMENT_TIMEOUT": "10",
        "PAYMENT_DEBUG": "true",
    })
})

result = await machine.run(ProcessPaymentAction(), params)
```

Тесты не зависят от реальных переменных окружения. Каждый тест может задать свой набор значений, включая отсутствие переменных для проверки обработки ошибок.

Расширяемость

EnvManager определяется как интерфейс (порт). Реализация по умолчанию читает os.environ, но при необходимости можно создать адаптеры для других источников конфигурации:

OsEnvManager — стандартная реализация, читает os.environ.

VaultEnvManager — получает секреты из HashiCorp Vault.

EtcdEnvManager — читает конфигурацию из etcd.

DotenvEnvManager — загружает переменные из .env файла.

FileEnvManager — читает из YAML/JSON конфигурационного файла.

Действия не меняются при смене источника — они работают с интерфейсом EnvManager, а конкретная реализация подставляется через DI при создании машины или при регистрации в транспортном слое [6].

Влияние

Прозрачность — зависимости от переменных окружения становятся частью декларации действия через @depends [2], а не размазаны по коду. Открывая класс действия, разработчик сразу видит, что оно зависит от EnvManager.

Тестируемость — в тестах можно подставить MockEnvManager с фиксированными значениями, не трогая реальные переменные окружения. Не нужно манипулировать os.environ в setUp/tearDown.

Безопасность — исключается случайное чтение несуществующих переменных. Метод с required=True гарантирует, что отсутствие критической переменной будет обнаружено немедленно, с понятным сообщением об ошибке.

Единообразие — все действия получают конфигурацию одинаково, через deps.get(EnvManager). Нет разброса между os.getenv, settings.VARIABLE, config["key"] в разных частях кода.

Гибкость — при необходимости перейти на централизованное хранилище конфигурации (Vault, etcd) достаточно заменить реализацию EnvManager, не трогая ни одного действия.

Типизация — get_int, get_bool, get_float автоматически преобразуют строковые значения в нужные типы и выбрасывают понятные ошибки при невозможности преобразования. Это устраняет целый класс багов, связанных с ручным парсингом строк.

Уникальность

В mainstream-фреймворках работа с переменными окружения остаётся «сырой» [6]. Django предлагает settings.py, FastAPI — Pydantic BaseSettings, но ни один из них не интегрирует конфигурацию с DI на уровне бизнес-действий. В AOA EnvManager — это полноценная зависимость, объявленная через @depends, подменяемая в тестах через ActionTestMachine [2] и заменяемая на альтернативный источник без изменения кода действий.

Связь с другими идеями

EnvManager естественно интегрируется с декларативной генерацией транспорта (idea_04): при регистрации действия в FastAPIRegistry или MCPRegistry можно передать конкретную реализацию EnvManager через resources, что делает конфигурацию частью композиционного корня [6]. В сочетании с координатором логеров (idea_02) значения из EnvManager (например, environment, service_name) могут автоматически подставляться в шаблоны логов через контекст [6].

idea_07_action_timeouts.md

Таймауты действий и контроль времени выполнения

Проблема

В реальных приложениях бизнес-операции зависят от внешних систем: баз данных, API, файловых хранилищ. Любой из этих ресурсов может «зависнуть» — ответить через 30 секунд вместо 100 миллисекунд, или не ответить вовсе. В традиционных фреймворках таймауты настраиваются на уровне отдельных вызовов (timeout в HTTP-клиенте, statement_timeout в PostgreSQL), но нет механизма контроля времени выполнения целой бизнес-операции с учётом всех её шагов и вложенных вызовов [6].

Если действие CreateOrderAction состоит из пяти аспектов и двух вложенных вызовов, каждый из которых укладывается в свой таймаут, но суммарно они занимают 15 секунд — пользователь всё равно ждёт. При этом ни один из отдельных вызовов не нарушил свой лимит. Проблема в том, что никто не контролирует общее время бизнес-операции.

Решение

Для любого Action можно задать максимальное время выполнения (timeout) на уровне декоратора класса или при регистрации в транспортном слое. ActionMachine проверяет общее время после каждого аспекта и при превышении реагирует в одном из двух режимов [6]:

Режим логирования (soft timeout) — машина записывает предупреждение через плагинную систему, но выполнение продолжается. Это полезно для мониторинга производительности без влияния на пользователей.

Режим исключения (hard timeout) — машина выбрасывает TimeoutException, действие прерывается. Если действие управляет транзакцией, корневое действие или обработчик ошибок (idea_01) может выполнить rollback.

Как это работает

Декоратор задаёт параметры таймаута на уровне класса:

```python
@timeout(max_duration=5.0, mode="exception")
@depends(PaymentGateway, description="Платёжный шлюз")
@CheckRoles(["admin", "operator"], desc="Доступно операторам")
class ProcessPaymentAction(BaseAction):

    @aspect("Валидация карты")
    async def validate_card(self, params, state, deps, connections):
        gateway = deps.get(PaymentGateway)
        result = await gateway.validate(params.card_token)
        return {"card_valid": result}

    @aspect("Списание средств")
    async def charge(self, params, state, deps, connections):
        gateway = deps.get(PaymentGateway)
        txn = await gateway.charge(params.card_token, params.amount)
        return {"txn_id": txn}

    @summary_aspect("Завершение платежа")
    async def finish(self, params, state, deps, connections):
        return ProcessPaymentAction.Result(
            txn_id=state["txn_id"],
            success=True
        )
```

ActionMachine проверяет время после завершения каждого аспекта. Если суммарное время превысило max_duration:

В режиме "log" — генерируется событие on_timeout для плагинов с полным контекстом (action_name, params, state, duration, nest_level, aspect_name), и выполнение продолжается.

В режиме "exception" — выбрасывается TimeoutException с указанием, на каком аспекте произошло превышение и сколько времени было затрачено. Если в действии есть обработчик @catch(TimeoutException), он получит управление и сможет выполнить компенсирующие действия.

Проверка выполняется между аспектами, а не внутри них. Это означает, что если один аспект занимает 10 секунд, а таймаут — 5, то аспект завершится полностью, и только после этого машина обнаружит превышение. Для прерывания долгих операций внутри аспекта разработчик использует стандартные механизмы asyncio (asyncio.wait_for, asyncio.timeout) [6].

Таймауты для вложенных действий

Когда корневое действие вызывает дочернее через deps.run_action(), оставшееся время передаётся как ограничение для дочернего действия. Если корневому действию осталось 2 секунды из 5, дочернее действие получит лимит в 2 секунды, даже если на нём самом стоит таймаут в 10 секунд. Минимальное значение побеждает [6].

Это гарантирует, что суммарное время всего дерева выполнения не превысит таймаут корневого действия.

Переопределение при регистрации

При регистрации действия в транспортном слое (FastAPIRegistry, MCPRegistry) можно переопределить таймаут:

```python
registry.register(
    ProcessPaymentAction,
    path="/payments",
    timeout=10.0,
    timeout_mode="exception",
    resources={...}
)
```

Это позволяет иметь разные таймауты для одного и того же действия в разных контекстах: например, 5 секунд для API и 60 секунд для фонового задания.

События для плагинов

ActionMachine генерирует событие on_timeout, которое содержит:

- action_name — полное имя действия
- params — входные параметры
- state — текущее состояние на момент превышения
- duration — фактическое время выполнения
- max_duration — установленный лимит
- aspect_name — имя последнего завершённого аспекта
- nest_level — уровень вложенности
- mode — "log" или "exception"

Плагин может подписаться на это событие для мониторинга:

```python
@on('on_timeout', '.*', ignore_exceptions=True)
async def on_timeout(self, state_plugin, event):
    alert(f"⏱ {event.action_name} превысил {event.max_duration}s "
          f"(фактически {event.duration:.2f}s, "
          f"последний аспект: {event.aspect_name})")
    return state_plugin
```

Влияние

Защита от зависаний — внешние системы могут отвечать непредсказуемо долго. Таймаут на уровне бизнес-операции гарантирует, что пользователь не будет ждать бесконечно.

Соблюдение SLA — если API должен отвечать за 3 секунды, таймаут в 2.5 секунды на действие с запасом на сериализацию обеспечивает выполнение обязательств.

Раннее обнаружение регрессий — в режиме "log" можно отслеживать, какие действия начали работать медленнее, до того как это станет проблемой для пользователей.

Тестирование производительности — в тестовом окружении таймауты помогают находить медленные сценарии до попадания в production. ActionTestMachine может запускать действия с теми же таймаутами, что и production.

Уникальность

Существующие фреймворки контролируют таймауты на уровне отдельных вызовов (HTTP-запрос, SQL-запрос), но не на уровне бизнес-операции целиком [6]. В AOA таймаут охватывает всё дерево выполнения: все аспекты, все вложенные вызовы, все обращения к ресурсам — и при этом работает через ту же плагинную систему, что и остальная наблюдаемость [2].

Интеграция с @catch (idea_01) позволяет обрабатывать TimeoutException контекстно-зависимо: выполнить rollback, отправить уведомление, вернуть частичный результат.

Интеграция с координатором логеров (idea_02) позволяет настроить отдельный шаблон логирования для таймаутов с полным контекстом.

Связь с другими идеями

Перехватчики исключений (idea_01) обрабатывают TimeoutException как любое другое исключение — с доступом к deps, state и connections. Обработчик может выполнить rollback транзакции и вернуть альтернативный Result.

Координатор логеров (idea_02) может иметь отдельный шаблон для таймаутов: "{time} | TIMEOUT | {action_name} | {duration}s > {max_duration}s | aspect={aspect_name}".

PostgresUnitOfWork (idea_03) при TimeoutException в режиме "exception" автоматически сбрасывает буфер через rollback, предотвращая потерю данных.

Action Interrogation (idea_05) — мультиассерт-тесты могут включать проверку времени выполнения как один из независимых assert-ов.

idea_08_production_test_generation.md

Генерация тестов из production-логов (Self-Healing Tests)

Проблема

В большинстве проектов тесты пишутся на основе предположений разработчика о том, как будет использоваться система. Эти предположения часто расходятся с реальностью: пользователи вводят данные, которые никто не предвидел, вызывают операции в порядке, который не был запланирован, и сталкиваются с ошибками в сценариях, которые не покрыты тестами. Когда в production происходит инцидент, команда тратит часы на расследование и написание регрессионного теста вручную. К этому моменту контекст уже частично утрачен [6].

Решение

Плагин ProductionCollectorPlugin собирает все выполнения действий в production в виде структурированных JSON-деревьев с полным контекстом: параметры, состояние на каждом шаге, вложенные вызовы, ошибки, роли пользователей. LLM-агент анализирует эти данные и автоматически генерирует unit-тесты, которые воспроизводят реальные сценарии [5][6].

Как это работает

Сбор данных

ProductionCollectorPlugin подписывается на все события ActionMachine через стандартный декоратор @on [2]. На каждое выполнение действия он собирает:

- полное имя действия (action_name)
- типизированные Params
- state после каждого аспекта (state_aspect)
- Result (для успешных выполнений)
- исключение (для неуспешных)
- duration каждого аспекта и общая
- nest_level для вложенных вызовов
- роли пользователя из Context [2]

Все эти данные уже доступны через PluginEvent [2] — плагину не нужно ничего дополнительно извлекать.

Результат сбора — JSON-дерево одного выполнения:

```json
{
  "action": "myapp.actions.ProcessPaymentAction",
  "params": {"user_id": 42, "card_token": "tok_live_abc", "amount": 150.0},
  "roles": ["user", "premium"],
  "aspects": [
    {
      "name": "validate_user",
      "state_after": {"user_valid": true},
      "duration": 0.003
    },
    {
      "name": "charge_card",
      "state_after": {"txn_id": "txn_789"},
      "duration": 0.450
    }
  ],
  "result": {"transaction_id": "txn_789", "amount": 153.0, "success": true},
  "duration": 0.512,
  "children": [
    {
      "action": "myapp.actions.CalculateCommissionAction",
      "params": {"amount": 150.0},
      "result": {"commission": 3.0},
      "duration": 0.002
    }
  ]
}
```

Генерация тестов

LLM-агент получает дерево и генерирует тест, используя ActionTestMachine [2]:

```python
@pytest.mark.asyncio
async def test_process_payment_real_scenario_42():
    """
    Сгенерировано из production-лога.
    Пользователь 42 (premium) оплатил 150.0, комиссия 3.0.
    """
    machine = ActionTestMachine(
        {CalculateCommissionAction: CalculateCommissionAction.Result(commission=3.0)},
        context=Context(user=UserInfo(user_id="42", roles=["user", "premium"]))
    )

    result = await machine.run(
        ProcessPaymentAction(),
        ProcessPaymentAction.Params(
            user_id=42, card_token="tok_test", amount=150.0
        )
    )

    multi_assert(
        lambda: assert_eq(result.transaction_id, "txn_789",
                          "ID транзакции"),
        lambda: assert_eq(result.amount, 153.0,
                          "Сумма с комиссией"),
        lambda: assert_eq(result.success, True,
                          "Успешность операции"),
    )
```

LLM знает структуру теста, потому что:

- Params типизированы — он видит имена полей и типы [2]
- Result типизирован — он знает, что проверять [2]
- Вложенные вызовы видны — он может подставить MockAction с правильным результатом [2]
- CheckRoles видны — он задаёт правильные роли в Context [2]
- State на каждом шаге виден — он может сгенерировать тесты для отдельных аспектов [5]

Три категории генерируемых тестов

Регрессионные тесты из ошибок. Каждое неуспешное выполнение (где дерево содержит исключение) автоматически превращается в тест, который воспроизводит ошибку. Когда разработчик исправит баг, тест начнёт проходить — и останется как регрессионный.

Тесты из редких путей. LLM анализирует распределение параметров и находит необычные комбинации: граничные значения, редкие роли, нетипичные последовательности вложенных вызовов. Эти сценарии превращаются в тесты, которые разработчик вряд ли написал бы сам.

Тесты из успешных сценариев. Типичные успешные выполнения превращаются в «золотые тесты» — эталонные сценарии, которые должны проходить всегда. Если после рефакторинга один из них упадёт, значит поведение изменилось.

Влияние

Покрытие тестами растёт автоматически без усилий разработчика. Каждый день production генерирует новые сценарии, и каждый из них может стать тестом.

Регрессионные тесты появляются одновременно с багами, а не после расследования. Время между «баг обнаружен» и «тест написан» сокращается с часов до минут.

Тесты основаны на реальных данных, а не на предположениях. Разработчик больше не гадает «а какие параметры тут могут быть» — он видит реальные данные из production.

Качество тестов выше, потому что LLM видит полный контекст: не только вход и выход, но и промежуточные состояния, длительности, вложенные вызовы [5].

Уникальность

Ни один другой фреймворк не имеет структурированного семантического дерева выполнения с типизированными параметрами и состоянием, необходимыми для такой генерации [5]. Обычные логи — это поток текста, из которого невозможно восстановить структуру бизнес-процесса. AOA даёт полное дерево с семантикой каждого узла из коробки [5].

ActionTestMachine [2] с MockAction [2] и поддержкой альтернативных реальностей делает сгенерированные тесты корректными — они проходят тот же конвейер (аспекты, чекеры, роли), что и production-код.

В сочетании с Action Interrogation (idea_05) каждый сгенерированный тест может содержать multi_assert блок, проверяющий сразу все поля Result и все побочные эффекты — один запуск, полная проверка.

Востребованность

Огромная — каждая команда тратит часы на расследование инцидентов и написание регрессионных тестов. Автоматизация этого процесса экономит дни разработки и повышает качество. Особенно ценно для команд с высокой частотой релизов, где ручное написание регрессионных тестов не успевает за скоростью изменений.

Связь с другими идеями

Координатор логеров (idea_02) обеспечивает доставку данных от ProductionCollectorPlugin в нужные каналы с нужным форматом [6].

Action Interrogation (idea_05) — LLM автоматически генерирует multi_assert блоки, проверяющие все аспекты результата независимо.

Обнаружение мёртвых тестов (idea_13) — данные из того же ProductionCollectorPlugin используются для обратного анализа: какие существующие тесты покрывают сценарии, которые никогда не встречаются в production.

Декларативная генерация транспорта (idea_04) — JSON-схемы Params и Result, используемые для генерации API, также используются LLM для понимания структуры параметров при генерации тестов [6].

idea_09_auto_rollback_mode.md

Режим автоматического отката транзакций для тестов (Auto-Rollback Mode)

Проблема

При тестировании действий, которые работают с реальной базой данных, каждый тест оставляет после себя записи: вставленные строки, обновлённые поля, созданные заказы. Это приводит к нескольким болезненным последствиям. Тесты перестают быть идемпотентными — повторный запуск может упасть из-за дублирования данных. Нужен отдельный механизм очистки базы после каждого прогона (truncate, fixtures, snapshots). Невозможно безопасно запускать тесты на копии production-базы — есть риск засорить реальные данные. В CI/CD тесты начинают влиять друг на друга, если запускаются параллельно [6].

Решение

При использовании ActionTestMachine все ресурсные менеджеры, поддерживающие транзакции, работают в режиме автоматического отката. Когда действие вызывает commit(), фактически выполняется rollback(). Бизнес-логика проходит полный конвейер — аспекты, чекеры, авторизация, вложенные вызовы — но транзакция никогда не фиксируется [6].

Как это работает

ActionTestMachine оборачивает каждый ресурсный менеджер, поддерживающий транзакции, в специальный прокси AutoRollbackWrapper. Этот прокси реализует тот же интерфейс (IConnectionManager), но подменяет поведение одного метода:

commit() вместо фиксации выполняет rollback(). Все остальные методы — open(), execute(), rollback() — делегируются реальному менеджеру без изменений [6].

Это означает, что:

Все INSERT, UPDATE, DELETE выполняются реально — данные попадают в базу и видны внутри транзакции. SELECT после INSERT вернёт вставленные строки. Вложенные действия, получающие прокси через DependencyFactory._wrap_connections() [2], также работают внутри той же транзакции. Но при завершении действия commit() превращается в rollback(), и все изменения исчезают.

Пример использования

```python
@pytest.mark.asyncio
async def test_create_order_auto_rollback():
    # Реальное соединение с тестовой БД
    real_db = PostgresConnectionManager({
        "host": "localhost",
        "port": 5432,
        "user": "test",
        "password": "test",
        "database": "shop_test"
    })

    # ActionTestMachine автоматически оборачивает в AutoRollbackWrapper
    machine = ActionTestMachine(
        mocks={},
        context=Context(user=UserInfo(roles=["admin"])),
        auto_rollback=True,  # включает режим автоотката
        connections={"connection": real_db}
    )

    result = await machine.run(
        CreateOrderAction(),
        CreateOrderAction.Params(
            user_id=1, product_id=10, quantity=2
        )
    )

    # Проверяем результат — действие выполнилось полностью
    assert result.order_id > 0
    assert result.notification_sent is True

    # Но в базе ничего не осталось — commit был заменён на rollback
    # Следующий тест начинает с чистой базы
```

Взаимодействие с PostgresUnitOfWork

Если используется PostgresUnitOfWork (idea_03), автооткат работает на уровне финального commit(). Буферизация команд происходит как обычно — execute() накапливает команды, select() вызывает flush, но итоговый commit() подменяется на rollback() [6]. Все накопленные и сброшенные команды откатываются вместе с транзакцией.

Взаимодействие с иерархией ресурсов

Прокси-обёртки для вложенных действий [2] работают поверх AutoRollbackWrapper. Цепочка выглядит так:

Корневое действие получает AutoRollbackWrapper (commit → rollback). Дочернее действие получает WrapperConnectionManager поверх AutoRollbackWrapper (commit → TransactionProhibitedError). execute() делегируется через цепочку к реальному соединению. Реальное соединение выполняет запрос внутри транзакции.

На любом уровне вложенности данные видны внутри транзакции, но никогда не фиксируются.

Режимы активации

Явный — через параметр auto_rollback=True при создании ActionTestMachine.

Через конфигурацию — через переменную окружения AOA_TEST_AUTO_ROLLBACK=true, которую читает EnvManager (idea_06).

Через декоратор теста — если добавить декоратор @auto_rollback на тестовую функцию, который настраивает машину автоматически.

Влияние

Идемпотентность тестов без дополнительного кода. Не нужны fixtures, truncate, tearDown — каждый тест начинает с чистой базы.

Безопасность. Можно запускать тесты на копии production-базы без риска засорить данные. Это полезно для воспроизведения production-багов в тестовом окружении.

Параллелизм. Тесты не влияют друг на друга, потому что каждый работает в своей транзакции, которая откатывается.

Реалистичность. В отличие от полного мокирования, бизнес-логика выполняется с реальной базой данных — реальные SQL-запросы, реальные constraint-проверки, реальные индексы. Это ловит баги, которые моки пропускают.

Скорость. rollback() быстрее, чем truncate + re-seed, особенно на больших базах.

Уникальность

Идея автоотката транзакций в тестах не нова — Django TestCase использует похожий механизм. Но в AOA она приобретает особую ценность благодаря интеграции с иерархией ресурсов [2]. Прокси-обёртки автоматически обеспечивают безопасность на вложенных уровнях: дочернее действие не может случайно зафиксировать транзакцию ни в обычном режиме (TransactionProhibitedError), ни в режиме автоотката [2][6].

В сочетании с PostgresUnitOfWork (idea_03) автооткат покрывает даже буферизованные batch-операции — они выполняются при flush, но откатываются при финальном commit.

В сочетании с Action Interrogation (idea_05) один тест с автооткатом может проверить десятки бизнес-инвариантов — все результаты SELECT внутри транзакции видны, все assert-ы выполняются независимо, и при этом база остаётся чистой.

Связь с другими идеями

PostgresUnitOfWork (idea_03) — буферизованные команды сбрасываются при flush, но откатываются при финальном commit.

Перехватчики исключений (idea_01) — если действие падает с ошибкой, обработчик @catch может выполнить rollback явно. В режиме автоотката это безвредно — rollback на уже откатываемой транзакции не вызывает ошибок.

Action Interrogation (idea_05) — мультиассерт идеально сочетается с автооткатом: один запуск действия с реальной базой, десять независимых проверок, ноль следов в базе.

Генерация тестов из production-логов (idea_08) — сгенерированные тесты могут автоматически использовать режим автоотката, чтобы безопасно воспроизводить production-сценарии на копии базы.

Таймауты действий (idea_07) — если действие превысило таймаут, незафиксированная транзакция автоматически откатывается при закрытии соединения, обеспечивая ту же идемпотентность.

idea_10_post_checks_effects.md

Постпроверки эффектов (Post-checks for Resources)

Проблема

В классическом тестировании разработчик проверяет только то, что вернуло действие — Result. Но бизнес-операция часто производит побочные эффекты: вставляет записи в базу данных, отправляет письма, обновляет кэш, создаёт файлы. Эти эффекты остаются непроверенными, если разработчик не написал отдельные assert-ы вручную после каждого вызова. В результате тест может пройти успешно (Result корректен), но в базе данных может не оказаться нужной записи, письмо может не уйти, а файл может быть пустым [6].

Проблема усугубляется тем, что для проверки побочных эффектов нужен доступ к ресурсам — к тем же репозиториям, сервисам и менеджерам соединений, которые использовало действие. В обычных фреймворках этот доступ приходится организовывать вручную, дублируя конфигурацию DI из основного кода в тестах.

Решение

В AOA все ресурсы уже доступны через DI [2], а ActionTestMachine даёт удобный способ их подмены [2]. Постпроверки используют тот же механизм: после выполнения действия разработчик может обратиться к любой зависимости через фабрику и проверить её состояние. Это не требует отдельной инфраструктуры — достаточно написать assert-ы к мокам или фейкам, которые уже переданы в ActionTestMachine.

Как это работает

Постпроверки выполняются после machine.run() и имеют доступ ко всем зависимостям, переданным в ActionTestMachine. Поскольку моки и фейки — обычные Python-объекты с состоянием, разработчик может проверить любое поле, список вызовов, содержимое буфера.

Пример

```python
@pytest.mark.asyncio
async def test_create_order_with_post_checks():
    fake_order_repo = FakeOrderRepository()
    fake_email = FakeEmailService()
    fake_inventory = FakeInventoryService()

    machine = ActionTestMachine(
        {
            OrderRepository: fake_order_repo,
            EmailService: fake_email,
            InventoryService: fake_inventory,
        },
        context=Context(user=UserInfo(roles=["manager"]))
    )

    result = await machine.run(
        CreateOrderAction(),
        CreateOrderAction.Params(
            user_id=1, product_id=10, quantity=2
        )
    )

    # Проверка Result — как обычно
    assert result.order_id > 0
    assert result.success is True

    # Постпроверки побочных эффектов
    # 1. Запись в базе данных действительно создана
    assert fake_order_repo.last_saved_order is not None
    assert fake_order_repo.last_saved_order["user_id"] == 1
    assert fake_order_repo.last_saved_order["product_id"] == 10
    assert fake_order_repo.last_saved_order["quantity"] == 2

    # 2. Письмо отправлено правильному получателю
    assert fake_email.sent_count == 1
    assert fake_email.last_recipient == "[EMAIL_REDACTED]"
    assert "заказ" in fake_email.last_subject.lower()

    # 3. Склад уменьшен на правильное количество
    assert fake_inventory.last_deducted_product == 10
    assert fake_inventory.last_deducted_quantity == 2
```

Как выглядят фейки для постпроверок

Фейки — это простые классы, которые реализуют интерфейс ресурса и записывают все вызовы во внутренние поля:

```python
class FakeOrderRepository:
    def __init__(self):
        self.last_saved_order = None
        self.save_count = 0

    def save(self, order_data):
        self.last_saved_order = order_data
        self.save_count += 1
        return {"id": 42, **order_data}


class FakeEmailService:
    def __init__(self):
        self.sent_count = 0
        self.last_recipient = None
        self.last_subject = None

    def send(self, to, subject, body):
        self.sent_count += 1
        self.last_recipient = to
        self.last_subject = subject


class FakeInventoryService:
    def __init__(self):
        self.last_deducted_product = None
        self.last_deducted_quantity = None

    def deduct(self, product_id, quantity):
        self.last_deducted_product = product_id
        self.last_deducted_quantity = quantity
```

Фейки не требуют специального базового класса или регистрации — они просто передаются в ActionTestMachine как обычные зависимости [2].

Сочетание с Action Interrogation

Постпроверки идеально сочетаются с мультиассерт-режимом (idea_05). Каждый побочный эффект проверяется как независимый assert, и все ошибки видны сразу:

```python
multi_assert(
    # Result
    lambda: assert_eq(result.order_id, 42,
                      "ID заказа"),
    lambda: assert_eq(result.success, True,
                      "Успешность"),

    # Побочный эффект: база данных
    lambda: assert_eq(fake_order_repo.save_count, 1,
                      "Заказ сохранён в БД"),
    lambda: assert_eq(fake_order_repo.last_saved_order["quantity"], 2,
                      "Количество в заказе"),

    # Побочный эффект: email
    lambda: assert_eq(fake_email.sent_count, 1,
                      "Письмо отправлено"),
    lambda: assert_eq(fake_email.last_recipient, "[EMAIL_REDACTED]",
                      "Получатель письма"),

    # Побочный эффект: склад
    lambda: assert_eq(fake_inventory.last_deducted_quantity, 2,
                      "Списание со склада"),
)
```

Один запуск действия, семь независимых проверок — три на Result и четыре на побочные эффекты. Все ошибки видны в одном отчёте.

Влияние

Тесты становятся глубже — проверяются не только возвращаемые значения, но и реальные последствия бизнес-операции.

Упрощается написание интеграционных тестов, потому что не нужно отдельно конфигурировать доступ к ресурсам — они уже доступны через DI [2].

Повышается уверенность в корректности взаимодействия с ресурсами. Если действие вернуло правильный Result, но не записало заказ в базу — постпроверка это обнаружит.

Фейки служат двойную роль: они и подменяют реальные ресурсы (позволяя действию выполниться без инфраструктуры), и записывают все обращения (позволяя проверить побочные эффекты).

Уникальность

В AOA все ресурсы уже доступны через DI [2], а ActionTestMachine даёт удобный способ их подмены [2]. Постпроверки не требуют отдельного фреймворка или специальной инфраструктуры — они являются естественным продолжением тестирования в AOA. Фейк — это одновременно мок для DI и журнал побочных эффектов для проверки.

В других фреймворках для проверки побочных эффектов обычно нужно либо поднимать реальную базу данных и делать SELECT после теста, либо использовать сложные mock-библиотеки (unittest.mock, MagicMock) с assert_called_with. В AOA достаточно простого фейка с полями, потому что DI-система прозрачна и подконтрольна разработчику [2].

Связь с другими идеями

Action Interrogation (idea_05) — мультиассерт превращает постпроверки из последовательности assert-ов в параллельный «допрос», где каждый побочный эффект проверяется независимо.

Auto-rollback Mode (idea_09) — при использовании реальной базы данных постпроверки могут выполнять SELECT внутри транзакции, которая откатится после теста. Данные видны внутри транзакции, но не загрязняют базу.

Генерация тестов из production-логов (idea_08) — LLM-агент, генерирующий тесты, может автоматически добавлять постпроверки для каждого ресурса, который использовало действие. Он видит, какие зависимости объявлены через @depends [2], и создаёт фейки с проверками.

Таймауты действий (idea_07) — постпроверки могут включать проверку того, что таймаут не был превышен, как один из независимых assert-ов.

Востребованность

Средне-высокая. Полезна для любых систем с побочными эффектами — а это практически все реальные приложения. Особенно ценна для систем с множеством ресурсов (БД + email + очередь + файлы), где ошибка в одном побочном эффекте может остаться незамеченной при проверке только Result.

idea_11_cyclic_call_prevention.md

Запрет циклических вызовов действий (Cyclic Call Prevention)

Проблема

Если ActionA вызывает ActionB через deps.run_action(), а ActionB прямо или косвенно вызывает ActionA, возникает бесконечная рекурсия. В обычных фреймворках это приводит к stack overflow в production — ошибка, которую сложно отловить на этапе тестирования, потому что она проявляется только при определённых комбинациях вызовов.

В контексте AOA циклические вызовы разрушают сразу несколько фундаментальных свойств архитектуры [5][6]:

Предсказуемость. Бизнес-процесс перестаёт быть линейным деревом и превращается в граф с циклами, который невозможно прочитать сверху вниз.

Принцип Минто. Пирамидальная структура (корень → дети → внуки) превращается в лабиринт. Уровни вложенности теряют смысл [5].

Транзакции. Иерархия ресурсов предполагает, что родитель владеет транзакцией, а ребёнок получает прокси [2]. При цикле невозможно определить, кто реальный владелец — транзакции «схлопнутся» [6].

Плагины. nest_level будет расти бесконечно, логи перестанут отражать реальную структуру вызовов, трассировка станет бессмысленной [2][5].

Семантическое дерево выполнения. Вся интроспекция, генерация тестов из production-логов, визуализация бизнес-процессов — всё это работает только с деревом, а не с графом [5].

Решение

Явная проверка в runtime без магии. В ActionProductMachine добавляется множество _call_stack, которое хранит классы действий, находящихся в текущем стеке вызовов. Перед выполнением каждого действия машина проверяет, нет ли его класса уже в стеке [5][6].

Как это работает

При каждом вызове machine.run() происходит следующее [5][6]:

Машина проверяет, содержится ли класс действия в _call_stack. Если да — немедленно выбрасывается TypeError с полной цепочкой вызовов.

Если нет — класс добавляется в _call_stack, и nest_level увеличивается.

Выполняется обычный конвейер: проверка ролей, аспекты, плагины, summary.

В блоке finally класс удаляется из _call_stack, и nest_level уменьшается.

Реализация:

```python
# В ActionProductMachine.__init__
self._call_stack: Set[Type] = set()

# В ActionProductMachine.run()
async def run(self, action, params, resources=None, connections=None):
    action_class = action.__class__

    if action_class in self._call_stack:
        chain = ' → '.join(
            c.__name__ for c in self._call_stack
        ) + f" → {action_class.__name__}"
        raise TypeError(
            f"Циклический вызов действий запрещён: {chain}"
        )

    self._call_stack.add(action_class)
    self._nest_level += 1
    try:
        # ... обычный конвейер ...
        return result
    finally:
        self._call_stack.discard(action_class)
        self._nest_level -= 1
```

Детали реализации:

_call_stack — множество (set), а не список, потому что важно только наличие класса в стеке, а не порядок. Это снижает накладные расходы [5].

Проверка выполняется перед добавлением класса в стек, чтобы обнаружить цикл до того, как он начнёт раскручиваться [5].

В сообщении об ошибке выводится полная цепочка вызовов, что сразу указывает разработчику на источник проблемы [5][6].

Сообщение об ошибке как документация

Сообщение содержит полную цепочку вызовов:

```
TypeError: Циклический вызов действий запрещён:
ProcessOrder → ChargeCard → ValidatePayment → ProcessOrder
```

Разработчик сразу видит, где и как возник цикл. Без дебаггера. Без анализа стека. Это продолжение философии AOA — «нет магии, всё явно» [5].

Что это даёт в совокупности

Три свойства вместе — одна точка входа/выхода, линейный конвейер аспектов, ацикличное дерево вызовов — создают архитектуру, которую можно полностью визуализировать [5]:

```
machine.run(ProcessOrder, params)
│
├─ [aspect] validate_user
├─ [aspect] check_products
├─ [aspect] calculate
│   └─ machine.run(CalculateDiscount, ...)
│       ├─ [aspect] load_tier
│       └─ [summary] apply_discount
├─ [aspect] charge
│   └─ machine.run(ChargeCard, ...)
│       ├─ [aspect] validate_card
│       └─ [summary] process_payment
└─ [summary] create_order
```

Каждый узел — самодостаточная единица с контрактом (Params/Result), с линейным конвейером, с проверкой ролей, с плагинами. И всё это читается как документ — сверху вниз, от корня к листьям [5].

Формально это DAG (directed acyclic graph), но на практике — именно дерево, потому что одно и то же действие может вызываться из разных мест (например, ValidateCardAction используется и в ProcessOrder, и в RecurringPayment), но рекурсивные вызовы запрещены [5][6].

Влияние

Гарантия предсказуемости. Бизнес-процесс всегда читается как документ — сверху вниз, от корня к листьям [5].

Безопасность транзакций. Иерархия ресурсов с прокси-обёртками работает корректно только при древовидной структуре вызовов [2].

Корректность плагинов. nest_level всегда отражает реальную глубину вложенности [2].

Тестируемость. Каждый узел дерева можно тестировать отдельно через ActionTestMachine [2].

Интроспекция. Статический анализ бизнес-процессов, генерация диаграмм, diff между версиями — всё это работает только с деревом [5].

Уникальность

Ни один Python-фреймворк не обеспечивает древовидную структуру бизнес-процессов на уровне runtime [5]. В Spring, Django, FastAPI вложенные вызовы сервисов могут образовывать произвольные графы — и разработчик узнает об этом только когда получит stack overflow или deadlock.

В AOA ацикличность — не рекомендация, а гарантия, обеспеченная проверкой в ActionProductMachine.run() [5][6]. Это добавляется к существующим гарантиям: линейность конвейера, обязательный CheckRoles, обязательный summary_aspect, контрактность state через чекеры [2][5].

Связь с другими идеями

Иерархия ресурсов с прокси-обёртками [2] — работает корректно только при древовидной структуре вызовов, потому что прокси предполагает, что родитель владеет транзакцией, а ребёнок получает безопасную обёртку.

Плагинная система с nest_level [2] — визуализация дерева выполнения имеет смысл только если структура действительно древовидная.

Генерация тестов из production-логов (idea_08) — ProductionCollectorPlugin собирает деревья выполнения; циклы сделали бы эти деревья бесконечными и непригодными для анализа.

Декларативная генерация транспорта (idea_04) — MCP-инструменты и FastAPI-эндпоинты строятся из метаданных Actions; ацикличность гарантирует, что граф зависимостей между Actions конечен и визуализируем.

Таймауты действий (idea_07) — оставшееся время передаётся дочерним действиям через дерево вызовов; циклы сделали бы этот механизм бесполезным.

idea_12_introspector.md

Интроспектор бизнес-процессов (AOA Introspector)

Проблема

В типичных проектах бизнес-логика размазана по контроллерам, сервисам, моделям и middleware. Чтобы ответить на простой вопрос — «какие бизнес-операции есть в системе, кто имеет к ним доступ и какие данные через них проходят» — нужно вручную прочитать десятки файлов, мысленно собрать картину и записать её в Confluence, где она устареет через неделю.

Ни один существующий инструмент не анализирует бизнес-архитектуру приложения, потому что в обычном коде бизнес-намерения не формализованы. Они существуют только в головах разработчиков и в устаревшей документации.

Суть

В AOA каждое действие — это полностью интроспектируемая программа [2][5]. Все метаданные доступны через простой getattr, потому что они являются обязательными атрибутами, без которых код не запустится [5]. Интроспектор — это утилита, которая обходит все Action-классы в проекте и строит полную карту бизнес-процессов из метаданных, не выполняя ни одного действия.

Что доступно для интроспекции прямо сейчас

Для любого Action-класса можно машинно извлечь:

Список зависимостей с описаниями — из атрибута _dependencies, заполняемого декоратором @depends [2]. Каждая зависимость содержит класс, description и опциональную factory.

Ролевую модель — из атрибута _role_spec, заполняемого декоратором @CheckRoles [2]. Значение может быть "NO_ROLE", "ANY_ROLE", конкретной строкой роли или списком ролей.

Полный упорядоченный список шагов — из методов с атрибутом _is_aspect, отсортированных по co_firstlineno [2]. Каждый метод содержит _aspect_description и _aspect_type ('regular' или 'summary').

Контракт каждого шага — из атрибута _result_checkers на методе [2]. Каждый чекер содержит field_name, required, desc и типоспецифичные ограничения (min_value, max_value, min_length, max_length, format, not_empty).

Контракт входных данных — из атрибута _field_checkers на классе Params [2].

Объявленные connections — из атрибута _connections, заполняемого декоратором @connection [2]. Каждый элемент содержит key, class и description.

Полное имя класса — через метод get_full_class_name(), который возвращает строку вида module.path.ClassName [2].

Docstring класса — через стандартный __doc__.

Что строит интроспектор

Полная карта бизнес-процессов в JSON или Markdown

Интроспектор обходит все Action-классы в указанных модулях и для каждого строит структурированное описание:

```json
{
  "action": "myapp.actions.CreateOrderAction",
  "docstring": "Создание заказа с оплатой и уведомлением",
  "roles": ["admin", "manager"],
  "dependencies": [
    {"class": "UserRepository", "description": "Репозиторий пользователей"},
    {"class": "PaymentGateway", "description": "Платёжный шлюз"},
    {"class": "SendNotificationAction", "description": "Дочернее действие уведомлений"}
  ],
  "connections": [
    {"key": "connection", "class": "PostgresConnectionManager", "description": "Основная БД"}
  ],
  "aspects": [
    {
      "name": "validate_user",
      "description": "Проверка пользователя",
      "type": "regular",
      "checkers": [
        {"field": "user_id", "type": "IntFieldChecker", "required": true, "desc": "ID пользователя"}
      ]
    },
    {
      "name": "charge_card",
      "description": "Списание средств",
      "type": "regular",
      "checkers": [
        {"field": "txn_id", "type": "StringFieldChecker", "required": true, "desc": "ID транзакции"}
      ]
    },
    {
      "name": "finish",
      "description": "Создание заказа",
      "type": "summary",
      "checkers": []
    }
  ],
  "params_checkers": [
    {"field": "user_id", "type": "IntFieldChecker", "required": true, "desc": "ID пользователя", "min_value": 1},
    {"field": "card_token", "type": "StringFieldChecker", "required": true, "desc": "Токен карты", "not_empty": true}
  ]
}
```

Матрица доступа

Интроспектор автоматически генерирует таблицу Action × Role, собранную из _role_spec всех действий [2]:

```
Action                        Roles
CreateOrderAction             [admin, manager]
SendNotificationAction        ANY
DeleteUserAction              [admin]
PublicHealthCheck             NONE
```

Эта таблица генерируется из кода и всегда актуальна. Аудитор может проверить безопасность системы, не читая ни строки Python.

Граф зависимостей

Интроспектор строит направленный граф, где узлы — Action-классы, а рёбра — зависимости из _dependencies [2]:

```
CreateOrderAction
  → UserRepository (ресурс)
  → PaymentGateway (ресурс)
  → SendNotificationAction (действие)
    → EmailService (ресурс)

ProcessPaymentAction
  → PaymentGateway (ресурс)
  → ValidateCardAction (действие)
```

Граф позволяет визуализировать связи между бизнес-операциями и ресурсами, обнаруживать общие зависимости и оценивать связность системы.

Автоматическое обнаружение архитектурных проблем

Интроспектор может обнаруживать подозрительные паттерны на основе метаданных:

Действие с CheckRoles.NONE, зависящее от PaymentGateway — подозрительная безопасность. Платёжная операция доступна без аутентификации.

Действие без зависимостей, но со сложной логикой (8+ аспектов) — кандидат на декомпозицию.

Два действия с идентичным набором аспектов и чекеров — возможный дубликат.

Действие, зависящее от другого действия, которое имеет более строгие роли — потенциальная эскалация привилегий.

Connection с описанием, не совпадающим ни с одной зависимостью — возможная ошибка конфигурации.

Diff бизнес-процессов между версиями

Если генерировать карту при каждом коммите (например, в CI/CD), можно автоматически показывать изменения в бизнес-логике:

```
v1.2 → v1.3:

  CreateOrderAction:
    + added aspect: check_fraud → {risk_score: float}
    + added dependency: FraudDetectionService
    ~ changed roles: [admin, manager] → [admin, manager, operator]

  DeleteUserAction:
    - removed aspect: soft_delete
    ~ changed summary contract: added field 'archived_at'

  + NEW: RecurringPaymentAction
    roles: [admin]
    aspects: validate_subscription, charge, schedule_next
    depends: PaymentGateway, SubscriptionRepository
```

Это changelog бизнес-логики, а не кода. Ревьюер видит не «изменено 47 файлов», а «добавлен шаг проверки фрода, расширен доступ на операторов, появилась новая операция рекуррентных платежей».

Генерация диаграмм бизнес-процессов

Из метаданных можно автоматически сгенерировать PlantUML или Mermaid:

```
CreateOrderAction [admin, manager]
  ├── validate_user → {user_id: int}
  │     depends: UserRepository
  ├── charge_card → {txn_id: str}
  │     depends: PaymentGateway
  │     └── ValidateCardAction [ANY]
  │           └── check → {valid: bool}
  └── [summary] finish → Result
```

Это не лог выполнения (как у плагинов). Это статическая карта процесса, построенная до первого запуска. Менеджер продукта может прочитать её без знания Python.

Почему это возможно только в AOA

Все существующие инструменты анализируют код: типы, зависимости, покрытие, цикломатическую сложность. Но никто не анализирует бизнес-намерения, потому что в обычном коде они не формализованы [5].

В AOA бизнес-намерения формализованы принудительно — через обязательные desc на чекерах, обязательные description на depends, обязательный @CheckRoles [2][5]. И они верифицированы — машина не запустит код без них [5].

Это означает, что AOA — первый фреймворк, где возможен семантический анализ бизнес-архитектуры: не «сколько строк в файле», а «какие бизнес-операции существуют, кто имеет к ним доступ, какие данные проходят через каждый шаг, и как операции связаны друг с другом» [5].

Влияние

Документация всегда актуальна — генерируется из кода, не может устареть, потому что метаданные являются частью кода [5].

Аудит безопасности за минуты — матрица доступа показывает всё сразу.

Onboarding ускоряется радикально — новый разработчик видит полную карту бизнес-процессов, не читая код.

Code review на уровне бизнес-логики — diff показывает, что изменилось в бизнес-процессах, а не в файлах.

AI-интеграция — JSON-карта бизнес-процессов идеально подходит для LLM, который может объяснить систему, найти проблемы и предложить оптимизации [5].

Уникальность

Ни один Python-фреймворк не даёт формальную метамодель бизнес-процессов, доступную для машинного анализа. В Django, Spring, FastAPI бизнес-логика не имеет формальной структуры — она размазана по сервисам, контроллерам и моделям. AOA заставляет разработчика формализовать бизнес-намерения в машиночитаемом виде, и интроспектор превращает эти данные в инструменты, которых не существует в природе [5].

Связь с другими идеями

Декларативная генерация транспорта (idea_04) использует те же метаданные для генерации API-эндпоинтов — интроспектор и регистратор транспорта работают с одним и тем же набором данных [6].

Координатор логеров (idea_02) может использовать карту из интроспектора для автоматической настройки шаблонов логирования по модулям [6].

Генерация тестов из production-логов (idea_08) — LLM использует карту бизнес-процессов из интроспектора для понимания структуры действий при генерации тестов [5].

Запрет циклических вызовов (idea_11) — интроспектор может статически обнаружить потенциальные циклы в графе зависимостей до запуска кода [5][6].

Action Interrogation (idea_05) — интроспектор может автоматически генерировать шаблоны мультиассерт-тестов, зная контракты каждого аспекта.

Востребованность

Огромная — для любого проекта с более чем десятью Actions. Особенно ценно для команд, где есть разделение между разработчиками и аналитиками, для аудита безопасности и для проектов с высокими требованиями к документации (финтех, медтех, государственные системы).

idea_13_dead_test_detection.md

Обнаружение холостых тестов (Dead Test Detection)

Проблема

В зрелых проектах тестовая база разрастается до сотен и тысяч тестов. Часть из них проверяет сценарии, которые никогда не встречаются в production: граничные случаи, которые были актуальны два года назад, но давно перекрыты валидацией на фронтенде, или пути выполнения, которые исчезли после рефакторинга, но тесты остались. Эти «холостые» тесты создают несколько проблем: замедляют прогон CI/CD, создают ложное чувство покрытия, отвлекают внимание при падениях и увеличивают стоимость поддержки [6].

При этом разработчик не может отличить холостой тест от важного регрессионного без глубокого анализа production-данных. В обычных фреймворках таких данных просто нет — логи представляют собой поток текста, из которого невозможно восстановить структуру бизнес-процесса.

Решение

Используя данные из production, собранные тем же ProductionCollectorPlugin (idea_08), AOA анализирует, какие пути выполнения реально встречаются в production. Затем эти данные сравниваются с покрытием unit-тестов. Тесты, которые проверяют сценарии, ни разу не случившиеся в production и не являющиеся явно негативными, помечаются как холостые и предлагаются к удалению или пересмотру [6].

Как это работает

Сбор production-данных

ProductionCollectorPlugin уже собирает для каждого выполнения действия полный контекст [2][5]:

- полное имя действия (action_name)
- типизированные Params
- state после каждого аспекта
- Result или исключение
- вложенные вызовы с nest_level
- роли пользователя

Из этих данных извлекаются «пути выполнения» — уникальные комбинации:

- какое действие вызвано
- какие аспекты выполнились
- какие ветвления произошли (по значениям state после каждого аспекта)
- какие вложенные действия вызваны
- какой результат получен (успех/ошибка, тип ошибки)

Каждый путь представляется как отпечаток (fingerprint) — хеш от последовательности аспектов, ключей state и типа результата.

Сбор тестовых данных

При запуске тестов через ActionTestMachine с включённым TestCoveragePlugin каждый тест также генерирует отпечаток. Плагин подписывается на те же события, что и ProductionCollectorPlugin, и записывает:

- какое действие тестируется
- какие аспекты выполнились
- какие значения в state после каждого аспекта
- какой Result получен
- какие MockAction использовались

Сравнение

Утилита DeadTestDetector принимает два набора отпечатков — production и test — и классифицирует каждый тест:

Живой тест — его отпечаток совпадает с хотя бы одним production-путём. Этот тест проверяет реальный сценарий.

Негативный тест — тест явно проверяет ошибочный сценарий (assert raises, assert result.success is False). Такие тесты всегда считаются ценными, даже если соответствующая ошибка никогда не встречалась в production.

Холостой тест — отпечаток не совпадает ни с одним production-путём, и тест не является негативным. Это кандидат на удаление или пересмотр.

Редкий тест — отпечаток совпадает с production-путём, который встречался менее N раз за период. Такие тесты помечаются как «потенциально устаревающие».

Пример отчёта

```
Dead Test Detection Report
Period: 2025-01-01 — 2025-06-01
Production executions analyzed: 142,847

Tests analyzed: 347

  LIVE: 203 (58.5%)
    Покрывают реальные production-пути

  NEGATIVE: 61 (17.6%)
    Проверяют ошибочные сценарии (всегда ценны)

  DEAD: 52 (15.0%)
    Не встречались в production за 6 месяцев

  RARE: 31 (8.9%)
    Встречались менее 10 раз за период

Top 10 dead tests:
  1. test_create_order_with_expired_coupon
     Action: CreateOrderAction
     Last production match: never
     Recommendation: удалить или пересмотреть

  2. test_payment_with_zero_amount
     Action: ProcessPaymentAction
     Last production match: never
     Note: валидация на фронтенде блокирует этот сценарий с 2024-03
     Recommendation: удалить, если валидация фронтенда стабильна

  3. test_notification_sms_channel_unknown
     Action: SendNotificationAction
     Last production match: 2024-08-15 (294 days ago)
     Recommendation: пересмотреть актуальность
```

Интеграция с CI/CD

DeadTestDetector может запускаться как шаг CI/CD pipeline:

```yaml
- name: Detect dead tests
  run: python -m aoa_tools.dead_test_detector
    --production-data s3://logs/production/
    --test-results ./test-coverage.json
    --period 180  # дней
    --threshold 10  # минимум production-совпадений для RARE
    --output dead-tests-report.md
```

Отчёт может быть:

- записан как артефакт pipeline
- отправлен в Slack/Telegram как уведомление
- добавлен как комментарий к pull request
- использован для автоматической пометки тестов декоратором @pytest.mark.dead

Автоматическая пометка

При желании DeadTestDetector может генерировать файл конфигурации pytest, который помечает холостые тесты:

```python
# generated by dead_test_detector
dead_tests = [
    "test_create_order_with_expired_coupon",
    "test_payment_with_zero_amount",
]
```

Разработчик решает, что с ними делать: удалить, переписать или оставить с пометкой.

Влияние

Снижается стоимость поддержки тестовой базы. Холостые тесты — это не только время прогона, но и когнитивная нагрузка: при падении такого теста разработчик тратит время на расследование несуществующей проблемы.

Ускоряется CI/CD. Удаление 50 холостых тестов из 350 может сократить время прогона на 15-20%.

Команда фокусируется на действительно важных сценариях. Вместо «у нас 95% покрытие» разработчик видит «у нас 58% живых тестов, 17% негативных и 15% холостых» — это честная картина.

Обнаруживаются «тестовые долги» — тесты, которые проверяют устаревшую логику. Это сигнал, что код тоже может содержать мёртвые ветки.

Уникальность

Только AOA может предоставить структурированные данные о реальном выполнении бизнес-процессов с типизированными параметрами, состоянием и вложенностью, необходимые для такого анализа [5][6]. В обычных фреймворках для сравнения тестов с production-данными пришлось бы:

- парсить текстовые логи (ненадёжно)
- инструментировать каждый сервис отдельно (дорого)
- вручную маппить пути выполнения на тесты (нереалистично)

В AOA ProductionCollectorPlugin и TestCoveragePlugin используют одну и ту же плагинную систему [2], одни и те же события (global_start, before/after aspect, global_finish) [2] и одну и ту же структуру данных (PluginEvent с action_name, params, state_aspect, result, nest_level) [2]. Сравнение становится тривиальным, потому что оба набора данных уже структурированы одинаково.

Связь с другими идеями

Генерация тестов из production-логов (idea_08) — использует тот же ProductionCollectorPlugin. Dead Test Detection — обратная сторона той же медали: idea_08 создаёт новые тесты из production-данных, а Dead Test Detection находит старые тесты, которые больше не соответствуют production-реальности [6].

Action Interrogation (idea_05) — мультиассерт-тесты снижают количество тестовых функций, что уменьшает «шум» и упрощает анализ: меньше функций — проще классифицировать [5].

Интроспектор бизнес-процессов (idea_12) — карта всех Actions из интроспектора может использоваться для обнаружения Actions, которые вообще не покрыты тестами — обратная задача к Dead Test Detection [5].

Координатор логеров (idea_02) — DeadTestDetector может отправлять уведомления о холостых тестах через координатор логеров с контекстом (имя теста, последнее совпадение с production, рекомендация) [6].

Востребованность

Высокая для зрелых проектов с разросшейся тестовой базой. Особенно ценно для команд, которые тратят значительное время на поддержку тестов и расследование ложных падений. Помогает бороться с «тестовым шумом» — ситуацией, когда количество тестов создаёт иллюзию качества, но реальное покрытие бизнес-сценариев остаётся низким.

idea_14_feature_flags_actions.md

Управление фичами на уровне бизнес-операций (Feature Flags for Actions)

Проблема

В традиционных системах feature flags реализуются на уровне HTTP-эндпоинтов, конфигурационных файлов или условных ветвлений внутри кода. Это приводит к нескольким проблемам. Флаги размазаны по коду — условие `if feature_enabled("new_checkout")` появляется в контроллерах, сервисах, шаблонах. Нет централизованного понимания, какие бизнес-операции затронуты флагом. Невозможно быстро ответить на вопрос «какие фичи сейчас включены и на какие операции они влияют». При удалении флага приходится искать и удалять условия по всему коду. Тестирование с флагами требует дублирования setup-фазы для каждой комбинации.

В AOA бизнес-операции уже являются атомарными единицами с чёткими границами — Actions [2][5]. Это создаёт естественную точку для управления фичами: не на уровне строк кода, а на уровне целых бизнес-операций.

Решение

Декоратор @feature на уровне класса Action, который связывает действие с именованным флагом. ActionMachine проверяет состояние флага перед выполнением действия — до проверки ролей, до запуска аспектов, до вызова плагинов. Если флаг выключен, машина выбрасывает FeatureDisabledError с понятным сообщением, или возвращает альтернативный результат — в зависимости от конфигурации.

Источник состояния флагов — FeatureFlagProvider — это ресурсный менеджер, объявляемый через @depends [2]. Реализация может быть любой: файл конфигурации, переменная окружения через EnvManager (idea_06), Redis, LaunchDarkly, база данных. Действие не знает, откуда берётся состояние флага — оно просто объявляет зависимость от фичи.

Как это работает

Разработчик помечает Action декоратором @feature с именем флага:

```python
@feature("new_checkout_flow", fallback_result=None)
@depends(OrderRepository, description="Репозиторий заказов")
@CheckRoles(["user"], desc="Доступно пользователям")
class NewCheckoutAction(BaseAction):

    @aspect("Валидация корзины")
    @IntFieldChecker("items_count", desc="Количество товаров")
    async def validate(self, params, state, deps, connections):
        ...
        return {"items_count": len(params.items)}

    @summary_aspect("Оформление заказа")
    async def checkout(self, params, state, deps, connections):
        ...
        return NewCheckoutAction.Result(order_id=42)
```

ActionMachine при вызове run() выполняет проверку в следующем порядке:

1. Проверка feature flag — если флаг выключен, действие не выполняется.
2. Проверка ролей — @CheckRoles [2].
3. Проверка connections — @connection [2].
4. Событие global_start для плагинов.
5. Выполнение аспектов.

Проверка флага происходит раньше всего остального, потому что если фича выключена, нет смысла проверять роли или готовить зависимости.

Поведение при выключенном флаге

Разработчик выбирает один из двух режимов при объявлении @feature:

Режим исключения (по умолчанию) — если fallback_result не указан, машина выбрасывает FeatureDisabledError с именем флага и именем действия. Транспортный слой (FastAPI, MCP) может перехватить это исключение и вернуть HTTP 404, 501 или соответствующий ответ.

Режим альтернативного результата — если указан fallback_result, машина возвращает его как Result действия. Аспекты не выполняются, плагины получают событие feature_disabled вместо global_start/global_finish.

```python
# Режим исключения — эндпоинт отвечает 501
@feature("experimental_ai_search")
class AISearchAction(BaseAction):
    ...

# Режим альтернативного результата — тихий откат на старую версию
@feature("new_pricing", fallback_result=PricingAction.Result(price=0, source="legacy"))
class NewPricingAction(BaseAction):
    ...
```

FeatureFlagProvider — источник состояния флагов

Состояние флагов абстрагировано через интерфейс:

```python
class FeatureFlagProvider(ABC):
    @abstractmethod
    def is_enabled(self, flag_name: str, context: Optional[Context] = None) -> bool:
        """
        Проверяет, включён ли флаг.
        
        Может учитывать контекст: роли пользователя, окружение,
        процент раскатки и т.д.
        """
        pass
```

Реализации:

EnvFeatureFlagProvider — читает переменные окружения через EnvManager (idea_06). Имя переменной формируется из имени флага: FEATURE_NEW_CHECKOUT_FLOW=true.

ConfigFileFeatureFlagProvider — читает YAML/JSON файл с флагами. Удобно для локальной разработки.

RemoteFeatureFlagProvider — обращается к внешнему сервису (LaunchDarkly, Unleash, Flagsmith). Кэширует результат на время выполнения действия.

PercentageFeatureFlagProvider — включает флаг для заданного процента пользователей, используя user_id из Context для детерминированного хэширования. Один и тот же пользователь всегда получает одинаковый результат.

FeatureFlagProvider регистрируется как зависимость машины, а не отдельных действий. Машина получает его при инициализации и использует для проверки всех флагов.

Событие для плагинов

Когда действие заблокировано флагом, ActionMachine генерирует событие feature_disabled:

```python
@on('feature_disabled', '.*', ignore_exceptions=True)
async def on_feature_disabled(self, state_plugin, event):
    log(f"⛔ {event.action_name} blocked by feature flag '{event.feature_name}' "
        f"for user {event.context.user.user_id}")
    return state_plugin
```

Событие содержит: action_name, feature_name, context, fallback_used (True/False). Это позволяет мониторить, как часто срабатывают флаги и какие пользователи их «попадают».

Интеграция с интроспектором

Интроспектор бизнес-процессов (idea_12) может извлекать информацию о флагах из атрибута _feature_spec, добавляемого декоратором @feature [5]. Это позволяет автоматически генерировать:

Карту фич — какие Actions привязаны к каким флагам.

Отчёт по раскатке — какие фичи включены, какие выключены, какие в процентной раскатке.

Предупреждения об устаревших флагах — если флаг включён для всех пользователей более 30 дней, интроспектор предлагает удалить декоратор @feature и оставить Action как обычный.

Влияние

Централизованное управление. Все feature flags привязаны к Actions — атомарным бизнес-операциям. Нет разбросанных if-условий по коду. Включить или выключить фичу — это включить или выключить конкретное действие.

Безопасная раскатка. Canary deployments на уровне бизнес-операций: новая версия CreateOrderAction для 5% пользователей, старая — для остальных. Прокси-слой не нужен — всё решается внутри машины.

Чистый код. Действие не содержит условной логики «если фича включена — делай так, иначе — эдак». Есть два Action: старый и новый. Флаг определяет, какой из них вызывается.

Мониторинг. Через плагинную систему [2] видно, как часто срабатывает каждый флаг, для каких пользователей, и есть ли ошибки в новой версии.

Тестируемость. Оба варианта (включён/выключен) тестируются одинаково через ActionTestMachine, без манипуляций с глобальным состоянием.

Уникальность

Существующие системы feature flags (LaunchDarkly, Unleash, django-waffle) работают на уровне HTTP-запросов или произвольных точек в коде. Ни одна из них не интегрирована с формальной моделью бизнес-операций. В AOA feature flag — это атрибут бизнес-операции, а не строчка кода. Это значит, что:

Интроспектор может построить полную карту фич из метаданных [5].
Декларативная генерация транспорта (idea_04) может автоматически скрывать эндпоинты выключенных фич из OpenAPI.
Генерация тестов из production-логов (idea_08) может группировать тесты по вариантам фич.
LLM-агент может спросить «какие фичи сейчас в тестировании» и получить ответ из метаданных Actions.

Связь с другими идеями

EnvManager (idea_06) — простейшая реализация FeatureFlagProvider через переменные окружения.

Декларативная генерация транспорта (idea_04) — регистратор может автоматически не создавать эндпоинт для действия с выключенным флагом, или добавлять пометку «experimental» в OpenAPI.

Интроспектор бизнес-процессов (idea_12) — извлекает информацию о флагах и генерирует карту фич.

Координатор логеров (idea_02) — отдельный шаблон для логирования срабатываний флагов.

Action Interrogation (idea_05) — мультиассерт-тесты для обоих вариантов фичи.

Генерация тестов из production-логов (idea_08) — LLM видит в дереве выполнения, какой вариант фичи использовался, и генерирует тесты для обоих.

idea_15_circuit_breaker_actions.md

Автоматический предохранитель на уровне бизнес-операций (Circuit Breaker for Actions)

Проблема

В реальных приложениях бизнес-операции зависят от внешних систем: платёжных шлюзов, API доставки, почтовых сервисов, баз данных. Когда внешняя система начинает сбоить, каждый запрос к ней занимает время (timeout), потребляет ресурсы (потоки, соединения) и ухудшает пользовательский опыт. Если платёжный шлюз не отвечает, сотни пользователей ждут по 30 секунд вместо мгновенного ответа «сервис временно недоступен».

Существующие реализации Circuit Breaker (pybreaker, resilience4j, Polly) работают на уровне отдельных вызовов — оборачивают конкретную функцию или HTTP-запрос. Но бизнес-операция — это не один вызов. Это конвейер аспектов, где сбой может произойти на любом шаге, и где важен контекст: какие параметры были, какой пользователь, какое действие. Существующие решения не знают ничего о бизнес-семантике вызова.

Решение

В AOA Circuit Breaker работает на уровне Action — атомарной бизнес-операции. Декоратор @circuit_breaker на классе действия задаёт параметры предохранителя: порог срабатывания, время восстановления и поведение при разомкнутой цепи. ActionMachine проверяет состояние предохранителя до начала выполнения — до проверки ролей, до запуска аспектов, до вызова плагинов.

Это принципиально другой уровень, чем обёртка вокруг HTTP-вызова. Предохранитель защищает не конкретный ресурс, а целую бизнес-операцию со всеми её шагами, зависимостями и вложенными вызовами.

Как это работает

Разработчик помечает Action декоратором @circuit_breaker:

```python
@circuit_breaker(
    failure_threshold=5,
    recovery_timeout=60.0,
    fallback_result=None
)
@depends(PaymentGateway, description="Платёжный шлюз")
@CheckRoles(["operator"], desc="Доступно операторам")
class ChargeCardAction(BaseAction):

    @aspect("Валидация карты")
    @BoolFieldChecker("valid", desc="Карта валидна")
    async def validate(self, params, state, deps, connections):
        gateway = deps.get(PaymentGateway)
        result = await gateway.validate(params.card_token)
        return {"valid": result}

    @summary_aspect("Списание")
    async def charge(self, params, state, deps, connections):
        gateway = deps.get(PaymentGateway)
        txn = await gateway.charge(params.card_token, params.amount)
        return ChargeCardAction.Result(txn_id=txn, success=True)
```

ActionMachine при вызове run() выполняет следующую последовательность:

1. Проверка circuit breaker — если предохранитель разомкнут (OPEN), действие не выполняется.
2. Проверка feature flag (idea_14) — если фича выключена, действие не выполняется.
3. Проверка ролей — @CheckRoles [2].
4. Проверка connections — @connection [2].
5. Событие global_start для плагинов.
6. Выполнение аспектов.

Три состояния предохранителя

CLOSED (замкнут, нормальная работа). Все вызовы проходят. Каждое неуспешное выполнение (исключение, не перехваченное @catch) увеличивает счётчик ошибок. Когда счётчик достигает failure_threshold — предохранитель переходит в OPEN.

OPEN (разомкнут, защита активна). Вызовы не проходят. ActionMachine немедленно возвращает fallback_result (если указан) или выбрасывает CircuitBreakerOpenError. Аспекты не выполняются, зависимости не создаются, ресурсы не расходуются. Через recovery_timeout секунд предохранитель переходит в HALF_OPEN.

HALF_OPEN (полуоткрыт, пробный вызов). Один вызов пропускается для проверки. Если он успешен — предохранитель возвращается в CLOSED, счётчик ошибок сбрасывается. Если неуспешен — предохранитель возвращается в OPEN, таймер recovery_timeout начинается заново.

Поведение при разомкнутом предохранителе

Разработчик выбирает один из двух режимов при объявлении @circuit_breaker:

Режим исключения (по умолчанию). Если fallback_result не указан, машина выбрасывает CircuitBreakerOpenError с именем действия и временем до следующей попытки. Транспортный слой (FastAPI, MCP) может перехватить это исключение и вернуть HTTP 503 Service Unavailable.

Режим альтернативного результата. Если указан fallback_result, машина возвращает его как Result действия. Аспекты не выполняются. Плагины получают событие circuit_breaker_open вместо global_start/global_finish.

```python
# Режим исключения — клиент получает 503
@circuit_breaker(failure_threshold=5, recovery_timeout=60.0)
class ChargeCardAction(BaseAction):
    ...

# Режим альтернативного результата — тихий откат
@circuit_breaker(
    failure_threshold=3,
    recovery_timeout=30.0,
    fallback_result=ChargeCardAction.Result(txn_id="", success=False)
)
class ChargeCardAction(BaseAction):
    ...
```

Хранение состояния предохранителя

Состояние предохранителя (текущее состояние, счётчик ошибок, время последнего сбоя) хранится в ActionProductMachine и привязано к классу действия. Это означает, что один экземпляр машины отслеживает состояние всех предохранителей для всех действий. При перезапуске приложения состояние сбрасывается — предохранители начинают с CLOSED.

Для распределённых систем (несколько инстансов приложения) состояние может быть вынесено в Redis или другое хранилище через CircuitBreakerStateProvider — ресурсный менеджер, объявляемый через @depends на уровне машины, а не отдельных действий.

События для плагинов

ActionMachine генерирует три типа событий, связанных с предохранителем:

circuit_breaker_open — предохранитель разомкнулся. Содержит: action_name, failure_count, threshold, context. Сигнал для мониторинга: внешняя система сбоит.

circuit_breaker_half_open — предохранитель перешёл в пробный режим. Содержит: action_name, recovery_timeout, context. Сигнал: система пытается восстановиться.

circuit_breaker_closed — предохранитель замкнулся. Содержит: action_name, context. Сигнал: система восстановилась.

```python
@on('circuit_breaker_open', '.*', ignore_exceptions=True)
async def on_open(self, state_plugin, event):
    alert(f"🔴 {event.action_name} circuit breaker OPEN "
          f"(failures: {event.failure_count}/{event.threshold})")
    return state_plugin
```

Интеграция с интроспектором

Интроспектор бизнес-процессов (idea_12) может извлекать параметры предохранителя из атрибута _circuit_breaker_spec, добавляемого декоратором @circuit_breaker [5]. Это позволяет автоматически генерировать:

Карту устойчивости системы — какие Actions защищены предохранителями, с какими порогами.

Отчёт по текущему состоянию — какие предохранители сейчас в OPEN, когда ожидается восстановление.

Предупреждения — действие зависит от внешней системы (@depends), но не имеет @circuit_breaker.

Тестирование

В ActionTestMachine предохранители можно контролировать явно:

```python
# Тест поведения при разомкнутом предохранителе

    mocks={PaymentGateway: FakePaymentGateway()},
    circuit_breaker_states={"ChargeCardAction": "OPEN"}
)
with pytest.raises(CircuitBreakerOpenError):
    await machine.run(ChargeCardAction(), params)

# Тест с fallback
machine = ActionTestMachine(
    mocks={PaymentGateway: FakePaymentGateway()},
    circuit_breaker_states={"ChargeCardAction": "OPEN"}
)
result = await machine.run(ChargeCardWithFallback(), params)

```

В сочетании с Action Interrogation (idea_05) можно проверить оба варианта в одном тесте через multi_assert.

Влияние

Защита пользователей. Вместо 30-секундного ожидания таймаута пользователь мгновенно получает ответ «сервис временно недоступен» или альтернативный результат.

Защита ресурсов. Когда предохранитель разомкнут, машина не создаёт зависимости, не открывает соединения, не запускает аспекты — нулевое потребление ресурсов на заведомо обречённые запросы.

Автоматическое восстановление. Через recovery_timeout система автоматически пробует восстановить работу. Не нужно ручное вмешательство.

Наблюдаемость. Через плагинную систему [2] видно, какие предохранители в каком состоянии, как часто они срабатывают и когда система восстанавливается.

Уникальность

Существующие Circuit Breaker работают на уровне отдельных вызовов — оборачивают функцию или метод. Они не знают, что вызов является частью бизнес-операции с авторизацией, контрактами, вложенными действиями и наблюдаемостью.

В AOA Circuit Breaker работает на уровне Action — атомарной бизнес-операции [5]. Это означает:

Предохранитель проверяется до выполнения любой логики — до ролей, до аспектов, до DI. Экономия ресурсов максимальна.

Интроспектор может построить карту устойчивости всей системы из метаданных [5].

Плагины видят состояние предохранителей с полным бизнес-контекстом — не просто «вызов X упал», а «действие ChargeCardAction для пользователя Y упало 5 раз за 10 секунд» [2].

Декларативная генерация транспорта (idea_04) может автоматически добавлять информацию о Circuit Breaker в OpenAPI — например, возможный ответ 503 с описанием.

Feature Flags (idea_14) и Circuit Breaker работают вместе: если фича выключена — действие не вызывается по решению разработчика; если предохранитель разомкнут — действие не вызывается по решению системы. Два уровня защиты через один механизм.

Связь с другими идеями

Перехватчики исключений (idea_01) — @catch обработчики не вызываются, когда предохранитель разомкнут, потому что аспекты не выполняются. Но если действие выполняется в режиме HALF_OPEN и падает, исключение проходит через @catch как обычно — и если @catch возвращает Result, предохранитель считает вызов успешным.

Таймауты действий (idea_07) — TimeoutException считается сбоем для счётчика предохранителя. Если действие регулярно превышает таймаут, предохранитель разомкнётся.

Координатор логеров (idea_02) — отдельный шаблон для событий предохранителя.

Интроспектор бизнес-процессов (idea_12) — автоматическая карта устойчивости системы.

Feature Flags (idea_14) — два уровня защиты (ручной и автоматический) через один механизм проверки перед выполнением.

Генерация тестов из production-логов (idea_08) — LLM видит события circuit_breaker_open в дереве выполнения и может генерировать тесты для сценариев деградации.

Востребованность

Очень высокая для любых систем, зависящих от внешних сервисов. Особенно ценно для e-commerce (платёжные шлюзы), логистики (API доставки), финтеха (банковские API) и любых микросервисных архитектур, где отказ одного сервиса не должен каскадно ронять остальные.

idea_16_rate_limiting_actions.md

Ограничение частоты вызовов на уровне бизнес-операций (Rate Limiting for Actions)

Проблема

В традиционных системах rate limiting реализуется на уровне HTTP-эндпоинтов, IP-адресов или API-ключей. Это работает для защиты инфраструктуры, но не для защиты бизнес-логики. Один и тот же эндпоинт может обслуживать разные бизнес-операции, а одна бизнес-операция может вызываться через разные транспорты (HTTP, MCP, CLI, внутренний вызов из другого действия).

Примеры проблем, которые не решает традиционный rate limiting:

Пользователь вызывает CreateOrderAction 100 раз в минуту через API — эндпоинт-лимит на /orders сработает, но если тот же пользователь вызывает то же действие через MCP-инструмент — лимит не применяется.

Внутренний сервис вызывает SendNotificationAction в цикле, создавая спам — нет никакого лимита, потому что вызов не проходит через HTTP.

Действие ChargeCardAction вызывается из пяти разных родительских действий — лимит на каждый эндпоинт считается отдельно, хотя бизнес-операция одна.

В AOA бизнес-операции уже являются атомарными единицами с чёткими границами — Actions [2][5]. Это создаёт естественную точку для ограничения частоты: не на уровне транспорта, а на уровне самой бизнес-операции, независимо от способа её вызова.

Решение

Декоратор @rate_limit на уровне класса Action, который задаёт максимальное количество вызовов за временной интервал. ActionMachine проверяет лимит перед выполнением действия — до проверки ролей, до запуска аспектов, до вызова плагинов. Если лимит превышен, машина выбрасывает RateLimitExceededError или возвращает альтернативный результат.

Ключевое отличие от традиционного подхода: лимит привязан не к транспорту (HTTP-эндпоинт), а к бизнес-операции (Action). Вызов через FastAPI, MCP, CLI или deps.run_action() — всё считается одинаково.

Как это работает

Разработчик помечает Action декоратором @rate_limit:

```python
@rate_limit(
    max_calls=10,
    period=60.0,
    scope="user",
    fallback_result=None
)
@depends(PaymentGateway, description="Платёжный шлюз")
@CheckRoles(["user"], desc="Доступно пользователям")
class ChargeCardAction(BaseAction):

    @aspect("Валидация карты")
    @BoolFieldChecker("valid", desc="Карта валидна")
    async def validate(self, params, state, deps, connections):
        gateway = deps.get(PaymentGateway)
        result = await gateway.validate(params.card_token)
        return {"valid": result}

    @summary_aspect("Списание")
    async def charge(self, params, state, deps, connections):
        gateway = deps.get(PaymentGateway)
        txn = await gateway.charge(params.card_token, params.amount)
        return ChargeCardAction.Result(txn_id=txn, success=True)
```

ActionMachine при вызове run() выполняет следующую последовательность:

1. Проверка rate limit — если лимит превышен, действие не выполняется.
2. Проверка circuit breaker (idea_15) — если предохранитель разомкнут, действие не выполняется.
3. Проверка feature flag (idea_14) — если фича выключена, действие не выполняется.
4. Проверка ролей — @CheckRoles [2].
5. Проверка connections — @connection [2].
6. Событие global_start для плагинов.
7. Выполнение аспектов.

Области действия лимита (scope)

Декоратор @rate_limit принимает параметр scope, который определяет, как группируются вызовы для подсчёта:

scope="global" — общий лимит на всё приложение. Все вызовы этого действия, от всех пользователей, суммируются. Полезно для защиты внешних API с жёстким лимитом.

scope="user" — лимит на одного пользователя. Используется user_id из Context для идентификации. Каждый пользователь имеет свой отдельный счётчик. Полезно для защиты от спама и абьюза.

scope="role" — лимит на роль. Все пользователи с ролью "operator" делят один счётчик. Полезно для ограничения команды операторов.

scope="action" — лимит на класс действия без привязки к пользователю. Эквивалентно "global", но семантически отличается — подчёркивает, что ограничение относится к самой операции.

scope — callable — функция, принимающая Context и возвращающая строку-ключ для группировки. Позволяет создавать произвольные области: по IP, по организации, по подписке.

```python
# Лимит по организации пользователя
@rate_limit(
    max_calls=100,
    period=3600.0,
    scope=lambda ctx: f"org:{ctx.get_extra('org_id')}"
)
class BulkExportAction(BaseAction):
    ...
```

Поведение при превышении лимита

Разработчик выбирает один из двух режимов при объявлении @rate_limit:

Режим исключения (по умолчанию). Если fallback_result не указан, машина выбрасывает RateLimitExceededError с именем действия, текущим значением счётчика и временем до сброса. Транспортный слой (FastAPI, MCP) может перехватить это исключение и вернуть HTTP 429 Too Many Requests.

Режим альтернативного результата. Если указан fallback_result, машина возвращает его как Result действия. Аспекты не выполняются. Плагины получают событие rate_limit_exceeded.

```python
# Режим исключения — клиент получает 429
@rate_limit(max_calls=5, period=60.0, scope="user")
class SendSMSAction(BaseAction):
    ...

# Режим альтернативного результата — тихое ограничение
@rate_limit(
    max_calls=100,
    period=3600.0,
    scope="global",
    fallback_result=ExportAction.Result(status="rate_limited", data=None)
)
class ExportAction(BaseAction):
    ...
```

Хранение состояния счётчиков

Состояние счётчиков (количество вызовов за период) хранится в RateLimitProvider — ресурсном менеджере, который регистрируется на уровне машины.

InMemoryRateLimitProvider — простейшая реализация с хранением в словаре. Подходит для одного инстанса приложения. При перезапуске счётчики сбрасываются.

RedisRateLimitProvider — реализация на основе Redis с атомарными инкрементами и TTL. Подходит для распределённых систем с несколькими инстансами.

SlidingWindowRateLimitProvider — реализация со скользящим окном вместо фиксированных интервалов. Даёт более плавное ограничение.

RateLimitProvider определяется как интерфейс:

```python
class RateLimitProvider(ABC):
    @abstractmethod
    async def check_and_increment(
        self,
        key: str,
        max_calls: int,
        period: float
    ) -> RateLimitResult:
        """
        Проверяет лимит и увеличивает счётчик.

        Возвращает RateLimitResult с полями:
        - allowed: bool — разрешён ли вызов
        - current_count: int — текущее количество вызовов
        - reset_at: float — timestamp сброса счётчика
        """
        pass
```

Машина получает RateLimitProvider при инициализации, а не через @depends на каждом действии — потому что rate limiting является инфраструктурой машины, а не зависимостью бизнес-логики.

События для плагинов

ActionMachine генерирует событие rate_limit_exceeded, которое содержит:

- action_name — полное имя действия
- scope_key — ключ области (например, "user:42")
- current_count — текущее количество вызовов
- max_calls — установленный лимит
- period — период в секундах
- reset_at — время сброса счётчика
- context — контекст выполнения
- fallback_used — True/False

```python
@on('rate_limit_exceeded', '.*', ignore_exceptions=True)
async def on_rate_limited(self, state_plugin, event):
    alert(f"⚠️ {event.action_name} rate limited for {event.scope_key} "
          f"({event.current_count}/{event.max_calls} in {event.period}s)")
    return state_plugin
```

Интеграция с интроспектором

Интроспектор бизнес-процессов (idea_12) может извлекать параметры rate limit из атрибута _rate_limit_spec, добавляемого декоратором @rate_limit [5]. Это позволяет автоматически генерировать:

Карту лимитов — какие Actions имеют ограничения, с какими параметрами, в каких областях.

Отчёт по текущему использованию — какие лимиты близки к исчерпанию.

Предупреждения — действие зависит от внешнего API (@depends), но не имеет @rate_limit, хотя внешний API имеет собственные ограничения.

Тестирование

Влияние

Защита бизнес-операций, а не транспорта. Лимит применяется независимо от того, как вызвано действие — через HTTP, MCP, CLI или из другого действия. Один пользователь не может обойти лимит, переключившись на другой канал.

Гранулярность. Можно установить разные лимиты для разных операций: SendSMSAction — 5 в минуту, CreateOrderAction — 100 в час, ExportReportAction — 3 в день. В традиционных системах для этого нужно настраивать каждый эндпоинт отдельно.

Прозрачность. Через плагинную систему [2] видно, какие лимиты срабатывают, для каких пользователей и как часто. Это данные для принятия решений о масштабировании или изменении тарифов.

Защита внешних API. Если ChargeCardAction вызывает PaymentGateway с лимитом 100 запросов в минуту, @rate_limit на действии гарантирует, что лимит не будет превышен — независимо от количества пользователей и транспортов.

Уникальность

Существующие решения для rate limiting (nginx, Kong, FastAPI Limiter, django-ratelimit) работают на уровне транспорта — HTTP-запросов, IP-адресов, API-ключей. Они не знают о бизнес-операциях, потому что в обычных фреймворках бизнес-операции не формализованы.

В AOA rate limiting работает на уровне Action — атомарной бизнес-операции [5]. Это означает:

Лимит применяется ко всем путям вызова одновременно — HTTP, MCP, CLI, deps.run_action().

Интроспектор может построить полную карту лимитов из метаданных [5].

Плагины видят срабатывания лимитов с полным бизнес-контекстом [2].

Декларативная генерация транспорта (idea_04) может автоматически добавлять заголовки X-RateLimit-Limit, X-RateLimit-Remaining и X-RateLimit-Reset в HTTP-ответы на основе метаданных @rate_limit.

Feature Flags (idea_14), Circuit Breaker (idea_15) и Rate Limiting работают вместе как три уровня защиты: ручная (флаги), автоматическая при сбоях (предохранитель), автоматическая при нагрузке (лимит). Все три проверяются в одном месте — перед выполнением действия — через единый механизм.

Связь с другими идеями

Circuit Breaker (idea_15) — защищает от сбоев, rate limiting — от перегрузки. Оба проверяются до выполнения аспектов. Если circuit breaker разомкнут, rate limit не увеличивает счётчик — действие не было выполнено.

Feature Flags (idea_14) — если фича выключена, rate limit не применяется. Три уровня защиты: ручная (флаги), автоматическая при сбоях (предохранитель), автоматическая при нагрузке (лимит).

Перехватчики исключений (idea_01) — @catch обработчики не вызываются при превышении лимита, потому что аспекты не выполняются.

Таймауты действий (idea_07) — таймаут и rate limit — комплементарные механизмы. Таймаут ограничивает время одного вызова, rate limit — количество вызовов. Оба применяются на уровне Action.

Координатор логеров (idea_02) — отдельный шаблон для событий rate limiting с контекстом (scope_key, current_count, max_calls).

Интроспектор бизнес-процессов (idea_12) — автоматическая карта лимитов всех Actions.

Декларативная генерация транспорта (idea_04) — автоматическое добавление заголовков rate limit в HTTP-ответы и информации о лимитах в OpenAPI.

Востребованность

Очень высокая для любых систем с пользователями: SaaS-платформы (лимиты по тарифам), финтех (защита платёжных API), e-commerce (защита от ботов), API-сервисы (монетизация через тарифы). Особенно ценно для систем, доступных через несколько транспортов — где традиционный HTTP-level rate limiting создаёт дыры в защите.

idea_17_execution_tree_plugin.md

Плагин семантического дерева выполнения (ExecutionTreePlugin)

Проблема

Плагинная система AOA уже даёт доступ к событиям выполнения: global_start, before/after каждого аспекта, global_finish с полным контекстом — params, state, result, duration, nest_level [2]. Однако каждый плагин форматирует эти данные по-своему. ConsoleLoggingPlugin выводит текст с ANSI-цветами [2], AnotherLoggingPlugin — другой текст. Ни один из них не производит структурированное, машиночитаемое дерево выполнения, пригодное для анализа, визуализации и AI-интеграции.

При этом все данные для построения такого дерева уже есть — они передаются в каждый обработчик через PluginEvent [2]. Не хватает одного: плагина, который соберёт эти данные в единую структуру.

Решение

ExecutionTreePlugin — плагин на 50-70 строк, который подписывается на все события ActionMachine и строит JSON-дерево выполнения бизнес-процесса в реальном времени. Дерево содержит полную семантику каждого узла: имя действия, типизированные параметры, состояние после каждого аспекта, результат, длительность, вложенные вызовы.

Результат работы плагина — один JSON-объект на каждый вызов machine.run(), который представляет собой полное семантическое дерево бизнес-процесса.

Как это работает

Плагин использует стек для отслеживания вложенности. При global_start создаётся новый узел и помещается на вершину стека. При after каждого аспекта к текущему узлу добавляется запись с именем аспекта, состоянием после выполнения и длительностью. При global_start вложенного действия новый узел помещается на стек, а при global_finish снимается со стека и добавляется как дочерний к родительскому узлу [5].

Реализация:

```python
class ExecutionTreePlugin(Plugin):

    def get_initial_state(self):
        return {"nodes": [], "stack": []}

    @on('global_start', '.*', ignore_exceptions=True)
    async def on_start(self, state_plugin, event):
        node = {
            "action": event.action_name,
            "params": repr(event.params),
            "nest_level": event.nest_level,
            "aspects": [],
            "children": [],
            "result": None,
            "duration": None
        }
        state_plugin["stack"].append(node)
        return state_plugin

    @on('after:.*', '.*', ignore_exceptions=True)
    async def on_after(self, state_plugin, event):
        if state_plugin["stack"]:
            current = state_plugin["stack"][-1]
            aspect_name = event.event_name.split(":")[1]
            current["aspects"].append({
                "name": aspect_name,
                "state_after": event.state_aspect,
                "duration": event.duration
            })
        return state_plugin

    @on('global_finish', '.*', ignore_exceptions=True)
    async def on_finish(self, state_plugin, event):
        if state_plugin["stack"]:
            node = state_plugin["stack"].pop()
            node["result"] = repr(event.result)
            node["duration"] = event.duration
            if state_plugin["stack"]:
                state_plugin["stack"][-1]["children"].append(node)
            else:
                state_plugin["nodes"].append(node)
        return state_plugin
```

Что это открывает

Для LLM и AI-агентов

Дерево — это готовый chain of thought для LLM [5]. Модель может прочитать его и объяснить бизнес-процесс на человеческом языке, найти точку сбоя по оборвавшемуся дереву, предложить оптимизацию по duration каждого аспекта, сгенерировать тесты из реальных execution trees [5].

Action как MCP-tool: Params — типизированный вход, Result — типизированный выход, CheckRoles — permissions, дерево выполнения — отладочная информация [5]. Один плагин превращает каждое Action в полноценный инструмент для LLM-агента с трассировкой.

Для мониторинга и дашбордов

Дерево можно отправить в ELK, Grafana, Datadog как структурированный лог. В отличие от текстовых логов, JSON-дерево позволяет строить запросы: «покажи все выполнения ProcessOrderAction, где аспект charge_card занял более 1 секунды» или «покажи все деревья с ошибками на втором уровне вложенности».

Для генерации тестов из production

ProductionCollectorPlugin (idea_08) может быть реализован как расширение ExecutionTreePlugin. Вместо того чтобы собирать данные заново, он просто сохраняет деревья, которые ExecutionTreePlugin уже построил. LLM анализирует сохранённые деревья и генерирует тесты через ActionTestMachine [2][5].

Для визуализации бизнес-процессов

Из JSON-дерева автоматически генерируются PlantUML или Mermaid диаграммы:

```
ProcessOrderAction (0.512s)
├── validate_user → {user_valid: true} [0.003s]
├── charge_card → {txn_id: "txn_789"} [0.450s]
│   └── CalculateCommissionAction (0.002s)
│       └── calculate → {commission: 3.0} [0.001s]
└── Result: order_id=42, total=153.0
```

Для детерминированного replay

Дерево содержит params, state на каждом шаге и result — это достаточно для воспроизведения бизнес-операции. В сочетании с ActionTestMachine [2] и MockAction [2] можно подставить результаты вложенных вызовов из сохранённого дерева и проверить, что при тех же входных данных система ведёт себя так же [5].

Для diff между версиями

Два дерева (до и после рефакторинга) можно сравнить структурно: появились ли новые аспекты, изменилась ли вложенность, изменились ли длительности. Это даёт changelog бизнес-логики, а не кода [5].

Почему это возможно только в AOA

В обычных фреймворках для получения такого дерева нужно вручную инструментировать каждый сервис, контроллер и middleware. В AOA дерево строится автоматически, потому что:

Все бизнес-операции проходят через единую точку machine.run() [2].

Каждый шаг (аспект) генерирует before/after события с полным контекстом [2].

Вложенные вызовы через deps.run_action() автоматически увеличивают nest_level [2].

Params и Result типизированы и доступны плагинам [2].

Описания аспектов (desc) и зависимостей (description) обязательны [2][5].

Плагины не могут изменить бизнес-данные — они только наблюдают [2].

Ни один Python-фреймворк не производит семантическое дерево выполнения бизнес-процесса с типизированными параметрами и состоянием из коробки [5]. ExecutionTreePlugin — это один плагин на 50-70 строк, который превращает эту архитектурную возможность в конкретный инструмент.

Влияние

Это самый короткий путь к killer feature для adoption. AI-интеграция, мониторинг, генерация тестов, визуализация — всё это начинается с одного JSON-дерева, которое AOA может производить без изменения ядра, без изменения действий, без изменения плагинной системы — просто добавлением одного плагина.

Связь с другими идеями

Генерация тестов из production-логов (idea_08) — ExecutionTreePlugin производит деревья, которые ProductionCollectorPlugin сохраняет, а LLM превращает в тесты.

Координатор логеров (idea_02) — координатор может использовать дерево из ExecutionTreePlugin для структурированного логирования с полным контекстом [6].

Декларативная генерация транспорта (idea_04) — MCP-регистратор использует дерево как отладочную информацию для AI-агентов [5][6].

Обнаружение холостых тестов (idea_13) — сравнение production-деревьев с тестовыми деревьями для классификации тестов.

Интроспектор бизнес-процессов (idea_12) — интроспектор строит статическую карту из метаданных, ExecutionTreePlugin строит динамическое дерево из реального выполнения. Вместе они дают полную картину: «как должно работать» и «как работает на самом деле».

Action Interrogation (idea_05) — мультиассерт-тесты могут включать проверки структуры дерева как один из независимых assert-ов: «дерево содержит ровно два дочерних вызова», «аспект charge_card выполнился за менее чем 1 секунду».

Востребованность

Максимальная. ExecutionTreePlugin — это точка схождения всех остальных идей. Он одновременно является: основой для AI-интеграции, инструментом мониторинга, источником данных для генерации тестов, форматом для визуализации и базой для сравнения версий. При этом его реализация — 50-70 строк кода, использующих уже существующую плагинную инфраструктуру без каких-либо изменений в ядре [2].

Application-Aware Database Observability — телеметрия PostgreSQL с бизнес-контекстом



Проблема

Индустрия наблюдаемости разделена на два мира, между которыми — пропасть.

Мир приложения (APM) — Datadog APM, New Relic, Dynatrace, OpenTelemetry. Они отвечают на вопросы: «Какой endpoint тормозит?», «Какой пользователь?», «Какая версия API?». Они видят, что запрос занял 50 мс, но не видят почему: сколько из этих 50 мс ушло на чтение с диска, сколько блоков было прочитано из кэша, сколько WAL сгенерировано, была ли сортировка на диске.

Мир базы данных (Database Monitoring) — pganalyze, Datadog DBM, Percona PMM, pg_stat_statements. Они отвечают на вопросы: «Какой запрос тормозит?», «Какой queryid?», «Какая таблица?». Они видят, что запрос SELECT * FROM orders WHERE user_id = $1 вызывается 10 000 раз в секунду, но не видят какой функционал приложения за этим стоит: 8000 вызовов приходят с экрана корзины, а 2000 — с экрана истории, и экран корзины делает в три раза больше дисковых чтений на запрос.

DBA видит медленный запрос, но не знает откуда он приходит — с экрана корзины или из фонового задания. Backend-разработчик видит медленный эндпоинт, но не знает почему база тормозит — не хватает индекса или кончается work_mem. Ни один существующий инструмент не связывает бизнес-контекст приложения (пользователь, экран, кнопка, A/B вариант, версия API) с низкоуровневой телеметрией базы данных (блоки буферного кэша, CPU, WAL, планы запросов) 01_draft.txt.

Ближайший аналог — sqlcommenter от Google — добавляет в SQL-комментарии controller, action, framework, route. Но он работает на уровне отдельных запросов, не поддерживает Unit of Work как единицу, не собирает EXPLAIN ANALYZE per-trace, не даёт метрик CPU и памяти, не умеет динамически включать/выключать детализацию и не поддерживает произвольный бизнес-контекст (A/B варианты, версию клиента). sqlcommenter делает 10% от того, что описано в этой идее 02_draft.txt.

В обычном Python-приложении эта задача требует значительной инфраструктуры. В AOA она решается практически бесплатно — потому что весь необходимый контекст уже собран и формализован в единой точке code.txt.



Что AOA даёт бесплатно

Это ключевое отличие от любого другого фреймворка. В AOA не нужно инструментировать код — контекст уже собран:

Context с user_id, ролями, trace_id, request_path, request_method, user_agent, tags (включая A/B варианты) формируется AuthCoordinator до начала выполнения любого действия и доступен плагинам code.txt.

Плагины получают полный PluginEvent на каждом событии — params, state, result, duration, nest_level, context — и при этом read-only, не могут нарушить бизнес-логику code.txt.

Resource Managers инкапсулируют работу с БД. PostgresProfilingConnectionManager — это просто расширенный PostgresConnectionManager, который добавляет инструментирование к каждому вызову execute code.txt.

Единственная точка входа machine.run() — все операции с БД проходят через неё, поэтому телеметрия гарантированно охватывает весь трафик code.txt.

В Django context размазан по middleware, request и session. В FastAPI — по Depends, State и Background Tasks. В Spring — по аннотациям, AOP и TransactionManager. Нигде бизнес-контекст не собран в одном типизированном объекте, доступном наблюдателям без изменения бизнес-логики. В AOA это фундаментальное свойство архитектуры.



Суть решения

Каждый SQL-запрос внутри пакета команд (Unit of Work) оборачивается в EXPLAIN (ANALYZE, BUFFERS, WAL, FORMAT JSON). PostgreSQL реально выполняет запрос и возвращает детальную статистику — буферный кеш, WAL, время планирования и выполнения. Параллельно расширение pg_stat_kcache вызывает системный getrusage() до и после пакета, что даёт CPU time и page faults на уровне ОС. В каждый SQL-запрос вшивается trace_id из Context.request. Все метрики собираются в Python, складываются в asyncio.Queue, и background-таск батчами отправляет их в Kafka (или любой другой sink).

Результат — для каждого вызова machine.run() система знает одновременно кто вызвал (пользователь, роль, экран, A/B вариант, версия API) и сколько это стоило (блоки диска, WAL bytes, CPU time, сортировки на диске). Эти два потока данных связаны через один trace_id code.txt.

class PostgresProfilingConnectionManager(PostgresConnectionManager):

def init(self, connection_params, config=None):

        super().__init__(connection_params)

        self._config = config or ProfilingConfig.standard()

        self._query_metrics: List[QueryMetrics] = []

        self._trace_id: Optional[str] = None



def set_trace_id(self, trace_id: str) -> None:

        self._trace_id = trace_id



def set_profiling_config(self, config: ProfilingConfig) -> None:

        self._config = config



async def execute(self, query, params=None):

        tagged_sql = self._tag_query(query)

if self._should_profile():

return await self._execute_with_explain(tagged_sql, params)

else:

            result = await super().execute(tagged_sql, params)

            self._query_metrics.append(QueryMetrics(

                sql_hash=hash(query),

                rows_affected=self._parse_rows(result),

                wall_time_ms=0,

                sampled=False,

            ))

return result



def _tag_query(self, sql: str) -> str:

if self._trace_id:

return f"/* trace_id='{self._trace_id}' */ {sql}"

return sql



def _should_profile(self) -> bool:

import random

return random.random() < self._config.sample_rate



async def _execute_with_explain(self, sql, params):

        flags = ["ANALYZE"]

if self._config.buffers:

            flags.append("BUFFERS")

if self._config.wal:

            flags.append("WAL")

        flags.append("FORMAT JSON")

        explain_sql = f"EXPLAIN ({', '.join(flags)}) {sql}"

        rows = await super().execute(explain_sql, params)

        metrics = self._parse_plan(rows)

        self._query_metrics.append(metrics)

return rows



def get_query_metrics(self) -> List[QueryMetrics]:

return list(self._query_metrics)



def get_wrapper_class(self):

return ProfilingWrapperConnectionManager

Детальная карта флагов

Флаг TIMING (overhead БД ~0.05мс/запрос). Wall clock time, planning_time, execution_time. Вызов gettimeofday() на каждом узле плана — на современном CPU это rdtsc, наносекунды. База для всего остального. Если выключить — используем просто conn.execute() без EXPLAIN, остаётся только wall_time из Python. Рекомендуется всегда включённым.

Флаг ROWS (overhead 0 дополнительно). actual_rows на каждом узле, rows_removed_by_filter, total_rows_affected из статуса команды. Уже включено в EXPLAIN ANALYZE бесплатно. Смысл выключать: нет, если TIMING включён.

Флаг BUFFERS (overhead БД ~0.02мс/запрос). shared/local/temp блоки hit/read/dirtied/written. PostgreSQL и так ведёт эти счётчики — флаг просто копирует их в вывод EXPLAIN. Самые ценные данные при минимальном overhead: hit ratio per-query, обнаружение промахов мимо кеша, temp_blks как индикатор нехватки work_mem. Рекомендуется всегда включённым.

Флаг IO_TIMING (overhead ~0.01–0.02мс на каждый IO-вызов). blk_read_time и blk_write_time, per-node I/O Read/Write Time. Требует track_io_timing = on. Два вызова clock_gettime() на каждое обращение к диску. PostgreSQL официально рекомендует включать. Позволяет отделить CPU от IO: estimated_cpu = execution_time - blk_read_time - blk_write_time. Смысл выключать: только на крайне старом железе.

Флаг WAL (overhead ~0.01мс/запрос). wal_records, wal_fpi, wal_bytes. Доступно с PostgreSQL 13. Счётчики WAL и так ведутся — флаг просто включает их в вывод. Стоимость околонулевая. Ключевая метрика нагрузки на репликацию и дисковую подсистему.

Флаг CPU (overhead ~0.6мс/пакет фиксированно). cpu_user_time_sec и cpu_system_time_sec через pg_stat_kcache diff. getrusage() возвращает CPU time процесса (backend). Поскольку один backend обслуживает одно соединение, а UoW выполняется в одной транзакции — diff даёт именно CPU данного пакета, без загрязнения от других сессий. Фиксированный overhead на пакет, не на запрос: для пакета из 1 запроса дорого, для пакета из 50 — дёшево. На managed PostgreSQL (RDS) недоступен. Единственный способ узнать реальное CPU time.

Флаг MEMORY (overhead на БД: 0 если BUFFERS и CPU уже включены). minor_page_faults и major_page_faults из pg_stat_kcache, hash_peak_memory_kb из EXPLAIN (Hash-узлы), sort_space_used_kb, sort_space_type (Memory/Disk) и hash_batches. Overhead на Python: ~0.05мс на парсинг Hash/Sort узлов. Обнаружение нехватки work_mem: sort on disk, hash batches > 1. major_page_faults > 0 — сервер свопит, критическая проблема.

Флаг PLAN (overhead на БД: 0 дополнительно, overhead Python: 0.1–0.5мс/запрос). Полный JSON-план каждого запроса, node_types_used, index_name, plan_rows vs actual_rows ratio. Данные уже есть в EXPLAIN JSON — вопрос только сохранять или отбрасывать. Overhead хранения: план одного запроса — 2–50KB. Пакет из 50 запросов — до 2.5MB. При 100 UoW/сек и 100% sampling — 250MB/сек в Kafka. Главный источник overhead по объёму данных, не по CPU. По умолчанию выключен. Включать при sampling ≤5%, или targeted для конкретного эндпоинта, или только для запросов медленнее порога.

Флаг PLAN_DETAILS (overhead Python: 0.5–2мс/запрос). Per-node buffers, per-node timing, per-worker метрики, JIT-метрики, filter selectivity per-node, полное дерево с вложенностью. Рекурсивный обход JSON-дерева EXPLAIN. Только для глубокой диагностики конкретных проблем.

Готовые пресеты

MINIMAL — TIMING, ROWS, без EXPLAIN вообще, только wall_time из Python. Overhead ~0.02мс/запрос. Для 100% трафика постоянно.

STANDARD — TIMING, ROWS, BUFFERS, IO_TIMING, WAL. EXPLAIN ANALYZE с этими опциями. Overhead БД ~0.1мс/запрос, Python ~0.03мс/запрос. Рекомендуется для 100% трафика.

FULL — STANDARD + CPU + MEMORY. EXPLAIN + pg_stat_kcache diff. Overhead БД ~0.1мс/запрос + 0.6мс/пакет. Для sampling 10–20%.

DIAGNOSTIC — все флаги включены. EXPLAIN + pg_stat_kcache + полный парсинг дерева плана. Overhead Python 1–3мс/запрос. Только для расследования конкретных проблем, 1–5% трафика или по запросу.

cpu_system_time_sec — время CPU в kernel space (системные вызовы read/write, выделение памяти, сетевые операции)



cpu_total_time_sec — сумма; единственный способ получить реальное CPU time без загрязнения от параллельных сессий, потому что getrusage() работает per-process

Память (из pg_stat_kcache diff):





minor_page_faults — обращения к памяти, уже в RAM. Прокси для «сколько новой памяти запрос затронул»



major_page_faults — обращения к памяти, потребовавшие чтения с диска (swap). Высокое значение — критическая проблема

Физический IO уровня ОС (из pg_stat_kcache diff):





os_reads_bytes — физические чтения на уровне ОС, включая то что прошло мимо shared_buffers



os_writes_bytes — физические записи на уровне ОС, включая WAL и direct IO

Буферный кеш (агрегат из EXPLAIN по всем запросам пакета):





shared_blks_hit — блоки, найденные в shared_buffers. Чем больше — тем лучше



shared_blks_read — блоки, прочитанные с диска. Много — данные не влезают в кеш



shared_blks_dirtied — блоки, изменённые в памяти (запишутся позже bgwriter)



shared_blks_written — блоки, записанные самим запросом (bgwriter не успевал)



local_blks_hit, local_blks_read, local_blks_dirtied, local_blks_written — аналогично для временных таблиц



temp_blks_read, temp_blks_written — временные файлы (сортировки/хеши не влезли в work_mem). Ненулевые значения — сигнал увеличить work_mem

IO timing (требует track_io_timing = on):





blk_read_time_ms — суммарное время физического чтения блоков с диска



blk_write_time_ms — суммарное время физической записи блоков на диск

WAL — Write-Ahead Log (PG13+):





wal_records — количество WAL-записей. Каждое изменение данных создаёт запись



wal_fpi — Full Page Images. Высокое значение после checkpoint — нормально



wal_bytes — объём данных в WAL. Ключевая метрика нагрузки на репликацию

Планировщик (агрегат по всем запросам):





total_planning_time_ms — суммарное время работы оптимизатора



total_execution_time_ms — суммарное серверное время выполнения



max_row_estimate_ratio — максимальное отношение Actual Rows / Plan Rows. Значение > 10 означает устаревшую статистику таблицы



worst_estimate_node — какой узел плана ошибся сильнее всего

Временные метрики из EXPLAIN (дополнительно):





actual_total_time — суммарное время на корневом узле плана, включает все дочерние узлы



actual_startup_time — время до получения первой строки результата

Память операций (из EXPLAIN):





hash_peak_memory_kb — пиковое использование памяти хеш-таблицами. Единственное место где PostgreSQL явно показывает потребление памяти конкретной операцией



hash_batches_disk — сколько батчей хеш-таблицы ушло на диск



sort_operations_memory — сколько сортировок уместилось в RAM



sort_operations_disk — сколько сортировок ушло на диск

JIT-компиляция (PostgreSQL 11+, если сработал):





jit_functions_compiled — количество функций, скомпилированных LLVM



jit_total_time_ms — суммарное время JIT. Может быть десятки мс на первый вызов — важно знать, стоил ли JIT своих затрат



jit_generation_time_ms, jit_inlining_time_ms, jit_optimization_time_ms, jit_emission_time_ms — времена стадий компиляции

Параллелизм:





workers_planned — сколько параллельных worker-ов планировщик хотел запустить



workers_launched — сколько реально запустилось. Расхождение означает нехватку max_parallel_workers

Per-node метрики (при флаге PLAN_DETAILS):





Node Type — тип операции: Seq Scan, Index Scan, Hash Join, Sort и другие



Plan Rows vs Actual Rows — ключевой диагностический приём: сильное расхождение означает устаревшую статистику



Sort Method и Sort Space Type — Memory или Disk (не влезло в work_mem)



Hash Buckets, Hash Batches, Peak Memory Usage — пиковое использование памяти хеш-таблицей в KB



Workers Planned vs Workers Launched — для параллельных запросов

Per-query детали (массив, включается флагом plan):





sql_hash — хеш нормализованного SQL



planning_time_ms, execution_time_ms — время на запрос



shared_blks_hit, shared_blks_read, wal_bytes — буферы на запрос



estimated_cpu_ms — приблизительный CPU per-query через execution_time - io_time



node_types_used — какие операции использовались: Seq Scan, Index Scan, Hash Join, Sort



Что невозможно получить

RSS / heap memory per-query — PostgreSQL выделяет память через palloc в контекстах, но не трекает суммарный RSS per-statement. Доступна только Hash Peak Memory для хеш-узлов.

Lock wait time per-query — wait_event это снимок, не накопительная метрика. Планируется в PG17 через pg_stat_lock_waits.

Точный CPU per-query (не per-UoW) — getrusage() дорого вызывать на каждый запрос. pg_stat_kcache даёт per-statement через pg_stat_statements, но при diff по backend — только per-UoW.

Network bytes per-query — протокол не считает это на стороне сервера.

GPU utilization — PostgreSQL не использует GPU.

Когда данные собраны, их можно передать LLM вместе с кодом Actions. LLM получает одновременно три структурированных потока, связанных через trace_id:

LLM не угадывает связи — связи формализованы архитектурой. Это открывает класс анализа, недоступный при работе с обычными фреймворками:

Бизнес-стоимость — видит что GenerateReportAction генерирует 15MB WAL на вызов, A/B вариант B вызывает его в 4 раза чаще → конкретные цифры для product manager



Генерация SQL-миграций с обоснованием через телеметрию



Предсказание регрессий до деплоя на основе трендов



Почему это возможно только в AOA

Context с бизнес-данными формируется ДО выполнения — AuthCoordinator собирает user_id, roles, trace_id, tags, path, method в единый объект. Плагины получают полный контекст автоматически через PluginEvent. Actions не видят Context — профилирование не загрязняет бизнес-логику. Ресурсные менеджеры с прокси-иерархией — метрики дочерних действий попадают в общий буфер безопасно. Всё проходит через machine.run() — единая точка инструментирования. Метаданные формализованы — интроспектор может построить карту «Action → средний IO-профиль» из кода.

В Django context размазан по middleware, request, session. В FastAPI — по Depends, State, Background Tasks. В Spring — по аннотациям, AOP и TransactionManager. Нигде бизнес-контекст не собран в одном типизированном объекте, доступном наблюдателям без изменения бизнес-логики. AOA делает связывание бизнес-контекста с телеметрией БД тривиальным — потому что оба уже существуют в формализованном виде и встречаются в одной точке: PluginEvent в global_finish.



Уникальность

Это не просто «мониторинг базы данных». Это первый инструмент в Python-экосистеме который связывает бизнес-намерения (кто, зачем, какой экран, какая версия A/B теста) с технической ценой (блоки диска, CPU, WAL, планы запросов) через единый trace_id на уровне каждой транзакции.

## idea_19_llm.md — LLM + телеметрия AOA + код Actions = новый класс анализа

### Проблема

Современные LLM-инструменты для анализа кода работают с одним потоком данных — исходным кодом. Модель читает файлы, строит предположения о том как система работает в реальности, и выдаёт рекомендации основанные на статическом анализе. Это принципиально ограниченный подход.

Разрыв между тем что написано в коде и тем что происходит в production — это нормальное состояние любой живой системы. Запрос который теоретически должен использовать индекс на практике делает Seq Scan. Метод который выглядит лёгким генерирует 2MB WAL при каждом вызове. Аспект который занимает три строки кода тратит 800мс из которых 795мс — ожидание блокировки. Статический анализ этого не видит.

В обычных фреймворках устранить этот разрыв невозможно — нет формализованного представления бизнес-операций, нет структурированного дерева выполнения, нет привязки технической телеметрии к бизнес-контексту. LLM получает либо код, либо логи — никогда оба потока вместе в структурированном виде.

В AOA этот разрыв устраняется архитектурно.

---

### Три потока данных в одной точке

Когда телеметрия PostgreSQL (idea_18) и семантическое дерево выполнения (idea_17) работают вместе, для каждого вызова `machine.run()` формируются три связанных структуры:

**Намерение** — код Action как машиночитаемая спецификация. Не просто исходный текст, а формализованные метаданные: зависимости с описаниями из `_dependencies`, ролевая модель из `_role_spec`, упорядоченный список шагов с контрактами каждого, объявленные соединения. Всё это верифицировано машиной — без этих метаданных код не запустится [4].

**Выполнение** — семантическое дерево из ExecutionTreePlugin. JSON-граф с params, state после каждого аспекта, duration каждого шага, вложенными вызовами и итоговым result. Не плоский лог — структурированный граф бизнес-процесса с полной семантикой каждого узла [3].

**Цена** — телеметрия PostgreSQL привязанная к trace_id из Context.request. Для каждого узла дерева: блоки буферного кеша, WAL bytes, CPU time (из pg_stat_kcache), планы запросов, IO timing, сортировки на диске, пиковое потребление памяти хеш-операциями [3].

LLM получает все три потока одновременно. Связь между ними формализована архитектурой через единый trace_id — модель не угадывает отношения между кодом, выполнением и стоимостью, она работает с явными структурами.

---

### Почему это возможно только в AOA

В Django бизнес-логика размазана по контроллерам, сервисам и моделям. Контекст — по middleware, request и session. Нет единой точки входа, нет формализованного конвейера, нет `nest_level`.

В FastAPI — по Depends, State и Background Tasks. В Spring — по аннотациям, AOP и TransactionManager.

В AOA все операции проходят через `machine.run()` [4]. Context с user_id, trace_id, ролями, tags формируется до начала выполнения и доступен плагинам [4]. Actions физически не видят Context — он не загрязняет бизнес-логику [4]. Плагины read-only — телеметрия не может нарушить бизнес-процесс [4]. Метаданные Actions обязательны и верифицированы — `@CheckRoles` без описания, `@depends` без description не пройдут машину [4].

Это три условия без которых LLM-анализ описанного класса невозможен: единая точка исполнения, формализованный конвейер с метаданными, изоляция контекста от бизнес-логики.

---

### Что получает LLM на входе

Пакет данных для анализа одного Action или одного блока выполнения содержит:

**Код Action** — класс целиком с декораторами, Params, Result, всеми аспектами, описаниями зависимостей и контрактами чекеров. Машиночитаемый контракт каждого шага.

**Дерево выполнения** (из ExecutionTreePlugin) — JSON с реальными значениями params при конкретном вызове, state после каждого аспекта, duration каждого шага в мс, вложенными Action-вызовами со своими поддеревьями.

**Телеметрия PostgreSQL** (из DbTelemetryPlugin, idea_18) — для каждого SQL-запроса в пакете: planning_time, execution_time, shared_blks_hit, shared_blks_read, wal_bytes, blk_read_time, sort_on_disk, hash_peak_memory_kb, план запроса с типами операций (Seq Scan / Index Scan / Hash Join). Плюс cpu_user_time и cpu_system_time на весь пакет из pg_stat_kcache.

**Бизнес-контекст** (из Context) — user_id, роли, request_path, request_method, tags включая ab_variant и client_type, service_version.

---

### Оптимизация SQL-запросов

LLM видит не только время выполнения — она видит почему запрос работает именно так.

Аспект `load_products` читает 4000 блоков с диска (`shared_blks_read=4000`) при каждом вызове, `shared_blks_hit=12` — почти нулевой hit ratio. Из плана запроса виден Seq Scan по таблице products. Из кода Action видно что в Params есть `product_ids: List[int]` с чекером `max_length=20` и аспект делает SELECT по списку ID.

**LLM предлагает:** индекс на `products.id` вероятно есть — проблема в том что планировщик выбирает Seq Scan из-за устаревшей статистики. Рекомендуется `ANALYZE products`. Альтернативно — поскольку из чекера видно `max_length=20`, список всегда маленький, можно форсировать Index Scan через `enable_seqscan = off` в рамках транзакции.

Это невозможно без одновременного доступа к коду (чекер max_length) и телеметрии (shared_blks_read, план). В обычном инструменте есть либо одно, либо другое.

---

### Обнаружение N+1 через паттерн выполнения

Дерево выполнения показывает что аспект `create_items` вызывает execute 47 раз с одинаковым шаблоном INSERT — один раз на каждый элемент в цикле. Суммарный wal_bytes пакета аномально высокий.

**LLM предлагает:** использовать PostgresUnitOfWork (idea_03) для batch-вставки. Вместо 47 round-trips к БД — один пакет. Предоставляет конкретный рефакторинг аспекта с буферизацией команд.

Без дерева выполнения с duration каждого execute этот паттерн не виден — только итоговое время.

---

### Диагностика блокировок

Аспект `update_inventory` занимает 800мс. Из телеметрии: `blk_read_time=2мс`, `cpu_user_time_sec=0.003` — остальные ~795мс это не IO и не CPU.

**LLM диагностирует:** почти всё время — ожидание. Вероятная причина — contention на строках inventory при параллельных запросах. Предлагает `SELECT FOR UPDATE SKIP LOCKED` для оптимистичной обработки, или переход на optimistic locking с версионированием.

Без разделения cpu_time и io_time от wall_clock такая диагностика невозможна — всё выглядит как «медленный запрос».

---

### Обнаружение длинных транзакций

Дерево выполнения показывает что между `open_connection` и `commit` в конвейере проходит 5 аспектов за 3.2 секунды. Из breakdown по аспектам: `validate` — 0.003с, `load_data` — 0.1с, `calculate` — 0.05с, `notify_external_api` — 2.8с, `save` — 0.05с.

**LLM предлагает:** вынести `notify_external_api` за пределы транзакции. Пока соединение удерживает блокировку 3.2 секунды, другие сессии не могут получить доступ к изменённым строкам. Транзакция должна содержать только атомарные изменения данных, уведомления — через фоновую задачу или очередь после commit.

Это видно только когда есть одновременно структура конвейера (из дерева) и время каждого шага.

---

### Бизнес-стоимость операций

LLM получает телеметрию за день и агрегирует по бизнес-контексту из Context.tags:

Action `GenerateReportAction` вызывается 200 раз в день, каждый вызов генерирует 15MB WAL и читает 80000 блоков. Из Context.tags видно что 180 из 200 вызовов приходят от A/B варианта `dashboard_v2`. Вариант `dashboard_v1` вызывает тот же Action 20 раз и генерирует в 4 раза меньше WAL на вызов — потому что запрашивает данные за неделю, а не за месяц.

**Вывод LLM:** вариант B дашборда стоит в 36 раз дороже варианта A в ресурсах БД. Рекомендует кеширование агрегатов или предрасчёт при сохранении данных.

Этот вопрос не имеет ответа ни в одном существующем APM или инструменте мониторинга БД — потому что нигде нет одновременно A/B контекста (бизнес) и WAL bytes (техника) в одной привязке.

---

### Предиктивный анализ

LLM получает телеметрию за неделю и видит тренд: `shared_blks_read` для `CreateOrderAction` растёт на 3% в день. Из плана запроса видно что индекс `orders_user_id_idx` имеет `Index Cond: (user_id = $1)` но `Rows Removed by Filter: 847` — индекс становится менее избирательным по мере роста таблицы.

**Вывод LLM:** через 30 дней этот запрос начнёт превышать SLA в 200мс. Рекомендует партиционирование таблицы orders по дате или составной индекс `(user_id, created_at DESC)`. Генерирует готовую миграцию с обоснованием через телеметрию.

---

### Автоматическая декомпозиция Actions

LLM видит что Action имеет 8 аспектов, 4 вложенных вызова и 12 запросов к БД суммарно. Из телеметрии видно что 3 аспекта из 8 генерируют 95% нагрузки. Из дерева выполнения видно что эти 3 аспекта не зависят друг от друга по state — между ними нет передачи данных.

**Вывод LLM:** аспекты `load_inventory`, `load_pricing`, `load_promotions` могут быть вынесены в отдельные Actions и выполнены параллельно через три независимых вызова `deps.run_action()`. Это сократит wall time с 1.2с до ~0.5с. Предоставляет рефакторинг с конкретным кодом.

### Связь телеметрии с тестами

LLM видит что в production аспект `charge_card` генерирует 500 WAL bytes и читает 200 блоков, выполняя 12 SQL-запросов. В тестах (через ActionTestMachine [4]) этот аспект замокан через MockAction — побочные эффекты БД не проверяются вообще.

**Вывод LLM:** рекомендует добавить интеграционный тест с auto-rollback (idea_09) для `charge_card` который проверит реальные SQL-запросы. Текущие моки скрывают N+1 проблему — в production аспект выполняет 12 запросов, тест проверяет только Result. Генерирует шаблон теста с multi_assert (idea_05) включающим проверку количества SQL-операций.

#### Пример сгенерированного теста

```python
@pytest.mark.asyncio
async def test_charge_card_should_not_generate_Nplus1():
    # Сгенерировано из production-лога с N+1 проблемой
    machine = ActionTestMachine({
        PaymentGateway: FakePaymentGateway(),
    })
    params = ChargeCardAction.Params(card_token="tok_test", amount=100.0)
    result = await machine.run(ChargeCardAction(), params)
    assert result.success is True
    # Проверка через ExecutionTreePlugin, что аспект выполнил ровно один SQL-запрос
```

### Обнаружение аномалий по пользователям

LLM видит что один user_id генерирует в 50 раз больше `shared_blks_read` чем медианный пользователь. Из Params видно что он передаёт `date_range="2020-01-01:2025-06-01"` — пятилетний диапазон, в то время как 99% пользователей запрашивают последние 30 дней.

**Вывод LLM:** добавить в чекер Params ограничение `max_date_range=90 days` с понятным сообщением об ошибке. Или создать отдельный `HistoricalReportAction` с предупреждением о длительности и явным согласием пользователя.

Это пример cost attribution на уровне пользователей — возможность которой нет нигде кроме системы где бизнес-контекст и техническая телеметрия связаны в одной точке.

### Генерация SQL-миграций с обоснованием

LLM видит проблемные запросы, знает схему из планов выполнения, понимает бизнес-контекст из кода Actions. Генерирует не просто миграцию — а миграцию с обоснованием через реальные данные:

```sql
-- Сгенерировано на основе анализа телеметрии CreateOrderAction
-- Проблема: Seq Scan на orders при фильтрации по user_id + status
-- Наблюдаемый overhead: 4000 shared_blks_read на вызов, 100 вызовов/сек
-- Ожидаемый эффект: снижение до ~50 shared_blks_read (оценка по selectivity из плана)
-- Безопасность: CONCURRENTLY — без блокировки на запись

CREATE INDEX CONCURRENTLY idx_orders_user_status
ON orders (user_id, status)
WHERE status IN ('pending', 'processing');
```

Ни один существующий инструмент не генерирует миграции с такой цепочкой обоснований — потому что для этого нужны одновременно реальные планы запросов, бизнес-контекст и статистика вызовов.

### Ролевой анализ LLM

В зависимости от того какие данные передаются и какой вопрос задаётся, LLM выступает в разных ролях:

**DBA-аналитик** — когда передаются планы запросов, buffer statistics, WAL metrics. Предлагает индексы, партиционирование, настройки work_mem.

**Архитектор** — когда передаётся дерево выполнения с duration по аспектам. Предлагает декомпозицию, параллелизацию, вынос логики за пределы транзакций.

**Бизнес-аналитик** — когда добавляется бизнес-контекст из Context.tags. Связывает нагрузку с A/B вариантами, версиями API, типами клиентов.

**QA-инженер** — когда сравнивается production-телеметрия с тестовым покрытием. Находит что моки скрывают, генерирует интеграционные тесты.

**SRE** — когда передаётся временной ряд телеметрии. Обнаруживает тренды деградации, предсказывает нарушения SLA, предлагает circuit breaker (idea_15) на проблемные операции.

Одна архитектура, один пакет данных, несколько уровней анализа — в зависимости от того какой вопрос задан.

---

### Влияние

- **Скорость диагностики** — вместо часов разбора логов минуты на запрос к LLM.
- **Качество рекомендаций** — LLM видит полный контекст: код, выполнение, нагрузку на БД.
- **Проактивность** — система может сама генерировать отчёты о потенциальных проблемах до того, как они станут критическими.
- **Документирование** — инсайты LLM могут сохраняться как комментарии к Actions или в базу знаний.
- **Улучшение тестового покрытия** — автоматическая генерация тестов из реальных сценариев.

---

### Уникальность

В любом другом фреймворке LLM получил бы код — разрозненные файлы без формальной структуры, или логи — поток текста без семантики, или метрики — графики без привязки к бизнес-операциям.

В AOA LLM получает три структурированных потока, связанных через trace_id и формализованные метаданные. Каждый Action — это спецификация с обязательными описаниями, контрактами и ролями [4]. Каждое выполнение — это семантическое дерево [3]. Каждый запрос к БД — это телеметрия с бизнес-контекстом [3].

LLM не угадывает связи — **связи формализованы архитектурой**.

Особая ценность — возможность задавать вопросы на естественном языке и получать ответы, которые невозможно получить ни из одного существующего инструмента по отдельности. Например:

- *«Почему версия B нашего A/B теста потребляет на 40% больше дисковых чтений?»*
- *«Какой экран приложения генерирует больше всего WAL и убивает репликацию?»*
- *«Покажи IO-профиль запросов пользователя, который жалуется на тормоза.»*

# idea_20_living_cell_business_organism.md

## Action как живая клетка — самоадаптирующийся бизнес-организм

---

## Проблема

Современные системы мониторинга и адаптации разделены на три изолированных мира, между которыми — пропасть.

**Мир инфраструктуры** — Kubernetes, Prometheus, Grafana. Они отвечают на вопрос «работает ли сервис». CPU 90%, диск полный, pod упал — перезапустить. Но система не знает **почему** ей плохо, и уж тем более не знает **достигает ли она бизнес-целей**.

**Мир технической наблюдаемости** — Datadog APM, pganalyze, OpenTelemetry. Они отвечают на вопрос «работает ли сервис хорошо». Медленный запрос, N+1 проблема, Seq Scan. Но они не знают, что замедление `charge_card` означает падение конверсии на 40%.

**Мир бизнес-аналитики** — Amplitude, Mixpanel, Grafana с SQL-запросами. Они отвечают на вопрос «достигаем ли мы бизнес-целей». Конверсия упала, заказов меньше. Но они **отделены от кода** — связь между техническими метриками и бизнес-показателями поддерживается людьми вручную, устаревает при каждом рефакторинге и требует отдельной инфраструктуры.

Ни одна существующая система не умеет одновременно:
- Понимать что **заказов стало меньше** (бизнес-семантика)
- Знать **почему** — какой конкретно аспект деградировал (техническая семантика)
- **Адаптировать бизнес-поведение** автоматически — запускать A/B тесты, менять таргетинг, переключать каналы
- Делать всё это **без отдельной инфраструктуры**, как встроенное свойство архитектуры

В обычном фреймворке для такого сценария нужно:
1. Отдельная система аналитики (Amplitude) — **не связана с кодом**
2. Отдельная система A/B тестов (LaunchDarkly) — **не связана с бизнес-логикой**
3. Отдельная система мониторинга (Datadog) — **не связана с бизнес-метриками**
4. Отдельная система оркестрации (Airflow) — **не связана с Actions**
5. Люди, которые склеивают всё это вместе — **и всё равно работают с запаздыванием**

**Это архитектурная проблема, а не проблема инструментов.** В обычных фреймворках бизнес-операции не формализованы — они размазаны по контроллерам, сервисам, моделям. Нет единой точки где можно наблюдать **поток бизнес-событий**. Нет формальной связи между кодом и бизнес-метриками.

---

## Суть

В AOA каждый вызов `machine.run()` — это формализованное бизнес-событие с типизированным Params, Result, duration, Context с тегами [1]. Это означает что **поток бизнес-событий уже формализован архитектурой** — он является побочным продуктом конвейера, а не отдельной подсистемой.

Идея 20 формализует эту возможность в архитектурный принцип: **Action как живая клетка, система как самоадаптирующийся бизнес-организм**.

Это не метафора. Это инженерное следствие трёх архитектурных условий AOA [1]:
1. **Единая точка исполнения** — `machine.run()`. Все бизнес-операции проходят через неё
2. **Формализованный конвейер с метаданными** — `action_name`, `aspect_name`, `duration`, `context`, `result` — гарантированные данные
3. **Context невидим для Actions** — клетка не управляет организмом, она работает только со своими входами и выходами

Убери любое из этих условий — и организм превращается обратно в набор кирпичей.

---

## Три уровня самосознания системы

**Первый уровень — инфраструктурный.** CPU 90%, память заканчивается. Kubernetes умеет перезапускать поды. Система знает что ей **плохо**, но не знает **почему** [1].

**Второй уровень — технико-семантический.** Система знает что «аспект `charge_card` в `ProcessPaymentAction` читает 4000 блоков с диска из-за Seq Scan» [1]. Связь между бизнес-операцией и технической ценой формализована через trace_id, привязанный к Context. Это даёт idea_18 + idea_19 [1].

**Третий уровень — бизнес-семантический.** Система знает не просто что запрос тормозит, а что **заказов стало меньше**. Это не техническая метрика — это бизнес-метрика, **извлечённая из потока выполнения Actions**. И именно этот уровень делает систему живой [1].

---

## Биологическая метафора как архитектурный принцип

| Свойство живой клетки | Механизм в AOA |
|---|---|
| ДНК — клетка знает свою структуру | `_dependencies`, `_role_spec`, `_result_checkers` — метамодель, без которой код не запустится |
| Метаболизм — предсказуемое преобразование | Конвейер аспектов: `state₀ → aspect₁ → state₁ → aspect₂ → result` |
| Мембрана — контроль входа/выхода | Params (неизменяемый вход), Result (неизменяемый выход), CheckRoles (иммунный барьер) |
| Нервная система — обратная связь | Плагинная система: `global_start`, `before/after:aspect`, `global_finish` с полным контекстом |
| Иммунитет — защита от угроз | CheckRoles + Circuit Breaker (idea_15) + Rate Limiting (idea_16) |
| Гомеостаз — возврат к норме | `BusinessMetricsPlugin` со скользящим окном + автоматическая коррекция |
| Размножение с мутацией | LLM-генерация вариантов Actions из метамодели + A/B тесты (idea_14) |
| Апоптоз — программируемая гибель | Обнаружение неиспользуемых Actions через поток событий (idea_13) |
| Сигнальные каскады | `PluginEvent` как межсервисные сигнальные молекулы через MCP |

---

## Решение

### Компонент 1: BusinessMetricsPlugin — бизнес-метаболизм системы

Плагин подписывается на `global_finish` и ведёт **скользящее окно бизнес-метрик** для каждого Action. Это не отдельная система аналитики — это тот же плагинный механизм, который уже используется для логирования и трассировки [1].

```python
from dataclasses import dataclass, field
from collections import deque
from typing import Dict, Optional
import time

@dataclass
class ActionWindow:
    """Скользящее окно метрик для одного Action."""
    action_name: str
    window_seconds: int = 3600  # 1 час
    
    # Временные ряды вызовов
    calls: deque = field(default_factory=lambda: deque(maxlen=10000))
    
    # Разбивка по бизнес-контексту
    by_channel: Dict[str, deque] = field(default_factory=dict)
    by_ab_variant: Dict[str, deque] = field(default_factory=dict)
    by_segment: Dict[str, deque] = field(default_factory=dict)
    
    def record(self, result, duration, tags: dict, timestamp: float):
        record = {
            "ts": timestamp,
            "success": result is not None and not getattr(result, 'error', None),
            "duration": duration,
            "tags": tags
        }
        self.calls.append(record)
        
        # Разбивка по каналам из Context.request.tags
        channel = tags.get("source_channel", "unknown")
        if channel not in self.by_channel:
            self.by_channel[channel] = deque(maxlen=1000)
        self.by_channel[channel].append(record)
        
        # Разбивка по A/B вариантам
        ab_variant = tags.get("ab_variant")
        if ab_variant:
            if ab_variant not in self.by_ab_variant:
                self.by_ab_variant[ab_variant] = deque(maxlen=1000)
            self.by_ab_variant[ab_variant].append(record)
    
    def get_rate(self, window_seconds: int = 3600) -> float:
        """Количество успешных вызовов за период."""
        now = time.time()
        cutoff = now - window_seconds
        return sum(
            1 for r in self.calls 
            if r["ts"] > cutoff and r["success"]
        )
    
    def get_trend(self, compare_window: int = 3600) -> float:
        """
        Тренд: соотношение текущего окна к предыдущему.
        < 1.0 — деградация, > 1.0 — рост, 1.0 — стабильно.
        """
        now = time.time()
        current = sum(
            1 for r in self.calls
            if r["ts"] > now - compare_window and r["success"]
        )
        previous = sum(
            1 for r in self.calls
            if now - 2 * compare_window < r["ts"] <= now - compare_window
            and r["success"]
        )
        if previous == 0:
            return 1.0
        return current / previous
    
    def get_breakdown_by_channel(self, window_seconds: int = 3600) -> dict:
        """Разбивка успешных вызовов по каналам за период."""
        now = time.time()
        cutoff = now - window_seconds
        result = {}
        for channel, records in self.by_channel.items():
            result[channel] = sum(
                1 for r in records
                if r["ts"] > cutoff and r["success"]
            )
        return result
    
    def get_conversion_by_ab_variant(self, window_seconds: int = 3600) -> dict:
        """Конверсия по A/B вариантам."""
        now = time.time()
        cutoff = now - window_seconds
        result = {}
        for variant, records in self.by_ab_variant.items():
            recent = [r for r in records if r["ts"] > cutoff]
            if recent:
                success_count = sum(1 for r in recent if r["success"])
                result[variant] = {
                    "total": len(recent),
                    "success": success_count,
                    "conversion": success_count / len(recent) if recent else 0
                }
        return result


class BusinessMetricsPlugin(Plugin):
    """
    Плагин бизнес-метаболизма системы.
    
    Превращает поток вызовов machine.run() в наблюдаемый бизнес-поток
    без единой строчки дополнительного кода в бизнес-логике.
    """
    
    # Пороги для иммунного ответа
    DEGRADATION_THRESHOLD = 0.6    # падение на 40% → тревога
    CRITICAL_THRESHOLD = 0.4       # падение на 60% → критично
    WATCH_ACTIONS = {              # Action-ы, которые мы считаем ключевыми бизнес-KPI
        "CreateOrderAction",
        "ProcessPaymentAction",
        "RegisterUserAction",
    }
    
    def get_initial_state(self):
        return {
            "windows": {},          # action_name → ActionWindow
            "funnels": {},          # trace_id → список шагов воронки
            "ltv_data": {},         # user_id → история покупок
            "alerts_sent": set(),   # дедупликация алертов
        }
    
    @on('global_finish', '.*', ignore_exceptions=True)
    async def on_finish(self, state_plugin, event: PluginEvent):
        action_name = event.action_name.split(".")[-1]  # короткое имя
        
        # Получаем или создаём окно для этого Action
        if action_name not in state_plugin["windows"]:
            state_plugin["windows"][action_name] = ActionWindow(action_name)
        
        window = state_plugin["windows"][action_name]
        
        # Записываем вызов с тегами из Context
        tags = dict(event.context.request.tags or {})
        window.record(
            result=event.result,
            duration=event.duration,
            tags=tags,
            timestamp=time.time()
        )
        
        # Строим воронку по trace_id
        # ViewProductAction → AddToCartAction → CreateOrderAction = воронка
        trace_id = event.context.request.trace_id
        if trace_id not in state_plugin["funnels"]:
            state_plugin["funnels"][trace_id] = []
        state_plugin["funnels"][trace_id].append({
            "action": action_name,
            "success": event.result is not None,
            "ts": time.time()
        })
        
        # Проверяем гомеостаз для ключевых бизнес-KPI
        if action_name in self.WATCH_ACTIONS:
            await self._check_homeostasis(state_plugin, action_name, window)
        
        return state_plugin
    
    async def _check_homeostasis(
        self, 
        state_plugin: dict, 
        action_name: str, 
        window: ActionWindow
    ):
        """
        Гомеостаз: проверяем отклонение от нормы и активируем
        иммунный ответ пропорционально угрозе.
        """
        trend = window.get_trend(compare_window=3600)
        
        # Дедупликация: не спамим алертами
        alert_key = f"{action_name}:{int(trend * 10)}"
        if alert_key in state_plugin["alerts_sent"]:
            return
        
        if trend < self.CRITICAL_THRESHOLD:
            # Уровень CRITICAL: вызов AI-скорой через MCP
            state_plugin["alerts_sent"].add(alert_key)
            await self._call_emergency(action_name, window, level="CRITICAL")
        
        elif trend < self.DEGRADATION_THRESHOLD:
            # Уровень ALERT: детальная диагностика
            state_plugin["alerts_sent"].add(alert_key)
            await self._call_emergency(action_name, window, level="ALERT")
    
    async def _call_emergency(
        self, 
        action_name: str, 
        window: ActionWindow, 
        level: str
    ):
        """
        Вызов AI-скорой через MCP.
        
        Передаём структурированную бизнес-телеметрию — не сырые логи,
        а машиночитаемую историю болезни с полным контекстом.
        """
        breakdown_channel = window.get_breakdown_by_channel()
        breakdown_ab = window.get_conversion_by_ab_variant()
        
        # Строим воронку по последним данным
        current_rate = window.get_rate(window_seconds=3600)
        previous_rate = current_rate / (window.get_trend() or 1)
        drop_percent = round((1 - window.get_trend()) * 100, 1)
        
        business_telemetry = {
            "level": level,
            "symptom": f"{action_name} drop {drop_percent}% in 1h",
            "action_name": action_name,
            "current_rate_per_hour": current_rate,
            "previous_rate_per_hour": previous_rate,
            "trend": window.get_trend(),
            
            # Разбивка по каналам — ключ для диагностики
            "breakdown_by_channel": {
                channel: {
                    "before": int(count / window.get_trend()),
                    "after": count,
                    "drop": f"{round((1 - window.get_trend()) * 100)}%"
                }
                for channel, count in breakdown_channel.items()
            },
            
            # A/B варианты — ключ для диагностики новых фич
            "breakdown_by_ab_variant": breakdown_ab,
            
            # Воронка конверсии
            "funnel": self._build_funnel_report(),
            
            # Рекомендация по типу диагностики
            "recommended_action": (
                "rollback_recent_deploy" if level == "CRITICAL"
                else "analyze_ab_variants"
            )
        }
        
        # Вызов через MCP — остальную часть делает AI-скорая (idea_19)
        await self.mcp_client.call("ai_business_emergency", business_telemetry)
    
    def _build_funnel_report(self) -> dict:
        """Строит воронку конверсии из накопленных данных."""
        # Упрощённая воронка — реальная реализация считает по trace_id
        return {
            "note": "Built from trace_id correlation across Actions"
        }
    
    def get_metabolic_map(self) -> dict:
        """
        Метаболическая карта системы.
        
        Соотношение вызовов разных Actions — как в биологии
        соотношение метаболических путей.
        Изменение соотношения = сигнал о здоровье бизнеса.
        """
        return {
            name: {
                "rate_1h": window.get_rate(3600),
                "rate_24h": window.get_rate(86400),
                "trend_1h": window.get_trend(3600),
                "channels": window.get_breakdown_by_channel(3600),
                "ab_variants": window.get_conversion_by_ab_variant(3600),
            }
            for name, window in self.state["windows"].items()
        }
```

### Компонент 2: ImmunityPlugin — адаптивный иммунный ответ

Иммунная система AOA — это не просто Circuit Breaker (idea_15). Это **обучающийся иммунитет**, который запоминает паттерны болезней и автоматически адаптирует уровень наблюдения.

```python
from enum import Enum
from dataclasses import dataclass
from typing import Optional

class ImmunityLevel(Enum):
    HEALTHY  = "healthy"    # норма
    WATCH    = "watch"      # p99 выросло в 1.5x → подробные логи
    ALERT    = "alert"      # p99 выросло в 3x → DIAGNOSTIC профиль БД
    CRITICAL = "critical"   # p99 выросло в 5x → вызов AI-скорой


@dataclass
class ImmunityMemoryRecord:
    """Запись иммунной памяти — задокументированный инцидент."""
    action_name: str
    symptoms: dict          # метрики в момент инцидента
    diagnosis: str          # поставленный диагноз
    treatment: str          # применённое лечение
    treatment_result: str   # результат: healed / no_effect / worsened
    timestamp: float
    
    def similarity(self, current_symptoms: dict) -> float:
        """
        Насколько текущие симптомы похожи на прошлый инцидент.
        Простая реализация — можно заменить ML-моделью.
        """
        score = 0.0
        checks = [
            ("p99_growth_rate", 0.3),
            ("error_rate_delta", 0.3),
            ("timeout_rate", 0.2),
            ("channel_breakdown_anomaly", 0.2),
        ]
        for key, weight in checks:
            if key in self.symptoms and key in current_symptoms:
                ratio = min(
                    self.symptoms[key], current_symptoms[key]
                ) / max(
                    self.symptoms[key], current_symptoms[key], 0.001
                )
                score += ratio * weight
        return score


class ImmunityPlugin(Plugin):
    """
    Адаптивная иммунная система AOA.
    
    Аналог биологического иммунитета:
    - Врождённый иммунитет: Circuit Breaker + Rate Limiting (idea_15, idea_16)
    - Приобретённый иммунитет: обучение на инцидентах, иммунная память
    - Адаптивный ответ: пропорциональное усиление наблюдения
    - Вакцинация: применение известных лечений до развёртывания болезни
    """
    
    # Пороги иммунного ответа
    THRESHOLDS = {
        ImmunityLevel.WATCH:    {"p99_growth": 1.5, "error_delta": 0.02, "timeout": 0.05},
        ImmunityLevel.ALERT:    {"p99_growth": 3.0, "error_delta": 0.05, "timeout": 0.10},
        ImmunityLevel.CRITICAL: {"p99_growth": 5.0, "error_delta": 0.10, "timeout": 0.20},
    }
    
    def get_initial_state(self):
        return {
            "stats": {},            # action_name → статистика
            "immunity_levels": {},  # action_name → ImmunityLevel
            "memory": [],           # список ImmunityMemoryRecord
            "active_treatments": {} # action_name → активное лечение
        }
    
    @on('global_finish', '.*', ignore_exceptions=True)
    async def on_finish(self, state_plugin, event: PluginEvent):
        action_name = event.action_name.split(".")[-1]
        
        # Обновляем статистику
        self._update_stats(state_plugin, action_name, event)
        
        # Оцениваем уровень угрозы
        current_symptoms = self._calculate_symptoms(
            state_plugin["stats"].get(action_name, {})
        )
        new_level = self._classify_threat(current_symptoms)
        old_level = state_plugin["immunity_levels"].get(
            action_name, ImmunityLevel.HEALTHY
        )
        
        # Иммунный ответ при изменении уровня
        if new_level != old_level:
            state_plugin["immunity_levels"][action_name] = new_level
            await self._mount_immune_response(
                state_plugin, action_name, new_level, current_symptoms
            )
        
        # Снятие диагностики при выздоровлении
        if new_level == ImmunityLevel.HEALTHY and old_level != ImmunityLevel.HEALTHY:
            await self._deactivate_diagnostics(state_plugin, action_name)
        
        return state_plugin
    
    def _update_stats(self, state_plugin, action_name, event):
        """Обновляем скользящую статистику."""
        if action_name not in state_plugin["stats"]:
            state_plugin["stats"][action_name] = {
                "durations": deque(maxlen=1000),
                "errors": deque(maxlen=1000),
                "timeouts": deque(maxlen=100),
            }
        
        stats = state_plugin["stats"][action_name]
        stats["durations"].append(event.duration or 0)
        stats["errors"].append(event.result is None)
        if event.duration and event.duration > getattr(event, 'max_duration', float('inf')):
            stats["timeouts"].append(time.time())
    
    def _calculate_symptoms(self, stats: dict) -> dict:
        """Вычисляем симптомы из статистики."""
        if not stats or not stats.get("durations"):
            return {}
        
        durations = list(stats["durations"])
        errors = list(stats["errors"])
        
        if len(durations) < 10:
            return {}
        
        # p99 последних 100 вызовов vs предыдущих 100
        recent = sorted(durations[-100:])
        previous = sorted(durations[-200:-100]) if len(durations) > 100 else recent
        
        p99_recent = recent[int(len(recent) * 0.99)]
        p99_previous = previous[int(len(previous) * 0.99)] if previous else p99_recent
        
        return {
            "p99_growth_rate": p99_recent / max(p99_previous, 0.001),
            "error_rate_delta": (
                sum(errors[-50:]) / 50 - sum(errors[-100:-50]) / 50
            ) if len(errors) >= 100 else 0,
            "timeout_rate": len([
                t for t in stats.get("timeouts", [])
                if t > time.time() - 300
            ]) / max(len(durations[-100:]), 1),
        }
    
    def _classify_threat(self, symptoms: dict) -> ImmunityLevel:
        """Классифицируем уровень угрозы."""
        if not symptoms:
            return ImmunityLevel.HEALTHY
        
        for level in [ImmunityLevel.CRITICAL, ImmunityLevel.ALERT, ImmunityLevel.WATCH]:
            thresholds = self.THRESHOLDS[level]
            if (
                symptoms.get("p99_growth_rate", 1.0) >= thresholds["p99_growth"]
                or symptoms.get("error_rate_delta", 0) >= thresholds["error_delta"]
                or symptoms.get("timeout_rate", 0) >= thresholds["timeout"]
            ):
                return level
        
        return ImmunityLevel.HEALTHY
    
    async def _mount_immune_response(
        self,
        state_plugin: dict,
        action_name: str,
        level: ImmunityLevel,
        current_symptoms: dict
    ):
        """
        Пропорциональный иммунный ответ.
        
        Организм не включает весь иммунитет сразу — реакция
        пропорциональна угрозе. Так же и здесь.
        """
        # Проверяем иммунную память — знаем ли мы эту болезнь?
        known_treatment = self._find_known_treatment(
            state_plugin, action_name, current_symptoms
        )
        
        if known_treatment and level != ImmunityLevel.CRITICAL:
            # Вакцинация: применяем известное лечение сразу
            await self._apply_treatment(action_name, known_treatment)
            state_plugin["active_treatments"][action_name] = known_treatment
            return
        
        if level == ImmunityLevel.WATCH:
            # Повышаем уровень логирования для этого Action точечно
            await self._set_verbose_logging(action_name, enabled=True)
        
        elif level == ImmunityLevel.ALERT:
            # Включаем DIAGNOSTIC профиль БД (idea_18)
            await self._set_db_profiling(action_name, profile="DIAGNOSTIC")
            await self._set_verbose_logging(action_name, enabled=True)
        
        elif level == ImmunityLevel.CRITICAL:
            # Вызываем AI-скорую с полной медицинской картой
            await self._call_ai_emergency(action_name, current_symptoms, level)
    
    async def _deactivate_diagnostics(self, state_plugin, action_name: str):
        """
        Снятие диагностики при выздоровлении.
        Адаптивное затухание — как снижение температуры после инфекции.
        """
        await self._set_verbose_logging(action_name, enabled=False)
        await self._set_db_profiling(action_name, profile="STANDARD")
        
        # Сохраняем в иммунную память
        if action_name in state_plugin["active_treatments"]:
            treatment = state_plugin["active_treatments"].pop(action_name)
            # Записываем успешное лечение
            state_plugin["memory"].append(ImmunityMemoryRecord(
                action_name=action_name,
                symptoms={},  # заполняется из статистики
                diagnosis="auto-resolved",
                treatment=treatment,
                treatment_result="healed",
                timestamp=time.time()
            ))
    
    def _find_known_treatment(
        self, 
        state_plugin: dict, 
        action_name: str, 
        symptoms: dict
    ) -> Optional[str]:
        """
        Поиск в иммунной памяти.
        
        Если болезнь уже встречалась — применяем известное лечение
        без вызова AI-скорой. Это экономит время и ресурсы.
        """
        best_match = None
        best_score = 0.7  # минимальный порог схожести
        
        for record in state_plugin["memory"]:
            if (
                record.action_name == action_name
                and record.treatment_result == "healed"
            ):
                score = record.similarity(symptoms)
                if score > best_score:
                    best_score = score
                    best_match = record.treatment
        
        return best_match
    
    async def _set_verbose_logging(self, action_name: str, enabled: bool):
        """Хирургически точное включение/выключение детального логирования."""
        # Через Redis или общий конфиг — только для конкретного Action
        await self.config_store.set(
            f"logging:{action_name}:verbose",
            enabled,
            ttl=3600 if enabled else None
        )
    
    async def _set_db_profiling(self, action_name: str, profile: str):
        """Динамическое переключение профиля БД (idea_18)."""
        await self.config_store.set(
            f"db_profiling:{action_name}",
            profile,
            ttl=1800 if profile == "DIAGNOSTIC" else None
        )
    
    async def _call_ai_emergency(
        self, 
        action_name: str, 
        symptoms: dict, 
        level: str
    ):
        """Вызов AI-скорой с структурированной медицинской картой."""
        await self.mcp_client.call("ai_immunity_emergency", {
            "action_name": action_name,
            "level": level,
            "symptoms": symptoms,
            "medical_history": [
                {
                    "diagnosis": r.diagnosis,
                    "treatment": r.treatment,
                    "result": r.treatment_result,
                    "days_ago": int((time.time() - r.timestamp) / 86400)
                }
                for r in self.immunity_memory
                if r.action_name == action_name
            ]
        })
    
    async def _apply_treatment(self, action_name: str, treatment: str):
        """Применение известного лечения."""
        # Конкретные реализации зависят от типа лечения
        treatment_handlers = {
            "increase_rate_limit": self._increase_rate_limit,
            "enable_cache": self._enable_cache,
            "switch_to_replica": self._switch_to_replica,
        }
        handler = treatment_handlers.get(treatment)
        if handler:
            await handler(action_name)
```

### Компонент 3: BusinessEvolutionPlugin — самоадаптация бизнес-поведения

Это четвёртый уровень — уже не самолечение, а **самооптимизация**. Система не просто чинит себя — она адаптирует бизнес-стратегию [1].

```python
from dataclasses import dataclass
from typing import List, Optional
import hashlib

@dataclass
class ABExperiment:
    """Описание активного A/B эксперимента."""
    experiment_id: str
    action_name: str
    feature_flag: str       # idea_14
    hypothesis: str
    traffic_percent: int    # сколько % трафика видит эксперимент
    start_time: float
    target_metric: str      # "conversion", "revenue", "ltv"
    baseline: float         # значение метрики до эксперимента
    
    # Накопленные результаты
    control_metrics: dict = None    # метрики контрольной группы
    variant_metrics: dict = None    # метрики варианта


class BusinessEvolutionPlugin(Plugin):
    """
    Плагин самоадаптации бизнес-поведения.
    
    Аналог эволюции в биологии:
    - Гомеостаз: система замечает отклонение от бизнес-целей
    - Мутация: LLM предлагает вариации (A/B тесты)
    - Отбор: автоматический анализ результатов
    - Закрепление: масштабирование успешных вариантов
    
    Ключевое: адаптация бизнес-поведения, а не технических параметров.
    Система не просто работает — она ДОСТИГАЕТ БИЗНЕС-ЦЕЛЕЙ.
    """
    
    # Минимальная статистическая значимость для принятия решения
    MIN_SAMPLES_PER_VARIANT = 100
    CONFIDENCE_THRESHOLD = 0.95
    
    def get_initial_state(self):
        return {
            "active_experiments": {},   # experiment_id → ABExperiment
            "completed_experiments": [],
            "ltv_cohorts": {},          # channel → LTV данные
            "evolution_log": [],        # история адаптаций
        }
    
    @on('global_finish', '.*', ignore_exceptions=True)
    async def on_finish(self, state_plugin, event: PluginEvent):
        action_name = event.action_name.split(".")[-1]
        tags = dict(event.context.request.tags or {})
        
        # Обновляем LTV-когорты (для стратегических решений)
        channel = tags.get("source_channel")
        if channel and event.result:
            await self._update_ltv_cohort(
                state_plugin, 
                channel, 
                event.context.user.user_id,
                event.result
            )
        
        # Обновляем активные эксперименты
        ab_variant = tags.get("ab_variant")
        experiment_id = tags.get("experiment_id")
        if ab_variant and experiment_id:
            await self._update_experiment(
                state_plugin, experiment_id, ab_variant, event
            )
        
        return state_plugin
    
    async def _update_ltv_cohort(
        self, 
        state_plugin: dict, 
        channel: str, 
        user_id: str, 
        result
    ):
        """
        Отслеживаем LTV по каналам привлечения.
        
        Это позволяет AI-скорой делать стратегические выводы:
        не просто "конверсия упала", а "Instagram-аудитория имеет
        LTV в 9 раз ниже Google-аудитории — перераспределить бюджет".
        """
        if channel not in state_plugin["ltv_cohorts"]:
            state_plugin["ltv_cohorts"][channel] = {
                "users": {},
                "total_revenue": 0.0,
                "total_orders": 0,
                "repeat_rate": 0.0,
            }
        
        cohort = state_plugin["ltv_cohorts"][channel]
        
        # Извлекаем бизнес-метрики из Result
        order_total = getattr(result, 'total', 0) or 0
        cohort["total_revenue"] += order_total
        cohort["total_orders"] += 1
        
        if user_id not in cohort["users"]:
            cohort["users"][user_id] = {"orders": 0, "revenue": 0}
        
        cohort["users"][user_id]["orders"] += 1
        cohort["users"][user_id]["revenue"] += order_total
        
        # Обновляем repeat rate
        repeat_users = sum(
            1 for u in cohort["users"].values() if u["orders"] > 1
        )
        cohort["repeat_rate"] = repeat_users / max(len(cohort["users"]), 1)
    
    async def _update_experiment(
        self,
        state_plugin: dict,
        experiment_id: str,
        ab_variant: str,
        event: PluginEvent
    ):
        """Обновляем метрики активного эксперимента."""
        if experiment_id not in state_plugin["active_experiments"]:
            return
        
        experiment = state_plugin["active_experiments"][experiment_id]
        
        # Определяем в какую группу попал вызов
        is_variant = ab_variant != "control"
        target = (
            experiment.variant_metrics if is_variant
            else experiment.control_metrics
        )
        if target is None:
            if is_variant:
                experiment.variant_metrics = {"count": 0, "success": 0, "revenue": 0}
            else:
                experiment.control_metrics = {"count": 0, "success": 0, "revenue": 0}
            target = experiment.variant_metrics if is_variant else experiment.control_metrics
        
        target["count"] += 1
        if event.result:
            target["success"] += 1
            target["revenue"] += getattr(event.result, 'total', 0) or 0
        
        # Проверяем готовность принять решение
        await self._maybe_conclude_experiment(state_plugin, experiment_id)
    
    async def _maybe_conclude_experiment(
        self, 
        state_plugin: dict, 
        experiment_id: str
    ):
        """
        Автоматическое принятие решения по эксперименту.
        
        Это и есть 'естественный отбор' — лучший вариант
        масштабируется автоматически, неудачные — отбрасываются.
        """
        experiment = state_plugin["active_experiments"].get(experiment_id)
        if not experiment:
            return
        
        control = experiment.control_metrics or {}
        variant = experiment.variant_metrics or {}
        
        if (
            control.get("count", 0) < self.MIN_SAMPLES_PER_VARIANT
            or variant.get("count", 0) < self.MIN_SAMPLES_PER_VARIANT
        ):
            return  # Недостаточно данных
        
        # Вычисляем конверсию по целевой метрике
        control_conv = control["success"] / max(control["count"], 1)
        variant_conv = variant["success"] / max(variant["count"], 1)
        
        winner = None
        if variant_conv > control_conv * 1.05:  # +5% → вариант победил
            winner = "variant"
        elif control_conv > variant_conv * 1.05:  # -5% → контроль победил
            winner = "control"
        
        if winner:
            await self._conclude_experiment(
                state_plugin, experiment_id, winner,
                control_conv, variant_conv
            )
    
    async def _conclude_experiment(
        self,
        state_plugin: dict,
        experiment_id: str,
        winner: str,
        control_conv: float,
        variant_conv: float
    ):
        """
        Закрепление результата эксперимента.
        
        Победитель масштабируется на 100% трафика.
        Это 'закрепление мутации' в эволюционном смысле.
        """
        experiment = state_plugin["active_experiments"].pop(experiment_id)
        
        result = {
            "experiment_id": experiment_id,
            "hypothesis": experiment.hypothesis,
            "winner": winner,
            "control_conversion": control_conv,
            "variant_conversion": variant_conv,
            "lift": (variant_conv - control_conv) / max(control_conv, 0.001),
            "concluded_at": time.time()
        }
        
        state_plugin["completed_experiments"].append(result)
        state_plugin["evolution_log"].append({
            "type": "experiment_concluded",
            "result": result,
            "ts": time.time()
        })
        
        # Если вариант победил — масштабируем через feature flag (idea_14)
        if winner == "variant":
            await self.feature_flag_provider.set_rollout(
                experiment.feature_flag, percent=100
            )
        else:
            # Контроль победил — отключаем вариант
            await self.feature_flag_provider.set_rollout(
                experiment.feature_flag, percent=0
            )
        
        # Уведомляем AI-аналитика для стратегических выводов
        await self.mcp_client.call("ai_evolution_report", {
            "experiment": result,
            "ltv_cohorts": self._get_ltv_summary(state_plugin),
            "recommendation_needed": True
        })
    
    def _get_ltv_summary(self, state_plugin: dict) -> dict:
        """
        LTV-сводка по каналам — стратегические данные для AI.
        
        Именно на основе этих данных AI может сказать:
        'Instagram LTV в 9 раз ниже Google — перераспределить бюджет'.
        """
        summary = {}
        for channel, cohort in state_plugin["ltv_cohorts"].items():
            user_count = len(cohort["users"])
            if user_count > 0:
                summary[channel] = {
                    "users": user_count,
                    "avg_ltv": cohort["total_revenue"] / user_count,
                    "avg_order_value": (
                        cohort["total_revenue"] / max(cohort["total_orders"], 1)
                    ),
                    "repeat_rate": cohort["repeat_rate"],
                    "total_revenue": cohort["total_revenue"],
                }
        return summary
    
    def launch_experiment(
        self,
        action_name: str,
        feature_flag: str,
        hypothesis: str,
        traffic_percent: int = 5,
        target_metric: str = "conversion"
    ) -> str:
        """
        Запуск нового A/B эксперимента.
        
        Может вызываться как вручную, так и автоматически
        AI-агентом при обнаружении бизнес-проблемы.
        """
        experiment_id = hashlib.md5(
            f"{action_name}:{feature_flag}:{time.time()}".encode()
        ).hexdigest()[:8]
        
        self.state["active_experiments"][experiment_id] = ABExperiment(
            experiment_id=experiment_id,
            action_name=action_name,
            feature_flag=feature_flag,
            hypothesis=hypothesis,
            traffic_percent=traffic_percent,
            start_time=time.time(),
            target_metric=target_metric,
            baseline=0.0,  # заполняется из текущих метрик
        )
        
        return experiment_id
```

### Компонент 4: Подключение — ноль изменений в бизнес-логике

## Уникальность

### 1. Единственный фреймворк с бизнес-семантическим уровнем самосознания

В обычных системах мониторинга алерты — это всегда про железо: «диск переполнен», «CPU 90%». Чтобы понять что происходит с бизнесом, инженер делает ручную «ментальную трансляцию» [1].

В AOA система знает не просто «запрос тормозит», а **«заказов стало меньше, и вот почему»**. Это третий уровень самосознания — бизнес-семантический [1]:

```
Уровень 1 (все): CPU 90% → перезапустить pod
Уровень 2 (idea_18+19): charge_card читает 4000 блоков → добавить индекс  
Уровень 3 (idea_20): заказов −40%, Instagram-лиды неэффективны → сменить таргетинг
```

### 2. Бизнес-события как побочный продукт архитектуры

В обычном фреймворке для бизнес-аналитики нужны отдельные подсистемы [1]. В AOA плагин на `global_finish` получает `action_name`, `result`, `duration`, `context` с тегами — это уже структурированное бизнес-событие [1].

**Не нужно строить аналитику** — она является следствием того, что все бизнес-операции проходят через `machine.run()`.

### 3. A/B тестирование встроено в архитектуру, а не прибито сбоку

В традиционных системах A/B тесты — это отдельная инфраструктура (LaunchDarkly, Optimizely), дашборды, ручной анализ [1]. В AOA:

- `Context.request.tags` уже несут `ab_variant` — без дополнительного кода
- `feature_flags` (idea_14) управляют процентом трафика декларативно
- `ExecutionTreePlugin` (idea_17) автоматически разбивает телеметрию по вариантам
- `BusinessEvolutionPlugin` делает **статистически обоснованные решения автоматически**

### 4. Иммунная память — система учится, а не реагирует

Обычный Circuit Breaker (idea_15) реагирует на каждый инцидент одинаково. `ImmunityPlugin` из idea_20 **запоминает** прошлые инциденты и при повторении применяет известное лечение без вызова AI-скорой. Это разница между реактивным мониторингом и проактивным иммунитетом.

### 5. LTV-осознание — система понимает долгосрочную ценность

Ни один APM или система мониторинга не умеет сказать: «Instagram-аудитория имеет LTV в 9 раз ниже Google — перераспределить бюджет». Это возможно только когда:

- `CreateOrderAction.Result` содержит `total` (типизированный выход)
- `RepeatOrderAction` привязан к тому же `user_id` из Context
- Плагин автоматически строит когорты по `source_channel` из `Context.request.tags`

Всё это является следствием архитектуры AOA, а не отдельной системой аналитики.

### 6. Три уровня адаптивности в одной точке

```
machine.run()
    ↓
[Rate Limiting] — защита от перегрузки (idea_16)
    ↓
[Circuit Breaker] — защита от сбоев (idea_15)
    ↓
[Feature Flags] — ручное управление (idea_14)
    ↓
[ImmunityPlugin] — технический иммунитет (idea_20)
    ↓
[BusinessMetricsPlugin] — бизнес-гомеостаз (idea_20)
    ↓
[BusinessEvolutionPlugin] — бизнес-эволюция (idea_20)
    ↓
[Бизнес-логика Action]
```

Все шесть уровней защиты и адаптации проверяются **в одной точке**. В обычном фреймворке каждый из них — отдельная система с отдельной инфраструктурой.

## Влияние

- **Скорость реакции на бизнес-проблемы** — с часов до минут. Не нужен инженер который делает «ментальную трансляцию»
- **Автоматический маркетинг** — система сама ищет лучшие каналы, запускает A/B тесты, масштабирует победителей
- **LTV-осознание** — стратегические решения о каналах привлечения принимаются на основе реальных данных, а не интуиции
- **Самообучение** — иммунная память накапливает знания об инцидентах, со временем система лечится быстрее
- **Нулевые изменения в бизнес-логике** — все механизмы работают как плагины поверх существующей архитектуры

## Заключение

Idea_20 — это не просто новая фича. Это **смена биологического статуса кода** [1].

В классической инженерии код пассивен: он выполняет инструкции, а люди занимаются мониторингом, анализом и адаптацией. В AOA код становится **активным участником собственного жизненного цикла** — он чувствует бизнес-проблему, диагностирует причину, экспериментирует с решениями и масштабирует успешное.

Обычный разработчик пишет скрипты чтобы поддерживать работоспособность кирпичей. Idea_20 создаёт архитектуру в которой **система сама заинтересована в своём выживании и развитии** [1].

Это уже не программирование — это создание цифровой жизни [2].
