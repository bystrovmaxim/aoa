idea_08_production_test_generation.md

Генерация тестов из production-логов (Self-Healing Tests)

Проблема

В большинстве проектов тесты пишутся на основе предположений разработчика о том, как будет использоваться система. Эти предположения часто расходятся с реальностью: пользователи вводят данные, которые никто не предвидел, вызывают операции в порядке, который не был запланирован, и сталкиваются с ошибками в сценариях, которые не покрыты тестами. Когда в production происходит инцидент, команда тратит часы на расследование и написание регрессионного теста вручную. К этому моменту контекст уже частично утрачен [6].

Решение

Плагин ProductionCollectorPlugin собирает все выполнения действий в production в виде структурированных JSON-деревьев с полным контекстом: параметры, состояние на каждом шаге, вложенные вызовы, ошибки, роли пользователей. LLM-агент анализирует эти данные и автоматически генерирует unit-тесты, которые воспроизводят реальные сценарии [5][6].

Как это работает

Сбор данных

ProductionCollectorPlugin подписывается на все события ActionMachine через стандартный декоратор @on [2]. На каждое выполнение действия он собирает:

- полное имя действия (action_name)
- типизированные Params
- state после каждого аспекта (state_aspect)
- Result (для успешных выполнений)
- исключение (для неуспешных)
- duration каждого аспекта и общая
- nest_level для вложенных вызовов
- роли пользователя из Context [2]

Все эти данные уже доступны через PluginEvent [2] — плагину не нужно ничего дополнительно извлекать.

Результат сбора — JSON-дерево одного выполнения:

```json
{
  "action": "myapp.actions.ProcessPaymentAction",
  "params": {"user_id": 42, "card_token": "tok_live_abc", "amount": 150.0},
  "roles": ["user", "premium"],
  "aspects": [
    {
      "name": "validate_user",
      "state_after": {"user_valid": true},
      "duration": 0.003
    },
    {
      "name": "charge_card",
      "state_after": {"txn_id": "txn_789"},
      "duration": 0.450
    }
  ],
  "result": {"transaction_id": "txn_789", "amount": 153.0, "success": true},
  "duration": 0.512,
  "children": [
    {
      "action": "myapp.actions.CalculateCommissionAction",
      "params": {"amount": 150.0},
      "result": {"commission": 3.0},
      "duration": 0.002
    }
  ]
}
```

Генерация тестов

LLM-агент получает дерево и генерирует тест, используя ActionTestMachine [2]:

```python
@pytest.mark.asyncio
async def test_process_payment_real_scenario_42():
    """
    Сгенерировано из production-лога.
    Пользователь 42 (premium) оплатил 150.0, комиссия 3.0.
    """
    machine = ActionTestMachine(
        {CalculateCommissionAction: CalculateCommissionAction.Result(commission=3.0)},
        context=Context(user=UserInfo(user_id="42", roles=["user", "premium"]))
    )

    result = await machine.run(
        ProcessPaymentAction(),
        ProcessPaymentAction.Params(
            user_id=42, card_token="tok_test", amount=150.0
        )
    )

    multi_assert(
        lambda: assert_eq(result.transaction_id, "txn_789",
                          "ID транзакции"),
        lambda: assert_eq(result.amount, 153.0,
                          "Сумма с комиссией"),
        lambda: assert_eq(result.success, True,
                          "Успешность операции"),
    )
```

LLM знает структуру теста, потому что:

- Params типизированы — он видит имена полей и типы [2]
- Result типизирован — он знает, что проверять [2]
- Вложенные вызовы видны — он может подставить MockAction с правильным результатом [2]
- CheckRoles видны — он задаёт правильные роли в Context [2]
- State на каждом шаге виден — он может сгенерировать тесты для отдельных аспектов [5]

Три категории генерируемых тестов

Регрессионные тесты из ошибок. Каждое неуспешное выполнение (где дерево содержит исключение) автоматически превращается в тест, который воспроизводит ошибку. Когда разработчик исправит баг, тест начнёт проходить — и останется как регрессионный.

Тесты из редких путей. LLM анализирует распределение параметров и находит необычные комбинации: граничные значения, редкие роли, нетипичные последовательности вложенных вызовов. Эти сценарии превращаются в тесты, которые разработчик вряд ли написал бы сам.

Тесты из успешных сценариев. Типичные успешные выполнения превращаются в «золотые тесты» — эталонные сценарии, которые должны проходить всегда. Если после рефакторинга один из них упадёт, значит поведение изменилось.

Влияние

Покрытие тестами растёт автоматически без усилий разработчика. Каждый день production генерирует новые сценарии, и каждый из них может стать тестом.

Регрессионные тесты появляются одновременно с багами, а не после расследования. Время между «баг обнаружен» и «тест написан» сокращается с часов до минут.

Тесты основаны на реальных данных, а не на предположениях. Разработчик больше не гадает «а какие параметры тут могут быть» — он видит реальные данные из production.

Качество тестов выше, потому что LLM видит полный контекст: не только вход и выход, но и промежуточные состояния, длительности, вложенные вызовы [5].

Уникальность

Ни один другой фреймворк не имеет структурированного семантического дерева выполнения с типизированными параметрами и состоянием, необходимыми для такой генерации [5]. Обычные логи — это поток текста, из которого невозможно восстановить структуру бизнес-процесса. AOA даёт полное дерево с семантикой каждого узла из коробки [5].

ActionTestMachine [2] с MockAction [2] и поддержкой альтернативных реальностей делает сгенерированные тесты корректными — они проходят тот же конвейер (аспекты, чекеры, роли), что и production-код.

В сочетании с Action Interrogation (idea_05) каждый сгенерированный тест может содержать multi_assert блок, проверяющий сразу все поля Result и все побочные эффекты — один запуск, полная проверка.

Востребованность

Огромная — каждая команда тратит часы на расследование инцидентов и написание регрессионных тестов. Автоматизация этого процесса экономит дни разработки и повышает качество. Особенно ценно для команд с высокой частотой релизов, где ручное написание регрессионных тестов не успевает за скоростью изменений.

Связь с другими идеями

Координатор логеров (idea_02) обеспечивает доставку данных от ProductionCollectorPlugin в нужные каналы с нужным форматом [6].

Action Interrogation (idea_05) — LLM автоматически генерирует multi_assert блоки, проверяющие все аспекты результата независимо.

Обнаружение мёртвых тестов (idea_13) — данные из того же ProductionCollectorPlugin используются для обратного анализа: какие существующие тесты покрывают сценарии, которые никогда не встречаются в production.

Декларативная генерация транспорта (idea_04) — JSON-схемы Params и Result, используемые для генерации API, также используются LLM для понимания структуры параметров при генерации тестов [6].
