class TelemetryCollector:
    """
    Собирает телеметрию через pg_stat_statements.
    
    Подход:
    1. Каждый SQL в пакете получает комментарий с trace_id
    2. Перед пакетом — snapshot pg_stat_statements для backend_pid
    3. После пакета — diff
    4. Результат пишем асинхронно в отдельную таблицу/отправляем в collector
    """

    def __init__(self, pool: asyncpg.Pool):
        self.pool = pool

    def _tag_sql(self, sql: str, trace_id: str, seq: int) -> str:
        """
        Добавляем комментарий в SQL.
        pg_stat_statements нормализует запросы, но комментарии
        в некоторых конфигурациях сохраняются.
        
        ВАЖНО: pg_stat_statements.track_comments = on (PG14+)
        """
        return f"/* trace_id={trace_id},seq={seq} */ {sql}"

    async def _get_backend_stats(
        self, conn: asyncpg.Connection
    ) -> dict:
        """Снимаем IO-статистику текущего бэкенда"""

                s.tup_inserted,
                s.tup_updated,

Исследование: телеметрия PostgreSQL для Unit of Work
По твоим ответам
1. Про 100 пакетов/сек
100 UoW/сек — это средняя нагрузка для бизнес-приложения. Не низкая, но и не экстремальная. Для контекста:

Низкая: 1-10 UoW/сек — внутренний CRM, бухгалтерия
Средняя: 50-200 UoW/сек — e-commerce, SaaS-платформа, логистика
Высокая: 1000+ UoW/сек — платёжные системы, ad-tech
Экстремальная: 10000+ UoW/сек — соцсети, телеком
При 100 UoW/сек у тебя есть 10мс бюджета на телеметрию одного пакета прежде чем она станет заметной. Это очень комфортно — можно позволить себе довольно подробный сбор.

2. Стратегии телеметрии — от лёгкой к тяжёлой
Уровень 0 — Только клиентская сторона (overhead ~0%)
Что получаем:

Wall clock time (замер в Python)
Количество запросов в пакете
Rows affected (из ответа PostgreSQL на каждую команду)
Что НЕ получаем:

Ничего со стороны базы
Как: просто time.monotonic() вокруг транзакции

Уровень 1 — pg_stat_statements diff (overhead < 1%)
Что получаем:

Всё из уровня 0
shared/local/temp blocks hit/read/write
blk_read_time, blk_write_time (если track_io_timing = on)
WAL bytes generated (PG13+)
total_exec_time со стороны сервера (точнее чем wall clock)
Что НЕ получаем:

CPU отдельно от IO
Потребление памяти
Как работает:

Перед транзакцией — SELECT из pg_stat_statements агрегат по текущему user+database
Выполняем пакет
После — ещё один SELECT, считаем diff
Проблема точности: pg_stat_statements — глобальная статистика. Если параллельно работают другие соединения от того же пользователя, diff будет "загрязнён" их метриками. При 100 UoW/сек это реальная проблема.

Решение проблемы: фильтровать по queryid. Каждый нормализованный запрос имеет уникальный queryid. Если ты знаешь свои запросы — можешь брать diff только по ним. Но всё равно не идеально если один и тот же запрос выполняется параллельно.

Уровень 2 — EXPLAIN (ANALYZE, BUFFERS, WAL) (overhead 2-5%)
Что получаем:

Всё из уровня 1
Точные данные по каждому запросу, без загрязнения
Shared/local/temp buffers hit/read/dirtied/written — для конкретного выполнения
WAL records и bytes
Planning time vs Execution time отдельно
Полный план выполнения (для диагностики)
Что НЕ получаем:

CPU (отдельно)
Память
Как работает: каждый запрос в пакете оборачивается в EXPLAIN (ANALYZE, BUFFERS, WAL, FORMAT JSON) <sql>. Запрос реально выполняется (это не dry-run), но к нему добавляется инструментирование.

Overhead: ~2-5% потому что:

Планировщик делает дополнительную работу по сбору runtime-статистики
JSON-сериализация плана
Парсинг JSON на стороне Python
Критический нюанс: EXPLAIN ANALYZE для DML (INSERT/UPDATE/DELETE) реально выполняет изменения. Это нормально внутри транзакции — они либо коммитятся, либо откатываются. Но нужно аккуратно обрабатывать RETURNING и побочные эффекты.

Уровень 3 — pg_stat_kcache (overhead 2-3%, но нужно расширение)
Что получаем:

Всё из уровня 1
CPU user time и system time (из getrusage)
Page faults (minor/major) — прокси для использования памяти
Физические reads/writes на уровне ОС (не только buffer pool)
Что НЕ получаем:

RSS / heap использование конкретного бэкенда (этого PostgreSQL просто не умеет отдавать per-query)
Как работает: расширение pg_stat_kcache хукается в те же точки что и pg_stat_statements и вызывает getrusage() до и после каждого запроса. Данные доступны через view pg_stat_kcache.

Важно: это расширение от Dalibo, не входит в core. Нужно компилировать или ставить из пакетов. На managed PostgreSQL (RDS, Cloud SQL) — недоступно.

Уровень 4 — Комбинированный (мой рекомендуемый для тебя)
EXPLAIN ANALYZE + pg_stat_kcache если доступен, иначе EXPLAIN ANALYZE + pg_stat_statements diff

Что получаем:

Точные buffer-метрики на запрос (из EXPLAIN)
CPU и page faults (из pg_stat_kcache, если есть)
WAL bytes
Planning vs Execution time
trace_id привязан через SQL-комментарий

Если очередь переполнена — дропаем (телеметрия не стоит того чтобы тормозить бизнес-логику)
4. asyncpg vs psycopg3 (async)
asyncpg
Архитектура: написан на Cython, использует нативный PostgreSQL binary protocol напрямую, без libpq.

Плюсы для нашей задачи:

Быстрее на ~2-3x для простых запросов
conn.get_server_pid() — мгновенно, без запроса
prepared statements кешируются автоматически — EXPLAIN ANALYZE для каждого запроса не нуждается в повторном planning
Бинарный протокол — меньше парсинга
Минусы для нашей задачи:

EXPLAIN ANALYZE возвращает текст, парсить нужно самим
Нет встроенного хука "до/после каждого запроса" — нужно оборачивать вручную
Пул соединений (asyncpg.Pool) — свой, не совместим с SQLAlchemy напрямую
copy протокол — свой API
Нюанс точности: asyncpg кеширует prepared statements. Когда ты отправляешь /* trace_id=abc */ SELECT * FROM users WHERE id = $1, pg_stat_statements увидит нормализованную версию. С track_comments = on (PG14+) комментарий сохранится в pg_stat_statements.query, но queryid будет одинаковым для разных trace_id — это хорошо для агрегации, но нужно иметь в виду.

psycopg3 (async)
Архитектура: обёртка над libpq, async через psycopg.AsyncConnection, использует текстовый или бинарный протокол.

Плюсы для нашей задачи:

Pipeline mode (PG14+) — можно отправить все запросы пакетом без ожидания ответа на каждый, потом собрать все результаты
cursor.statusmessage — rows affected из коробки
conn.execute() поддерживает prepare=True/False — контроль над кешированием
Лучше совместимость с SQLAlchemy 2.0
conn.notifies() — удобно для получения уведомлений от БД
Минусы для нашей задачи:

Медленнее на ~30-50% для мелких запросов (из-за libpq overhead)
Пул (psycopg_pool.AsyncConnectionPool) чуть менее зрелый
Критическая разница для телеметрии: Pipeline mode

asyncpg (обычный режим):
  Query 1 → Response 1 → Query 2 → Response 2 → Query 3 → Response 3
  [         Round Trip          ][         Round Trip          ][...]

psycopg3 pipeline:
  Query 1 → Query 2 → Query 3 → Response 1 → Response 2 → Response 3
  [     Одна пачка отправки    ][     Одна пачка получения    ]
Pipeline экономит round-trips. Для пакета из 10 запросов при latency 1ms к БД — это 9ms экономии. Но: в pipeline нельзя использовать EXPLAIN ANALYZE (потому что каждый ответ меняет контекст следующего запроса).

Значит:

Для уровня 0-1 телеметрии → psycopg3 с pipeline будет быстрее
Для уровня 2-3 телеметрии → asyncpg и psycopg3 примерно одинаковы (pipeline невозможен)

Суть решения в одном абзаце
Каждый SQL-запрос внутри пакета (Unit of Work) оборачивается в EXPLAIN (ANALYZE, BUFFERS, WAL, FORMAT JSON). Это заставляет PostgreSQL реально выполнить запрос, но при этом собрать детальную статистику по буферам, WAL и времени. Параллельно расширение pg_stat_kcache автоматически вызывает системный getrusage() до и после каждого запроса, что даёт CPU time и page faults на уровне ОС. В SQL-комментарий вшивается trace_id. Всё это собирается в Python, складывается в asyncio.Queue, и background-таск батчами отправляет в Kafka.

Почему asyncpg, а не psycopg3
Для уровня 4 pipeline mode psycopg3 невозможен — EXPLAIN ANALYZE требует последовательного выполнения (каждый запрос меняет состояние данных, следующий может от этого зависеть). Раз pipeline отпадает, asyncpg выигрывает за счёт:

Cython + нативный binary protocol — парсинг JSON-планов быстрее
Автоматический кеш prepared statements — планировщик PostgreSQL не тратит время на повторное планирование одинаковых запросов
Меньше overhead на каждый round-trip — нет прослойки libpq

Методология оценки
EXPLAIN ANALYZE добавляет инструментирование к каждому узлу плана выполнения. Для каждого узла PostgreSQL вызывает InstrStartNode() / InstrStopNode() — это gettimeofday() или clock_gettime(). pg_stat_kcache добавляет getrusage() на каждый запрос (не узел — это важно).

Проблема и решение
При 100 команд/сек overhead ~3.6% — укладываемся в 5%.

При 500+ команд/сек — не укладываемся.

Решение: адаптивный sampling

Sampling решается тривиально — для каждого пакета кидаем кубик, и если не попал в выборку, выполняем запросы обычным conn.execute() без EXPLAIN-обёртки. trace_id всё равно остаётся в SQL-комментарии.

Ты прав, я перечислил параметры pg_stat_kcache (user_time, system_time), но не показал их явно в итоговой схеме отправки в Kafka. Но проблема глубже — давай разберём, откуда вообще берётся CPU и что именно мы можем получить.

Источники CPU-метрик в PostgreSQL
Источник А: pg_stat_kcache (прямое измерение через getrusage)
Параметр
Что именно измеряет
Точность
user_time_sec
Время, которое ядра CPU потратили на код PostgreSQL в user space — парсинг SQL, построение плана, сортировки в памяти, вычисление выражений, обход индексов, сериализация результатов
Высокая, но это суммарно по процессу за период, а не per-query. Мы берём diff — получаем per-UoW
system_time_sec
Время CPU в kernel space — системные вызовы read()/write()/mmap()/brk(). Высокое значение означает что запросы много взаимодействуют с ОС: дисковый IO, выделение памяти, сетевые буферы

user_time + system_time
Полное CPU time данного пакета. Это прямой аналог того, что показывает time в bash или profiler
—
Нюанс точности: getrusage() возвращает CPU time процесса (backend). Поскольку один backend обслуживает одно соединение, а наш UoW выполняется в одной транзакции на одном соединении — diff между двумя вызовами getrusage() даёт именно CPU нашего пакета. Загрязнения от других сессий нет — это per-process метрика.

Источник Б: EXPLAIN ANALYZE (косвенное через timing)
EXPLAIN ANALYZE даёт actual_time на каждом узле плана. Это wall clock time, не CPU time. Но:

Если blk_read_time и blk_write_time доступны (track_io_timing=on), то:


CPU time ≈ actual_time − blk_read_time − blk_write_time
Это приблизительная оценка CPU per-query. Неточная потому что не учитывает ожидание локов, сетевые задержки внутри сервера и прочее. Но для большинства запросов — разумное приближение.

RSS / heap memory per-query

Оценка идеи
По вопросу о сэмплировании
Да, абсолютно верно. Сэмплирование — это ручка громкости для overhead. Можешь крутить от 0% (никакой телеметрии) до 100% (каждый запрос), и overhead масштабируется линейно. Более того — ты можешь сэмплировать неравномерно: 100% для подозрительного эндпоинта и 5% для всего остального.

Оценка идеи
Что ты описываешь
Ты описываешь Application-Aware Database Observability — связывание бизнес-контекста приложения (пользователь, экран, кнопка, A/B вариант, версия API) с низкоуровневой телеметрией базы данных (блоки, CPU, WAL, планы запросов) через единый trace_id.

Это не просто мониторинг БД. Это возможность спросить:

«Почему версия B нашего A/B теста даёт на 40% больше disk reads чем версия A?»

«Какой экран приложения генерирует больше всего WAL-записей и убивает репликацию?»

«Клиент X жалуется на тормоза — покажи мне IO-профиль всех его запросов за последний час»

Есть ли такое на рынке
Прямых аналогов — нет
Вот что существует и чем оно отличается:

Категория 1: APM (Application Performance Monitoring)


Datadog APM, New Relic, Dynatrace, Elastic APM
Что умеют:

Трейсинг запросов от HTTP до SQL
Время выполнения запроса
Slow query detection
Чего НЕ умеют:

❌ Блоки буферного кеша per-trace
❌ WAL bytes per-trace
❌ CPU time per-trace со стороны БД
❌ Планы запросов, привязанные к бизнес-контексту
❌ Hash peak memory, sort on disk per-trace
Они видят «запрос занял 50мс», но не видят «из них 30мс — disk read потому что shared_buffers промахнулся на 4000 блоков».

Категория 2: Database Monitoring


pganalyze, Datadog DBM, SolarWinds DPA, 
pgMustard, Percona PMM
Что умеют:

pg_stat_statements аналитика
Планы запросов
Wait events
Рекомендации по индексам
Чего НЕ умеют:

❌ Привязка к бизнес-контексту (пользователь, экран, A/B вариант)
❌ Фильтрация телеметрии по бизнес-атрибутам
❌ Динамическое включение детального профилирования для конкретного сценария
Они видят «запрос SELECT * FROM orders WHERE user_id = $1 вызывается 10000 раз/сек», но не видят «из них 8000 приходят с экрана корзины, а 2000 с экрана истории, и экран корзины делает в 3 раза больше disk reads на запрос».

Категория 3: sqlcommenter (Google)


Проект от Google Cloud — именно SQL-комментарии с контекстом
Что умеет:

Добавляет в SQL-комментарии: controller, action, framework, route
Работает с pg_stat_statements
Интегрируется с Cloud SQL Insights (Google)
Чего НЕ умеет:

❌ EXPLAIN ANALYZE per-trace
❌ CPU/memory метрики
❌ WAL bytes
❌ Динамическое сэмплирование
❌ Произвольный бизнес-контекст (A/B вариант, версия клиента)
❌ Unit of Work как единица — работает на уровне отдельных запросов
sqlcommenter — ближайший аналог, но он делает 10% от того, что ты описываешь.

Детальная карта флагов
Флаг 1: TIMING — базовое время
Что включает:

wall_time (Python time.monotonic())
planning_time + execution_time (из EXPLAIN ANALYZE)
Как собирается:

sql


EXPLAIN (ANALYZE, FORMAT JSON) <sql>
Что даёт:

Время каждого запроса и пакета в целом
Разделение на время планирования и выполнения
Overhead на БД: ~0.05мс на запрос

Вызов gettimeofday() на каждом узле плана. На современном CPU это rdtsc — наносекунды.

Overhead на Python: ~0.01мс

Смысл выключать: практически нет. Это база для всего остального. Если выключить — используем просто conn.execute() без EXPLAIN вообще, остаётся только wall_time из Python.

Флаг 2: ROWS — счётчики строк
Что включает:

actual_rows на каждом узле
rows_removed_by_filter
total_rows_affected (из command status)
Как собирается:
Уже включено в EXPLAIN ANALYZE — бесплатно если Флаг 1 включён. Без EXPLAIN — только rows affected из статуса команды (INSERT 0 5).

Что даёт:

Сколько строк обработал каждый запрос
Расхождение plan_rows vs actual_rows (диагностика устаревшей статистики)
Overhead: 0 дополнительно к Флагу 1

Смысл выключать: нет, если Флаг 1 включён

Флаг 3: BUFFERS — буферный кеш
Что включает:

shared_blks_hit / read / dirtied / written
local_blks_hit / read / dirtied / written
temp_blks_read / written
Как собирается:

sql


EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) <sql>
Что даёт:

Главная метрика эффективности: hit ratio per-query
Обнаружение запросов, которые "промахиваются" мимо кеша
temp_blks — обнаружение нехватки work_mem
Overhead на БД: ~0.02мс на запрос

PostgreSQL и так ведёт эти счётчики для pg_stat_statements. Флаг BUFFERS просто копирует их в вывод EXPLAIN. Почти бесплатно.

Overhead на Python: ~0.005мс (парсинг 6 доп. полей из JSON)

Смысл выключать: минимальный. Это самые ценные данные при минимальном overhead. Рекомендую держать всегда включённым вместе с ANALYZE.

Флаг 4: IO TIMING — время дисковых операций
Что включает:

blk_read_time (мс на чтение блоков с диска)
blk_write_time (мс на запись)
per-node I/O Read Time / I/O Write Time
Как собирается:
Требует настройку PostgreSQL:

ini


track_io_timing = on    # в postgresql.conf
После этого данные появляются в EXPLAIN BUFFERS автоматически.

Что даёт:

Разделение: запрос медленный из-за CPU или из-за диска?
estimated_cpu = execution_time - blk_read_time - blk_write_time
Обнаружение медленного storage
Overhead на БД: ~0.01-0.02мс на каждый IO-вызов

Два вызова clock_gettime() на каждое обращение к диску. На SSD с 10000 IOPS это ~0.1-0.2мс/сек суммарно. На практике overhead менее 1% даже под нагрузкой. PostgreSQL официально рекомендует включать.

Overhead на Python: 0 дополнительно (данные уже в том же JSON)

Смысл выключать: только если у тебя крайне старый сервер с дорогим clock_gettime() (до 2010 года). На современном железе — всегда включай.

Флаг 5: WAL — метрики журнала транзакций
Что включает:

wal_records (количество WAL-записей)
wal_fpi (full page images)
wal_bytes (объём WAL)
Как собирается:

sql


EXPLAIN (ANALYZE, BUFFERS, WAL, FORMAT JSON) <sql>
Доступно с PostgreSQL 13.

Что даёт:

Оценка нагрузки на репликацию: какие операции генерируют больше всего WAL
Оценка нагрузки на дисковую подсистему (WAL пишется синхронно)
Оптимизация bulk-операций
Overhead на БД: ~0.01мс на запрос

Счётчики WAL и так ведутся. Флаг просто включает их в вывод.

Overhead на Python: ~0.002мс (3 доп. поля)

Смысл выключать: если БД младше PG13 (флаг не поддерживается). В остальных случаях — держать включённым. Стоимость околонулевая.

Флаг 6: CPU — процессорное время
Что включает:

cpu_user_time_sec
cpu_system_time_sec
Как собирается:
Отдельные SQL-запросы к pg_stat_kcache до и после пакета:

sql


⌄
-- ДО пакета
SELECT user_time, system_time 
FROM pg_stat_kcache 
WHERE ...

-- ПОСЛЕ пакета  
SELECT user_time, system_time 
FROM pg_stat_kcache 
WHERE ...
Что даёт:

Единственный способ узнать реальное CPU time
Разделение user/system — CPU-bound vs IO-bound
Cost attribution: какой бизнес-сценарий сжигает CPU
Overhead на БД: ~0.3мс на SELECT из pg_stat_kcache × 2 = ~0.6мс на пакет

Это фиксированный overhead на пакет, не на запрос. Для пакета из 1 запроса — дорого (30% при 2мс запросе). Для пакета из 50 запросов — дёшево (0.6%).

Overhead на Python: ~0.1мс (два round-trip)

Смысл выключать: ДА, это самый дорогой флаг в расчёте на мелкие пакеты.

Рекомендация:

Пакет ≥ 10 запросов → включать
Пакет < 10 запросов → включать при sampling ≤ 20%
Managed PostgreSQL (RDS) → выключать (pg_stat_kcache недоступен)
Флаг 7: MEMORY — информация о памяти
Что включает:

minor_page_faults, major_page_faults (из pg_stat_kcache)
hash_peak_memory_kb (из EXPLAIN — Hash-узлы)
sort_space_used_kb (из EXPLAIN — Sort-узлы)
sort_space_type (Memory / Disk)
hash_batches (сколько ушло на диск)
Как собирается:

Page faults — из того же pg_stat_kcache (тот же SELECT что и CPU)
Hash/Sort — из JSON-плана EXPLAIN ANALYZE (парсинг узлов)
Что даёт:

Обнаружение нехватки work_mem (sort on disk, hash batches > 1)
major_page_faults → сервер свопит (критическая проблема)
Cost attribution по памяти
Overhead на БД: 0 дополнительно если Флаги 3 и 6 уже включены

Page faults — из того же getrusage(). Hash/Sort метрики — из того же EXPLAIN JSON.

Overhead на Python: ~0.05мс (парсинг Hash/Sort узлов из JSON-дерева)

Смысл выключать: выключается вместе с Флагом 6 (для page faults) или отдельно для парсинга Hash/Sort узлов. Парсинг узлов — это чисто Python-overhead, не трогает БД.

Флаг 8: PLAN — сохранение плана запроса
Что включает:

Полный JSON-план каждого запроса в пакете
node_types_used (какие операции: Seq Scan, Index Scan, Hash Join...)
index_name (какой индекс использован)
plan_rows vs actual_rows ratio (диагностика)
Как собирается:
Данные уже есть в EXPLAIN ANALYZE JSON — вопрос только сохранять или отбрасывать полный план.

Что даёт:

Диагностика: почему запрос медленный, а не только насколько
Обнаружение Seq Scan вместо Index Scan
Обнаружение плохих джойнов (Nested Loop на больших таблицах)
Исторические планы: «план изменился после ANALYZE / после обновления PG»
Overhead на БД: 0 дополнительно

Overhead на Python: ~0.1-0.5мс на запрос (json.loads полного плана + сериализация для Kafka)

Overhead на трафик/хранение: план одного запроса — 2-50KB JSON. Пакет из 50 запросов — до 2.5MB. При 100 UoW/сек и 100% sampling — 250MB/сек в Kafka. Это много.

Смысл выключать: ДА. Это главный источник overhead по объёму данных, не по CPU.

Рекомендация:

По умолчанию — выключен
Включать при sampling ≤ 5%
Или включать targeted: только для конкретного эндпоинта/пользователя
Или сохранять только для запросов медленнее порога (> 100мс)
Флаг 9: PLAN DETAILS — глубокий анализ плана
Что включает:

Per-node buffers (на каком узле сколько блоков прочитано)
Per-node timing (на каком узле сколько времени)
Per-worker метрики (для параллельных запросов)
JIT-метрики
Filter selectivity per-node
Полное дерево с вложенностью
Как собирается:
Полный парсинг JSON-дерева EXPLAIN с рекурсивным обходом всех узлов.

Что даёт:

Точная локализация: «90% времени — на Seq Scan таблицы orders на узле 7 из 15»
JIT-анализ: «JIT-компиляция заняла 40мс — стоило ли оно того?»
Параллелизм: «запланировано 4 worker, запустилось 2 — bottleneck»
Overhead на БД: 0 дополнительно

Overhead на Python: ~0.5-2мс на запрос (рекурсивный обход дерева + сериализация)

Смысл выключать: ДА. Это нужно только для глубокой диагностики. Включать по запросу для конкретных проблемных сценариев.

# LLM + телеметрия AOA + код Actions = новый класс анализа

## Почему это работает именно в AOA

В обычном фреймворке если отдать LLM логи и код — модель получит поток текста и разрозненные файлы. Она будет **угадывать** связи между ними.

В AOA модель получает **три связанных структуры** [1]:

**Первое — код Action как машиночитаемую спецификацию.** Не просто код, а формализованные метаданные: список шагов с описаниями, зависимости, роли, контракты каждого аспекта через чекеры — всё это доступно через атрибуты класса и верифицировано машиной [1].

**Второе — семантическое дерево выполнения.** ExecutionTreePlugin строит JSON-дерево с params, state после каждого аспекта, duration, вложенными вызовами и result [2]. Это не плоский лог — это граф бизнес-процесса.

**Третье — телеметрию PostgreSQL привязанную к trace_id.** Для каждого узла дерева — блоки буферного кеша, WAL bytes, CPU time, планы запросов, IO timing. Всё связано через один trace_id из Context.request [1].

LLM видит одновременно **намерение** (код), **выполнение** (дерево), **цену** (телеметрия БД) — и может рассуждать о всех трёх.

---

## 1. Оптимизация

LLM видит что аспект `load_products` читает 4000 блоков с диска (`shared_blks_read=4000`) при каждом вызове, а `shared_blks_hit=12` — почти нулевой hit ratio. При этом из плана запроса видно Seq Scan по таблице products. Из кода Action видно что в Params есть `product_ids: List[int]` и аспект делает SELECT по списку ID.

**LLM предлагает:** «Добавить индекс на `products.id` (хотя он вероятно есть — проблема в том что планировщик выбирает Seq Scan из-за устаревшей статистики). Рекомендую выполнить `ANALYZE products`. Альтернативно — если список ID всегда маленький (из чекера видно `max_length=20`), можно форсировать Index Scan через `SET enable_seqscan = off` в рамках транзакции.»

**Что ещё LLM может предложить по оптимизации:**

- **N+1 детекция.** Видит что аспект `create_items` вызывает execute 47 раз с одинаковым шаблоном INSERT — предлагает использовать PostgresUnitOfWork [2] для batch-вставки
- **Избыточные запросы между аспектами.** Аспект `validate` загружает пользователя, а аспект `check_permissions` загружает того же пользователя повторно — предлагает передать через state
- **Sort on Disk.** Видит в плане запроса `Sort Method: external merge` с `Sort Space Type: Disk` — предлагает увеличить `work_mem` или добавить индекс для устранения сортировки
- **Неиспользуемые данные.** SELECT возвращает 30 столбцов, но из state после аспекта видно что используются только 3 — предлагает SELECT только нужных полей
- **WAL pressure.** Один Action генерирует 2MB WAL — предлагает COPY вместо множественных INSERT или отложить VACUUM

---

## 2. Поиск проблем

- **Рассогласование plan/actual.** Планировщик ожидал 10 строк, получил 50000 — LLM видит это в EXPLAIN ANALYZE и предлагает ANALYZE таблицы или пересмотр условий WHERE
- **Lock contention.** Аспект `update_inventory` занимает 800мс, из них `blk_read_time=2мс` и `cpu_time=3мс` — остальные 795мс это ожидание. LLM предполагает блокировку и предлагает SELECT FOR UPDATE SKIP LOCKED или оптимистичную блокировку
- **Транзакция слишком длинная.** Дерево показывает что между `open()` и `commit()` проходит 5 аспектов за 3 секунды, из которых 2.5с — вызов внешнего API в аспекте `notify`. LLM предлагает вынести уведомление за пределы транзакции
- **Memory spill.** Hash Join с `Peak Memory: 256MB` и `Batches: 16` (ушёл на диск) — предлагает увеличить `work_mem` для этого конкретного запроса
- **Регрессия после деплоя.** Сравнивая телеметрию за вчера и сегодня для того же Action — видит что `shared_blks_read` вырос в 10 раз после миграции

---

## 3. Инсайты — то, что не видно ни в одном существующем инструменте

### Бизнес-стоимость каждой операции

LLM может посчитать: «Action `GenerateReportAction` вызывается 200 раз в день, каждый вызов генерирует 15MB WAL и читает 80000 блоков. Это 3GB WAL и 1.2GB disk reads в день **только на отчёты**. При этом из Context.tags видно что 180 из 200 вызовов приходят от A/B варианта `dashboard_v2`. Вариант `dashboard_v1` вызывает тот же Action 20 раз и генерирует в 4 раза меньше WAL на вызов — потому что запрашивает данные за неделю, а не за месяц.»

**Вывод LLM:** «Вариант B вашего дашборда стоит в 36 раз дороже варианта A в ресурсах БД. Рассмотрите кеширование или предагрегацию.»

### Автоматическое предложение декомпозиции Action

LLM видит что Action имеет 8 аспектов, 4 вложенных вызова и 12 запросов к БД суммарно. Из телеметрии видно что 3 аспекта из 8 генерируют 95% нагрузки. Из дерева выполнения видно что эти 3 аспекта не зависят друг от друга по state.

**Вывод LLM:** «Аспекты `load_inventory`, `load_pricing`, `load_promotions` могут быть вынесены в отдельные Actions и выполнены параллельно. Это сократит wall time с 1.2с до ~0.5с.»

### Обнаружение аномалий по бизнес-контексту

LLM видит что один user_id генерирует в 50 раз больше `shared_blks_read` чем средний пользователь. Из Params видно что он передаёт `date_range="2020-01-01:2025-06-01"` — пятилетний диапазон, в то время как 99% пользователей запрашивают последний месяц.

**Вывод LLM:** «Добавьте ограничение в чекер Params — `max_date_range=90 days`. Или создайте отдельный Action для исторических отчётов с предупреждением о длительности.»

### Предиктивный анализ

LLM получает телеметрию за неделю и видит тренд: `shared_blks_read` для `CreateOrderAction` растёт на 3% в день. Из плана запроса видно что индекс `orders_user_id_idx` имеет `Index Cond: (user_id = $1)` но `Rows Removed by Filter: 847` — индекс становится менее избирательным по мере роста таблицы.

**Вывод LLM:** «Через 30 дней этот запрос начнёт превышать ваш SLA в 200мс. Рассмотрите партиционирование таблицы orders по дате или составной индекс `(user_id, created_at DESC)`.»

### Автоматическая генерация миграций

LLM видит проблемные запросы, знает схему из планов, понимает бизнес-контекст из Action.

**Может сгенерировать:**
```sql
-- Сгенерировано на основе анализа телеметрии CreateOrderAction
-- Проблема: Seq Scan на orders при фильтрации по user_id + status
-- Текущий overhead: 4000 shared_blks_read per call, 100 calls/sec
-- Ожидаемый эффект: снижение до ~50 shared_blks_read

CREATE INDEX CONCURRENTLY idx_orders_user_status 
ON orders (user_id, status) 
WHERE status IN ('pending', 'processing');
```

### Связь телеметрии с тестами

LLM видит что в production аспект `charge_card` генерирует 500 WAL bytes и читает 200 блоков. В тестах (через ActionTestMachine [1]) этот аспект замокан через MockAction [1] — побочные эффекты БД не проверяются.

**Вывод LLM:** «Рекомендую добавить интеграционный тест с auto-rollback [2] для `charge_card`, который проверит реальные SQL-запросы. Текущие моки скрывают N+1 проблему — в production аспект выполняет 12 запросов, а тест проверяет только Result.»

---

## Почему это возможно только здесь

В любом другом фреймворке LLM получил бы:
- **Код** — сервисы, контроллеры, модели без формальной структуры
- **Логи** — поток текста без семантики
- **Метрики** — графики без привязки к бизнес-операциям

В AOA LLM получает **три структурированных потока**, связанных через trace_id и формализованные метаданные. Каждый Action — это спецификация с обязательными описаниями, контрактами и ролями [1]. Каждое выполнение — это семантическое дерево [2]. Каждый запрос к БД — это телеметрия с бизнес-контекстом.

LLM не угадывает связи — **связи формализованы архитектурой**.
