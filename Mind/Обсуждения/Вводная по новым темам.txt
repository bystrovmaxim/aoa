В ActionMachine эти менеджеры лежат в ResourceManagers/ и наследуются от BaseResourceManager/BaseConnectionManager. Они идеально выполняют роль адаптеров.

Нет состояния – Action получает всё необходимое через параметры (params) и зависимости (deps). Он не хранит соединения, не открывает файлы. Это позволяет легко тестировать каждый аспект изолированно.

Ядро – именно здесь живёт уникальная ценность приложения. Ядро определяет что делать, а адаптеры – как.

В ActionMachine действия (BaseAction) не имеют состояния и не знают о деталях реализации сервисов – они только вызывают их через интерфейс (например, EmailService).

3. Как это связано в ActionMachine
Твоя архитектура чётко разделяет эти две роли:

Ресурсные менеджеры (ResourceManagers) управляют состоянием и доступом к внешним системам. Они реализуют конкретные адаптеры.

Действия (Core/BaseAction) содержат бизнес-логику. Они получают доступ к ресурсам через фабрику зависимостей (DependencyFactory), которая предоставляет готовые экземпляры менеджеров (или их моков в тестах).

Пример:

PostgresConnectionManager – знает, как подключиться к PostgreSQL, выполнить коммит, откат. У него есть состояние (соединение).

UserRepository – может использовать этот менеджер для выполнения запросов. Сам репозиторий – тоже тонкий слой, почти без логики (только маппинг). Его можно рассматривать как часть порта.

В идеале, управление транзакциями должно происходить вне действия, на уровне машины или специального декоратора. В ActionMachine это можно сделать через плагины или обёртки, но пока явного механизма нет – ты планируешь его добавить (пункт 1 в «следующих сериях»).

Ресурсные менеджеры vs Действия (Actions)
В любой нетривиальной системе есть два принципиально разных типа сущностей:

Ресурсные менеджеры — инкапсулируют доступ к внешним системам (базы данных, файлы, API). Они имеют состояние (соединение, открытый файл, сессия) и занимаются только «транспортом» — передачей данных, управлением транзакциями, но не содержат бизнес-логики.

Действия (Actions) — реализуют бизнес-логику. Они не имеют состояния, получают всё необходимое через параметры и зависимости, и их единственная цель — принять решение, что сделать с данными.

Ресурсные менеджеры: порты и адаптеры
Ресурсный менеджер — это адаптер в терминах гексагональной архитектуры. Он реализует интерфейс (порт), через который ядро приложения общается с внешним миром.

Характеристики:

Имеет состояние — например, открытое соединение с БД, дескриптор файла, авторизованную сессию API.

Не содержит бизнес-логики — максимум, что допускается, это преобразование форматов (маппинг из результата запроса в модель данных).

Управляет жизненным циклом ресурса — открытие, закрытие, коммит, откат транзакций.

Используется действиями через Dependency Injection.

Примеры в ActionMachine:

PostgresConnectionManager — управляет соединением с PostgreSQL, транзакциями.

CsvConnectionManager — работает с CSV-файлами, записывает строки.

BaseConnectionManager — абстрактный класс для всех менеджеров соединений.

python
# Пример использования ресурсного менеджера (не в действии, а в вызывающем коде)
mgr = PostgresConnectionManager(db_params)
mgr.open()
# выполнить операции...
mgr.commit()
Действия (Actions): ядро бизнес-логики
Действие — это ядро приложения. Оно содержит правила, которые делают систему уникальной.

Характеристики:

Не имеет состояния — все данные, необходимые для выполнения, приходят через параметры (params) и зависимости (deps).

Содержит бизнес-логику — проверки, расчёты, вызовы других действий или сервисов.

Ничего не знает о транспорте — не открывает соединения, не управляет транзакциями. Всё, что ему нужно от внешнего мира, он получает через зависимости.

Легко тестируется — можно подменить зависимости моками и проверить каждый аспект изолированно.

Пример в ActionMachine:

python
@depends(EmailService, SmsService)
@CheckRoles(CheckRoles.ANY)
class NotificationAction(BaseAction['Params', 'Result']):
    @dataclass(frozen=True)
    class Params(BaseParams):
        channel: str
        message: str
        recipient: str

    @dataclass(frozen=True)
    class Result(BaseResult):
        success: bool

    @aspect("Выбор канала")
    def choose_channel(self, params, state, deps):
        if params.channel == 'email':
            state['service'] = deps.get(EmailService)
        else:
            state['service'] = deps.get(SmsService)
        return state

    @summary_aspect("Отправка")
    def send(self, params, state, deps):
        state['service'].send(params.recipient, params.message)
        return NotificationAction.Result(success=True)
Здесь EmailService и SmsService — это тоже ресурсные менеджеры (адаптеры). Действие не знает, как именно они отправляют сообщения — это их внутреннее дело.

Взаимодействие: DependencyFactory
Связующим звеном между действиями и ресурсными менеджерами служит DependencyFactory. Она:

Хранит информацию о зависимостях, объявленных через @depends.

Кэширует созданные экземпляры, чтобы повторно использовать их в рамках одного выполнения.

Предоставляет действие через run_action для запуска вложенных действий.

Благодаря этому действию не нужно самому заботиться о том, откуда взять EmailService — фабрика предоставит готовый экземпляр.

Когда использовать ресурсный менеджер, а когда действие?
Простое правило:

Сущность	Используйте, если...
Ресурсный менеджер	Нужно управлять состоянием (соединение, файл, сессия) и/или предоставить «трубу» для данных без принятия бизнес-решений.
Действие	Нужно реализовать бизнес-правило, которое можно выразить в коде, и которое не зависит от того, как именно данные будут сохранены или переданы.
Более формально:

Если код содержит логику типа «если сумма > 1000, то запросить подтверждение» — это действие.

Если код открывает сокет, выполняет SQL-запрос и закрывает соединение — это ресурсный менеджер.

Преимущества разделения
Тестируемость — действия тестируются с моками, ресурсные менеджеры — с реальными интеграционными тестами.

Поддерживаемость — изменение способа хранения данных (например, переход с PostgreSQL на MongoDB) не требует переписывания бизнес-логики.

Гибкость — можно комбинировать разные ресурсные менеджеры с одним и тем же действием (например, отправлять уведомления через email, SMS или Telegram, просто подставив другой адаптер).

Чистота кода — каждый класс отвечает за что-то одно.

ActionMachine жёстко закрепляет это разделение: ресурсные менеджеры лежат в ResourceManagers/, действия — в вашем приложении (например, actions/). Фабрика зависимостей и машина действий связывают их воедино, сохраняя прозрачность и контроль.

Что важно:

Каждый ресурсный менеджер управляет состоянием (соединение, сессия) и предоставляет низкоуровневые операции (создать задачу, получить задачу, обновить статус). Он не содержит бизнес-логики, только вызовы API и маппинг.

Ресурсные менеджеры vs Действия (Actions)
В хорошо структурированном приложении существуют два принципиально разных типа компонентов:

Ресурсные менеджеры – адаптеры, которые управляют доступом к внешним системам (базы данных, API, файлы). Они имеют состояние (соединение, сессия) и занимаются только «транспортом», но не содержат бизнес-логики.

Действия (Actions) – ядро бизнес-логики. Они не имеют состояния, получают всё необходимое через параметры и зависимости, и принимают решения, что делать с данными.

Разделение этих ролей – ключ к чистой, тестируемой и гибкой архитектуре, которую реализует ActionMachine.

1. Ресурсные менеджеры (адаптеры)
Ресурсный менеджер – это конкретная реализация порта (интерфейса) для доступа к внешнему ресурсу. Его задача – предоставить ядру унифицированный способ взаимодействия с конкретной технологией (PostgreSQL, YouTrack API, файловая система и т.д.).

Характеристики:
Имеет состояние – открытое соединение, дескриптор файла, авторизованная сессия.

Не содержит бизнес-логики – максимум преобразование форматов (маппинг) и обработка ошибок ввода-вывода.

Управляет жизненным циклом – открытие, закрытие, транзакции (где применимо).

Реализует порт – интерфейс, определённый в слое ядра.

Пример в ActionMachine:
python
class PostgresConnectionManager(BaseConnectionManager):
    def __init__(self, params):
        self.conn = None
        ...

    def _doOpenConnection(self):
        self.conn = psycopg2.connect(...)

    def execute(self, query, params=None):
        with self.conn.cursor() as cur:
            cur.execute(query, params)
            return cur.fetchall()
2. Действия (Actions)
Действие – это атомарная единица бизнес-логики. Оно не знает, откуда пришли данные и куда они пойдут; оно только реализует правила предметной области.

Характеристики:
Не имеет состояния – все необходимые данные передаются через params (бизнес-данные) и deps (зависимости).

Содержит бизнес-логику – проверки, расчёты, вызовы других действий или сервисов.

Ничего не знает о транспорте – не управляет соединениями, не открывает файлы.

Легко тестируется – зависимости подменяются моками.

Пример в ActionMachine:
python
@depends(UserRepository, EmailService)
class NotifyUserAction(BaseAction['Params', 'Result']):
    @dataclass
    class Params:
        user_id: int
        message: str

    @dataclass
    class Result:
        sent: bool

    @summary_aspect("Отправка уведомления")
    def handle(self, params, state, deps):
        user = deps.get(UserRepository).get(params.user_id)
        deps.get(EmailService).send(user.email, params.message)
        return NotifyUserAction.Result(sent=True)
3. Взаимодействие: DependencyFactory
Связующим звеном служит DependencyFactory. Она:

Хранит информацию о зависимостях, объявленных через @depends.

Кэширует созданные экземпляры ресурсных менеджеров.

Предоставляет действие через run_action для запуска вложенных действий.

Благодаря этому Action не нужно самому заботиться о том, откуда взять UserRepository – фабрика предоставит готовый экземпляр (или его мок в тестах).

4.3. Адаптеры с внутренними моделями
Каждый адаптер может иметь собственные внутренние модели, отражающие структуру конкретного API, но наружу он отдаёт только общие модели домена.

python
# resource_managers/issue_trackers/youtrack/client.py
import requests
from ..interface import IssueTracker
from ..models import Issue
from .internal_models import YouTrackIssue  # специфичная для YouTrack модель

class YouTrackTracker(IssueTracker):
    def __init__(self, base_url: str, token: str):
        self.base_url = base_url
        self.token = token
        self.session = requests.Session()
        self.session.headers.update({'Authorization': f'Bearer {token}'})

    def get_issue(self, issue_id: str) -> Issue:
        response = self.session.get(f"{self.base_url}/issues/{issue_id}")
        response.raise_for_status()
        # Преобразуем ответ в общую модель
        return self._to_domain(response.json())

    def _to_domain(self, raw: dict) -> Issue:
        # маппинг YouTrackIssue → Issue
        return Issue(
            id=raw['id'],
            title=raw['summary'],
            status=raw['state']['name'],
            assignee=raw.get('assignee', {}).get('login')
        )
4.4. Единый интерфейс – единая модель
Если два адаптера реализуют один порт, они обязаны использовать общие модели домена во всех публичных методах. Иначе теряется смысл интерфейса – ядро не сможет работать с ними единообразно.

Правило: Адаптер может иметь свои внутренние структуры, но преобразование в общие модели должно происходить внутри адаптера, чтобы снаружи он всегда оперировал понятиями домена.

5. Взаимодействие с действиями
Действие никогда не импортирует адаптер напрямую. Оно зависит только от порта (интерфейса), а конкретный адаптер подставляется через DI.

python
# actions/notify_user.py
from resource_managers.issue_trackers.interface import IssueTracker
from resource_managers.issue_trackers.models import Issue

@depends(IssueTracker)   # зависит от порта, а не от конкретной реализации
class NotifyUserAction(...):
    ...
Фабрика зависимостей при создании действия подставит нужный адаптер (YouTrackTracker, JiraTracker и т.д.) в зависимости от конфигурации приложения или тестового окружения.

6. Преимущества такого подхода
Заменяемость – можно легко переключиться с одного сервиса на другой, просто изменив конфигурацию DI.

Тестируемость – бизнес-логика тестируется с моками портов, адаптеры – в интеграционных тестах.

Чистота кода – каждый класс отвечает за одно: либо за бизнес-правила, либо за взаимодействие с внешним миром.

Масштабируемость – новые сервисы добавляются без изменения существующего кода, только новые адаптеры.

7. Заключение
Разделение на ресурсные менеджеры (адаптеры) и действия (ядро) – это фундамент чистой архитектуры. Правильная организация папок, выделение портов и общих моделей, а также использование единого интерфейса для взаимозаменяемых реализаций позволяют строить гибкие, тестируемые и долгоживущие приложения.

ActionMachine предоставляет все необходимые инструменты для реализации этого подхода: декларативные зависимости через @depends, фабрику DependencyFactory, поддержку вложенных действий и мощную систему плагинов. Опираясь на эти принципы, вы сможете создать проект, который будет легко поддерживать и развивать годами.

ActionMachine предлагает структуру: каждый аспект — это один шаг, который легко понять и проверить.

Как начать мыслить аспектами?
Возьми любую свою функцию и мысленно разбей её на части:

Что нужно проверить перед выполнением? (валидация, права)

Какие данные нужно подготовить? (загрузить из БД, преобразовать)

Какая основная операция? (расчёт, изменение)

Что нужно сделать после? (логирование, уведомления)

Каждый из этих шагов — кандидат в аспекты. На тестовых примерах ты уже видел, как ParentAction разбит на delay и extra_check. Это и есть тот самый конвейр.

Со временем ты начнёшь видеть эти этапы автоматически. Главное — первый шаг сделан. Ты уже задаёшь правильные вопросы.

🔨 Шаг 1: Выделяем этапы (аспекты)
Посмотрим на код и мысленно разобьём на логические шаги:

python
from dataclasses import dataclass
from typing import List
from ActionMachine.Core import BaseAction, aspect, summary_aspect, depends
from resource_managers.products.interface import ProductRepository
from resource_managers.orders.interface import OrderRepository
from resource_managers.email.interface import EmailService
from ActionMachine.Auth import CheckRoles

# Параметры действия
class ProcessOrderParams(BaseParams):
    user_id: int
    items: List[dict]  # список [{'product_id': 1, 'quantity': 2}, ...]
    payment_method: str

# Результат действия
class ProcessOrderResult(BaseResult):
    order_id: int

@depends(ProductRepository)
@depends(OrderRepository)
@depends(EmailService)
@CheckRoles(CheckRoles.ANY)  # потом проверим конкретную роль в аспекте
class ProcessOrderAction(BaseAction[ProcessOrderParams, ProcessOrderResult]):

    @aspect("Проверка ролей")
    def check_roles(self, params, state, deps):
        # Здесь мы могли бы использовать контекст пользователя, но для простоты
        # будем считать, что роль уже проверена в CheckRoles.
        # В реальном проекте можно добавить дополнительную проверку.
        return state

    @aspect("Валидация заказа")
    def validate(self, params, state, deps):
        if not params.items:
            raise ValueError("Заказ пуст")
        product_repo = deps.get(ProductRepository)
        # Создаём список продуктов для последующего использования
        products = []
        for item in params.items:
            product = product_repo.get_product(item['product_id'])
            if not product:
                raise ValueError(f"Товар {item['product_id']} не найден")
            if product['stock'] < item['quantity']:
                raise ValueError(f"Недостаточно товара {product['name']}")
            products.append(product)
        # Сохраняем продукты в state для следующего аспекта
        state['products'] = products
        return state

    @aspect("Расчёт итоговой цены")
    def calculate_total(self, params, state, deps):
        total = 0
        products = state['products']
        for i, item in enumerate(params.items):
            total += products[i]['price'] * item['quantity']
        # Здесь можно было бы проверить VIP-статус пользователя (из контекста),
        # но контекст недоступен. Вместо этого передадим информацию через params
        # или через state. Для простоты предположим, что скидка зависит от количества:
        if len(params.items) > 5:
            total *= 0.95  # скидка 5% за большой заказ
        state['total'] = total
        return state

    @aspect("Списание со склада")
    def decrease_stock(self, params, state, deps):
        product_repo = deps.get(ProductRepository)
        for item in params.items:
            product_repo.decrease_stock(item['product_id'], item['quantity'])
        return state

    @aspect("Создание заказа")
    def create_order(self, params, state, deps):
        order_repo = deps.get(OrderRepository)
        order_id = order_repo.create_order(
            user_id=params.user_id,
            items=params.items,
            total=state['total'],
            payment_method=params.payment_method
        )
        state['order_id'] = order_id
        return state

    @summary_aspect("Отправка уведомления и возврат результата")
    def finish(self, params, state, deps):
        email_service = deps.get(EmailService)
        user_email = f"user_{params.user_id}@example.com"  # в реальности из UserRepository
        email_service.send(user_email, f"Заказ {state['order_id']} оформлен")
        return ProcessOrderResult(order_id=state['order_id'])

Представьте класс, который обрабатывает заказы. Он пытается делать всё сразу: проверять права, валидировать корзину, рассчитывать скидки, обновлять склад, отправлять уведомления, логировать. python
import logging
from typing import List, Dict, Optional

class OrderProcessor:
    """Универсальный обработчик заказов (спагетти-версия)"""
    
    def __init__(self, db_connection, email_client, logger=None):
        self.db = db_connection
        self.email = email_client
        self.logger = logger or logging.getLogger(__name__)
        self.current_user = None
        self.cart_items = []
        self.total = 0.0
        self.discount_applied = False
        self.order_id = None
        self._initialized = False

    def authenticate(self, user_id: int) -> bool:
        """Метод для аутентификации пользователя (можно вызвать отдельно)"""
        user = self.db.fetch_one("SELECT * FROM users WHERE id = %s", (user_id,))
        if user and 'customer' in user['roles']:
            self.current_user = user
            self.logger.info(f"User {user_id} authenticated")
            return True
        self.logger.warning(f"Authentication failed for user {user_id}")
        return False

    def load_cart(self, items: List[Dict]) -> None:
        """Загрузить товары в корзину (можно вызвать отдельно)"""
        self.cart_items = items
        self.logger.debug(f"Cart loaded with {len(items)} items")

    def validate_cart(self) -> bool:
        """Проверить наличие товаров и их доступность"""
        if not self.cart_items:
            raise ValueError("Cart is empty")
        for item in self.cart_items:
            product = self.db.fetch_one(
                "SELECT * FROM products WHERE id = %s", (item['product_id'],)
            )
            if not product:
                raise ValueError(f"Product {item['product_id']} not found")
            if product['stock'] < item['quantity']:
                raise ValueError(f"Insufficient stock for product {product['name']}")
            # Сохраняем продукт для последующего использования (внутреннее состояние)
            item['product_details'] = product
        self.logger.info("Cart validated")
        return True

    def calculate_total(self) -> float:
        """Рассчитать общую стоимость"""
        total = 0.0
        for item in self.cart_items:
            product = item.get('product_details')
            if not product:
                # Если продукт не загружен – подгружаем (костыль)
                product = self.db.fetch_one(
                    "SELECT * FROM products WHERE id = %s", (item['product_id'],)
                )
                item['product_details'] = product
            total += product['price'] * item['quantity']
        self.total = total
        return total

    def apply_discount(self) -> None:
        """Применить скидку, если пользователь VIP"""
        if self.current_user and self.current_user.get('is_vip'):
            self.total *= 0.9
            self.discount_applied = True
            self.logger.info("VIP discount applied")

    def update_stock(self) -> None:
        """Обновить остатки на складе"""
        for item in self.cart_items:
            self.db.execute(
                "UPDATE products SET stock = stock - %s WHERE id = %s",
                (item['quantity'], item['product_id'])
            )
        self.logger.info("Stock updated")

    def create_order(self) -> int:
        """Создать заказ в БД и вернуть ID"""
        order_id = self.db.execute_returning(
            "INSERT INTO orders (user_id, items, total, payment_method) VALUES (%s, %s, %s, %s) RETURNING id",
            (self.current_user['id'], self.cart_items, self.total, self.payment_method)
        )
        self.order_id = order_id
        self.logger.info(f"Order {order_id} created")
        return order_id

    def send_confirmation(self) -> None:
        """Отправить подтверждение на email"""
        if not self.order_id:
            raise RuntimeError("Order not created yet")
        self.email.send(
            to=self.current_user['email'],
            subject="Order confirmation",
            body=f"Your order #{self.order_id} has been placed."
        )
        self.logger.info(f"Confirmation sent for order {self.order_id}")

    def process_full_order(self, user_id: int, items: List[Dict], payment_method: str) -> int:
        """Попытка сделать всё сразу (но можно вызывать и отдельные методы)"""
        self.payment_method = payment_method
        if not self.authenticate(user_id):
            raise PermissionError("Authentication failed")
        self.load_cart(items)
        self.validate_cart()
        self.calculate_total()
        self.apply_discount()
        self.update_stock()
        order_id = self.create_order()
        self.send_confirmation()
        return order_id
Что здесь плохо?
Несколько точек входа – можно вызывать отдельные методы в любом порядке, что легко приводит к некорректному состоянию (например, send_confirmation до create_order).

Смешение ответственности – класс знает и про аутентификацию, и про БД, и про email, и про логику скидок.

Скрытые зависимости – методы используют внутренние поля, которые могут быть не инициализированы.

Дублирование кода – в calculate_total снова подгружаем продукт, если его нет в item.

Тестировать сложно – нужно мокать и БД, и email, и правильно выставлять внутреннее состояние.

Логика размазана – неясно, где начинается и заканчивается обработка конкретного заказа.

python
from dataclasses import dataclass
from typing import List, Dict
from ActionMachine.Core import BaseAction, aspect, summary_aspect, depends
from ActionMachine.Auth import CheckRoles

@dataclass
class ProcessOrderParams(BaseParams):
    user_id: int
    items: List[Dict]   # список [{'product_id': int, 'quantity': int}]
    payment_method: str

@dataclass
class ProcessOrderResult(BaseResult):
    order_id: int

@depends(UserRepository)
@depends(ProductRepository)
@depends(OrderRepository)
@depends(EmailService)
@CheckRoles(CheckRoles.ANY)  # Роль проверим в аспекте
class ProcessOrderAction(BaseAction[ProcessOrderParams, ProcessOrderResult]):

    @aspect("Проверка прав доступа")
    def check_authorization(self, params: ProcessOrderParams, state: Dict, deps) -> Dict:
        user_repo = deps.get(UserRepository)
        if not user_repo.user_has_role(params.user_id, 'customer'):
            raise PermissionError("User is not a customer")
        state['user_id'] = params.user_id
        return state

    @aspect("Валидация товаров и загрузка данных")
    def validate_and_load_products(self, params: ProcessOrderParams, state: Dict, deps) -> Dict:
        if not params.items:
            raise ValueError("Cart is empty")
        product_repo = deps.get(ProductRepository)
        products = []
        for item in params.items:
            product = product_repo.get_product(item['product_id'])
            if not product:
                raise ValueError(f"Product {item['product_id']} not found")
            if product['stock'] < item['quantity']:
                raise ValueError(f"Insufficient stock for product {product['name']}")
            products.append(product)
        state['products'] = products
        return state

    @aspect("Расчёт стоимости со скидкой")
    def calculate_total(self, params: ProcessOrderParams, state: Dict, deps) -> Dict:
        total = 0.0
        products = state['products']
        for i, item in enumerate(params.items):
            total += products[i]['price'] * item['quantity']
        user_repo = deps.get(UserRepository)
        if user_repo.is_vip(params.user_id):
            total *= 0.9
            state['discount_applied'] = True
        state['total'] = total
        return state

    @aspect("Списание со склада")
    def decrease_stock(self, params: ProcessOrderParams, state: Dict, deps) -> Dict:
        product_repo = deps.get(ProductRepository)
        for item in params.items:
            product_repo.decrease_stock(item['product_id'], item['quantity'])
        return state

    @aspect("Создание заказа")
    def create_order(self, params: ProcessOrderParams, state: Dict, deps) -> Dict:
        order_repo = deps.get(OrderRepository)
        order_id = order_repo.create_order(
            user_id=params.user_id,
            items=params.items,
            total=state['total'],
            payment_method=params.payment_method
        )
        state['order_id'] = order_id
        return state

    @summary_aspect("Отправка уведомления и возврат результата")
    def send_notification_and_return(self, params: ProcessOrderParams, state: Dict, deps) -> ProcessOrderResult:
        email_service = deps.get(EmailService)
        user_repo = deps.get(UserRepository)
        user = user_repo.get_user(params.user_id)
        email_service.send_order_confirmation(user['email'], state['order_id'])
        return ProcessOrderResult(order_id=state['order_id'])

Мы видим, что в этом фрагменте описаны интерфейсы (порты) для различных ресурсных менеджеров. Это правильный подход: каждый домен (user, product, order, email) имеет свой порт. Эти интерфейсы будут реализованы конкретными адаптерами (например, PostgresUserRepository, YouTrackProductRepository и т.д.). Такая структура соответствует принципам гексагональной архитектуры и чистой архитектуры: ядро (бизнес-логика) зависит только от интерфейсов, а не от конкретных реализаций.

Пользователь спрашивает: "это же по сути пространства" – вероятно, он имеет в виду, что эти интерфейсы описывают пространства имён или контракты для разных доменных областей. Да, это именно так: каждый интерфейс определяет контракт для работы с определённым ресурсом (пользователи, товары, заказы, email). Это позволяет ядру (действиям) работать с ними единообразно, не заботясь о конкретных реализациях.

Да, это именно пространства в том смысле, что каждый интерфейс описывает контракт для работы с определённой доменной областью (пользователи, товары, заказы, уведомления). Такое разделение даёт несколько ключевых преимуществ:

Явные границы – каждый порт чётко отделяет свою ответственность. Ядро (Action) не знает, как именно реализован UserRepository, оно знает только, что у него есть методы get_user, user_has_role, is_vip.

Лёгкая замена – можно подменить реализацию (например, заменить PostgreSQL на MongoDB) без изменения бизнес-логики. Достаточно написать новый адаптер, реализующий тот же интерфейс.

Тестируемость – в тестах мы можем подставить мок, реализующий интерфейс, и проверить, что действие вызывает нужные методы с правильными параметрами.

Изоляция – изменения в одном адаптере (например, в способе хранения пользователей) не влияют на другие части системы.

В ActionMachine эти интерфейсы используются в декораторе @depends. Действие не знает о конкретных реализациях, только о портах. Фабрика зависимостей (DependencyFactory) при выполнении действия подставит нужный адаптер (или мок в тестах).

Такая организация — это основа чистой архитектуры, где бизнес-логика остаётся независимой от инфраструктуры. Ты всё делаешь правильно.

# legacy/order_processor.py
import logging
from typing import List, Dict, Optional

class OrderProcessor:
    def __init__(self, db_connection, email_client, logger=None):
        self.db = db_connection
        self.email = email_client
        self.logger = logger or logging.getLogger(__name__)
        self.current_user = None
        self.cart_items = []
        self.total = 0.0
        self.order_id = None

    def process_full_order(self, user_id: int, items: List[Dict], payment_method: str) -> int:
        # Шаг 1: аутентификация
        user = self.db.fetch_one("SELECT * FROM users WHERE id = %s", (user_id,))
        if not user or 'customer' not in user['roles']:
            raise PermissionError("Authentication failed")
        self.current_user = user
        self.logger.info(f"User {user_id} authenticated")

        # Шаг 2: валидация корзины
        if not items:
            raise ValueError("Cart is empty")
        self.cart_items = items
        for item in items:
            product = self.db.fetch_one("SELECT * FROM products WHERE id = %s", (item['product_id'],))
            if not product:
                raise ValueError(f"Product {item['product_id']} not found")
            if product['stock'] < item['quantity']:
                raise ValueError(f"Insufficient stock for product {product['name']}")
            item['product_details'] = product  # добавляем в исходный dict (опасно!)
        self.logger.info("Cart validated")

        # Шаг 3: расчёт суммы
        total = 0.0
        for item in items:
            product = item['product_details']
            total += product['price'] * item['quantity']
        if user.get('is_vip'):
            total *= 0.9
            self.logger.info("VIP discount applied")
        self.total = total

        # Шаг 4: списание со склада
        for item in items:
            self.db.execute(
                "UPDATE products SET stock = stock - %s WHERE id = %s",
                (item['quantity'], item['product_id'])
            )
        self.logger.info("Stock updated")

        # Шаг 5: создание заказа
        order_id = self.db.execute_returning(
            "INSERT INTO orders (user_id, items, total, payment_method) VALUES (%s, %s, %s, %s) RETURNING id",
            (user_id, items, total, payment_method)
        )
        self.order_id = order_id
        self.logger.info(f"Order {order_id} created")

        # Шаг 6: отправка email
        self.email.send(
            to=user['email'],
            subject="Order confirmation",
            body=f"Your order #{order_id} has been placed."
        )
        self.logger.info(f"Confirmation sent for order {order_id}")

        return order_id

            raise ValueError(f"Insufficient stock")
        products.append(product)
    state['products'] = products
    return state
Теперь в легаси-процессор мы передаём items уже без дублирования запросов.

Итог: пошаговый план рефакторинга
Обёртка – создаём действие с одним аспектом, который вызывает легаси. Добавляем порты и адаптеры-обёртки.

Тесты – пишем тесты для действия, мокая зависимости.

Постепенное выделение аспектов – по одному выносим куски логики в аспекты, каждый раз проверяя, что тесты проходят.

Удаление легаси – когда все шаги выделены, убираем вызов старого кода и делаем summary-аспект.

Чистка – удаляем временные адаптеры, оставляем только чистые реализации портов.

Этот подход позволяет рефакторить легаси-код без остановки разработки и с постоянной обратной связью от тестов.

Ты абсолютно прав. В процессе рефакторинга легаси-класса (или просто при проектировании новой функциональности) возникает вопрос: «Где должна жить эта логика — в аспекте текущего действия или в отдельном действии?»

Вот простое практическое правило, которое я вывел из твоих же мыслей и нашего опыта:

Сначала делай аспектом. Если логика начинает использоваться в двух разных действиях — выделяй в отдельное действие.

Почему так?
Аспект — это шаг внутри конкретного бизнес-процесса. Он тесно связан с действием, не предполагает самостоятельного переиспользования. Выделять его сразу в отдельное действие — преждевременная абстракция.

Действие — это самостоятельная бизнес-операция. Его можно вызвать из любого другого действия через deps.run_action(), и у него есть собственная цепочка аспектов, валидация, права доступа.

Пример из жизни
Ты написал действие ProcessOrderAction. В нём есть аспект calculate_discount, который считает скидку на основе корзины и статуса пользователя.

Через месяц появляется новое действие CalculateEstimateAction (предварительный расчёт стоимости без оформления заказа). Оно тоже должно считать скидку по тем же правилам.

В этот момент ты выносишь логику расчёта скидки в отдельное действие (например, CalculateDiscountAction). Теперь оба «родительских» действия могут вызывать его через DI.

Что это даёт
Единый источник правды — правила расчёта скидки не дублируются.

Тестируемость — CalculateDiscountAction тестируется отдельно, со своими моками.

Композиция без дублирования — сохраняется чистота аспектов в каждом конкретном процессе.

Исключение
Иногда аспект может быть настолько общим, что его можно сразу делать отдельным действием. Например, «отправка email-уведомления» — это самостоятельная бизнес-операция, которая может вызываться из многих мест. Но в сомнительных случаях лучше начать с аспекта и вынести позже — это дешевле, чем потом бороться с лишней абстракцией.

Итог: следуй правилу «сначала аспект, выноси при повторном использовании». Оно спасает от overengineering и держит фокус на текущей задаче.

Пример трансформации легаси-кода в ActionMachine
Давай возьмём классический пример из интернет-магазина: обработка платежа. В легаси-коде это может быть один огромный класс, который делает всё: валидирует карту, списывает деньги, обновляет баланс, отправляет email и пишет в аудит.

Исходный легаси-монстр
python
# legacy/payment_processor.py
import logging
from typing import Dict

class PaymentProcessor:
    """Огромный класс, который делает всё сразу"""
    
    def __init__(self, db_conn, email_client, logger=None):
        self.db = db_conn
        self.email = email_client
        self.logger = logger or logging.getLogger(__name__)
    
    def process_payment(self, user_id: int, order_id: int, card_token: str, amount: float) -> Dict:
        # Шаг 1: проверка пользователя
        user = self.db.fetch_one("SELECT * FROM users WHERE id = %s", (user_id,))
        if not user:
            raise ValueError("User not found")
        self.logger.info(f"User {user_id} found")
        
        # Шаг 2: проверка заказа
        order = self.db.fetch_one("SELECT * FROM orders WHERE id = %s", (order_id,))
        if not order:
            raise ValueError("Order not found")
        if order['status'] != 'pending':
            raise ValueError("Order is not pending")
        self.logger.info(f"Order {order_id} is valid")
        
        # Шаг 3: валидация карты через внешний сервис
        card_valid = self._validate_card(card_token, amount)
        if not card_valid:
            raise ValueError("Card validation failed")
        
        # Шаг 4: списание средств (ещё один внешний вызов)
        transaction_id = self._charge_card(card_token, amount)
        self.logger.info(f"Payment charged, transaction: {transaction_id}")
        
        # Шаг 5: обновление статуса заказа в БД
        self.db.execute(
            "UPDATE orders SET status = 'paid', transaction_id = %s WHERE id = %s",
            (transaction_id, order_id)
        )
        
        # Шаг 6: отправка email
        self.email.send(
            to=user['email'],
            subject="Payment received",
            body=f"Your payment of {amount} for order {order_id} has been processed."
        )
        
        # Шаг 7: запись в аудит
        self.db.execute(
            "INSERT INTO audit (user_id, action, details) VALUES (%s, %s, %s)",
            (user_id, 'payment_processed', f"order:{order_id}, amount:{amount}")
        )
        
        return {
            'success': True,
            'transaction_id': transaction_id,
            'order_id': order_id
        }
    
    def _validate_card(self, card_token: str, amount: float) -> bool:
        # Вызов внешнего API валидации карт
        # (в реальности тут был бы HTTP-запрос)
        return True
    
    def _charge_card(self, card_token: str, amount: float) -> str:
        # Вызов платёжного шлюза
        # (в реальности тут был бы HTTP-запрос)
        return f"txn_{order_id}_{int(time.time())}"

Шаг 1.3: Создаём действие-обёртку
Теперь создадим действие, которое пока просто вызывает старый PaymentProcessor, но уже через порты.

python
from dataclasses import dataclass
from actionmachine import BaseAction, summary_aspect, depends
from ports.user_repository import UserRepository
from ports.order_repository import OrderRepository
from ports.payment_gateway import PaymentGateway
from ports.email_service import EmailService
from ports.audit_repository import AuditRepository

@dataclass
class ProcessPaymentParams:
    user_id: int
    order_id: int
    card_token: str
    amount: float

@dataclass
class ProcessPaymentResult:
    success: bool
    transaction_id: str
    order_id: int

@depends(UserRepository)
@depends(OrderRepository)
@depends(PaymentGateway)
@depends(EmailService)
@depends(AuditRepository)
class ProcessPaymentAction(BaseAction[ProcessPaymentParams, ProcessPaymentResult]):

    @summary_aspect("Временный аспект, вызывающий легаси")
    def handle(self, params: ProcessPaymentParams, state, deps) -> ProcessPaymentResult:
        # Создаём старый процессор, используя адаптеры
        # (это временно, потом уберём)
        from legacy.payment_processor import PaymentProcessor
        
        # Адаптеры хранят внутри старые зависимости
        user_repo = deps.get(UserRepository)
        order_repo = deps.get(OrderRepository)
        payment_gateway = deps.get(PaymentGateway)
        email_service = deps.get(EmailService)
        audit_repo = deps.get(AuditRepository)
        
        # Собираем всё в кучу для старого процессора (костыль)
        # В реальности пришлось бы немного подправить старый класс,
        # чтобы он принимал эти зависимости, но пока можно сделать так:
        processor = PaymentProcessor(
            db_conn=user_repo.db,  # доступ к внутренностям адаптера (временно)
            email_client=email_service.client
        )
        
        result = processor.process_payment(
            user_id=params.user_id,
            order_id=params.order_id,
            card_token=params.card_token,
            amount=params.amount
        )
        
        return ProcessPaymentResult(**result)

Итерация 3: Выделяем повторно используемую логику в отдельное действие
Представим, что в другом месте приложения (например, при предварительном расчёте стоимости) нам понадобилась та же логика валидации карты. Мы замечаем, что аспект validate_card сейчас привязан к действию ProcessPaymentAction, но его логика может быть полезной сама по себе.

Шаг 3.1: Создаём отдельное действие для валидации карты
python
@dataclass
class ValidateCardParams:
    card_token: str
    amount: float

@dataclass
class ValidateCardResult:
    valid: bool

@depends(PaymentGateway)
class ValidateCardAction(BaseAction[ValidateCardParams, ValidateCardResult]):

    @summary_aspect("Проверка карты")
    def validate(self, params, state, deps):
        payment_gateway = deps.get(PaymentGateway)
        valid = payment_gateway.validate_card(params.card_token, params.amount)
        return ValidateCardResult(valid=valid)
Шаг 3.2: Заменяем аспект вызовом действия
В ProcessPaymentAction вместо аспекта validate_card теперь используем вызов нового действия через deps.run_action().

python
@depends(ValidateCardAction)  # добавляем зависимость
class ProcessPaymentAction(...):

    @aspect("Валидация карты")
    def validate_card(self, params, state, deps):
        validate_action = deps.get(ValidateCardAction)
        result = deps.run_action(
            validate_action,
            ValidateCardParams(card_token=params.card_token, amount=params.amount)
        )
        if not result.valid:
            raise ValueError("Card validation failed")
        return state
Теперь логика валидации карты живёт в отдельном действии, которое можно переиспользовать где угодно.

Итог: пошаговая трансформация
Изоляция инфраструктуры – создаём порты и адаптеры-обёртки, действие вызывает старый код.

Выделение аспектов – по одному выносим логику в аспекты, каждый раз проверяя, что тесты проходят.

Удаление легаси – когда все шаги в аспектах, убираем старый код.

Выделение действий – если логика аспекта становится нужна в другом месте, превращаем его в отдельное действие.

Пошаговый пример: превращаем легаси-монстра в ресурсный менеджер и строим Action
Представьте, что в проекте живёт старый класс PaymentManager, который используется повсюду как синглтон. Он делает всё: и валидацию карт, и списание, и логирование, и даже обновление БД. Код огромный, запутанный, с состоянием. Мы хотим начать использовать ActionMachine, но не можем переписать весь класс сразу. Наша цель — изолировать его за интерфейсом, чтобы новый код (Action) зависел от абстракции, а потом постепенно откусывать куски логики.

Исходный легаси-монстр

Проблема: этот класс используется во многих местах, его сложно тестировать, он хранит состояние и делает слишком много.

Итерация 1: Изолируем монстра за интерфейсом (портом)
Создадим абстрактный класс (порт), который описывает только те методы, которые реально нужны для наших будущих Action. Пока реализация будет просто делегировать вызовы монстру.

Шаг 1.1: Определяем порт
python
# ports/payment_gateway.py
from abc import ABC, abstractmethod

class PaymentGateway(ABC):
    @abstractmethod
    def validate_card(self, card_token: str, amount: float) -> bool:
        pass
    
    @abstractmethod
    def charge(self, card_token: str, amount: float) -> str:
        pass
Шаг 1.2: Создаём адаптер-обёртку
Адаптер реализует порт и внутри использует синглтон PaymentManager. Это наш первый шаг — мы не меняем легаси, а просто даём ему «чистый интерфейс».

python
# adapters/legacy_payment_adapter.py
from ports.payment_gateway import PaymentGateway
from legacy.payment_manager import PaymentManager

class LegacyPaymentAdapter(PaymentGateway):
    def __init__(self):
        self._legacy = PaymentManager()  # получаем синглтон
    
    def validate_card(self, card_token: str, amount: float) -> bool:
        # Делегируем вызов, возможно с преобразованием форматов
        return self._legacy.validate_card(card_token, amount)
    
    def charge(self, card_token: str, amount: float) -> str:
        # Делегируем
        return self._legacy.charge(card_token, amount)
Теперь мы можем использовать этот адаптер везде, где раньше использовали PaymentManager. Но главное — мы можем написать Action, которое будет зависеть от порта PaymentGateway, а не от конкретного класса.

Итерация 2: Создаём первое действие, использующее порт
Теперь напишем простое действие, которое выполняет платеж, используя наш адаптер. Пока логика минимальна — просто вызывает методы порта.

python
from dataclasses import dataclass
from actionmachine import BaseAction, summary_aspect, depends
from ports.payment_gateway import PaymentGateway

@dataclass
class ChargeParams:
    card_token: str
    amount: float

@dataclass
class ChargeResult:
    success: bool
    transaction_id: str

@depends(PaymentGateway)
class ChargeAction(BaseAction[ChargeParams, ChargeResult]):

    @summary_aspect("Выполнить платеж")
    def handle(self, params, state, deps):
        gateway = deps.get(PaymentGateway)
        if not gateway.validate_card(params.card_token, params.amount):
            return ChargeResult(success=False, transaction_id="")
        txn_id = gateway.charge(params.card_token, params.amount)
        return ChargeResult(success=True, transaction_id=txn_id)
Что изменилось?

Мы теперь не используем PaymentManager напрямую.

Действие зависит только от интерфейса PaymentGateway.

В тестах мы можем подменить адаптер моком.

Легаси-монстр всё ещё живёт внутри адаптера, но уже не расползается по коду.

Итерация 3: Постепенно вытаскиваем логику из монстра
Теперь, когда у нас есть действие, мы можем начать «откусывать» куски логики из легаси-класса и переносить их в адаптер или даже в новые действия.

Предположим, мы заметили, что в методе charge легаси-класса есть дополнительная логика: обновление статистики и запись в БД. Это уже не чисто транспорт — это бизнес-логика. Мы можем решить, что статистика должна обновляться в отдельном действии, а БД — через отдельный ресурсный менеджер.

Шаг 3.1: Выделяем работу с БД в отдельный порт
Создаём интерфейс для работы с историей платежей:

python
# ports/payment_repository.py
class PaymentRepository(ABC):
    @abstractmethod
    def save_charge(self, transaction_id: str, amount: float) -> None:
        pass
Адаптер, использующий легаси PaymentManager, может пока делегировать вызовы, но мы уже отделили концепцию.

Шаг 3.2: Разбиваем действие на аспекты
Теперь наше действие может использовать несколько ресурсов:

python
@depends(PaymentGateway)
@depends(PaymentRepository)
class ChargeAction(BaseAction[ChargeParams, ChargeResult]):

    @aspect("Валидация карты")
    def validate(self, params, state, deps):
        gateway = deps.get(PaymentGateway)
        if not gateway.validate_card(params.card_token, params.amount):
            raise ValueError("Card validation failed")
        return state

    @aspect("Списание")
    def charge(self, params, state, deps):
        gateway = deps.get(PaymentGateway)
        txn_id = gateway.charge(params.card_token, params.amount)
        state['transaction_id'] = txn_id
        return state

    @summary_aspect("Сохранение результата")
    def save(self, params, state, deps):
        repo = deps.get(PaymentRepository)
        repo.save_charge(state['transaction_id'], params.amount)
        return ChargeResult(success=True, transaction_id=state['transaction_id'])
Теперь логика работы с БД вынесена в отдельный аспект, и мы можем легко заменить реализацию PaymentRepository (например, на реальную БД или мок).

Итерация 4: Если логика становится общей, выделяем в отдельное действие
Предположим, в другом месте (например, при возврате платежа) нам понадобилась та же валидация карты. Мы можем превратить аспект валидации в самостоятельное действие.

python
@depends(PaymentGateway)
class ValidateCardAction(BaseAction[ValidateCardParams, ValidateCardResult]):
    @summary_aspect("Проверить карту")
    def validate(self, params, state, deps):
        gateway = deps.get(PaymentGateway)
        valid = gateway.validate_card(params.card_token, params.amount)
        return ValidateCardResult(valid=valid)
Теперь ChargeAction может вызывать это действие через deps.run_action():

python
@depends(ValidateCardAction)
class ChargeAction(...):
    @aspect("Валидация карты")
    def validate(self, params, state, deps):
        validate_action = deps.get(ValidateCardAction)
        result = deps.run_action(validate_action, ValidateCardParams(...))
        if not result.valid:
            raise ValueError("Invalid card")
        return state
Итог: что мы получили
Изоляция легаси — старый PaymentManager спрятан за интерфейсом, новый код его не видит.

Постепенная миграция — мы можем переписывать отдельные части легаси, не ломая систему, заменяя адаптер на новую реализацию.

Чистая архитектура — наши Actions зависят только от портов, легко тестируются.

Возможность выделять общую логику — аспекты, ставшие слишком общими, превращаются в самостоятельные действия.

Этот подход идеально подходит для работы с большим легаси-кодом: сначала изолируем «монстра» за интерфейсом, потом строим вокруг него Actions, и только затем начинаем по частям переписывать внутренности монстра, не останавливая разработку.

✅ Почему это рабочее решение
1. Безопасность
Мы не трогаем старый код сразу. Мы только добавляем новые абстракции вокруг него.

Старый класс продолжает работать как раньше, все существующие вызовы к нему остаются.

Новый код пишется уже с использованием ActionMachine, опираясь на интерфейсы.

2. Постепенность
Можно двигаться маленькими шагами: сначала создать один порт, потом написать одно действие, потом отрезать кусок логики.

Каждый шаг завершается рабочим состоянием и может быть закоммичен.

3. Тестируемость
Появляется возможность тестировать новую логику изолированно, подменяя адаптеры моками.

Старый код пока тестируется как есть (если тесты были), но теперь можно постепенно покрывать тестами новые действия.

4. Независимость
Новый код не зависит от легаси-класса, только от порта. Можно заменять реализацию порта по мере переписывания.

5. Возможность выделять общую логику
Как мы показали в примере, аспекты, ставшие общими, легко превращаются в отдельные действия. Это естественная эволюция.

⚠️ Риски и сложности
1. Правильное выделение порта
Нужно выбрать правильный уровень абстракции. Если порт будет слишком детализирован (повторять все методы легаси), то мы не получим выигрыша. Если слишком общий — может не хватить методов.

Рекомендация: начинать с минимально необходимого набора методов для первого действия.

2. Сохранение состояния легаси
Легаси-класс может иметь внутреннее состояние (кэши, статистику). Если мы начинаем вызывать его методы через адаптер, состояние сохраняется, но если мы потом начнем переписывать куски, нужно аккуратно переносить и состояние.

3. Сложность координации
Если несколько команд работают над разными частями системы, может возникнуть путаница: кто за какой порт отвечает, какой адаптер использует легаси. Нужна документация и общее понимание.

4. Производительность
Лишний слой абстракции может добавить небольшие накладные расходы, но обычно они незначительны. В критических местах можно будет оптимизировать позже.

5. Длительность
Полная миграция может занять много времени. Нужно убедить менеджмент, что это необходимо, иначе процесс может быть остановлен на полпути.

📈 Реальный опыт
Подобные подходы успешно применяются в крупных проектах:

Strangler Fig Pattern — описан Мартином Фаулером и используется при переходе от монолита к микросервисам.

Anti-Corruption Layer — паттерн из книги Эванса по DDD, защищает новый домен от «заразы» легаси-кода.

В индустрии это стандартная практика при работе с унаследованными системами.

Многие современные фреймворки (Spring, .NET Core) предлагают механизмы для такой изоляции. ActionMachine предоставляет готовые инструменты: порты, адаптеры, фабрика зависимостей, аспекты и действия. Всё, что нужно — это следовать описанной стратегии.

🔧 Когда это решение может не сработать
Если легаси-код настолько связан, что невозможно выделить четкие границы (например, нет четких интерфейсов, все завязано на глобальные переменные).

Если нет возможности даже создать адаптер-обертку без модификации легаси (например, класс не позволяет подменять зависимости).

Если команда не готова инвестировать время в рефакторинг и будет соблазн всё бросить.

В таких случаях требуется сначала подготовить легаси-код: провести небольшой рефакторинг, чтобы выделить интерфейсы или хотя бы сделать класс более тестируемым. Но это уже выходит за рамки чистого применения ActionMachine.

когда монстра упоюаковывать в ресурс, а когда в набор action
Мысли пользователя: "когда монстра упоюаковывать в ресурс, а когда в набор action". Видимо, опечатка: "упоюаковывать" = "упаковывать". Вопрос: когда старый большой класс с логикой и состоянием превращать в ресурсный менеджер (адаптер), а когда в набор действий.

Нужно дать критерии. Вспомним обсуждения ранее: ресурсный менеджер — это адаптер, который предоставляет доступ к внешней системе, инкапсулирует состояние и не содержит бизнес-логики. Действие — бизнес-логика, stateless, использует ресурсы.

Для легаси-монстра: если его основная ответственность — взаимодействие с внешним миром (БД, API, файлы), то это кандидат в ресурс. Если внутри много бизнес-правил, то это бизнес-логика, которую нужно разбить на действия.

Критерии:

Есть состояние, управление ресурсами (соединения, открытые файлы) → ресурс.

Есть бизнес-правила, которые могут быть переиспользованы → действия.

Если монстр смешанный, то сначала выделяем ресурсную часть (доступ к внешним системам) через порты, а бизнес-логику постепенно вытаскиваем в действия, используя этот ресурс.

Ответ должен быть понятным, с примерами.

Выбор между тем, чтобы упаковать легаси-монстра в ресурсный менеджер или разбить на набор действий, зависит от того, что этот монстр на самом деле делает. Есть простое правило, которое поможет не ошибиться:

Ресурсный менеджер — если монстр управляет доступом к внешней системе (база данных, API, файловая система) и имеет состояние (соединения, сессии, кэши).
Набор действий — если монстр содержит бизнес-логику, правила и процессы, не привязанные к конкретному транспорту.

Но чаще всего легаси-монстр — это смесь того и другого. Тогда стратегия такая:

Сначала выделяем ресурсную часть — то, что отвечает за коммуникацию с внешним миром. Создаём для неё порт (интерфейс) и адаптер, который пока делегирует вызовы монстру. Это позволяет отделить «грязь» (работу с БД, API) от бизнес-логики.

Оставшуюся бизнес-логику начинаем вытаскивать в действия — по одному сценарию за раз. Каждое действие будет использовать ресурсный адаптер через порт.

Давай разберём на примерах.

🧱 Когда разбивать на действия
Признаки:

Монстр реализует последовательности шагов: проверить, рассчитать, сохранить, уведомить.

Внутри много условных операторов, циклов, вызовов других методов того же класса.

Состояние класса меняется по ходу выполнения (например, self.total, self.items).

🧠 Критерий: долгоживущее состояние vs временное состояние
Когда в ресурсный менеджер
Признаки:

Монстр хранит данные, которые должны сохраняться между вызовами (например, кэш результатов запросов, статистику использования, пул соединений).

Он управляет ресурсами, которые нельзя пересоздавать на каждый вызов (соединения с БД, HTTP-клиенты с keep-alive).

Вокруг него «танцы с бубном» — много кода завязано на его состояние, и потеря этого состояния приведёт к ошибкам или падению производительности.

Что делаем:
Оборачиваем его в адаптер, реализующий порт. Внутри адаптера монстр продолжает жить и хранить своё состояние. Все действия, которым нужен доступ к этому состоянию, используют порт и получают один и тот же экземпляр адаптера (через DI как синглтон).

Когда в действие
Признаки:

Класс создаётся с нуля при каждом вызове, его состояние ограничено рамками одной операции.

После завершения работы объект выбрасывается, состояние не нужно сохранять.

Логика состоит из последовательности шагов, которые можно разбить на аспекты.

Что делаем:
Превращаем его в действие (или несколько действий). Внутри действия используем ресурсные менеджеры для доступа к долгоживущим состояниям.

📦 Пример: монстр с долгоживущим состоянием
Представь класс, который собирает метрики по платежам и хранит их в памяти. Эти метрики важны для мониторинга в реальном времени, их нельзя сбрасывать при каждом запросе.

python
# legacy/payment_stats_monster.py
class PaymentStatsMonster:
    """Синглтон, который копит статистику по платежам"""
    _instance = None
    
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._init()
        return cls._instance
    
    def _init(self):
        self.total_charged = 0.0
        self.charge_count = 0

        self.connection = create_expensive_connection()

        self.total_charged += amount
        self.charge_count += 1
        self.last_hour_charges.append((time.time(), amount))
        # очистка старых записей
        cutoff = time.time() - 3600

    def get_stats(self):
        return {

            'count': self.charge_count,
            'last_hour_total': sum(x[1] for x in self.last_hour_charges)
        }
    
    def flush_to_db(self):

        self.connection.execute("...")
Этот монстр:

Хранит состояние (статистику) в памяти.

Имеет соединение с БД.

Его нельзя пересоздавать при каждом вызове — потеряем накопленные данные.

Шаг 1: Изолируем как ресурсный менеджер
Создаём порт, который описывает только нужные методы, и адаптер, который внутри держит монстра.

python
# ports/payment_stats_repository.py
class PaymentStatsRepository(ABC):
    @abstractmethod
    def record_charge(self, amount: float) -> None:
        pass
    
    @abstractmethod
    def get_stats(self) -> dict:
        pass

# adapters/stats_adapter.py
from legacy.payment_stats_monster import PaymentStatsMonster

class StatsAdapter(PaymentStatsRepository):
    def __init__(self):
        self._monster = PaymentStatsMonster()  # получаем синглтон
    
    def record_charge(self, amount: float) -> None:
        self._monster.record_charge(amount)
    
    def get_stats(self) -> dict:
        return self._monster.get_stats()
Теперь мы можем внедрять этот адаптер как синглтон в DI-контейнер. Все действия будут использовать один и тот же экземпляр, и статистика не потеряется.

Шаг 2: Пишем действие, использующее ресурс
python
@depends(PaymentStatsRepository)
class ChargeAction(BaseAction):
    # ... аспекты валидации и списания ...
    
    @aspect("Запись статистики")
    def record_stats(self, params, state, deps):
        stats_repo = deps.get(PaymentStatsRepository)
        stats_repo.record_charge(params.amount)
        return state
🔄 Пример: временное состояние — сразу действие
Другой класс, который создаётся каждый раз заново и не хранит долгоживущих данных:

class TemporaryCalculator:
    def __init__(self, multiplier):
        self.multiplier = multiplier  # передаётся при создании

    def add(self, x):

class CalculateAction(BaseAction):
    @aspect("Накопление")
    def accumulate(self, params, state, deps):
        state['total'] = sum(params.values)
        return state
    
    @summary_aspect("Умножение")
    def multiply(self, params, state, deps):
        return CalculateResult(state['total'] * params.multiplier)
🧩 Смешанный случай: монстр с долгоживущим состоянием и временными методами
Часто монстр делает и то, и другое: хранит глобальное состояние, но также имеет методы, которые можно вызывать без сохранения состояния между вызовами. В этом случае мы всё равно изолируем монстра как ресурс (чтобы сохранить его долгоживущие данные), а временные методы становятся частью действий, использующих этот ресурс.

Главное правило, которое ты сформулировал, идеально:

«Если монстр с долгоживущим состоянием, вокруг которого все танцы с бубном, нам нельзя потерять это состояние — изолируем его как ресурс. Если класс или набор классов каждый раз создаются с нуля и их состояние временное для вызова, то сразу action.»

Этот принцип позволяет чётко разделить ответственность и избежать ошибок при рефакторинге.
