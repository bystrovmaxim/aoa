🧠 **Философия и ключевые идеи**

ActionMachine родился из простого наблюдения: бизнес-логика в приложениях часто размазана по контроллерам, сервисам, хелперам, превращаясь в неподдерживаемую «лапшу». Мы предлагаем **структурированный подход**, где каждая бизнес-операция — это явное, самодостаточное действие.

🌟 **Основополагающие принципы**

| Принцип | Суть |
|--------|------|
| Действие как атом | Каждая бизнес-операция — отдельный класс с чёткими границами |
| Явное лучше неявного | Все зависимости, права доступа и этапы выполнения видны сразу |
| Разделение ответственности | Бизнес-данные (`params`), инфраструктура (DI), контекст запроса (плагины) |
| Наблюдатели не вмешиваются | Плагины читают контекст, но не могут менять бизнес-логику |
| Декларативность через декораторы | Действие описывает свои этапы и правила с помощью декораторов |
| Композиция вместо наследования | Для бизнес-логики используем композицию (через DI), наследование допустимо только для инфраструктурных классов |

🧩 **Архитектурные слои**

```
┌─────────────────────────────────────────────────────┐
│  Входные адаптеры (FastAPI, MCP, CLI)               │
│  • Сейчас пишутся вручную, в планах генерация       │
├─────────────────────────────────────────────────────┤
│  Action Machine (ядро оркестрации)                   │
│  • Управляет порядком выполнения аспектов            │
│  • Знает о плагинах и вызывает их                    │
│  • Генерирует события до/после каждого этапа         │
├─────────────────────────────────────────────────────┤
│  Плагины (читают контекст)                           │
│  • Логирование, метрики, аудит, трейсинг             │
│  • Подписываются через регулярные выражения          │
│  • Изолированы от ошибок бизнес-логики               │
├─────────────────────────────────────────────────────┤
│  Действия (бизнес-логика)                            │
│  • Не видят контекст запроса (user, ip и т.д.)       │
│  • Зависимости получают через конструктор (DI)       │
│  • Параметры — только бизнес-данные в `params`       │
└─────────────────────────────────────────────────────┘
```

🔒 **Чёткие границы между слоями**

| Слой | Видит контекст | Видит params | Создаёт зависимости |
|------|----------------|--------------|---------------------|
| Входные адаптеры | ✅ | ✅ | Нет, только передаёт |
| Action Machine | ✅ | ✅ | Нет, только оркестрирует |
| Плагины | ✅ | ✅ | Нет, только читают |
| Действия | ❌ | ✅ | Да, через DI |

⚙️ **Ядро: Action Machine**

🎬 **Что делает Machine**

1. Получает класс действия, параметры и контекст  
2. Собирает аспекты, помеченные декораторами `@aspect` и `@summary_aspect`  
3. Последовательно выполняет регулярные аспекты, передавая между ними состояние (`state`)  
4. Для каждого аспекта запускает before/after-события плагинов  
5. Выполняет summary-аспект, возвращающий результат  
6. Генерирует глобальные события `global_start` и `global_finish`

📍 **Ключевые особенности**

- Machine хранит список плагинов и вызывает их напрямую (упрощённо, но прозрачно)  
- Все события строятся динамически: `before:delay`, `after:prepare`  
- Порядок аспектов определяется порядком объявления в классе (по номеру строки)  
- Плагины получают `nest_level` — глубину вложенности вызова действий  

---

👁️ **Плагины: наблюдатели без вмешательства**

📌 **Ключевое решение**

Плагины **не могут изменять** параметры, состояние или результат действия — только читать и добавлять метаданные через собственное состояние. Это исключает случайное влияние на бизнес-логику.

🎯 **Подписка на события**

```python
@on('before:.*', '.*', ignore_exceptions=True)
async def log_before(self, state_plugin, event_name, action_name,
                     params, state_aspect, is_summary, deps, context,
                     result, duration, nest_level, **kwargs):
    print(f"{indent(nest_level)}[{event_name}] before aspect")
    return state_plugin
```

- Регулярные выражения для имени события и класса действия  
- Флаг `ignore_exceptions` позволяет продолжить работу при сбое плагина  
- Методы получают полный контекст выполнения  

✅ **Что это даёт**

| Преимущество | Объяснение |
|--------------|------------|
| Безопасность | Плагин не может испортить бизнес-данные |
| Изоляция | Ошибка в плагине не ломает действие (если `ignore_exceptions=True`) |
| Тестируемость | Каждый плагин тестируется отдельно |
| Гибкость | Можно комбинировать множество плагинов, они не конфликтуют |

📦 **Действия (Actions): кирпичики бизнес-логики**

🧱 **Структура действия**

```python
@depends(EmailService, SmsService)
@CheckRoles(CheckRoles.ANY)
class NotificationAction(BaseAction['Params', 'Result']):
    
    @dataclass(frozen=True)
    class Params(BaseParams):
        channel: str
        message: str
        recipient: str

    @dataclass(frozen=True)
    class Result(BaseResult):
        success: bool

    @aspect("Выбор канала")
    @InstanceOfChecker("service", (EmailService, SmsService))
    def choose_channel(self, params, state, deps):
        if params.channel == 'email':
            state['service'] = deps.get(EmailService)
        else:
            state['service'] = deps.get(SmsService)
        return state

    @summary_aspect("Отправка")
    def send(self, params, state, deps):
        state['service'].send(params.recipient, params.message)
        return NotificationAction.Result(success=True)
```

📋 **Аспекты**

- **Регулярные аспекты** (`@aspect`) — выполняются последовательно, получают и возвращают `state` (словарь). Могут добавлять данные в состояние.
- **Summary-аспект** (`@summary_aspect`) — завершающий этап, возвращает типизированный `Result`.

🎨 **Правила работы с состоянием**

- `state` передаётся от аспекта к аспекту **по ссылке**, мутировать его разрешено, но **не рекомендуется** — лучше возвращать новый словарь.
- Если у аспекта есть чекеры (`@InstanceOfChecker`, `@IntFieldChecker` и т.д.), они проверяют типы и наличие полей в возвращаемом `state`.  
- Если чекеров нет, аспект **обязан вернуть пустой словарь** — иначе будет исключение.

💉 **DI для инфраструктуры**

Зависимости, такие как репозитории, клиенты API, передаются через конструктор с помощью декоратора `@depends`. Фабрика `DependencyFactory` кэширует экземпляры.

🧪 **Тестирование как первоклассная абстракция**

`ActionTestMachine` позволяет:

- Подменять любые зависимости через словарь моков  
- Тестировать отдельные аспекты напрямую (через `build_factory`)  
- Отслеживать вызовы с помощью `MockAction`

```python
def test_choose_channel():
    machine = ActionTestMachine({EmailService: FakeEmail()})
    factory = machine.build_factory(NotificationAction)
    action = NotificationAction()
    result = action.choose_channel(params, {}, factory)
    assert result['service'] is fake_email
```

Благодаря чёткому разделению слоёв, тестирование бизнес-логики становится простым и естественным.

🚧 **Текущие ограничения и планы развития**

ActionMachine — это **работающий код**, который уже используют в проектах. Но, как и любой инструмент, он имеет свои недочёты, которые мы честно признаём и планируем исправить.

| Ограничение | Описание | План решения |
|-------------|----------|---------------|
| **Async/sync гибрид** | `_run_plugins_sync` может упасть в async-окружении | Переход на полностью асинхронный `run` (с обратной совместимостью через синхронную обёртку) |
| **Мутабельный state** | Аспекты могут мутировать входящий словарь, что усложняет отладку | Перейти на иммутабельный `state` (копирование) или строго требовать возврат нового словаря |
| **Кэширование плагинов** | Кэш `(event_name, class_name)` не обновляется при динамическом изменении плагинов | Добавить механизм инвалидации кэша при изменении списка плагинов |
| **Наследование `_role_spec`** | Ролевая спецификация наследуется, что может дать неявные права | Явно копировать `_role_spec` в каждом классе или использовать механизм `__init_subclass__` |
| **Автоматическая генерация транспорта** | В манифесте обещана, но пока не реализована | Добавить генераторы для FastAPI, MCP, CLI на основе метаданных действий |
| **DebugMachine** | Отсутствует, хотя полезна для трассировки | Реализовать как плагин или отдельную машину с подробным логированием |

🎯 **Ключевые достижения**

| Аспект | Решение |
|--------|---------|
| Структурированность | Бизнес-логика разбита на атомарные действия с явными этапами |
| Прозрачность зависимостей | Инфраструктура через DI, бизнес-данные через `params` |
| Расширяемость | Плагины подписываются на события и не влияют на логику |
| Тестируемость | `ActionTestMachine` и возможность тестировать отдельные аспекты |
| Качество кода | `mypy --strict` без ошибок, radon: почти все функции A (B в самых сложных местах), vulture чист [2] |

**ActionMachine — это не «ещё одна библиотека». Это способ организовать бизнес-логику так, чтобы её можно было поддерживать, расширять и тестировать годами.** Мы честно смотрим на свои недостатки и открыто говорим о планах. Код уже работает, а сообщество помогает делать его лучше. Присоединяйтесь! 🚀
