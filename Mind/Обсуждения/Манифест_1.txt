🧠 Философия и ключевые идеи    
    
ActionEngine родился из простого, но мощного наблюдения: бизнес-логика в современных приложениях размазана по контроллерам, сервисам, хелперам и утилитам, превращаясь в неподдерживаемую лапшу кода.    
    
🌟 Основополагающие принципы    
    
Принцип Суть    
Действие как атом Каждая бизнес-операция — отдельный, самодостаточный класс    
Явное лучше неявного Все зависимости действия видны сразу    
Разделение по природе данных Сквозные данные (user, trace_id) vs инфраструктура (db, cache)    
Наблюдатели не вмешиваются Плагины только читают, никогда не пишут    
Декларативность через интерфейсы Действие само говорит, что оно умеет    
Композиция вместо наследования Никаких базовых классов с тонной методов    
    
🎯 Что на самом деле происходит в запросе    
    
В любом запросе к системе есть три принципиально разных типа данных:    
    
```python    
# 1. Сквозные данные (request-level) — нужны везде    
- Кто: user_id, roles    
- Откуда: ip, user_agent    
- Для трассировки: trace_id, request_id    
    
# 2. Бизнес-данные (params) — то, с чем работает логика    
- product_id, quantity, price    
- from_account, to_account, amount    
    
# 3. Инфраструктура (service-level) — как работать с внешним миром    
- database connection    
- cache client    
- email sender    
- payment gateway    
```    
    
Гениальное открытие: эти данные должны жить в разных местах и обрабатываться по-разному.

🧩 Архитектурные слои    
    
📊 Четырёхуровневая архитектура    
    
```    
┌─────────────────────────────────────────────────────┐    
│  Транспортный слой (FastAPI, MCP, CLI)              │    
│  • Автоматически генерируется из действий           │    
│  • Никакой бизнес-логики, только маршрутизация      │    
├─────────────────────────────────────────────────────┤    
│  Action Machine (ядро оркестрации)                   │    
│  • Управляет порядком выполнения                     │    
│  • Не знает о плагинах, только генерирует события    │    
├─────────────────────────────────────────────────────┤    
│  Plugin Manager (управление плагинами)               │    
│  • Получает события от Machine                       │    
│  • Рассылает их плагинам                             │    
│  • Изолирует ошибки                                  │    
├─────────────────────────────────────────────────────┤    
│  Действия + Плагины (бизнес-логика и инфраструктура) │    
│  • Действия: чистая бизнес-логика, не видят контекст │    
│  • Плагины: read-only наблюдатели, видят контекст    │    
└─────────────────────────────────────────────────────┘    
```    
    
🔒 Чёткие границы между слоями    
    
Слой Видит контекст Видит params Создаёт зависимости    
Транспорт ✅ ✅ Нет, только передаёт    
Machine ✅ ✅ Нет, только оркестрирует    
Plugin Manager ✅ ✅ Нет, только рассылает    
Плагины ✅ ✅ Нет, только читают    
Действия ❌ ✅ Да, через DI

⚙️ Ядро: Action Machine    
    
🎬 Что делает Machine    
    
1. Получает класс действия, параметры и контекст    
2. Проверяет, какие аспекты реализованы в действии    
3. Генерирует события до и после каждого аспекта    
4. Вызывает методы действия в правильном порядке    
5. Собирает результат и возвращает его    
    
📍 Ключевая особенность    
    
Machine не знает о плагинах. Она только генерирует события:    
    
· before_authorization    
· after_authorization    
· before_validation    
· after_validation    
· before_handle    
· after_handle    
· и так далее    
    
Эти события получает Plugin Manager, который уже решает, какие плагины и как вызывать.    
    
🔄 Почему это гениально    
    
· Полное разделение оркестрации и расширения    
· Возможность подменять Machine для тестов    
· Прозрачность — можно включить трассировку и увидеть каждый шаг    
· Никакой магии — Machine просто вызывает методы в нужном порядке

👁️ Плагины: наблюдатели без вмешательства    
    
📌 Ключевое открытие    
    
Плагины никогда не должны влиять на бизнес-логику. Они только наблюдают и собирают данные. Это фундаментальное решение, которое исключает целый класс ошибок.    
    
🎯 Что могут плагины    
    
```python    
class TelemetryPlugin:    
    def before_handle(self, ctx, params):    
        # могут читать ctx    
        # могут читать params    
        # НЕ могут их менять    
        ctx._start = time.time()    
        
    def after_handle(self, ctx, params, result):    
        duration = time.time() - ctx._start    
        # отправляют метрики, но не трогают result    
```    
    
✅ Что это даёт    
    
Преимущество Объяснение    
Безопасность Плагин не может сломать бизнес-логику    
Изоляция Ошибка в плагине не влияет на действие    
Тестируемость Плагины тестируются отдельно    
Композиция Можно включить 10 плагинов, порядок не важен    
    
🛡️ Обработка ошибок    
    
Plugin Manager вызывает плагины с защитой:    
    
```python    
for plugin in plugins:    
    try:    
        plugin.before_handle(ctx, params)    
    except Exception:    
        log.error(f"Plugin {plugin.name} failed")    
        # бизнес-логика продолжается    
```    
    
⚡ Асинхронность    
    
Плагины могут выполняться асинхронно, не блокируя ответ. Plugin Manager может:    
    
· Вызывать их синхронно (просто)    
· Ставить в очередь (для надёжности)    
· Запускать в фоне (для скорости)

📦 Действия (Actions): чистые бизнес-кирпичики    
    
🧱 Что такое действие    
    
Действие — это класс, который реализует один или несколько интерфейсов-аспектов и содержит только бизнес-логику.    
    
🔥 Главное: действие не видит контекст!    
    
```python    
class TransferMoneyAction(IHandle):    
    def __init__(self, account_repo: AccountRepository):    
        self.account_repo = account_repo    
        
    def handle(self, params):    
        # ВСЁ, что нужно для логики — в params    
        from_id = params['from_account']    
        to_id = params['to_account']    
        amount = params['amount']    
        user_id = params['user_id']  # да, user_id тоже в params!    
            
        # чистая бизнес-логика    
        return self.account_repo.transfer(user_id, from_id, to_id, amount)    
```    
    
📋 Доступные аспекты (интерфейсы)    
    
Интерфейс Метод Когда вызывается    
IAuthorization authorize(params) До валидации    
IValidation validate(params) После авторизации    
IPreHandle pre_handle(params) Перед основной логикой    
IHandle handle(params) Основная логика    
IPostHandle post_handle(params, result) После основной логики    
IOnError on_error(error) При исключении    
    
🎨 Почему action не видит контекст    
    
Это ключевое архитектурное решение:    
    
Аспект С контекстом Без контекста    
Прозрачность Что в ctx? Всё явно в params    
Тестирование Мокай ctx Просто словарь    
Документация Скрытые зависимости Всё видно    
Соблазн халтурить "Добавлю-ка я сюда ещё" Невозможно    
    
💉 DI для инфраструктуры    
    
Инфраструктурные зависимости (БД, кэш, API) приходят через конструктор:    
    
```python    
class CreateOrderAction(IHandle):    
    def __init__(    
        self,    
        order_repo: OrderRepository,      # явно    
        payment_gateway: PaymentGateway,  # явно    
        email_sender: EmailSender         # явно    
    ):    
        self.order_repo = order_repo    
        self.payment_gateway = payment_gateway    
        self.email_sender = email_sender    
        
    def handle(self, params):    
        # используем self для инфраструктуры    
        # params для бизнес-данных    
        order = self.order_repo.create(params)    
        self.payment_gateway.charge(order)    
        self.email_sender.send(order)    
        return {"order_id": order.id}    
```

🤖 Разные движки для разных целей    
    
Action Machine — это не один класс, а семейство стратегий:    
    
🏭 ProductionMachine    
    
· Полная проверка всех аспектов    
· Строгий порядок выполнения    
· Интеграция с плагинами    
    
🧪 TestMachine    
    
· Без авторизации    
· Без плагинов    
· Для быстрого тестирования бизнес-логики    
    
🔍 DebugMachine    
    
· Подробное логирование каждого шага    
· Трассировка вызовов    
· Для отладки сложных сценариев    
    
📦 Как это используется    
    
```python    
# В продакшене    
machine = ProductionMachine(plugin_manager)    
    
# В тестах    
machine = TestMachine()    
result = machine.run(CreateOrderAction, params)    
    
# В отладке    
machine = DebugMachine(verbose=True)    
machine.run(CreateOrderAction, params)  # видим каждый шаг    
```

🚀 Транспортный слой: автоматическая генерация    
    
🔮 Магия декларативности    
    
Поскольку каждое действие декларативно описывает:    
    
· какие параметры принимает (через чекеры)    
· какой результат возвращает    
· какие роли требуются    
· какая у него бизнес-логика    
    
Мы можем автоматически сгенерировать для него интерфейсы!    
    
🌐 FastAPI    
    
```python    
# Реестр действий автоматически создаёт роутер    
router = generate_fastapi_router(registry)    
    
app.include_router(router)    
    
# Появляются эндпоинты:    
# POST /api/v1/create-order    
# POST /api/v1/transfer-money    
# POST /api/v1/cancel-order    
```    
    
🤖 MCP (Model Context Protocol)    
    
```python    
# Автоматически генерируются инструменты    
tools = generate_mcp_tools(registry)    
    
# Claude и другие LLM могут вызывать ваши действия!    
```    
    
🖥️ CLI    
    
```python    
# Автоматически создаются консольные команды    
!action-engine create-order --product-id=123 --quantity=2    
```    
    
📖 Документация    
    
```python    
# Автоматически генерируется OpenAPI    
# Автоматически генерируется Markdown для Wiki    
# Автоматически генерируются диаграммы зависимостей    
```    
    
🎯 Что это даёт    
    
· Один источник правды — код действия описывает всё    
· Никакого дублирования — не пишем контроллеры вручную    
· Всегда актуально — документация обновляется при каждом пуше    
· Множество протоколов — бесплатно!

🛡️ Ответы на критику    
    
📌 Критика 1: «Контекст станет God Object»    
    
Ответ: Контекст read-only и содержит только сквозные данные (user, trace_id). Инфраструктура идёт через явный DI. Это не God Object, а паспорт запроса — стандартная практика в индустрии.    
    
📌 Критика 2: «Движок станет чёрным ящиком»    
    
Ответ: Machine только генерирует события. Вся логика расширения — в плагинах, которые можно отключить или заменить. Можно подключить трассировку и видеть каждый шаг.    
    
📌 Критика 3: «Рефлексия медленная»    
    
Ответ: Метаданные кэшируются. При 500 действиях рефлексия занимает 0.1–0.3 секунды при старте. Узкие места не здесь.    
    
📌 Критика 4: «Нелинейные процессы»    
    
Ответ: 90% бизнес-процессов линейны. Для остальных — действия могут вызывать другие действия через DI, либо создаются специальные координаторы (Saga, Process Manager).    
    
📌 Критика 5: «Overengineering для простых задач»    
    
Ответ: ActionEngine — для бизнес-логики, не для health-check'ов. Простые вещи можно оставить обычными функциями.    
    
📌 Критика 6: «Версионирование сложно»    
    
Ответ: Версионирование решается на транспортном уровне. Действия имеют имя и версию, генератор API сам решает, что отдавать.    
    
📌 Критика 7: «context_requires — полумера»    
    
Ответ: Мы отказались от этой идеи. Контекст не передаётся в действия вообще. Зависимости — через DI, бизнес-данные — через params.

🏆 Итог: что мы построили    
    
🎯 Ключевые достижения    
    
Аспект Решение    
Чистота бизнес-логики Действия не видят контекст, только params    
Прозрачность зависимостей Инфраструктура через DI, бизнес-данные через params    
Расширяемость Read-only плагины через события    
Тестируемость Действия тестируются простым вызовом методов    
Производительность Кэширование метаданных, асинхронные плагины    
Документация Автоматическая генерация из деклараций    
Множество протоколов REST, MCP, CLI — бесплатно    
    
🏛️ Финальная архитектура    
    
```    
┌─────────────────────────────────────────────────────┐    
│  FastAPI    │  MCP       │  CLI      │  Документация │    
│  (генерация)│ (генерация)│(генерация)│  (генерация)  │    
├─────────────────────────────────────────────────────┤    
│  Action Machine (оркестрация аспектов)               │    
│  • before_auth → auth → after_auth                   │    
│  • before_validation → validation → after_validation │    
│  • before_handle → handle → after_handle             │    
│  • ...                                               │    
│  Генерирует события для Plugin Manager               │    
├─────────────────────────────────────────────────────┤    
│  Plugin Manager (управление плагинами)               │    
│  • получает события                                   │    
│  • вызывает плагины с защитой                         │    
│  • может работать асинхронно                          │    
├─────────────────────────────────────────────────────┤    
│  Плагины (читают ctx)                                 │    
│  • Телеметрия │ Аудит │ Логирование │ Трассировка   │    
├─────────────────────────────────────────────────────┤    
│  Действия (НЕ видят ctx)                              │    
│  • Бизнес-логика                                      │    
│  • Зависимости через DI                               │    
│  • Параметры через params                             │    
└─────────────────────────────────────────────────────┘    
```    
    
🌟 Главное достижение    
    
Мы создали не просто фреймворк, а новую парадигму разработки, где:    
    
· Бизнес-логика — это набор независимых, тестируемых кирпичиков    
· Инфраструктура подключается автоматически через плагины    
· Документация и API генерируются из кода    
· Разработчик пишет только то, что уникально для его бизнеса    
· Новичок понимает код с первого взгляда    
    
ActionEngine — это не «ещё одна библиотека». Это способ думать о разработке. 🚀
