# Changelog

All notable changes to `aoa-fastapi-adapter` are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Fixed

- **`resolve_verdicts()` — one `synthetic` dict replaces two parallel `{key set + builder function}` pairs.** Unknown-endpoint and route-level-auth-rejection isolation (both per-item, no real `access_decide` call) each had their own `set[_DedupKey]` plus a zero-argument verdict-builder function, duplicating the same idea — "this key's answer is already known" — twice (fix-audit finding 13, second document). Both builders always returned the same value regardless of key, and `BaseVerdict` is frozen, so one shared instance per synthetic outcome (`_UNKNOWN_ENDPOINT_VERDICT`/`_UNAUTHORIZED_VERDICT`, module-level constants) is safe to hand out for every key that needs it. `unknown_keys`/`unauthorized_keys` collapse into one `synthetic: dict[_DedupKey, BaseVerdict]`.
- **`build_manifest_from_route_index()` — `FastApiAdapter.build()` no longer computes the same route index twice.** `build_manifest(routes)` internally called `build_route_index(routes)` to deduplicate routes by operation, but `adapter.py` had already computed that exact index moments earlier for its own use, then handed the raw `self._routes` list back to `build_manifest` — silently redoing the same dedup pass on the same input (fix-audit finding 16, second document). `build_manifest` is now a thin wrapper around a new `build_manifest_from_route_index(route_index: Mapping[str, FastApiRouteRecord]) -> Manifest`, and `adapter.py`'s `_register_permissions_endpoints` calls the latter directly with the `route_index` it already has. `build_manifest(routes)` itself is unchanged for every existing caller/test.

- **`FastApiRouteRecord.operation` — one canonical `"{method} {path}"` format, not three independent copies.** `adapter.py` (route-lookup key and error-message `route_label`), `permissions.py`'s `build_route_index()`, and `manifest.py`'s `_build_endpoint()` each formatted `f"{record.method} {record.path}"` themselves — they agreed only because `FastApiRouteRecord.__post_init__` already normalizes `method`/`path` on the one shared record, not because any of them called shared code (fix-audit finding 10, second document — the same class of bug finding 10 of the first document fixed for a different pair of call sites). New `FastApiRouteRecord.operation` property is now the only place this format is written; all three call sites read it instead of reformatting.
- **`AuthorizationError` → `403` body now carries `reason`/`level`, not just `detail`.** A real `.call()` denial used to drop the developer-declared `reason=` a `grant(when=...)`/`check_roles(guard=...)` was rejected with (or the framework-fixed `"FORBIDDEN_ROLE"`) — `handle_authorization_error` only ever serialized `str(exc)`, and `AuthorizationError` has no `__str__` override, so `exc.reason` never reached the caller even though the resolver's own `.can()` prediction path already surfaced the same information. The `403` body now additively carries `reason: str | None` and `level: int | None` alongside the unchanged `detail` — existing consumers reading `detail` are unaffected. Both are `null` only for an entry-gate failure ("Authentication required"); a level-3 `access_decide` denial carries its own real `reason` too, since `access_decide` returns a `FailSecurityVerdict(reason=...)` directly (`aoa-action-machine`'s `BaseVerdict` hierarchy — see its own CHANGELOG), not a raw `bool`. This handler predates chapter 3.5 and covers every real action call in the app, not only the resolver/catalog endpoints. ([#130](https://github.com/bystrovmaxim/aoa/issues/130))

### Added

- **`POST /permissions/resolve` — minimal permissions resolver (role-gate only).** New bespoke route registered directly in `FastApiAdapter.build()` (like `_register_health_check`, but always authenticated): takes a list of `{operation, params}` questions and returns a same-order list of verdicts, backed by the existing `machine.check_access_decide` (the final wire shape for each verdict — a `BaseVerdict` subclass — is described below, under "Contract hardening"). List-shaped wire protocol (`items`/`results`) from day one — a single question is a batch of one, not a separate code path. `auth_coordinator.process(request)` is always called; a `403` follows only when it returns `None` — a resolved anonymous `Context` (e.g. `NoAuthCoordinator`) flows through normally, so `@check_roles(GuestRole)` actions resolve correctly for unauthenticated callers instead of being special-cased in the resolver. New modules: `aoa.fastapi.permissions_schema` (wire models), `aoa.fastapi.permissions` (`operation` name resolution, per-item verdict resolution). No changes to `aoa-action-machine`. ([#134](https://github.com/bystrovmaxim/aoa/issues/134) · part of [#130](https://github.com/bystrovmaxim/aoa/issues/130))
- **Reserved-route-path guard on `FastApiAdapter`.** `_RESERVED_PATHS` (`/health`, `/permissions/resolve`, and any other bespoke path the adapter later grows) is checked on every `.post/.get/.put/.delete/.patch(...)` call; registering an action on a reserved path now raises `ReservedRoutePathError` immediately instead of being silently shadowed by the adapter's own bespoke route at `build()` time. ([#134](https://github.com/bystrovmaxim/aoa/issues/134))
- **Contract hardening: one execution recipe, a flat result shape, cache partitions, route-shadow safety, a self-describing manifest, and a versioned wire language.** Consolidates the still-`[Unreleased]` resolver/catalog protocol's hardening work into one entry. **Breaking changes throughout** — the resolver wire protocol has not shipped in any released version yet, so none of this carries a deprecation period.
  - **One execution recipe for `.can()` and `.call()`.** New `aoa.fastapi.execution_plan` module: `EndpointExecutionPlan` (immutable, built once per route at `build()` — its route record plus the *effective* `AuthCoordinatorProtocol`) and `PreparedEndpointContext` (`.prepare(request)`'s per-request result — `Context` + resolved `connections`). Both real endpoint handlers and the resolver now build `Context`/`connections` exclusively through this plan — reading the *same* `EndpointExecutionPlan` object for a given route, not two independently-constructed ones that merely agreed by coincidence: `build()` now builds one shared `plan_index` and passes it to both `_register_endpoint` and `_register_permissions_endpoints`, rather than each building its own. **Fixes a real divergence:** the resolver used to authenticate every batch item with the adapter's single default `auth_coordinator` and never resolved `connections` at all, so a route with an `auth_coordinator=` override, or an `access_decide` reading `connections`, could answer "can I?" differently from what a real call would enforce. The resolver now runs each *distinct* `operation`'s own plan (reusing its own entry-gate `context` when a route doesn't override the default coordinator, to avoid authenticating twice) and passes each item's real `connections` into `check_access_decide`. `resolve_verdicts()` takes `plan_index`/`prepared_by_operation` instead of one shared `Context`. **A route-level `auth_coordinator=` override rejecting the caller for one operation is isolated to that operation's own positions in `results`** (`FailSecurityVerdict("UNAUTHORIZED")`), not left to fail the whole batch with a bare `403` and no `results` at all — this per-route auth path only became reachable from inside the resolver once it went through `EndpointExecutionPlan`, so the isolation gap could not exist before this same entry.
  - **`ResolveResponse.results` — one `BaseVerdict` subclass per answer, not one flat flag-shaped type for all of them, replacing `verdicts`/`Verdict`.** The response array is renamed `verdicts` → `results`; each element is a `BaseVerdict` subclass imported directly from `aoa-action-machine` (`intents.access_control` — see that package's own CHANGELOG for the hierarchy itself): `AllowedVerdict` (success, no `reason` field at all), or `FailSecurityVerdict`/`FailErrorVerdict` (both carry a mandatory, non-empty `reason`). `kind` is not a free-standing enum field a caller could set to a mismatched value — it is `type(self).__name__`, computed once on `BaseVerdict` and inherited by every subclass; the earlier `ResolveItemKind` `StrEnum` this package briefly imported (`SUCCESS`/`SECURITY`/`FLAG`/`MACHINE_RULE`/`CHECK_ERROR`) no longer exists in any form — the set of possible `kind` values is exactly the set of `BaseVerdict` subclasses that exist. `UNKNOWN_ENDPOINT` and any other check-side failure (timeout, crash) are a `FailErrorVerdict` — **not a denial**, and must never be cached as one; an unknown `operation` isolates to its own item (`FailErrorVerdict("UNKNOWN_ENDPOINT")`) instead of failing the whole batch. `reason` on a `FailSecurityVerdict` is a clean, developer-declared string (see `aoa-action-machine`'s own entry for the `reason=` mechanism this depends on): a role that never matched reports the fixed `"FORBIDDEN_ROLE"`; `access_decide` (level 3) now returns `FailSecurityVerdict(reason=...)` directly, so its denials carry a real developer-declared reason too, not raw cascade text. `Verdict` and `ResolveItemResult` are both removed — the wire type went from a shared struct this package once owned, to a shared struct `aoa-action-machine` owned instead, to no separate struct at all: `BaseVerdict` itself, imported and re-exported here, *is* the wire type. `to_wire()` is gone; `resolve_verdicts()` returns real `BaseVerdict` instances directly in `ResolveResponse.results`, no projection step. **`ResolveResponse.results` is typed `list[SerializeAsAny[BaseVerdict]]`, not plain `list[BaseVerdict]`** — pydantic v2 otherwise serializes a field by its *declared* container type, silently dropping subclass-only fields regardless of `Field(exclude=...)`; `SerializeAsAny` opts back into polymorphic, runtime-type serialization. The manifest's published reference schema for `BaseVerdict` (`schemas["BaseVerdict"]`, chapter 3.5 task 7) documents the guaranteed minimum shape — `{kind}` for `AllowedVerdict`, `{kind, reason}` for the two failure subclasses — not a closed one; a subclass that adds its own field (e.g. an object id on a specific denial) is still valid against it.
  - **`POST /permissions/resolve` deduplicates identical items.** `resolve_verdicts()` groups a batch's items by `(operation, canonical_key(params))`; only the first occurrence of a key triggers a real `check_access_decide` call — distinct keys run concurrently via `asyncio.gather` — and later occurrences copy that result, so `results` never shrinks.
  - **`GET /permissions/namespace` — `PermissionNamespace`/`cache_partition` for the client's cache key.** New bespoke route, same auth gate as the resolver (`403` only when `process(request)` returns `None`). Returns `{cache_partition: str}` — an opaque label the client attaches to every cached resolver answer so a cache entry can never be shared across two identities, derived fresh on every call via `aoa-action-machine`'s new `compute_cache_partition` (see its own CHANGELOG). Deliberately never on the manifest or on `ResolveResponse` — the client needs the label *before* it can look an answer up in its cache.
  - **Route uniqueness and shadowing.** `build_manifest()` now dedups an exact `(method, path)` duplicate first-wins (before `manifest_version` is computed), matching Starlette's own router behavior; registering an exact duplicate emits a non-fatal `UserWarning`. **The dedup itself is `build_route_index()`, reused directly, not reimplemented a second time:** `build_manifest()` used to carry its own local first-wins helper, keyed by a `(method, path)` tuple, next to `build_route_index()`'s independent one keyed by the `"{method} {path}"` operation string — the same rule, written twice, agreeing only because nothing had yet forced them apart (audit finding 10). `build_manifest()` now calls `build_route_index(routes).values()` directly; the local helper is gone. New: two *different* path templates that could match the same URL (`/users/me` alongside `/users/{id}`) now fail `build()` with a new `RouteShadowError`, checked once every route is known — the overlap check understands Starlette's typed converters (`{id:int}`, `{id:float}`, `{id:uuid}`, the greedy `{rest:path}`) and is deliberately conservative. **The check now also covers the adapter's own four bespoke routes** (`/health`, `/permissions/resolve`, `/client-manifest.json`, `/permissions/namespace`) — they used to be invisible to it, registered outside `self._routes` and never fed into the check, so an app-registered template like `/{item_id}` could structurally shadow `/health` with no error at `build()` time. `_check_for_route_shadowing` now takes plain `(method, path)` pairs instead of `FastApiRouteRecord`s, so the four reserved routes do not need a fake `action_class` to participate; the new `_RESERVED_ROUTES` constant (method + path per bespoke route) is the single source both this check and the pre-existing exact-path `_RESERVED_PATHS` guard read from, so the two can no longer drift apart.
  - **`GET /client-manifest.json` — HTTP caching contract and three separate version numbers.** `Manifest.protocol` is renamed `Manifest.version`; new `Manifest.manifest_schema_version` (the manifest's own shape) is independent of both `version` and of `manifest_version` (a `sha256:`-prefixed hash of the manifest body computed *without* the field itself — no circular fixed point; its hashed body now also covers `version`, `manifest_schema_version`, and `schemas`). The route now answers with `ETag: "<manifest_version>"` (quoted) and `Cache-Control: private, no-cache`; a matching `If-None-Match` (exact or `*`) now gets `304 Not Modified` with no body. **The response body is serialized once, at `build()` time, not per request:** the route used to call `manifest.model_dump(mode="json")` — and, inside `JSONResponse(...)`'s own constructor, `json.dumps()` on top of that — freshly for every non-304 request, even though `self._routes` (and therefore the manifest) never changes after `build()`. Both the dict conversion and the JSON byte-serialization now run exactly once. **What is cached is only the rendered JSON bytes, not a `Response` object** (fix-audit finding 2): every cache-miss request now gets a fresh `Response` wrapping those bytes, rather than one `JSONResponse` instance reused verbatim across every request — `Response.raw_headers` is a plain mutable list that `Response.__call__` hands by reference into the ASGI message, so a shared instance's headers could be permanently mutated in place by header-rewriting middleware (e.g. `GZipMiddleware`) on its first compressed response, corrupting every subsequent request regardless of whether that later request asked for compression.
  - **`Manifest.schemas` — reference schemas for the protocol's own fixed messages.** New key: `ResolveRequest`, `ResolveResponse`, `BaseVerdict`, `ErrorEnvelope`, and `Manifest` itself, each published as `{mode, json_schema}` — `json_schema` is JSON Schema **Draft 2020-12** (carries its own `$schema`), in a small agreed subset (plain types, lists, enums, in-document `$ref`s; no recursive types, no custom formats); `mode` is `"validation"` for `ResolveRequest` and `"serialization"` for everything else. `manifest_schema_version` bumped `1` → `2` for this addition.
  - **Error contract: `ErrorEnvelope` for whole-request failures.** New `{"error": {"code": "..."}}` envelope (`ErrorDetail`/`ErrorEnvelope` in `permissions_schema.py`, also published under `Manifest.schemas`) — reserved for failures that discard the entire request (no `results` array at all), never for a per-item problem (a `FailErrorVerdict` element stays inside a normal `200`).
  - **Versioning: an unsupported `version` is `400 unsupported_version`.** `ResolveRequest.protocol`/`ResolveResponse.protocol` are renamed `.version`, now reading the single source `aoa.fastapi.permissions_schema.SUPPORTED_VERSION` that `Manifest.version` also publishes. The resolver checks `body.version` first, *before* `auth_coordinator.process(request)` — a client speaking the wrong wire language is never made to authenticate just to be told to upgrade. A mismatch raises the new `UnsupportedVersionError`, mapped to `400` with the `ErrorEnvelope` body above.

  ([#130](https://github.com/bystrovmaxim/aoa/issues/130))

## [1.1.1] – 2026-07-11

### Changed

- **`auth_coordinator` is now typed `AuthCoordinatorProtocol` instead of `Any`.** Applies to `FastApiAdapter.__init__` and the `auth_coordinator=` parameter on `.post/.get/.put/.delete/.patch(...)`. Purely additive typing — structural (`Protocol`), so every coordinator that already works continues to work unchanged; mypy/IDEs now catch a mismatched custom coordinator instead of only failing at runtime. ([#108](https://github.com/bystrovmaxim/aoa/issues/108) · [aoa-action-machine CHANGELOG](../aoa-action-machine/CHANGELOG.md))

## [1.1.0] – 2026-07-10

### Added

- **`auth_coordinator=` on `.post/.get/.put/.delete/.patch(...)` — per-route authentication override.** Overrides the adapter's default coordinator for one route only; falls back to the adapter default when omitted, via the new `BaseAdapter.effective_auth_coordinator`. Lets a route like `/auth/login` opt out of a strict adapter-wide coordinator without weakening it for every other route. ([#66](https://github.com/bystrovmaxim/aoa/issues/66) · [aoa-action-machine CHANGELOG](../aoa-action-machine/CHANGELOG.md))

### Changed

- **Test suite relocated into the package (`packages/aoa-fastapi-adapter/tests/`).** FastAPI adapter tests moved out of the shared root `tests/action_machine/adapters/fastapi/` into the package's own `tests/` directory, with a per-package `[dependency-groups]` dev group (adding `httpx` for the FastAPI `TestClient`) and `[tool.pytest.ini_options]`. The cross-package dependencies on `tests/action_machine/scenarios` and the shared adapter-fixture modules were replaced by a self-contained `tests/support/` package (`domain_model`, `connections`, `adapter_fixtures` + facade) — faithful trimmed copies — so the package no longer reaches into another package's test tree. ([#82](https://github.com/bystrovmaxim/aoa/issues/82))

## [1.0.0] – 2026-06-24

### Added

- **Initial standalone release, extracted from `aoa-action-machine`.** `FastApiAdapter`, `FastApiRouteRecord`, and `query_field_before` helpers are now distributed under the `aoa.fastapi` namespace as a separate package (`pip install aoa-fastapi-adapter`). The package depends on `aoa-action-machine`, `fastapi`, and `uvicorn[standard]`. ([#63](https://github.com/bystrovmaxim/aoa/issues/63))

For the pre-extraction history of `FastApiAdapter` (originally introduced in the monorepo at `[0.5.5]`), see the [aoa-action-machine CHANGELOG](../aoa-action-machine/CHANGELOG.md).
