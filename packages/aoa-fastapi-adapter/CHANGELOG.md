# Changelog

All notable changes to `aoa-fastapi-adapter` are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- **`POST /permissions/resolve` — minimal permissions resolver (role-gate only).** New bespoke route registered directly in `FastApiAdapter.build()` (like `_register_health_check`, but always authenticated): takes a list of `{operation, params}` questions and returns a same-order list of `Verdict`s, backed by the existing `machine.check_access_decide`. List-shaped wire protocol (`items`/`verdicts`) from day one — a single question is a batch of one, not a separate code path. `auth_coordinator.process(request)` is always called; a `403` follows only when it returns `None` — a resolved anonymous `Context` (e.g. `NoAuthCoordinator`) flows through normally, so `@check_roles(GuestRole)` actions resolve correctly for unauthenticated callers instead of being special-cased in the resolver. `scope`/`entities`/`reason_code`/`expires_at` are reserved in the wire schema but deliberately unpopulated in this release — see the upcoming object-level verdict and rate-limiting work. `CheckAccessDecideBatchSizeExceededError` now maps to HTTP `413`. New modules: `aoa.fastapi.permissions_schema` (wire models), `aoa.fastapi.permissions` (`operation` name resolution, `AccessVerdict` -> wire projection). No changes to `aoa-action-machine`. ([#134](https://github.com/bystrovmaxim/aoa/issues/134) · part of [#130](https://github.com/bystrovmaxim/aoa/issues/130))
- **Reserved-route-path guard on `FastApiAdapter`.** `_RESERVED_PATHS` (`/health`, `/permissions/resolve`, and any other bespoke path the adapter later grows) is checked on every `.post/.get/.put/.delete/.patch(...)` call; registering an action on a reserved path now raises `ReservedRoutePathError` immediately instead of being silently shadowed by the adapter's own bespoke route at `build()` time. ([#134](https://github.com/bystrovmaxim/aoa/issues/134))
- **Contract hardening: one execution recipe, a flat result shape, cache partitions, route-shadow safety, a self-describing manifest, and a versioned wire language.** Consolidates the still-`[Unreleased]` resolver/catalog protocol's hardening work into one entry. **Breaking changes throughout** — the resolver wire protocol has not shipped in any released version yet, so none of this carries a deprecation period.
  - **One execution recipe for `.can()` and `.call()`.** New `aoa.fastapi.execution_plan` module: `EndpointExecutionPlan` (immutable, built once per route at `build()` — its route record plus the *effective* `AuthCoordinatorProtocol`) and `PreparedEndpointContext` (`.prepare(request)`'s per-request result — `Context` + resolved `connections`). Both real endpoint handlers and the resolver now build `Context`/`connections` exclusively through this plan. **Fixes a real divergence:** the resolver used to authenticate every batch item with the adapter's single default `auth_coordinator` and never resolved `connections` at all, so a route with an `auth_coordinator=` override, or an `access_decide` reading `connections`, could answer "can I?" differently from what a real call would enforce. The resolver now runs each *distinct* `operation`'s own plan (reusing its own entry-gate `context` when a route doesn't override the default coordinator, to avoid authenticating twice) and passes each item's real `connections` into `check_access_decide`. `resolve_verdicts()` takes `plan_index`/`prepared_by_operation` instead of one shared `Context`. **A route-level `auth_coordinator=` override rejecting the caller for one operation is isolated to that operation's own positions in `results`** (`kind=SECURITY, reason="UNAUTHORIZED"`), not left to fail the whole batch with a bare `403` and no `results` at all — this per-route auth path only became reachable from inside the resolver once it went through `EndpointExecutionPlan`, so the isolation gap could not exist before this same entry.
  - **`ResolveResponse.results` = `ResolveItemResult { kind: ResolveItemKind, reason: str }` — one flat shape for every answer, replacing `verdicts`/`Verdict`.** The response array is renamed `verdicts` → `results`; each element collapses from the old `Verdict` (`allowed`/`scope`/`level`/`reason`/`reason_code`/`entities`/`expires_at` — several independently-settable fields that could contradict each other) into exactly `{kind, reason}`. `kind` is the new `ResolveItemKind` `StrEnum` (`SUCCESS`/`SECURITY`/`FLAG`/`MACHINE_RULE`/`CHECK_ERROR`, canonically defined in `aoa-action-machine`, imported here); `SUCCESS` carries `reason=""`, every other channel a non-empty string. `UNKNOWN_ENDPOINT` and any other check-side failure (timeout, crash) are `CHECK_ERROR` — **not a denial**, and must never be cached as one; an unknown `operation` now isolates to its own item (`kind=CHECK_ERROR, reason="UNKNOWN_ENDPOINT"`) instead of failing the whole batch. `reason` on a `SECURITY` result is a clean, developer-declared string (see `aoa-action-machine`'s own entry for the `reason=` mechanism this depends on): a role that never matched reports the fixed `"FORBIDDEN_ROLE"`; `access_decide` (level 3) denials still report raw cascade text — its own clean-reason mechanism remains a separate, not-yet-done change. `to_wire()` now returns `ResolveItemResult` as a straight copy of `kind`/`reason`; `Verdict` is removed.
  - **`POST /permissions/resolve` deduplicates identical items.** `resolve_verdicts()` groups a batch's items by `(operation, canonical_key(params))`; only the first occurrence of a key triggers a real `check_access_decide` call — distinct keys run concurrently via `asyncio.gather` — and later occurrences copy that result, so `results` never shrinks. `ActionProductMachine.max_check_access_decide_batch_size` (`aoa-action-machine`) is now a public read-only property, since the resolver enforces the same cap itself against the batch size *after* deduplication.
  - **`GET /permissions/namespace` — `PermissionNamespace`/`cache_partition` for the client's cache key.** New bespoke route, same auth gate as the resolver (`403` only when `process(request)` returns `None`). Returns `{cache_partition: str}` — an opaque label the client attaches to every cached resolver answer so a cache entry can never be shared across two identities, derived fresh on every call via `aoa-action-machine`'s new `compute_cache_partition` (see its own CHANGELOG). Deliberately never on the manifest or on `ResolveResponse` — the client needs the label *before* it can look an answer up in its cache.
  - **Route uniqueness and shadowing.** `build_manifest()` now dedups an exact `(method, path)` duplicate first-wins (before `manifest_version` is computed), matching `build_route_index()` and Starlette's own router behavior; registering an exact duplicate emits a non-fatal `UserWarning`. New: two *different* path templates that could match the same URL (`/users/me` alongside `/users/{id}`) now fail `build()` with a new `RouteShadowError`, checked once every route is known — the overlap check understands Starlette's typed converters (`{id:int}`, `{id:float}`, `{id:uuid}`, the greedy `{rest:path}`) and is deliberately conservative.
  - **`GET /client-manifest.json` — HTTP caching contract and three separate version numbers.** `Manifest.protocol` is renamed `Manifest.version`; new `Manifest.manifest_schema_version` (the manifest's own shape) is independent of both `version` and of `manifest_version` (a `sha256:`-prefixed hash of the manifest body computed *without* the field itself — no circular fixed point; its hashed body now also covers `version`, `manifest_schema_version`, and `schemas`). The route now answers with `ETag: "<manifest_version>"` (quoted) and `Cache-Control: private, no-cache`; a matching `If-None-Match` (exact or `*`) now gets `304 Not Modified` with no body.
  - **`Manifest.schemas` — reference schemas for the protocol's own fixed messages.** New key: `ResolveRequest`, `ResolveResponse`, `ResolveItemResult`, `ErrorEnvelope`, and `Manifest` itself, each published as `{mode, json_schema}` — `json_schema` is JSON Schema **Draft 2020-12** (carries its own `$schema`), in a small agreed subset (plain types, lists, enums, in-document `$ref`s; no recursive types, no custom formats); `mode` is `"validation"` for `ResolveRequest` and `"serialization"` for everything else. `manifest_schema_version` bumped `1` → `2` for this addition.
  - **Error contract: `ErrorEnvelope` for whole-request failures.** New `{"error": {"code": "..."}}` envelope (`ErrorDetail`/`ErrorEnvelope` in `permissions_schema.py`, also published under `Manifest.schemas`) — reserved for failures that discard the entire request (no `results` array at all), never for a per-item problem (`CHECK_ERROR` stays inside a normal `200`).
  - **Versioning: an unsupported `version` is `400 unsupported_version`.** `ResolveRequest.protocol`/`ResolveResponse.protocol` are renamed `.version`, now reading the single source `aoa.fastapi.permissions_schema.SUPPORTED_VERSION` that `Manifest.version` also publishes. The resolver checks `body.version` first, *before* `auth_coordinator.process(request)` — a client speaking the wrong wire language is never made to authenticate just to be told to upgrade. A mismatch raises the new `UnsupportedVersionError`, mapped to `400` with the `ErrorEnvelope` body above.

  ([#130](https://github.com/bystrovmaxim/aoa/issues/130))

## [1.1.1] – 2026-07-11

### Changed

- **`auth_coordinator` is now typed `AuthCoordinatorProtocol` instead of `Any`.** Applies to `FastApiAdapter.__init__` and the `auth_coordinator=` parameter on `.post/.get/.put/.delete/.patch(...)`. Purely additive typing — structural (`Protocol`), so every coordinator that already works continues to work unchanged; mypy/IDEs now catch a mismatched custom coordinator instead of only failing at runtime. ([#108](https://github.com/bystrovmaxim/aoa/issues/108) · [aoa-action-machine CHANGELOG](../aoa-action-machine/CHANGELOG.md))

## [1.1.0] – 2026-07-10

### Added

- **`auth_coordinator=` on `.post/.get/.put/.delete/.patch(...)` — per-route authentication override.** Overrides the adapter's default coordinator for one route only; falls back to the adapter default when omitted, via the new `BaseAdapter.effective_auth_coordinator`. Lets a route like `/auth/login` opt out of a strict adapter-wide coordinator without weakening it for every other route. ([#66](https://github.com/bystrovmaxim/aoa/issues/66) · [aoa-action-machine CHANGELOG](../aoa-action-machine/CHANGELOG.md))

### Changed

- **Test suite relocated into the package (`packages/aoa-fastapi-adapter/tests/`).** FastAPI adapter tests moved out of the shared root `tests/action_machine/adapters/fastapi/` into the package's own `tests/` directory, with a per-package `[dependency-groups]` dev group (adding `httpx` for the FastAPI `TestClient`) and `[tool.pytest.ini_options]`. The cross-package dependencies on `tests/action_machine/scenarios` and the shared adapter-fixture modules were replaced by a self-contained `tests/support/` package (`domain_model`, `connections`, `adapter_fixtures` + facade) — faithful trimmed copies — so the package no longer reaches into another package's test tree. ([#82](https://github.com/bystrovmaxim/aoa/issues/82))

## [1.0.0] – 2026-06-24

### Added

- **Initial standalone release, extracted from `aoa-action-machine`.** `FastApiAdapter`, `FastApiRouteRecord`, and `query_field_before` helpers are now distributed under the `aoa.fastapi` namespace as a separate package (`pip install aoa-fastapi-adapter`). The package depends on `aoa-action-machine`, `fastapi`, and `uvicorn[standard]`. ([#63](https://github.com/bystrovmaxim/aoa/issues/63))

For the pre-extraction history of `FastApiAdapter` (originally introduced in the monorepo at `[0.5.5]`), see the [aoa-action-machine CHANGELOG](../aoa-action-machine/CHANGELOG.md).
