## [Unreleased]

### Added

- **`generateClient(url)` — static codegen entry point (`aoa-client-js/codegen`, separate from the runtime entry point).** Fetches a manifest and returns a TypeScript source string: a `Params`/`Result` interface pair per endpoint, plus a typed `api` object built by `createGateApi(engine)`/`createApi(engine, actionInvoker)`. Every JSON Schema this codegen touches — each endpoint's `params_schema`/`result_schema`, and the protocol's own `schemas.ResolveResponse` — is walked into the same small, bounded IR first (`json-schema-ir.ts`; string/integer/number/boolean, object with properties/required, array, enum — both `$ref`-to-`$defs` and a `Literal[...]` field inline, since pydantic emits those with no `$ref` — `anyOf`-null as `Optional`, and an untyped dict as an index signature; depth-guarded at 20 to reject a cyclic schema rather than hang), so both renderers below target the identical parse rather than each independently deciding which subset of JSON Schema to understand. Only two renderers consume that IR, though, and not symmetrically: `json-schema-to-ts.ts` renders TypeScript for *every* endpoint's `Params`/`Result`, while `json-schema-to-zod.ts` renders a runtime `zod` validator *only* for the wire-protocol's own `ResolveResponse` (`ResolveResponseSchema`) — no zod validator is generated for any individual endpoint's own parameters or result. `ManifestEndpoint.name` is documented as informational only, not guaranteed unique — `naming.ts`'s `NameRegistry.claimBase` derives a base type name by stripping a trailing `"Action"` and gives a second endpoint that collides on the same base a deterministic numeric suffix (`CancelOrder`, `CancelOrder2`, …) instead of silently shadowing the first endpoint's types.
- **Hybrid `api` layout: full bracket path always, dot alias only on a "clean" path (`path-layout.ts`, shared between static and dynamic codegen).** `api.post["/actions/cancel-order"]` always exists as an exact, closed object (a missing route is a `tsc` error, not a wrong guess); `api.get.orders` additionally exists only when every path segment is a valid identifier — no `{param}`, hyphen, or dot. A path that would collide with a deeper path through the same segment (branch/leaf collision) demotes only the shorter, colliding path to bracket-only; the deeper path keeps its alias — resolved via a two-phase trie insert plus post-hoc demotion, independent of registration order.
- **`Primitive` — the hand-written runtime shared by every generated endpoint object (`primitive.ts`), not re-templated per endpoint.** `GatePrimitive<TParams>` (`.verdict()`/`.can()`) and `CallablePrimitive<TParams, TResult>` (adds `.run()`) are two distinct interfaces, not one interface with an optional method: `createGateApi` builds an object whose `Primitive`s genuinely lack `.run()` at the type level — accessing it doesn't compile, it isn't merely unused — while `createApi(engine, actionInvoker)` builds the callable form. `.run()` builds its call (method, path, body) from the generated endpoint descriptor via `buildInvocation`; the injected `actionInvoker` only executes an already-fixed `Invocation` and cannot redirect it to a different route or body, so `.can()` and `.run()` are guaranteed to agree on which endpoint they mean. `ActionInvoker` is generic in `TResult` (`<TResult>(invocation) => Promise<TResult>`) so each generated `.run()` gets its own return type with no unsafe cast at the call site.
- **`AoaEngine.loadFrom(url)` — dynamic codegen, no build step (`dynamic-api.ts`, `buildDynamicGateApi`).** Fetches the same manifest already at runtime and builds the same gate-only `api` shape in memory, through the identical `buildLayout` (`path-layout.ts`) that `generateClient` consumes — cross-checked directly so the two codegen paths cannot silently diverge on shape. No `.run()` (there is no `actionInvoker` at load time) and no compile-time types: a path absent from the live manifest is a runtime `TypeError` on property access, not a `tsc` error. Manifest-shape failures surface as the same typed `ProtocolError`/`NetworkUnavailable` vocabulary `resolve()` already uses, not a separate error type.
- **`aoa-codegen` CLI + `--check` — thin wrapper around `generateClient`, with a schema-drift mode for CI (`bin.ts`, `check-drift.ts`).** The CLI (`--url`, `--out`, `--check`) contains no generation logic of its own — it only parses arguments and either writes what `generateClient` returned or diffs against it — so there is no second generator for the committed file to drift from. `--check` (`diffGeneratedSource`) is a declaration-level diff, not a line diff or a rebuild-and-`git diff`: it names which specific interface/type/function/const changed (e.g. `changed: CancelOrderParams`) instead of an opaque "files differ," reports a `manifest_version` change separately, and deliberately ignores the `// Source: <url>` header line (legitimately environment-dependent — `--check` may run against staging or localhost). This is the recommended CI default for a committed static client: schema drift (a renamed field, a removed route) is caught before deploy, not after. No server-side codegen endpoint (`GET /client.ts`) and no CI-only second generator to diff against were added — the server keeps publishing exactly one JSON manifest (chapter 3); codegen lives entirely in this package.
- **4 runnable examples under `examples/step_27_ui_permissions_codegen/`** (`.ts` + paired `.ipynb`): static `generateClient` + `Primitive` (bracket path vs. dot alias, same-instance check), the dynamic `engine.loadFrom` mode (identical call shapes, no compile-time types), a stale static client against a drifted manifest (`UNKNOWN_ENDPOINT`, `.can()` throwing rather than returning a silent `false`), and one JSON fixture (`contracts/fixtures/resolve_response_basic.json`) parsed by the real generated `ResolveResponseSchema` — the TypeScript half of a contract test whose Python half lives in `aoa-fastapi-adapter` (`tests/test_resolve_contract.py`); if `AllowedVerdict`/`FailSecurityVerdict`/`FailErrorVerdict` change shape on one side and not the other, one of the two goes red. ([#138](https://github.com/bystrovmaxim/aoa/issues/138))
- **186 tests across 15 files; 100% statement/branch/function/line coverage on every codegen and runtime module** except `bin.ts`'s process-level entry point (`main()`, `runCheck()`, the `argv`/`import.meta.url` guard) and the pre-existing, purely-declarative `types.ts`. `bin.ts`'s CLI behavior is verified end-to-end through a real spawned `node` subprocess (`execFile`, not `execFileSync` — a synchronous spawn from inside a test that also runs a same-process local manifest server deadlocks the parent's own event loop) rather than in-process, so v8's coverage instrumentation — attached to the test worker process — cannot observe it running, despite it being exercised by 7 real-process test cases; `parseArgs`, the one piece of `bin.ts` implemented as an importable pure function, is covered like everything else. ([#138](https://github.com/bystrovmaxim/aoa/issues/138))

- **New package: `AoaEngine.resolve()` — a minimal, framework-neutral HTTP client for the `/permissions/resolve` protocol.** First release of `aoa-client-js`, a self-contained TypeScript package (structural precedent: `aoa-maxitor/client`) with no dependency on the Python side beyond the network protocol. `AoaEngine.resolve(items)` makes exactly one `POST /permissions/resolve` request per call — no caching, no batching yet (chapters 6/7) — and validates the response before trusting any of it, down to each individual element: `res.ok`, an exact (not substring) `application/json` media-type match, a JSON-parseable non-null body, wire-language `version`, that `results.length` matches the request, and that every element of `results` carries a `kind` from the closed set (`AllowedVerdict`/`FailSecurityVerdict`/`FailErrorVerdict`) with a non-empty `reason` on the two `Fail*` kinds — a malformed element (missing/unrecognized `kind`, missing `reason`) is rejected with `ProtocolError` rather than returned as-is, where a caller's own `kind === "AllowedVerdict"` check would otherwise silently misread it as a denial. Any other malformed or unreachable response throws a typed error too (`Unauthorized` on `401`, `ProtocolError` on every other validation failure, `NetworkUnavailable` when `fetch` itself throws) — a transport failure never becomes a synthetic denial. `resolve()` returns `Verdict[]`, a discriminated union of the same three outcome classes the server uses (`aoa-action-machine`'s `BaseVerdict` hierarchy): `AllowedVerdict` (success, no `reason` field at all), and `FailSecurityVerdict`/`FailErrorVerdict` (both carry a mandatory `reason`; `FailErrorVerdict` is not a denial and must never be cached as one). Every call carries its own `trace_id` (`crypto.randomUUID()` by default, injectable via `TransportConfig.generateTraceId` for environments without it, overridable per call via `opts.traceId` to correlate several calls under one user gesture) sent as the `x-trace-id` header the server already reads; request headers are merged case-insensitively through a real `Headers` object, so a caller-supplied header — differently cased or not — actually replaces the default instead of joining it with a comma on the wire. `AoaEngine`'s identity (`cachePartition`) is fixed at construction and read-only — switching subjects means constructing a new `AoaEngine`, never mutating an existing one. `isRetryableCheckError(reason)` is a client-side judgment call on whether a `FailErrorVerdict` is worth retrying (`TIMEOUT` only) — the wire itself carries no such concept. `fetchImpl` is injected via the constructor rather than read off `window.fetch`, so `AoaEngine` runs in Node (SSR, chapter 10) as well as the browser. All four typed error classes (`Unauthorized`/`ProtocolError`/`NetworkUnavailable`/`AoaResolveError`) set their own `.name`; `AoaResolveError.message` is a sentence, with `.reason` carrying the raw wire code separately. Source avoids TypeScript parameter properties so every module also runs unmodified via `node <file>.ts` (Node's native type-stripping doesn't support them); internal relative imports carry explicit `.ts` extensions for the same reason. 41 tests; `engine.ts`/`index.ts` at 100% statement/branch/function/line coverage (`@vitest/coverage-v8`). ([#137](https://github.com/bystrovmaxim/aoa/issues/137), part of [#130](https://github.com/bystrovmaxim/aoa/issues/130))
- **4 runnable examples under `examples/step_27_ui_permissions_client/`** (`.ts` + paired `.ipynb`): a single question, a batch of questions in one call, request-level errors vs. a `FailErrorVerdict` inside `results`, and the `fetchImpl` dependency-injection pattern for testing. Each `.ts` file executes directly via `node`, no build step — **but, unlike this book's Python/Colab examples, the `.ipynb` pair is not yet portable to a fresh Google Colab session:** it imports `AoaEngine` via a relative filesystem path into the local monorepo checkout (`aoa-client-js` isn't published anywhere), and requires a Deno Jupyter kernel, not the default Python one. Tracked separately as [#152](https://github.com/bystrovmaxim/aoa/issues/152). ([#137](https://github.com/bystrovmaxim/aoa/issues/137))

- **`Primitive.run()` — a fresh, non-cached precheck right before the real invocation (chapter 5.5).** Before delegating to `actionInvoker`, `run(params)` now calls `engine.resolve([{ operation, params }], { skipCache: true })` — the only call site in the library that uses `skipCache` — so a stale cached "allowed" answer can never let a revoked action through. `AllowedVerdict` proceeds to the real call; `FailSecurityVerdict` throws a plain `Error` carrying the reason (no dedicated error class — the caller doesn't need to know a precheck happened at all); `FailErrorVerdict`/network errors propagate as-is (`AoaResolveError`/`NetworkUnavailable`/`ProtocolError`), never becoming a synthetic deny. `resolve()` itself gained a minimal cache in the same change (new `cache.ts`): a per-item read unless `skipCache` is set, only actual misses go to the network in one batch, and a fresh cacheable answer (never `FailErrorVerdict`) writes through to the cache regardless of `skipCache`. Deliberately narrower than chapter 6's own cache ([#139](https://github.com/bystrovmaxim/aoa/issues/139)) — a single fresh/stale cutover (`ttlMs`, default `3_000`ms, configurable via `transport.cache`), no canonical-JSON key, no stale-while-revalidate, no eviction limit — chapter 6 is expected to extend this module, not replace it. ([#157](https://github.com/bystrovmaxim/aoa/issues/157))
- **1 runnable example under `examples/step_27_ui_permissions_precheck/`** (`.ts` + paired `.ipynb`): a fresh cache entry still makes `run()` issue its own network call via `skipCache`, and a stale cached "allowed" answer never lets a revoked action through — `run()`'s precheck asks fresh and stops before `actionInvoker`. ([#157](https://github.com/bystrovmaxim/aoa/issues/157))
